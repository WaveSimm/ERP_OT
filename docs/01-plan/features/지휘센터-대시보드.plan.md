## 1. 개요 (Overview)

수백 개의 프로젝트를 운영하는 조직에서 **팀장급 이상이 회의 중에 실시간으로 공유하며 의사결정**할 수 있는 지휘센터형 대시보드.

- **당일 기준 ±1주일** 일정을 프로젝트별로 요약
- **이슈·위험·지연** 즉시 가시화
- **부서 / 고객사 / 커스텀** 단위로 프로젝트 그룹핑
- 수백 개 프로젝트를 **1-2초 내 렌더링**하는 고성능 뷰

---

## 2. 문제 정의 (Problem Statement)

| 문제 | 영향 |
|------|------|
| 프로젝트가 수백 개인데 한 화면에서 전체 현황을 볼 방법이 없음 | 회의 시 개별 보고 의존, 준비 시간 과다 |
| 이번 주 어떤 프로젝트에서 무슨 일이 생기는지 사전 파악 불가 | 이슈 사후 대응 |
| 팀별 / 고객사별 묶음 조회가 안 됨 | 부문장 단위 현황 파악 불가 |
| 지연·예산 초과·자원 과부하가 어디서 발생했는지 한눈에 안 보임 | 의사결정 지연 |
| 회의 중 빔 프로젝터에 띄워도 읽기 어려운 현재 화면 구성 | 회의 효율 저하 |

## 3. 목표 및 성공 지표 (Goals & Success Metrics)

### 목표
- 팀장 주간 회의 시 별도 보고서 없이 대시보드만으로 현황 공유
- 이슈 발생 후 24시간 이내 감지 → 실시간 감지로 개선
- 부문별 책임자가 자기 영역 프로젝트만 필터링해서 볼 수 있도록

### 성공 지표 (KPI)

| 지표 | 현재 | 목표 |
|------|------|------|
| 주간 회의 보고서 준비 시간 | 2~3시간 | 0 (대시보드로 대체) |
| 이슈 최초 인지 시간 | 1일+ | < 15분 (자동 감지) |
| 대시보드 초기 로딩 (500개 프로젝트) | 해당 없음 | < 2s |
| 대시보드 갱신 주기 | 해당 없음 | 5분 자동 (수동 새로고침 즉시) |

## 5. 사용자 스토리 (User Stories)

### 핵심 스토리

**US-D01**: 팀장으로서 대시보드에 접속하면 **내가 관할하는 프로젝트 그룹**이 기본으로 펼쳐져 있고, 각 프로젝트의 당일 기준 ±1주일 일정 요약과 RAG(Red/Amber/Green) 상태를 즉시 확인한다.

**US-D02**: 대시보드에서 **그룹핑 기준을 전환**할 수 있다 (부서별 ↔ 고객사별 ↔ 커스텀). 전환 시 즉시 재정렬된다.

**US-D03**: 각 프로젝트 행에서 **이번 주 시작·종료·마일스톤 예정 태스크**를 미니 타임라인 또는 이벤트 목록으로 확인한다.

**US-D04**: 이슈가 있는 프로젝트에 **이슈 뱃지(🔴 Critical / 🟠 Warning / 🟡 Info)**가 표시되고, 클릭하면 이슈 상세(어떤 태스크가 지연, 얼마나, 원인)를 팝업으로 확인한다.

**US-D05**: 부문장으로서 **전체 프로젝트를 부서별로 접어보기(accordion)**하여 부서 단위 집계 수치(진행 중, 지연, 이슈 수)를 본다.

**US-D06**: 회의 진행 중 **프레젠테이션 모드**로 전환하면 폰트가 커지고 사이드바가 숨겨지며 빔 프로젝터에 최적화된 레이아웃이 된다.

**US-D07**: 특정 프로젝트 행을 클릭하면 **해당 프로젝트의 간트 차트 상세 뷰**로 이동한다 (새 탭 또는 슬라이드 패널).

**US-D08**: 대시보드가 **5분마다 자동 갱신**되며, 새로운 이슈가 생기면 해당 행이 시각적으로 강조(pulse animation)된다.

**US-D09**: 팀장으로서 **프로젝트 그룹을 직접 정의**하고 (부서·고객사 외 커스텀), 내 그룹 설정을 저장하여 다음 접속 시에도 유지한다.

**US-D10**: 대시보드 상단에 **전체 집계 요약 카드** (전체 프로젝트 수, 정상/지연/위험 수, 이번 주 완료 예정 수)를 표시한다.

### 부가 스토리

**US-D11**: 이슈 유형별 필터 (지연만 보기 / 예산초과만 보기 / 자원과부하만 보기) 적용.

**US-D12**: 대시보드를 PDF 또는 이미지로 내보내기 (회의록 첨부용).

**US-D13**: 특정 프로젝트를 즐겨찾기하여 대시보드 상단에 고정.

## 7. 이슈 분류 체계 (Issue Classification)

### RAG 기준

| 등급 | 색상 | 조건 |
|------|------|------|
| **Critical** | 🔴 빨강 | 크리티컬 패스 태스크 지연 / 예산 > 110% / 자원 배정률 > 120% |
| **Warning** | 🟠 주황 | 비크리티컬 태스크 지연 / 예산 100~110% / 자원 100~120% / 3일 이상 진행률 미업데이트 |
| **Info** | 🟡 노랑 | 마일스톤 3일 이내 / 태스크 완료 예정 이번 주 / 신규 이슈 댓글 |
| **On Track** | 🟢 초록 | 위 조건 해당 없음 |

### 이슈 상세 팝업 내용

```
[프로젝트: A라인 배관 교체]  🔴 Critical × 1  🟠 Warning × 2

🔴 [크리티컬 패스 지연] 지상 배관 세그먼트
   기준: 2/20~2/25 → 현재: 2/23~3/1 (+4일)
   담당: 배관팀 김철수 (배정 100%)
   영향: 수압테스트 +4일 → 최종완료 3/11 → 3/15

🟠 [자원 과부하] 크레인 #3
   현재 배정: 142% (2/26~3/1 기간)
   중복 배정: A라인 100% + C동공사 42%

🟠 [진행률 미업데이트] 전기 배선 세그먼트
   마지막 업데이트: 4일 전 (2/22)
   담당자: 전기팀 이영희
   → [담당자에게 알림 보내기]   [프로젝트로 이동]
```

## 8. 그룹핑 설계 (Grouping)

### 그룹 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **DEPARTMENT** | 부서 기반 그룹 | 설비팀, 공무팀, 영업1팀 |
| **CLIENT** | 고객사 기반 그룹 | ㈜한국산업, ABC Corp |
| **CUSTOM** | 사용자 정의 그룹 | "신공장 건설 프로그램", "2026 외자 프로젝트" |
| **NONE** | 그룹 없이 전체 목록 | 상태/날짜 순 정렬 |

### 2단계 계층 구조 (확정)

최대 **2단계**로 제한. 별도 Program 엔티티 없이 `ProjectGroup`의 `parentGroupId`로 구현.

```
Level 1 (최상위 그룹)          Level 2 (하위 그룹)       소속 프로젝트
─────────────────────────────────────────────────────────────────────
신공장 건설 프로그램   ──────►  A동 건설          ──► P001, P002, P003
  (CUSTOM)                     B동 건설          ──► P004, P005
                                인프라 구축        ──► P006, P007, P008
                         └── 직접 소속                ► P009 (L2 없이 L1에 직접)

설비팀                   ──── (하위 그룹 없음) ────► P010, P011 ... P047
  (DEPARTMENT)
```

### 롤업 집계 표시 항목 (그룹 행)

| 항목 | 계산 방식 | 표시 |
|------|-----------|------|
| 총 프로젝트 수 | DISTINCT 소속 프로젝트 수 | `45개` |
| 이슈 수 | 소속 프로젝트 이슈 합산 | `🔴×2 🟠×6` |
| 진행률 | Σ(plannedBudget × progress) / Σ(plannedBudget) | `avg 64%` |
| 예산 | Σ(actualBudget) / Σ(plannedBudget) | `12.5억/18억` |
| 완료 예정일 | max(projectedEndDate) | `▶ 2026-06-30` |

> **중복 방지**: 프로젝트가 L1과 L2 양쪽에 소속된 경우 L1 집계 시 DISTINCT 처리

- 하나의 프로젝트가 **여러 그룹에 동시 소속** 가능 (N:M 관계)
- 그룹 설정은 Admin이 조직 레벨로 관리, 개인은 즐겨찾기·핀 설정만

## 9. 기술 고려사항 (Technical Considerations)

### 아키텍처 접근

대시보드는 **project-service 내 별도 모듈**로 구현 (신규 마이크로서비스 불필요).

```
project-service
  └── src/
      ├── api/
      │   └── dashboard/          ← 대시보드 전용 라우터
      ├── application/
      │   └── dashboard/
      │       ├── DashboardService.ts        ← 집계 오케스트레이터
      │       ├── ProjectGroupService.ts     ← 그룹 관리
      │       └── IssueDetectorService.ts    ← 이슈 감지 엔진
      └── infrastructure/
          └── cache/
              └── DashboardCacheRepository.ts ← Redis 캐시 레이어
```

### 핵심 도메인 엔티티 (신규)

```
ProjectGroup (최대 2단계)
  ├── id: string
  ├── name: string                    ← "신공장 건설 프로그램", "A동 건설"
  ├── type: "DEPARTMENT"|"CLIENT"|"CUSTOM"
  ├── level: 1 | 2                   ← 1=최상위, 2=하위 (3단계 생성 불가)
  ├── parentGroupId: string?         ← level=2만 사용
  ├── color: string
  ├── sortOrder: Int                 ← 동일 레벨·부모 내 표시 순서
  ├── description: string?
  └── ProjectGroupMembership (1:N)

ProjectGroupMembership
  ├── groupId: string
  ├── projectId: string
  └── displayOrder: Int              ← 그룹 내 프로젝트 표시 순서

DashboardConfig                  ← 사용자별 대시보드 설정
  ├── userId: string
  ├── defaultGroupBy: GroupType
  ├── pinnedProjectIds: string[] ← 즐겨찾기 프로젝트
  ├── expandedGroupIds: string[] ← 펼쳐진 그룹 상태 기억
  └── issueFilters: Json         ← 활성 필터 저장

DashboardCache (Redis)           ← 집계 캐시 (DB 엔티티 아님)
  Key: "dashboard:{projectId}:summary"
  TTL: 300s (5분)
  Value: {
    status: "GREEN"|"AMBER"|"RED",
    issues: Issue[],
    weeklyEvents: WeeklyEvent[],
    progress: number,
    projectedEndDate: Date
  }
```

### 성능 전략 (수백 개 프로젝트)

```
[요청 흐름]

Client → GET /api/v1/dashboard?groupBy=DEPARTMENT&date=today

         ↓
DashboardService
  1. 그룹 목록 조회 (DB, 캐시 가능)
  2. 각 프로젝트 요약 조회
       → Redis 캐시 HIT: 즉시 반환
       → Redis 캐시 MISS: DB 집계 후 캐시 저장 (TTL 5분)
  3. 그룹별 집계 (인메모리)
  4. 응답 반환

[백그라운드 갱신 Job]
  Cron: 매 5분
  - 활성 프로젝트 순회
  - 이슈 감지 (IssueDetectorService)
  - Redis 캐시 업데이트
  - 새 Critical 이슈 발생 시 → WebSocket broadcast
```

### ±1주일 미니 타임라인 데이터

```typescript
interface WeeklyEvent {
  date: Date;                           // 이벤트 날짜
  type: "SEGMENT_START" | "SEGMENT_END" | "MILESTONE" | "SEGMENT_ACTIVE";
  taskName: string;
  segmentName?: string;
  isDelayed: boolean;
  delayDays?: number;
  isCriticalPath: boolean;
}

// 프로젝트별 ±7일 이벤트 집계 쿼리 (세그먼트 기반)
SELECT
  ts.start_date, ts.end_date, ts.name,
  t.name as task_name,
  t.is_critical_path
FROM task_segments ts
JOIN tasks t ON t.id = ts.task_id
WHERE t.project_id = :projectId
  AND ts.start_date <= :endWindow    -- today + 7
  AND ts.end_date   >= :startWindow  -- today - 7
ORDER BY ts.start_date;
```
