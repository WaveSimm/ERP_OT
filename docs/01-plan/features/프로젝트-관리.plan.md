## 1. 개요 (Overview)

ERP/OT 플랫폼 내에서 산업 현장의 프로젝트(공사, 유지보수, 설비 설치 등)를 계획하고, 추적하며, 완료 처리하는 통합 프로젝트 관리 모듈. 팀 간 **실시간 협업**, **정밀 자원 할당(% / 시간)**, **자원 운영 현황 모니터링**, **일정 버전 관리 및 영향 분석**을 핵심 기능으로 포함한다.

### 배경 (Background)

OT(Operational Technology) 환경에서는 설비 설치, 정기 유지보수, 공장 확장, 품질 개선 등 다수의 프로젝트가 동시에 진행된다. 현재는 스프레드시트나 개별 툴로 분산 관리되어 실시간 현황 파악이 어렵고, ERP 재무/자원 데이터와 연계가 없다.

## 2. 문제 정의 (Problem Statement)

| 문제 | 영향 |
|------|------|
| 프로젝트 현황 파악을 위해 여러 시스템을 확인해야 함 | 의사결정 지연 |
| 작업 지시(Work Order)와 프로젝트가 연결되지 않음 | 공정 누락 위험 |
| 자원 배정이 "담당자 지정" 수준에 그쳐 실투입 시간/비율을 알 수 없음 | 자원 과부하 / 유휴 발생 |
| 동일 자원이 여러 프로젝트에 몇 % 투입되는지 한눈에 볼 수 없음 | 자원 중복 배정 |
| 프로젝트 내 의사소통이 이메일/메신저로 분산 → 맥락 유실 | 협업 효율 저하 |
| 예산 소비 현황을 실시간으로 볼 수 없음 | 예산 초과 위험 |
| 지연 프로젝트 조기 감지 불가 | 납기 리스크 |
| 일정이 바뀌어도 최초 계획 대비 얼마나 달라졌는지 추적 불가 | 원인 분석 불가 |
| 태스크 지연이 후속 태스크에 미치는 영향을 사전에 알 수 없음 | 연쇄 지연 / 납기 초과 |
| 하나의 작업이 날짜가 다른 여러 구간으로 나뉘는데 단일 일정으로만 표현됨 | 실제 공정 반영 불가 |
| 구간별로 투입 인원이 달라도 태스크 전체에 동일 자원이 배정된 것처럼 보임 | 자원 계획 오차 |
| 비슷한 유형의 프로젝트를 매번 처음부터 계획해야 함 | 초기 계획 수립 과다 소요 |
| 완료된 프로젝트의 WBS를 새 프로젝트에 재활용할 방법이 없음 | 경험치 누적 불가 |
| 대형 프로그램(예: 신공장 건설) 산하 세부 프로젝트들을 계층적으로 묶어볼 수 없음 | 프로그램 단위 현황 파악 불가 |
| 그룹 단위 예산·진행률 합산이 안 됨 | 부문장 보고 시 수작업 집계 |

## 5. 사용자 스토리 (User Stories)

### 핵심 스토리

**US-01**: Manager로서 신규 프로젝트를 생성하고, 마일스톤과 WBS(Work Breakdown Structure)를 정의할 수 있다.

**US-02**: Manager로서 팀원과 장비를 특정 태스크에 배정할 때, **투입 비율(%)** 또는 **시간(h/day, h/week)** 중 원하는 단위로 지정할 수 있다.
> 예) 김철수 → 이번 태스크에 50% 또는 4h/day 배정

**US-03**: Manager로서 자원을 배정하려 할 때, 해당 자원의 **현재 다른 프로젝트 포함 총 배정률**을 실시간으로 확인하고 초과 시 경고를 받는다.
> 예) 김철수: 프로젝트A 30% + 프로젝트B 40% = 현재 70% → 신규 30% 추가 가능

**US-04**: Admin/Manager로서 **자원 운영 현황 대시보드**에서 각 자원(인력/장비)의 기간별 가용율, 투입 프로젝트 목록, 과부하/유휴 상태를 한 화면에서 본다.

**US-05**: Operator로서 내 담당 태스크에 **댓글을 작성하고 팀원을 @멘션**할 수 있으며, 멘션된 사람에게 인앱 알림이 간다.

**US-06**: Manager/Operator로서 태스크에 **파일을 첨부**하고(도면, 작업지시서 등), 팀원이 동일 태스크 내에서 파일을 확인한다.

**US-07**: 프로젝트 참여자 전원이 **활동 피드(Activity Feed)**를 통해 "누가, 언제, 무엇을 변경했는지" 실시간으로 확인한다.
> 예) "홍길동이 태스크 [배관 설치]의 진행률을 60%로 업데이트했습니다"

**US-08**: Operator로서 내 담당 태스크의 진행률을 업데이트하고, 이슈가 있으면 보고할 수 있다.

**US-09**: Manager로서 간트 차트로 전체 일정을 시각화하고, 지연 태스크를 한눈에 확인한다.

**US-10**: Admin/Manager로서 프로젝트별 예산 투입/잔여를 실시간으로 확인한다.

**US-11**: 시스템이 태스크 지연 위험(예정일 - 오늘 < 3일, 진행률 < 50%)을 자동 감지하고 담당자에게 알림을 보낸다.

**US-12**: Manager로서 프로젝트 완료 후 실적 보고서(일정 준수율, 예산 집행률, **자원 투입 실적**)를 생성한다.

**US-13**: Manager로서 프로젝트 계획 확정 시점에 **기준선(Baseline)을 저장**하고, 이후 모든 일정 변경이 이 기준 대비 얼마나 달라졌는지 실시간으로 비교한다.
> 예) "배관 설치" 태스크: 기준 3/1→3/15 → 현재 3/5→3/22 (+4일 지연)

**US-14**: Manager로서 **다중 기준선**을 관리한다 (Baseline 1 = 최초, Baseline 2 = 1차 변경 승인 후).
> 변경 이력: "2026-03-01 홍길동이 Baseline 2를 저장했습니다. 사유: 자재 납품 지연 반영"

**US-15**: Manager/Viewer로서 **간트 차트에서 기준선 바(ghost bar)**와 현재 진행 바를 함께 보고, 태스크별 편차(일수)를 컬러로 구분한다.
```
[기준선 ░░░░░░]        ← 반투명 회색
[현재   ████████]  +5d ← 지연 시 빨강, 단축 시 파랑
```

**US-16**: Manager로서 **태스크 일정 변경 이력**을 조회한다. 누가 언제 어떤 필드를 변경했는지, 변경 사유가 함께 기록된다.
> 예) 2026-03-10 / 김철수 / endDate: 2026-03-15 → 2026-03-20 / 사유: "자재 조달 지연"

**US-17**: Manager로서 태스크 일정을 변경할 때, 시스템이 **후속 태스크(의존성 체인)에 미치는 영향을 즉시 계산**하여 보여준다.
```
[배관 설치] +5일 지연 시 영향:
  ├── [전기 배선] +5일 (FS 의존)     → 예상 완료: 4/10 → 4/15
  ├── [단열 공사] +5일 (FS 의존)     → 예상 완료: 4/12 → 4/17
  └── [최종 검수] +5일 (FS 의존)     → 예상 완료: 4/20 → 4/25
  📌 프로젝트 완료일: 4/20 → 4/25 (+5일 지연)
```

**US-18**: Manager로서 **What-if 시뮬레이션**을 통해 "만약 태스크 A가 N일 지연된다면?" 시나리오를 실제 저장 없이 미리 시뮬레이션하고 영향 리포트를 확인한다.

**US-19**: 시스템이 **크리티컬 패스(Critical Path)**를 자동 계산하고, 간트 차트에서 강조 표시한다. 크리티컬 패스 상의 태스크가 지연되면 즉시 경고한다.

**US-23**: Manager로서 하나의 태스크를 **여러 개의 세그먼트(구간)**로 나눌 수 있다. 각 세그먼트는 독립적인 일정(시작·종료일)과 인원 배정을 가진다.

> **예시 — "배관 설치" 태스크:**
> | 세그먼트 | 기간 | 배정 인원 |
> |---------|------|----------|
> | 1. 지하 배관 | 3/1 ~ 3/5 | 배관팀 3명 (50%) |
> | 2. 지상 배관 | 3/10 ~ 3/20 | 배관팀 5명 (100%) + 크레인 1대 |
> | 3. 수압 테스트 | 3/22 ~ 3/25 | QA팀 2명 (30%) |
> → 세그먼트 사이 공백(3/6~3/9)은 자재 대기 기간

**US-24**: Manager로서 세그먼트별 자원 배정을 독립적으로 설정한다. 같은 자원도 세그먼트마다 다른 투입 비율(% 또는 h/day)로 배정할 수 있다.
> 예) 김철수: 세그먼트1 → 50%, 세그먼트2 → 100%, 세그먼트3 → 미배정

**US-25**: 간트 차트에서 다중 세그먼트 태스크를 **하나의 행에 분리된 바(bar) 여러 개**로 표시하고, 공백 구간은 점선으로 연결한다.
```
배관 설치  [■■■■]···[■■■■■■■■■■]···[■■■]
           지하배관    지상배관       수압테스트
           3/1~3/5   3/10~3/20     3/22~3/25
```

**US-26**: Operator로서 태스크 전체 진행률과 **세그먼트별 진행률**을 각각 업데이트한다. 전체 진행률은 세그먼트 기간 가중 평균으로 자동 계산되며, 수동 오버라이드도 가능하다.

**US-27**: Manager로서 태스크에 세그먼트를 추가/삭제/순서 변경할 때, 해당 변경이 **Baseline 이력 및 영향 분석**에 자동으로 반영된다.

### 부가 스토리

**US-28**: Work Order와 프로젝트 태스크(또는 세그먼트)를 연결하여 현장 작업 현황이 프로젝트 진행률에 반영된다.

### 템플릿 & 복사 스토리 (v1.4)

**US-31**: Manager로서 **템플릿 라이브러리**를 열면 조직 공용 템플릿과 내 개인 템플릿이 카테고리별로 분류되어 보인다 (예: "배관 공사", "전기 설비", "정기 유지보수").

**US-32**: Manager로서 템플릿을 선택하고 **프로젝트 시작일만 입력**하면, 템플릿에 저장된 상대 오프셋(dayOffset)이 절대 날짜로 자동 변환되어 WBS 전체가 생성된다.
> 예) 템플릿 "A라인 배관 공사" 시작일 3/1 입력 →
> - 세그먼트1 "지하 배관": dayOffset 0~4 → 3/1~3/5
> - 세그먼트2 "지상 배관": dayOffset 9~19 → 3/10~3/20 (Gap 4일 자동 보존)
> - 세그먼트3 "수압 테스트": dayOffset 21~24 → 3/22~3/25

**US-33**: Manager로서 템플릿 인스턴스화 전 **미리보기 화면**에서 계산된 일정 전체를 확인하고, 개별 날짜를 조정한 뒤 생성한다.

**US-34**: Manager로서 템플릿에서 **원하는 마일스톤/태스크만 선택적으로 가져와** 현재 프로젝트에 추가한다 (부분 가져오기).

**US-35**: Manager로서 현재 프로젝트 또는 완료된 프로젝트의 WBS를 **템플릿으로 저장**한다. 저장 시 실적 데이터(진행률, 실제 투입 실적)는 제외되고 구조와 기간만 저장된다.
> 저장 옵션:
> - 자원 배정 포함 여부 (역할만 or 특정 인원)
> - 공용 공개 or 개인 저장
> - 카테고리 및 태그 지정

**US-36**: Manager로서 **태스크를 복사**한다. 복사 시 다음을 선택한다:
- 복사 대상: 같은 프로젝트 내 다른 마일스톤 / 다른 프로젝트
- 세그먼트 포함 여부 (포함 시 구조 동일하게 복사)
- 자원 배정 포함 여부 (포함 시 담당자 그대로 / 미포함 시 빈 상태)
- 날짜 오프셋: +N일 이동 (예: 원본보다 2주 뒤로)

**US-37**: Manager로서 **마일스톤 단위로 복사**한다. 하위 태스크 전체와 태스크 간 의존성이 그룹 내에서 보존된다.

**US-38**: Manager로서 **프로젝트 전체를 복제**한다 (새 프로젝트로). 시작일을 지정하면 모든 일정이 오프셋으로 재계산되며, 실적 데이터는 초기화된다.

**US-39**: 복사 또는 템플릿 인스턴스화 완료 후, 생성된 태스크들에 대해 **일괄 날짜 밀기/당기기**를 적용할 수 있다 (선택한 태스크를 +/-N일 일괄 이동).

**US-40**: Admin으로서 자주 쓰이는 템플릿에 **권장(Recommended) 뱃지**를 붙이고, 조직 표준 WBS를 사용하도록 가이드한다.

**US-30**: Manager로서 **자원 부하 히트맵**을 통해 월별/주별로 어떤 자원이 집중 투입되는지 시각화한다.

### 그룹 계층 스토리 (v1.5)

**US-41**: Admin으로서 **Level 1 그룹**(부서·고객사·커스텀 프로그램)을 생성하고, 그 아래 **Level 2 하위 그룹**을 추가한다. 계층은 최대 2단계로 제한된다.
> 예) L1: "신공장 건설 프로그램" → L2: "A동 건설", "B동 건설", "인프라 구축"

**US-42**: Manager로서 프로젝트를 특정 그룹(L1 또는 L2)에 소속시킨다. 하나의 프로젝트가 여러 그룹에 동시 소속될 수 있다.

**US-43**: 대시보드에서 **L1 그룹을 펼치면 L2 하위 그룹과 직접 소속 프로젝트**가 accordion으로 표시된다.
```
▼ [L1: 신공장 건설 프로그램]  45개  🔴×2  avg 64%  12.5억/18억
  ├── ▼ [L2: A동 건설]  12개  🔴×1  68%
  │     프로젝트 목록...
  ├── ▼ [L2: B동 건설]  18개  🟢    72%
  └── ▼ [L2: 인프라 구축]  15개  🟠×3  55%
```

**US-44**: L1·L2 그룹 행에 **롤업 집계**가 자동 표시된다.
- **진행률**: 소속 프로젝트 예산 가중 평균
- **예산**: 소속 프로젝트 계획 예산 합계 / 실적 합계
- **이슈**: 소속 프로젝트 이슈 수 합산 (Critical / Warning)
- **기간**: 가장 이른 시작일 ~ 가장 늦은 완료 예정일

**US-45**: Manager로서 그룹 내 **프로젝트 표시 순서(displayOrder)**를 드래그로 변경한다. (중요도·착수 순서 반영)

**US-46**: Admin으로서 그룹을 삭제할 때, 소속 프로젝트는 삭제되지 않고 **그룹 소속만 해제**된다. 하위 그룹(L2)이 있는 L1 그룹은 하위 그룹을 먼저 처리해야 삭제 가능하다.

**US-47**: Manager로서 L1 그룹 단위로 **그룹 대시보드 뷰**에 접근한다. 해당 그룹 소속 프로젝트만 필터링된 지휘센터 대시보드 뷰.

### 핵심 도메인 엔티티

> **v1.3 구조 변경**: `Task`는 논리적 컨테이너가 되고, 실제 일정·자원은 `TaskSegment`로 이동.

```
Project
  ├── Milestone (1:N)
  │   └── Task (1:N)                        ← 논리적 작업 단위 (컨테이너)
  │       │  ※ 직접적인 startDate/endDate 없음
  │       │  ※ effectiveStartDate = min(segments.startDate) [computed]
  │       │  ※ effectiveEndDate   = max(segments.endDate)   [computed]
  │       │  ※ overallProgress   = 가중평균(segments) 또는 수동 오버라이드
  │       │
  │       ├── TaskSegment (1:N)              ← 구간별 독립 일정 + 자원
  │       │     ├── id: string
  │       │     ├── name: string             ← "지하 배관", "지상 배관", "테스트"
  │       │     ├── sortOrder: Int           ← 표시 순서
  │       │     ├── startDate: Date
  │       │     ├── endDate: Date
  │       │     ├── progressPercent: Float   ← 세그먼트별 진행률
  │       │     └── SegmentAssignment (1:N)  ← 이 구간의 자원 배정
  │       │           ├── resourceId: string
  │       │           ├── allocationPercent: Float?      ← % 단위
  │       │           ├── allocationHoursPerDay: Float?  ← 시간 단위
  │       │           └── allocationMode: "PERCENT"|"HOURS"
  │       │
  │       ├── TaskDependency (1:N)           ← 의존성은 Task 레벨 유지
  │       ├── Comment (1:N)
  │       │     └── Mention (1:N) → User
  │       └── Attachment (1:N)
  │
  ├── Budget
  ├── ActivityLog (1:N)
  └── ProjectTemplate                         ← 상세 구조는 아래 별도 섹션 참조

Resource (인력 / 장비)
  ├── dailyCapacityHours: Float              ← 기준 근무시간 (기본 8h)
  └── ResourceAllocationSummary             ← 크로스-프로젝트 집계 (세그먼트 기반)
```

### Task vs TaskSegment 역할 분리

| 속성 | Task (컨테이너) | TaskSegment (실행 단위) |
|------|----------------|------------------------|
| 이름 | "배관 설치" | "지하 배관", "지상 배관", "테스트" |
| 일정 | effectiveStart/End (계산값) | startDate, endDate (직접 입력) |
| 자원 배정 | 없음 | SegmentAssignment (세그먼트별) |
| 진행률 | 가중평균 (자동) 또는 수동 | 세그먼트별 직접 입력 |
| 의존성 | Task 레벨 FS/SS/FF/SF | — (세그먼트 간 의존성은 v2.0) |
| 댓글/첨부 | Task 레벨 | — |

### 세그먼트 진행률 계산

```
Task.overallProgress =
  Σ (segment.progressPercent × segment.durationDays)
  ─────────────────────────────────────────────────
           Σ (segment.durationDays)

단, 공백 기간(Gap)은 durationDays 계산에서 제외
수동 오버라이드 시 isManualProgress = true → 자동계산 비활성화
```

─── 그룹 계층 (v1.5) ─────────────────────────────────

ProjectGroup (최대 2단계, 재귀 없음)
  ├── id: string
  ├── name: string                    ← "신공장 건설 프로그램", "A동 건설"
  ├── type: "DEPARTMENT"|"CLIENT"|"CUSTOM"
  ├── level: 1 | 2                   ← DB 단계 명시 (순환 방지)
  ├── parentGroupId: string?         ← level=2만 사용, level=1은 null
  ├── color: string                  ← 그룹 구분 색상
  ├── sortOrder: Int                 ← 동일 레벨·부모 내 표시 순서

### 의존성 유형 (Dependency Types)

| 유형 | 의미 | 예시 |
|------|------|------|
| **FS** (Finish-to-Start) | 선행 완료 후 후행 시작 | 기초공사 완료 → 골조 시작 |
| **SS** (Start-to-Start) | 선행 시작 후 후행 시작 | 배관 시작 → 단열 시작 |
| **FF** (Finish-to-Finish) | 선행 완료와 동시에 후행 완료 | 검사 완료 = 인수 완료 |
| **SF** (Start-to-Finish) | 선행 시작 후 후행 완료 | 신설비 가동 시작 → 구설비 운영 종료 |

### 자원 할당 모드 상세 (세그먼트 단위)

| 모드 | 입력 | 내부 저장 | 표시 |
|------|------|-----------|------|
| PERCENT | 50% | `allocationPercent = 50` | "50% (4.0h/day)" |
| HOURS | 4h/day | `allocationHoursPerDay = 4` | "4.0h/day (50%)" |

- 환산 공식: `percent = (hoursPerDay / dailyCapacityHours) × 100`
- `dailyCapacityHours` 기본값: 8h (조직 설정에서 변경 가능)
- **같은 자원, 세그먼트마다 다른 비율** 완전 지원
  ```
  배관팀 김철수:
    세그먼트1 (3/1~3/5)  → 50%  (4h/day)
    세그먼트2 (3/10~3/20) → 100% (8h/day)
    세그먼트3 (3/22~3/25) → 미배정
  ```

### 자원 운영 현황 집계 쿼리 (세그먼트 기반)

```sql
-- 특정 자원의 특정 기간 총 배정률 (세그먼트 기반)
SELECT
  r.id,
  r.name,
  SUM(sa.allocation_percent) AS total_allocation_percent,
  100 - SUM(sa.allocation_percent) AS available_percent
FROM resources r
JOIN segment_assignments sa ON sa.resource_id = r.id
JOIN task_segments ts ON ts.id = sa.segment_id   -- ← TaskSegment 기반
WHERE ts.start_date <= :end_date
  AND ts.end_date >= :start_date
GROUP BY r.id, r.name;
-- 세그먼트별 기간이 조회 기간과 겹치는 경우만 집계 → 정확한 실투입 산정
```

### CPM (Critical Path Method) 알고리즘 — 세그먼트 모델 적용

```
태스크의 CPM 입력값:
  ES(task) = min(segment.startDate)   ← effectiveStartDate
  EF(task) = max(segment.endDate)     ← effectiveEndDate
  duration(task) = EF - ES (공백 포함)

Forward Pass (ES/EF 계산):
  ES(i) = max(EF(predecessor) + lag)
  EF(i) = ES(i) + effectiveDuration(i)

Backward Pass (LS/LF 계산):
  LF(i) = min(LS(successor) - lag)
  LS(i) = LF(i) - effectiveDuration(i)

Total Float(i) = LS(i) - ES(i)
Critical Path  = Float == 0 인 태스크 경로

※ 세그먼트 간 공백(Gap)은 태스크의 effectiveDuration에 포함됨
   → 공백이 있는 태스크는 Float가 더 적게 계산될 수 있음 (보수적 접근)
```

**일정 변경 영향 전파 알고리즘**

```typescript
// 태스크 T가 N일 지연될 때 영향받는 후속 태스크 계산
function propagateDelay(taskId: string, delayDays: number): ImpactResult {
  const affected = [];
  const queue = [{ taskId, delay: delayDays }];

  while (queue.length > 0) {
    const { taskId, delay } = queue.shift();
    const successors = getDependencies(taskId); // FS/SS/FF/SF

    for (const dep of successors) {
      const propagatedDelay = calculatePropagatedDelay(dep.type, delay);
      if (propagatedDelay > dep.successor.totalFloat) {
        // Float 초과 → 실제 지연 발생
        affected.push({ taskId: dep.successorId, delayDays: propagatedDelay });
        queue.push({ taskId: dep.successorId, delay: propagatedDelay });
      }
    }
  }
  return { affectedTasks: affected, projectedEndDate: recalcEndDate() };
}
```

**What-if 시뮬레이션**: 실제 DB 변경 없이 메모리 내 가상 그래프로 계산 후 결과만 반환.
