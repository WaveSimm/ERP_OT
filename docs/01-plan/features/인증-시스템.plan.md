# Plan: 인증-시스템 (Authentication System)

**버전**: v1.0
**작성일**: 2026-03-19
**상태**: Plan 완료

---

## 1. 배경 및 목적

현재 로그인은 개발용 placeholder로, 어떤 아이디/비밀번호든 ADMIN 권한으로 통과됩니다.
시스템을 인터넷에 오픈하기 전에 실제 인증/인가 시스템이 필요합니다.

### 현재 문제점
- `POST /api/auth/login` — 비밀번호 검증 없이 JWT 발급
- JWT `sub` = `user-${username}` (임의 문자열, DB ID 아님)
- 모든 로그인 사용자 = ADMIN 권한
- auth-service 라우트 미구현 (health check만 존재)
- 사용자 관리 UI 없음

---

## 2. 범위 (Scope)

### 포함
- auth-service 완전 구현 (로그인/로그아웃/토큰 갱신)
- 사용자 CRUD (관리자 전용)
- 비밀번호 변경 (본인)
- 관리자 비밀번호 초기화
- 프론트엔드 로그인 페이지 실제 인증 연동
- 사용자 관리 페이지 (관리자 전용)
- AppLayout 현재 사용자 정보 표시
- 기존 Resource.userId를 실제 User ID로 마이그레이션

### 제외
- 소셜 로그인 (Google, GitHub)
- 2FA / OTP
- SSO / LDAP 연동
- 비밀번호 찾기 이메일 발송

## 3. 사용자 스토리

### 인증 (US-01 ~ US-05)

| ID | 스토리 | 우선순위 |
|----|--------|:--------:|
| US-01 | 사용자는 이메일+비밀번호로 로그인하고 JWT를 받을 수 있다 | P0 |
| US-02 | Access Token 만료 시 Refresh Token으로 자동 갱신된다 | P0 |
| US-03 | 사용자는 로그아웃하여 토큰을 무효화할 수 있다 | P0 |
| US-04 | 잘못된 이메일/비밀번호 입력 시 명확한 에러가 표시된다 | P0 |
| US-05 | 비활성화된 계정은 로그인이 거부된다 | P0 |

### 사용자 관리 (US-06 ~ US-10)

| ID | 스토리 | 우선순위 |
|----|--------|:--------:|
| US-06 | ADMIN은 새 사용자를 생성할 수 있다 (이름, 이메일, 역할) | P0 |
| US-07 | ADMIN은 사용자 역할과 활성/비활성을 변경할 수 있다 | P0 |
| US-08 | ADMIN은 사용자 비밀번호를 초기화할 수 있다 | P1 |
| US-09 | 사용자는 본인 비밀번호를 변경할 수 있다 | P1 |
| US-10 | ADMIN은 사용자 목록을 조회할 수 있다 | P0 |

### 마이그레이션 (US-11)

| ID | 스토리 | 우선순위 |
|----|--------|:--------:|
| US-11 | 기존 Resource의 userId를 실제 User DB ID로 매핑한다 | P0 |

## 4. 기술 결정사항

### 토큰 전략
- **Access Token**: JWT, 만료 8시간, Authorization 헤더
- **Refresh Token**: JWT, 만료 7일, httpOnly 쿠키 저장
- **JWT Secret**: 환경변수 `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET` (별도)
- **JWT Sub**: 실제 User DB ID (cuid)

### 비밀번호
- **해시**: bcrypt, salt rounds 12
- **최소 조건**: 8자 이상 (초기 단순화)

### 역할 (Role)
| 역할 | 권한 |
|------|------|
| ADMIN | 전체 관리 (사용자 생성/삭제 포함) |
| MANAGER | 프로젝트 생성/수정, 태스크 관리 |
| OPERATOR | 태스크 상태 변경, 진행률 업데이트 |
| VIEWER | 읽기 전용 |

### 초기 관리자 계정
- DB seed로 admin 계정 자동 생성
- 이메일: `admin@erp-ot.local`
- 초기 비밀번호: `.env`의 `ADMIN_INITIAL_PASSWORD`

## 5. 구현 범위

### Phase 1 — auth-service 구현
```
services/auth/src/
├── api/
│   ├── routes/
│   │   ├── auth.routes.ts       # 로그인/로그아웃/갱신
│   │   └── user.routes.ts       # 사용자 CRUD (admin)
│   ├── middleware/
│   │   └── auth.middleware.ts   # JWT 검증
│   └── dtos/
│       ├── auth.dto.ts
│       └── user.dto.ts
├── application/
│   ├── auth.service.ts          # 로그인/토큰 발급
│   └── user.service.ts          # 사용자 CRUD
├── infrastructure/
│   └── repositories/
│       └── user.repository.ts   # Prisma 구현
└── index.ts                     # 서버 진입점 완성
```

### Phase 2 — Next.js 연동
```
apps/web/src/
├── app/
│   ├── api/auth/login/route.ts  # auth-service 프록시로 교체
│   ├── login/page.tsx           # 이메일+비밀번호 폼
│   └── admin/
│       └── users/page.tsx       # 사용자 관리 페이지 (신규)
├── components/
│   └── AppLayout.tsx            # 현재 사용자 이름/역할 표시
└── lib/api.ts                   # authApi 추가
```

### Phase 3 — 마이그레이션
- Resource.userId: `user-${name}` → 실제 User.id 업데이트
- 로그인 이름 기반 자동 매핑 또는 관리자 수동 매핑 UI

## 6. API 명세 (예정)

| Method | Path | 설명 | 권한 |
|--------|------|------|------|
| POST | `/api/v1/auth/login` | 로그인 | 공개 |
| POST | `/api/v1/auth/refresh` | 토큰 갱신 | 공개 |
| POST | `/api/v1/auth/logout` | 로그아웃 | 인증 |
| GET | `/api/v1/auth/me` | 내 정보 | 인증 |
| PATCH | `/api/v1/auth/me/password` | 비밀번호 변경 | 인증 |
| GET | `/api/v1/users` | 사용자 목록 | ADMIN |
| POST | `/api/v1/users` | 사용자 생성 | ADMIN |
| PATCH | `/api/v1/users/:id` | 사용자 수정 | ADMIN |
| POST | `/api/v1/users/:id/reset-password` | 비밀번호 초기화 | ADMIN |
| PATCH | `/api/v1/users/:id/deactivate` | 비활성화 | ADMIN |

## 7. 영향 받는 기존 코드

| 파일 | 변경 내용 |
|------|-----------|
| `apps/web/src/app/api/auth/login/route.ts` | auth-service 실제 호출로 교체 |
| `apps/web/src/app/login/page.tsx` | 이메일 필드 추가, "개발 모드" 문구 제거 |
| `apps/web/src/components/AppLayout.tsx` | 현재 사용자 이름/역할 표시 |
| `apps/web/src/lib/api.ts` | `devLogin` → `authApi.login` 으로 교체 |
| `project-service` auth middleware | JWT 검증 로직은 동일 (secret 공유) |
| `project.resources.userId` | 실제 User ID로 업데이트 필요 |

## 8. 비고

- auth-service는 이미 Prisma schema, User 엔티티, 기본 구조가 스캐폴딩됨
- project-service의 JWT 검증 미들웨어는 동일 secret을 사용하므로 별도 수정 불필요
- 초기 배포 시 관리자가 직접 사용자 계정을 생성하는 방식 (셀프 회원가입 없음)
