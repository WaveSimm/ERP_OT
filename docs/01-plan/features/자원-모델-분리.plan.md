> **Feature**: 자원-모델-분리
> **Phase**: Plan

> **Status**: Draft (사용자 합의 완료, 검토 대기)

## 1. 개요 (Overview)

현재 `project.resources` 테이블이 사람·장비·차량·시설을 한 모델에 묶어 관리하면서 발생하는 **이중 ID 시스템**과 **모델 의미 혼란**을 해결한다.

핵심 변경:
- **PERSON Resource 폐기** → 사람은 `auth_users`로 직접 다룸
- **EquipmentResource 신규** → 비인력(장비/차량/시설) 전용
- **ExternalPerson 신규** → 외주/협력업체 전용
- **AuthUser status 확장** → 현직/퇴직자/정지 상태 관리
- **SegmentAssignment polymorphic** → personUserId / externalPersonId / equipmentResourceId 셋 중 하나

## 2. 배경 (Background)

### 2.1 현재 데이터 구조 (cleanup 후 2026-05-04)

```
auth_users          : 68명 (실제 직원 64 + 관리계정 2 + 잔재 2)
project.resources   : 68 PERSON + 1 VEHICLE + 1 FACILITY = 70개
SegmentAssignment   : 295건 (전부 PERSON 자원에만 배정)
```

비인력 자원: 2개만, 배정 0건 → **사실상 미사용**.
인력 자원: 156개 매핑 중 154개 email로 auth_users와 연결 → **이중 ID로 매번 변환 필요**.

### 2.2 문제

| 문제 | 증상 |
|---|---|
| **이중 ID 시스템** | `resources.userId`(email) ↔ `auth_users.id`(CUID) ↔ `attendance.userId`(CUID) 매번 변환 |
| **자원 모델의 의미 혼란** | "인력자원 탭"은 이미 `/admin/users`로 이동(2026-05-04). DB만 한 테이블 |
| **휴가/근태 머지 실패** | 인력현황 타임라인 Phase 3에서 발견 — Resource(email) → auth_users(CUID) 변환 필요 |
| **자동 동기화 hook 부담** | `apps/web/src/app/resources/page.tsx:251-260` auth→Resource 단방향 동기화 hook |
| **모델 발전 어려움** | 사람과 장비를 한 모델에 묶어 각각 필요한 컬럼(휴가·검교정 등) 추가 어려움 |
| **외주/퇴직자 처리 불명확** | "Resource로만 등록된 외부인" 케이스 처리 안 됨 |

### 2.3 합의된 결정 (2026-05-04 토론 결과)

- **B안**: 모델 자체를 분리 (단순 UI 분리가 아닌 근본 해결)
- **P-2**: 직원·퇴직자는 `auth_users`(status로 구분), 외주는 별도 `ExternalPerson` 테이블, 비인력은 `EquipmentResource`

이유:
- P-1(단일 auth_users): 외주가 auth_users 들어가는 게 어색
- P-3(통합 Person + AuthAccount): 마이그레이션 너무 큼 (모든 userId 컬럼 변경)
- **P-2**: 마이그레이션 부담 중간, 의미 분리 명확, 기존 인증·휴가 로직 영향 작음

## 4. 사용자 스토리

### US-1: 직원 계정 라이프사이클
- **As a** HR/관리자
- **I want to** 신입은 ACTIVE로 등록, 퇴직자는 RETIRED로 변경
- **So that** 퇴직자도 과거 데이터 흔적은 보존되고, 새 배정·로그인은 차단된다
- **AC**: status=RETIRED 사용자는 사용자 검색·결재라인·새 task 배정 후보에서 자동 제외

### US-2: 외주 등록·종료
- **As a** PM
- **I want to** 외주를 ExternalPerson에 등록(회사명·계약기간 포함), 계약 종료 시 ARCHIVED 처리
- **So that** 외주가 task에 배정되지만 회사 시스템(로그인·휴가·결재)에는 들어오지 않는다
- **AC**: 외주는 자원 배정 picker에 나오지만, 결재라인·휴가 신청 화면에는 안 나옴

### US-3: 비인력 자원 관리
- **As a** 자원 관리자
- **I want to** 장비/차량/시설을 EquipmentResource에 등록·관리
- **So that** 향후 자산 번호·검교정·점검 주기 등 비인력 전용 속성을 자유 추가할 수 있다
- **AC**: 자원관리 페이지의 "비인력 자원" 탭에서 EquipmentResource CRUD

### US-4: 자원 배정 (사람·외주·장비 통합 picker)
- **As a** PM
- **I want to** task segment에 직원/외주/장비를 배정
- **So that** 한 segment에 여러 종류 자원을 자연스럽게 묶을 수 있다
- **AC**: 배정 모달에 "직원" / "외주" / "장비" 탭 또는 통합 검색 (각자 별도 ID 보존)

### US-5: 인력현황 타임라인 (회귀 보장)
- **As a** PM/팀장
- **I want to** 직원·외주별 부하·휴가·휴일근무를 타임라인으로 본다
- **So that** 자원 모델 분리 후에도 기존 화면이 그대로 작동한다
- **AC**: 직원은 auth_users.id 기준 직접 조회 (email lookup 불필요), 외주는 외주 부하만 표시

### US-6: 마이그레이션 무손실
- **As a** 시스템 관리자
- **I want to** 마이그레이션 후 모든 SegmentAssignment·과거 배정 이력이 그대로 보존
- **So that** 진행 중 task의 자원 배정이 끊기지 않는다
- **AC**: 마이그레이션 전후 SegmentAssignment 295건의 표시 자원이 100% 일치

## 5. 데이터 모델 변경

### 5.1 Before

```prisma
model Resource {
  id                 String       @id @default(cuid())
  userId             String?      @unique // email
  name               String
  type               ResourceType @default(PERSON)
  dailyCapacityHours Float        @default(8.0)
  isActive           Boolean      @default(true)
  ...
  @@map("resources") @@schema("project")
}

enum ResourceType { PERSON, EQUIPMENT, VEHICLE, FACILITY }

model SegmentAssignment {
  resourceId String  // → Resource.id
  ...
}
```

### 5.2 After

```prisma
// ─── auth schema (services/auth) ───────────────────────────

model User {                         // 기존 auth_users — 직원 (현직 + 퇴직자)
  id              String              @id @default(cuid())
  email           String              @unique
  name            String
  // 신규 컬럼
  status          EmployeeStatus      @default(ACTIVE)
  hireDate        DateTime?
  retirementDate  DateTime?
  loginEnabled    Boolean             @default(true)
  ...
  @@map("auth_users")
}

enum EmployeeStatus {
  ACTIVE       // 현직
  RETIRED      // 퇴직
  SUSPENDED    // 휴직·정지
}

// ─── project schema (services/project) ─────────────────────

model EquipmentResource {            // 비인력만 (Resource → 분리·rename)
  id           String         @id @default(cuid())
  name         String
  type         EquipmentType  @default(EQUIPMENT)
  isActive     Boolean        @default(true)
  // 향후 확장: serialNumber, location, calibrationCycle 등
  groupMemberships ResourceGroupMember[]
  assignments      SegmentAssignment[]
  @@map("equipment_resources") @@schema("project")
}

enum EquipmentType { EQUIPMENT, VEHICLE, FACILITY @@schema("project") }

model ExternalPerson {               // 외주
  id            String           @id @default(cuid())
  name          String
  company       String?
  contactEmail  String?
  contactPhone  String?
  status        ExternalStatus   @default(ACTIVE)
  contractStart DateTime?
  contractEnd   DateTime?
  groupMemberships ResourceGroupMember[]
  assignments      SegmentAssignment[]
  @@map("external_persons") @@schema("project")
}

enum ExternalStatus { ACTIVE, ARCHIVED @@schema("project") }

model SegmentAssignment {            // polymorphic FK (셋 중 하나만)
  id                    String         @id @default(cuid())
  segmentId             String
  // 셋 중 정확히 하나
  personUserId          String?        // → auth_users.id (cross-service, FK 없음)
  externalPersonId      String?        // → ExternalPerson.id
  equipmentResourceId   String?        // → EquipmentResource.id
  allocationMode        AllocationMode @default(PERCENT)
  allocationPercent     Float?
  allocationHoursPerDay Float?
  ...
  segment   TaskSegment       @relation(fields: [segmentId], references: [id], onDelete: Cascade)
  external  ExternalPerson?   @relation(fields: [externalPersonId], references: [id])
  equipment EquipmentResource? @relation(fields: [equipmentResourceId], references: [id])
  // FK 무결성: DB CHECK constraint로 XOR 강제
  @@check("(personUserId IS NOT NULL)::int + (externalPersonId IS NOT NULL)::int + (equipmentResourceId IS NOT NULL)::int = 1")
  @@map("segment_assignments") @@schema("project")
}

model ResourceGroupMember {           // 그룹 멤버도 polymorphic
  groupId               String
  personUserId          String?
  externalPersonId      String?
  equipmentResourceId   String?
  ...
  @@check(...)  // XOR
  @@map("resource_group_members") @@schema("project")
}

// 폐기:
// - Resource 모델
// - ResourceType enum
```

### 5.3 핵심 약속

- **personUserId는 cross-service 참조** — auth-service 소유라 FK 안 걸음 (기존 패턴 유지)
- **externalPersonId / equipmentResourceId는 FK** — 같은 project schema
- **CHECK constraint로 XOR 강제** — 데이터 무결성 보장
- **groupMembers도 polymorphic** — ResourceGroup 자체는 그대로 유지

## 6. 마이그레이션 계획 (4 Phase)

### Phase 1: 스키마 확장 (downtime 없음)

```sql
-- 1. AuthUser status 추가
ALTER TABLE auth_users ADD COLUMN status VARCHAR(20) DEFAULT 'ACTIVE';
ALTER TABLE auth_users ADD COLUMN hire_date DATE;
ALTER TABLE auth_users ADD COLUMN retirement_date DATE;
ALTER TABLE auth_users ADD COLUMN login_enabled BOOLEAN DEFAULT true;

-- 2. 신규 테이블 생성
CREATE TABLE project.equipment_resources (...);
CREATE TABLE project.external_persons (...);

-- 3. SegmentAssignment에 새 컬럼 추가 (nullable, 기존 resourceId 유지)
ALTER TABLE project.segment_assignments
  ADD COLUMN person_user_id TEXT,
  ADD COLUMN external_person_id TEXT,
  ADD COLUMN equipment_resource_id TEXT;

-- 4. ResourceGroupMember에도 동일 추가
ALTER TABLE project.resource_group_members
  ADD COLUMN person_user_id TEXT,
  ADD COLUMN external_person_id TEXT,
  ADD COLUMN equipment_resource_id TEXT;
```

### Phase 2: 데이터 마이그레이션 (1회 스크립트)

```

   - 모든 SegmentAssignment에 person_user_id/external_person_id/equipment_resource_id 정확히 하나
```

| Phase 3 인력현황 타임라인 단기 fix 폐기로 표시 일시 중단 | 화면 회귀 | 본 PDCA에서 우선 구현, 1~2일 내 복구 |
| `resources.userId` 마이그레이션 안 된 2명 (test 잔재) | 매핑 실패 | TestWorker/test1/test2 사전 정리 (별도 작업) |
| Polymorphic FK CHECK constraint가 마이그레이션 중간 단계에서 위반 | 마이그레이션 실패 | Phase 1에서는 CHECK 미적용 → Phase 2 완료 후 추가 |
| ResourceGroup 멤버 변환 누락 | 그룹 표시 깨짐 | Phase 2에서 group_members도 동시 변환·검증 |
| 외주 부하·통계 표시 누락 | UX 회귀 | US-5에 외주 표시 명시, design에 외주 통계 정의 |
| 마이그레이션 후 cleanup-load-test.sh 작동 안 함 | 부하테스트 후 더미 정리 못함 | 본 PDCA 작업 일부로 cleanup 갱신 + `--yes` 옵션 |

## 9. 성공 기준 (Definition of Done)

1. **데이터**: 마이그레이션 후 SegmentAssignment 295건의 표시 자원 100% 일치
2. **DB 무결성**: SegmentAssignment XOR CHECK constraint 위반 0건
3. **화면**: 자원관리·인력현황·배정 모달이 새 모델 기준으로 정상 작동
4. **회귀**: 기존 휴가/근태/결재 화면에 영향 없음 (수동 검증 시나리오)
5. **부하테스트**: seed-load-test → run-round → cleanup 순환 작동
6. **Match Rate**: 90% 이상
7. **문서**: design v1.0 + analysis + report 작성 완료
