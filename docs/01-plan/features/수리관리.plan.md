## 1. 개요 (Overview)

해외 제조사에서 수입한 장비/센서를 판매하는 수입사로서, **자사 보유 장비**와 **고객 판매 장비** 모두에 대한 AS(After-Sales Service) 전 과정을 관리하는 모듈. AS 접수부터 진단, 수리, 견적/청구, 부품 관리, 제조사 발송까지의 워크플로우를 통합 관리한다.

### 배경 (Background)

- 해외 제조사로부터 장비를 수입하여 고객에게 판매
- 판매 후 AS/수리 요청이 들어오면 자체 수리 또는 제조사에 발송하여 수리
- 교체 부품은 제조사에 별도 발주하여 재고 관리
- **현재 엑셀 기반 관리 시스템** 사용 중 (AS관리_2022_V10.xlsx, 장비점검이력.xlsx)
  - AS관리: 391건 이력, 29개 컬럼으로 관리
  - 점검보고서: 124컬럼 구조화 양식 (수작업 작성)
  - 장비점검이력: 최신 간소화 양식 (2단계 점검 플로우)
- AS 이력, 부품 재고, 견적서 등이 분산 관리되어 추적이 어려움

### 기존 시스템 분석 (AS-IS)

#### AS관리_2022_V10.xlsx — 메인 관리 시트
- **관리시트**: 391건, 핵심 필드:
  - 접수: 순번, 접수일, 접수번호, 접수담당, 접수증상
  - 장비: 제품명, 제작사, 일련번호, OT재고NO, 제조년월, 구매일
  - 고객: 회사명, 담당자(부서), 담당연락처
  - 추적: 현재위치, 무상여부, 요청일
  - **이중 진행상황**: 점검진행상황(기술팀) + 영업부진행상황(영업팀) — 핵심 워크플로우
  - 완료: 완료일, 점검결과, 점검완료, Link
  - 비용: 직접경비, 공수, 해외발송, 수리후입고

- **점검진행상황 상태값**: 점검중, 수리완료, 수리불가, 수리보류, 제조사수리, 제조사수리필요, 제조사수리완료, 정상동작, 점검완료, 업데이트완료, LCD교체예정, 데이터점검중
- **영업부진행상황 상태값**: 진행중, 완료, 입고, 납품완료, 제조사수리, 제조사수리필요, 제조사수리중, 제조사원격수리, 수리불가

- **점검보고서(샘플)**: 구조화된 보고서 양식
  - 섹션: 접수번호, 장비이력(제작사/일련번호), 사용자(회사명/담당자), 수리점검담당자
  - 본문: 접수증상, 점검내용(번호매김 단계별), 점검결과, 제조사수리여부

#### 장비점검이력.xlsx — 최신 간소화 양식
- **핵심 필드**: 완결[V], 접수번호, 입고일, 점검종류(수리/납품), Customer, Equipment, Maker, Product, 접수증상, 접수자, 점검완료일, S.N, 점검소견
- **2단계 점검 플로우**: 1차점검자 → 제조사수리필요여부[V] → Maker Reference No. → 본사수령후점검소견 → 2차점검자

### 핵심 가치

1. **AS 워크플로우 통합**: 접수 → 1차점검 → (제조사수리) → 2차점검 → 완료 전 과정 추적
2. **이중 추적 체계**: 기술팀 점검상황 + 영업팀 진행상황 병행 관리
3. **고객 관리**: 고객사별 보유 장비, AS 이력, 연락처 통합 관리
4. **제조사 연동 추적**: 수리 발송/입고, 부품 발주/입고 이력 관리
5. **점검보고서 디지털화**: 기존 엑셀 보고서를 시스템 내 구조화된 보고서로 전환
6. **비용 관리**: 직접경비, 공수, 해외발송비 등 비용 산출
7. **통계/리포트**: 장비별 고장률, 평균 수리 기간, 비용 분석

## 3. 핵심 도메인 모델

### 3.1 고객 (Customer)

```
Customer
├── id, name (회사명)
├── contactPerson (담당자명)
├── phone, email
├── address
├── notes
└── assets[] ← 고객 보유 장비/센서 연결
```

### 3.2 고객 자산 (CustomerAsset)

고객에게 판매한 장비/센서를 추적. 기존 equipment 스키마의 장비와는 별개로, 고객 소유 자산을 관리.

```
CustomerAsset
├── id, customerId
├── assetType ("EQUIPMENT" | "SENSOR")
├── name, serialNumber
├── manufacturer, model
├── soldAt (판매일)
├── warrantyExpiry (보증 만료일)
├── notes
└── repairOrders[] ← AS 이력
```

### 3.3 수리 접수 (RepairOrder)

AS 워크플로우의 핵심 엔티티. 기존 엑셀의 "관리시트" 1행 = RepairOrder 1건.

```
RepairOrder
├── id, orderNumber (자동 채번: AS-2026-0001)
├── customerId? (고객 AS인 경우)
├── customerAssetId? (고객 자산)
├── equipmentId? (자사 장비)
├── sensorId? (자사 센서)
├── orderType: REPAIR | DELIVERY (점검종류: 수리/납품)
├── status: RECEIVED → INSPECTING_1ST → QUOTED → APPROVED → REPAIRING → SHIPPED_TO_MFG → RECEIVED_FROM_MFG → INSPECTING_2ND → COMPLETED → CLOSED → CANCELLED
├── techStatus (점검진행상황 — 기술팀 자유 입력/선택)
├── salesStatus (영업부진행상황 — 영업팀 자유 입력/선택)
├── priority: LOW | NORMAL | HIGH | URGENT
├── currentLocation (현재위치: 자사, 고객사, 제조사 등)
├── symptom (접수증상)
├── diagnosis1st (1차 점검소견)
├── diagnosis2nd (본사수령후 점검소견 — 제조사 수리 후)
├── repairDetails (수리 내용)
├── receivedAt (입고일), completedAt (점검완료일), closedAt
├── receivedBy (접수자/접수담당)
├── inspector1stId (1차 점검자)
├── inspector2ndId (2차 점검자)
├── needsMfgRepair (제조사수리필요여부)
├── mfgReferenceNo (Maker Reference No.)
├── isWarranty (무상여부)
├── otInventoryNo (OT재고NO)
├── estimatedDays (예상 수리 기간)
├── notes
├── inspectionReport ← 점검보고서
├── quotes[] ← 견적
├── usedParts[] ← 사용 부품
├── shipments[] ← 제조사 발송/입고
└── costs[] ← 비용 항목
```

### 3.4 점검보고서 (InspectionReport)

기존 엑셀 "점검보고서(샘플)" 시트를 디지털화. 구조화된 보고서.

```
InspectionReport
├── id, repairOrderId
├── reportNumber (보고서 번호)
├── equipmentHistory (장비이력: 제작사, 일련번호, 제조년월 등 — JSON)
├── customerInfo (사용자 정보: 회사명, 담당자 등 — JSON)
├── inspectorId (수리점검담당자)
├── symptom (접수증상)
├── inspectionSteps (점검내용 — 번호매김 단계별, JSON array)
├── result (점검결과)
├── needsMfgRepair (제조사수리여부)
├── mfgRepairReason (제조사수리 사유)
├── createdAt, updatedAt
└── attachments[] ← 첨부파일 (사진 등)
```

### 3.5 비용 (RepairCost)

## 4. AS 워크플로우

### 4.1 기본 플로우 (기존 시스템 기반)

```
접수/입고(RECEIVED)
  ↓
1차 점검(INSPECTING_1ST) ← 1차점검자 배정
  ↓
├── 자체 수리 가능 (needsMfgRepair=false)
│   ├── 유상 → 견적 발행(QUOTED) → 고객 승인(APPROVED) → 수리(REPAIRING) → 완료(COMPLETED)
│   └── 무상 → 수리(REPAIRING) → 완료(COMPLETED)
│
└── 제조사 수리 필요 (needsMfgRepair=true)
    → 제조사 발송(SHIPPED_TO_MFG)
    → 제조사 입고(RECEIVED_FROM_MFG)
    → 2차 점검(INSPECTING_2ND) ← 2차점검자 배정
    → 완료(COMPLETED)
  ↓
종료(CLOSED) ← 완결[V] 체크
```

### 4.2 이중 추적 체계

기존 시스템의 핵심 특징: **기술팀**과 **영업팀**이 동일 건을 각자의 관점에서 추적.

```
RepairOrder
├── status (시스템 상태 — FSM 자동 전이)
├── techStatus (점검진행상황 — 기술팀이 자유 기록)
│   예: 점검중, 수리완료, 수리불가, 수리보류, 제조사수리필요, 정상동작 등
└── salesStatus (영업부진행상황 — 영업팀이 자유 기록)
    예: 진행중, 완료, 입고, 납품완료, 제조사수리중, 수리불가 등
```

- `status`: FSM으로 제어되는 공식 상태 (상태 전이 규칙 적용)
- `techStatus` / `salesStatus`: 자유 텍스트 또는 프리셋 선택 (각 팀이 독립 업데이트)

### 4.3 점검종류

기존 장비점검이력의 "점검종류" 반영:
- **수리 (REPAIR)**: AS/수리 접수 — 일반적인 고장 수리
- **납품 (DELIVERY)**: 신규 납품 장비의 입고 점검/설치 확인

### 4.4 특수 케이스

- **자사 장비 수리**: customerId 없이, equipmentId/sensorId만 연결. 견적 불필요 시 QUOTED/APPROVED 스킵 가능.
- **무상 수리**: isWarranty=true → 비용 0원, 견적 생략
- **제조사 원격 수리**: 물리적 발송 없이 제조사가 원격으로 수리 (salesStatus로 추적)

## 5. UI 구성

### 5.1 메뉴 구조

```
자원관리
├── 장비운영 (기존)
├── 수리관리 ← NEW
│   ├── AS 접수/목록 (기본 탭)

│   └── 통계
```

### 5.2 주요 화면

#### AS 접수/목록 페이지
- 상단: 상태별 필터 (전체/접수/진단중/수리중/완료), 검색 (접수번호, 고객명, 시리얼)
- 목록: 접수번호, 고객명, 장비명, 시리얼, 상태, 우선도, 담당자, 접수일, 경과일
- FAB: "+ AS 접수" 버튼

#### AS 상세 페이지
- 헤더: 접수번호, 상태 배지(FSM), 현재위치, 우선도
- 이중 상태 표시: 점검진행상황 + 영업부진행상황 (각 팀 독립 편집)
- 탭 구성:
  - **기본정보**: 고객/장비 정보, 접수증상, OT재고NO
  - **점검**: 1차점검 소견, 제조사수리필요여부, 2차점검 소견
  - **점검보고서**: 구조화된 점검보고서 작성/조회 (단계별 점검내용)
  - **견적/비용**: 견적 목록, 직접경비/공수/해외발송비
  - **부품**: 사용 부품 목록, 부품 추가
  - **제조사 발송**: 발송/입고 이력, Maker Reference No., 운송장 번호
  - **이력**: 상태 변경 타임라인

#### 고객 관리 페이지
- 고객 목록 + 검색
- 고객 상세: 기본정보, 보유 자산 목록, AS 이력

#### 부품/재고 페이지
- 부품 목록: 부품번호, 이름, 재고, 최소재고 (부족 시 경고)
- 입출고 이력
- 발주 관리: 발주 목록, 발주 작성, 입고 처리

#### 통계 페이지
- 장비별 고장 건수 (Bar Chart)
- 월별 AS 접수/완료 추이 (Line Chart)
- 평균 수리 기간 (카드)
- 수리 비용 합계 (기간별)
- 부품 소모 TOP 10

## 6. 구현 단계 (Phases)

### Phase 1: Core — AS 접수/추적 + 고객 관리

**목표**: AS 워크플로우의 기본 CRUD + 이중 상태 관리 (기존 엑셀 "관리시트" 대체)

- [ ] DB 스키마: Customer, CustomerAsset, RepairOrder
- [ ] API: 고객 CRUD, AS 접수/조회/상태변경
- [ ] UI: AS 목록/상세, 고객 목록/상세, AS 접수 폼
- [ ] 상태 전이 로직 (FSM) + 이중 추적 (techStatus / salesStatus)
- [ ] 접수번호 자동 채번 (AS-YYYY-NNNN)
- [ ] 점검종류 (수리/납품) 구분
- [ ] 1차/2차 점검자 배정 및 점검소견 기록

### Phase 2: 점검보고서 + 비용 관리

**목표**: 기존 엑셀 "점검보고서(샘플)" 시트 디지털화 + 비용 추적

- [ ] DB 스키마: InspectionReport, RepairCost
- [ ] API: 점검보고서 CRUD, 비용 CRUD
- [ ] UI: 점검보고서 작성 폼 (단계별 점검내용), 보고서 미리보기/인쇄
- [ ] 비용 항목 관리 (직접경비, 공수, 해외발송비)
- [ ] 보고서 PDF 내보내기 (향후)

### Phase 3: 견적 관리

- [ ] DB 스키마: RepairQuote, QuoteItem
- [ ] API: 견적 CRUD, 견적 승인/반려
- [ ] UI: 견적 작성 폼, 견적 목록, 견적서 미리보기
- [ ] 환율 지원 (KRW/USD/EUR)

### Phase 4: 부품/재고 관리

- [ ] DB 스키마: Part, PartTransaction, PurchaseOrder, PurchaseOrderItem
- [ ] API: 부품 CRUD, 입출고, 발주 관리
- [ ] UI: 부품 목록, 입출고 이력, 발주 관리
- [ ] 최소 재고 미달 알림
- [ ] 수리에 사용된 부품 자동 출고 처리

### Phase 5: 제조사 연동

- [ ] DB 스키마: Shipment
- [ ] API: 발송/입고 CRUD
- [ ] UI: 발송/입고 이력, Maker Reference No., 운송장 입력
- [ ] AS 상태 자동 전이 (발송 시 SHIPPED_TO_MFG, 입고 시 RECEIVED_FROM_MFG)
- [ ] 2차 점검 플로우 연동

### Phase 6: 통계/리포트

- [ ] API: 통계 집계 엔드포인트
- [ ] UI: 대시보드 차트 (Chart.js or Recharts)
- [ ] 장비별 고장률, 월별 추이, 비용 분석, 부품 소모 TOP

## 7. 기술 결정

| 항목 | 결정 |
|------|------|
| **서비스 위치** | equipment-service 확장 (equipment 스키마에 테이블 추가) |
| **DB 스키마** | `equipment` 스키마 내 신규 테이블 |
| **자사 장비 연결** | 기존 Equipment/Sensor 모델 FK 참조 |
| **고객 자산** | CustomerAsset 별도 모델 (자사 장비와 분리) |
| **접수번호** | `AS-{년도}-{4자리순번}` 시퀀스 |
| **통화** | KRW 기본, USD/EUR 선택 시 환율 입력 |
| **상태 전이** | FSM 패턴 (기존 attendance FSM과 동일 방식) |

## 8. 기존 시스템 연동

| 대상 | 연동 방식 |
|------|-----------|
| **Equipment/Sensor** | equipmentId/sensorId FK로 자사 장비 AS 연결 |
| **MaintenanceRecord** | AS 완료 시 정비이력 자동 생성 |
| **AssetSchedule** | 수리 기간 동안 장비 일정에 반영 (IN_MAINTENANCE) |
| **장비 상태** | AS 접수 시 IN_MAINTENANCE, 완료 시 AVAILABLE 복원 |

## 10. 비기능 요구사항

- AS 목록 조회 < 500ms (인덱스 최적화)
- 접수번호 중복 방지 (DB 시퀀스 또는 트랜잭션 내 채번)
- 견적 금액 소수점 2자리 (Decimal)
- 부품 재고 음수 방지 (트랜잭션 내 검증)
