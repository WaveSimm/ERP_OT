## 1. Overview

### 1.1 배경

현재 시스템에서 "마일스톤"이라는 단어가 **두 가지 다른 의미로 혼재**하고 있어 데이터 무결성과 의미 정합성이 모두 깨져있다.

| 의미 | 위치 | 데이터 | 문제 |
|------|------|------:|------|
| 그룹/Phase 컨테이너 | `Milestone` 모델 | 0건 | 사용 안 됨에도 코드만 존재 |
| 단일 날짜 이정표 태스크 | `Task.isMilestone` 플래그 | 2건 | start=end 강제 검증 없음, segment 모델 우회 가능 |

추가로:
- 갠트는 `effectiveStartDate`만 보고 ◆ 표시 → endDate 정보 유령처럼 잔존
- AddTaskModal만 single-date input 강제 → API 직접 호출 시 우회 가능
- aggregate·baseline·CPM 곳곳에 `isMilestone` 분기 코드 산재

### 1.2 목표

**Milestone을 Task에서 분리한 일급 시민(first-class entity)으로 재설계.**

1. 데이터 모델 분리: Milestone과 Task는 서로 다른 도메인 개념
2. 자동 달성 추적: linkedTasks 완료 시 status 자동 갱신 (PMBOK 표준)
3. 통합 의존성 그래프: Task↔Milestone 양방향 의존 표현
4. CPM 제약: dueDate를 latestStart 제약으로 사용, INFEASIBLE 자동 감지
5. 사용자 직관 보존: Gantt에서는 여전히 task tree에 ◆로 한 행 차지
6. 의미 모호성 제거: Milestone(그룹) 모델 폐기, isMilestone 플래그 폐기

### 1.3 비목표 (Out of This PDCA)

- (c) 결재/외부알림/청구 트리거 → 별도 PDCA "마일스톤-트리거" (외부알림 인프라 의존)
- ProjectPhase 신규 모델 (기간 그룹) → 필요 시 별도
- Milestone 외부 공개 페이지 → 외부알림 PDCA와 함께
- Milestone baseline (계획 vs 실제) → ProjectBaseline 확장 별도

## 2. User Intent Discovery (Plan Plus)

### Q1. 핵심 문제
"마일스톤"이 시스템에서 두 의미(그룹 vs 시점)로 혼재. 데이터 무결성·의미 정합성 모두 깨짐. 릴리즈 전 정리가 비용·리스크 모두 최소.

### Q2. 주 사용자
내부 PM (외부 노출은 후속 PDCA로 분리)

### Q3. 성공 기준
1. Milestone과 Task가 데이터 모델 차원에서 분리됨
2. linkedTasks 완료 → 자동 ACHIEVED, dueDate 임박+진행률 부족 → AT_RISK 자동
3. Task↔Milestone 의존성 단일 그래프로 표현
4. 사용자 직관(Task-like 행 표시) 유지
5. CPM이 dueDate를 제약으로 사용해 INFEASIBLE 자동 감지

### Q4. 제약
- 릴리즈 전 (마이그레이션 부담 거의 없음, 데이터 isMilestone Task 2건만)
- 결재/외부알림 인프라 미완 → (c) 트리거 분리 결정
- 프로젝트-관리·프로젝트-진도율-캐시 PDCA Match Rate 보존 필요 (회귀 금지)

## 5. Functional Requirements

### FR-1. Milestone CRUD
- PM은 프로젝트 내 Milestone을 생성/수정/삭제할 수 있다
- 필드: name, description, dueDate(옵셔널), achievementCriteria, sortOrder, displayParentId

### FR-2. linkedTasks 관리
- Milestone에 0개 이상의 Task를 linkedTasks로 연결/해제할 수 있다
- linkedTasks 비어있으면 수동 ACHIEVED 마커로 동작 (a 모드)
- linkedTasks 있으면 자동 추적 (b 모드)

### FR-3. 자동 status 갱신
- 모든 linkedTasks가 DONE → status = ACHIEVED + achievedDate 자동 기록
- dueDate D-7 이내 + linkedTasks 평균 진행률 < 70% → AT_RISK
- dueDate 경과 + 미ACHIEVED → MISSED
- CPM totalFloat < 0 → INFEASIBLE

### FR-4. 통합 의존성 그래프
- Task → Task / Task → Milestone / Milestone → Task / Milestone → Milestone 의존 표현
- DependencyType: FS / SS / FF / SF
- lag (지연) 옵션
- DB CHECK 제약으로 predecessor/successor 정확히 각 1개 보장

### FR-5. CPM 제약
- Milestone은 0-duration 노드로 CPM 계산에 포함
- dueDate가 있으면 latestFinish 제약으로 사용
- latestStart > dueDate → INFEASIBLE 표시 + 알림

### FR-6. Gantt 표시 (6e+)
- Milestone은 task tree의 한 행으로 ◆ 아이콘 표시
- displayParentId·sortOrder로 위치 지정 (PM이 수동 배치)
- dueDate 위치에 ◆ 마커 (status별 색상)
- 의존선이 task와 milestone 사이 자연스럽게 그려짐
- isCritical Milestone은 빨간 테두리

### FR-7. 마일스톤 사이드패널
- Gantt 우측에 마일스톤 일람
- 필터: 프로젝트, status, 임박 정도
- 정렬: dueDate, sortOrder, status
- 각 행에 진척률(linkedTasks 평균) 표시

### FR-8. 알림
- AT_RISK / MISSED / INFEASIBLE / D-임박 → 기존 Notification 인프라
- linkType: MILESTONE
- 결재라인 보고는 내부 알림 채널만 사용

### FR-9. 마이그레이션
- 기존 isMilestone=true Task 2건 → Milestone 레코드로 변환
- segment의 startDate를 Milestone.dueDate로 이전
- 의존성 (TaskDependency) → Dependency 테이블 변환
- 기존 Milestone(그룹) 0건 → DROP

## 7. Technical Approach

### 7.1 데이터 모델 변경

```prisma
// 새 Milestone 모델 (이정표 = 시점)
model Milestone {
  id                  String              @id @default(cuid())
  projectId           String
  name                String
  description         String?

  // 시점 (옵셔널)
  dueDate             DateTime?
  achievedDate        DateTime?

  // 자동 추적
  status              MilestoneStatus     @default(PLANNED)
  achievementCriteria AchievementCriteria @default(ALL_LINKED_TASKS_DONE)

  // 위치 (Task-like 표시 - 6e+)
  sortOrder           Int                 @default(0)
  displayParentId     String?

  // CPM 캐시
  isCritical          Boolean             @default(false)
  earliestStart       DateTime?
  latestStart         DateTime?
  totalFloat          Float?

  // 진척률 캐시 (linkedTasks 평균)
  linkedProgress      Float?              @default(0)

  createdBy           String
  createdAt           DateTime            @default(now())
  updatedAt           DateTime            @updatedAt

  project             Project             @relation(fields: [projectId], references: [id], onDelete: Cascade)
  linkedTasks         Task[]              @relation("MilestoneLinkedTasks")

  @@map("milestones")
  @@schema("project")
}

enum MilestoneStatus {
  PLANNED
  AT_RISK
  ACHIEVED
  MISSED
  INFEASIBLE
}

enum AchievementCriteria {
  ALL_LINKED_TASKS_DONE
  MANUAL
}

// 통합 의존성 (4c)
model Dependency {
  id                       String         @id @default(cuid())
  predecessorTaskId        String?
  predecessorMilestoneId   String?
  successorTaskId          String?
  successorMilestoneId     String?
  dependencyType           DependencyType @default(FS)
  lag                      Int            @default(0)
  createdAt                DateTime       @default(now())

  predecessorTask          Task?          @relation("PredecessorTask", fields: [predecessorTaskId], references: [id], onDelete: Cascade)
  predecessorMilestone     Milestone?     @relation("PredecessorMilestone", fields: [predecessorMilestoneId], references: [id], onDelete: Cascade)
  successorTask            Task?          @relation("SuccessorTask", fields: [successorTaskId], references: [id], onDelete: Cascade)
  successorMilestone       Milestone?     @relation("SuccessorMilestone", fields: [successorMilestoneId], references: [id], onDelete: Cascade)

  @@map("dependencies")
  @@schema("project")
  // CHECK constraints (raw SQL migration):
  //   (predecessorTaskId IS NOT NULL) <> (predecessorMilestoneId IS NOT NULL)
  //   (successorTaskId IS NOT NULL) <> (successorMilestoneId IS NOT NULL)
}

// Task — isMilestone, milestoneId 제거
model Task {
  // ... existing fields
  // ❌ isMilestone (제거)
  // ❌ milestoneId (제거)
  linkedMilestones  Milestone[]  @relation("MilestoneLinkedTasks")
  predecessorDeps   Dependency[] @relation("PredecessorTask")
  successorDeps     Dependency[] @relation("SuccessorTask")
}
```

### 7.2 마이그레이션 단계

1. **Phase A — 신규 모델 추가** (병행 운영 가능)
   - Milestone 모델 신규 컬럼 추가 (dueDate 등)
   - Dependency 테이블 신규 생성
   - 기존 Milestone(그룹), TaskDependency, Task.isMilestone, Task.milestoneId 유지

2. **Phase B — 데이터 백필**
   - 기존 isMilestone Task 2건 → Milestone 레코드 생성
   - linkedTasks 매핑 (해당 Task가 자기 자신을 link 또는 별도 기준으로)
   - TaskDependency → Dependency 변환 (모두 task ↔ task로)

3. **Phase C — 코드 전환**
   - Milestone/Dependency 신규 API 활성화
   - 기존 isMilestone 분기 코드 제거
   - GanttChart Task-like 렌더링으로 전환
   - 사이드패널 추가

4. **Phase D — 구버전 제거**
   - Task.isMilestone, Task.milestoneId 컬럼 DROP
   - TaskDependency 테이블 DROP
   - 기존 Milestone(그룹) 의미 코드 제거

→ 데이터 0건이라 Phase A~D를 단일 PR로 통합 가능 (소규모 백필만 필요).
