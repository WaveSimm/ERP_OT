## 1. Overview

### 1.1 배경

`프로젝트-마일스톤-재설계` PDCA(2026-04-30 완료, Match 93%)에서 Milestone을 일급 시민으로 분리하고 linkedTasks·achievementCriteria·CPM 캐시 등 풍부한 인프라를 도입.

KHOA Planner 파일 import(2026-05-01) 후 사용자가 실제 갠트에서 마일스톤 사용해본 뒤 다음 두 가지 핵심 문제 발견:

1. **부모 Task와 의미 중복**: Milestone(linkedTasks 보유)과 Parent Task(자식 task 보유)가 기능적으로 거의 동일 — 둘 다 "여러 task 묶기 + 진척률 집계". 시각 강화로 해결 안 됨.
2. **본질에 비해 과한 추상**: 사용자의 멘탈 모델은 "시점 확인 이벤트"(입찰 확인, 투찰 확인, 입고 확인) — 단일 시점·이진 상태(됐냐/안됐냐). linkedTasks·자동 추적·status enum 5종 등은 과잉.

### 1.2 목표

PMBOK 표준 의미("checkpoint event with zero duration")로 회귀:
1. Milestone 별도 모델 폐기
2. Task에 `isMilestone` Boolean 컬럼 부활 (시점 task)
3. 시점 task = 단일 segment (start=end), 자동 추적 X, status는 Task 표준 (TODO/IN_PROGRESS/DONE)
4. 갠트에서 ◆ 표시는 그대로 유지 (status 색상 4단계)
5. 부모 Task와 시각·의미 명확히 분리

### 1.3 비목표 (Out of This PDCA)

- responsibleUserId 컬럼 신규 (별도 PDCA 후보)
- 외부 약속/계약 시점 도메인 (별도 PDCA "외부-마일스톤")
- CPM Milestone 노드화 (애초 미구현)
- Notification priority 변경

## 7. Technical Approach

### 7.1 Schema diff

```prisma
model Task {
  ...
+ isMilestone     Boolean   @default(false)  // 시점 task (◆), start=end 강제
  ...
- linkedMilestones Milestone[]   @relation("MilestoneLinkedTasks")
}

- model Milestone { ... }                       // 폐기
- enum MilestoneStatus { ... }                  // 폐기
- enum AchievementCriteria { ... }              // 폐기

model Dependency {
- predecessorMilestoneId   String?
- successorMilestoneId     String?
  predecessorTaskId        String   // NOT NULL
  successorTaskId          String   // NOT NULL
  ...
}
```

### 7.2 Migration SQL (단일 트랜잭션)

```sql
BEGIN;

-- 1. Task에 isMilestone 컬럼 추가
ALTER TABLE project.tasks ADD COLUMN "isMilestone" BOOLEAN NOT NULL DEFAULT false;

-- 2. Milestone → Task 변환 (KHOA 현장 납품 1건)
INSERT INTO project.tasks (id, "projectId", "parentId", name, description, "sortOrder",
  status, "overallProgress", "isManualProgress", "isMilestone", "createdBy", "createdAt", "updatedAt")
SELECT
  'tsk_ms_' || substring(id from 5),
  "projectId",
  "displayParentId",
  name,
  description,
  "sortOrder",
  CASE WHEN status = 'ACHIEVED' THEN 'DONE'::project."TaskStatus" ELSE 'TODO'::project."TaskStatus" END,
  CASE WHEN status = 'ACHIEVED' THEN 100 ELSE 0 END,
  true,  -- isManualProgress (시점 task는 PM 수동)
  true,  -- isMilestone
  "createdBy", "createdAt", "updatedAt"
FROM project.milestones;

-- 3. 변환된 task에 단일 segment (start=end=dueDate)
INSERT INTO project.task_segments (id, "taskId", name, "sortOrder", "startDate", "endDate", "progressPercent", "createdAt", "updatedAt")
SELECT
  'seg_ms_' || substring(m.id from 5),
  'tsk_ms_' || substring(m.id from 5),
  '시점',
  0,
  COALESCE(m."dueDate", CURRENT_DATE),
  COALESCE(m."dueDate", CURRENT_DATE),
  CASE WHEN m.status = 'ACHIEVED' THEN 100 ELSE 0 END,
  NOW(), NOW()
FROM project.milestones m;

-- 4. ACHIEVED 마일스톤 → system work log
INSERT INTO project.work_logs (id, task_id, author_id, author_name, content, worked_at, is_deleted, created_at, updated_at)
SELECT
  'wl_ms_' || substring(m.id from 5),
  'tsk_ms_' || substring(m.id from 5),
  'system', 'system',
  '[system] 마일스톤-시점태스크-회귀 마이그레이션. 이전 ACHIEVED milestone 변환.',
  COALESCE(m."achievedDate"::date, m."dueDate", CURRENT_DATE),
  false, NOW(), NOW()
FROM project.milestones m WHERE m.status = 'ACHIEVED';

-- 5. Dependency 단순화
DELETE FROM project.dependencies
WHERE "predecessorMilestoneId" IS NOT NULL OR "successorMilestoneId" IS NOT NULL;

ALTER TABLE project.dependencies
  DROP CONSTRAINT IF EXISTS dep_predecessor_xor,
  DROP CONSTRAINT IF EXISTS dep_successor_xor,
  DROP CONSTRAINT IF EXISTS dep_no_self_milestone;

ALTER TABLE project.dependencies
  DROP COLUMN "predecessorMilestoneId",
  DROP COLUMN "successorMilestoneId",
  ALTER COLUMN "predecessorTaskId" SET NOT NULL,
  ALTER COLUMN "successorTaskId" SET NOT NULL;

-- 6. 백필 검증
DO $$
DECLARE
  expected INT;
  actual_tasks INT;
  actual_segs INT;
BEGIN
  SELECT COUNT(*) INTO expected FROM project.milestones;
  SELECT COUNT(*) INTO actual_tasks FROM project.tasks WHERE "isMilestone" = true;
  SELECT COUNT(*) INTO actual_segs FROM project.task_segments WHERE "taskId" LIKE 'tsk_ms_%';
  IF expected != actual_tasks OR expected != actual_segs THEN
    RAISE EXCEPTION 'Migration mismatch: ms=%, tasks=%, segs=%', expected, actual_tasks, actual_segs;
  END IF;
  RAISE NOTICE 'Backfill verified: % milestones → % tasks + % segments', expected, actual_tasks, actual_segs;
END $$;

-- 7. 폐기
DROP TABLE project."_MilestoneLinkedTasks";
DROP TABLE project.milestones;
DROP TYPE project."MilestoneStatus";
DROP TYPE project."AchievementCriteria";

COMMIT;
```

- 갈래 1 (linkedTasks 제거)·갈래 2 (외부 약속) 검토 → 부모 task와 차이 여전히 약함
- 갈래 3 (모델 통합) 채택 — PMBOK 표준 부합

### Reverse 대상 (마일스톤-재설계 PDCA 결정)
- (D) Milestone 일급 시민 분리 → reverse (Task isMilestone 부활)

### Reverse 사유
1. 사용자 멘탈 모델("시점 확인")은 작업 분해보다 단순

3. 풍부한 인프라(linkedTasks, achievementCriteria, AT_RISK 자동 등)가 본 사용처에 과함
