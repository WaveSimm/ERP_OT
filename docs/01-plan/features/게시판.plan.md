## 1. 개요 (Overview)

회사 내부 정보 공유와 공식 발행을 위한 통합 게시판 시스템. 공지사항(top-down 발행)과 자료실(bottom-up 협업)을 하나의 도메인으로 통합 관리하되, **2-level 트리 구조(대분류 → 보드 → 글)**로 향후 부서별 게시판까지 자연스럽게 확장 가능.

### 1.1 배경 (Background)

- **현재 상태**: 회사 공지·기술 자료 공유 채널이 사내에 부재 (이메일/메신저로 산발 처리)
- **개인 알림(`Notification`)** 시스템은 있으나 출결·결재 트리거용으로, 회사 발행 콘텐츠와 성격 다름
- **수요**:
  - 회사·부서 공지의 일괄 전달 + 읽음 추적
  - 디버깅·트러블슈팅·현장 업무 노하우의 영구 축적
  - 기술 자료·매뉴얼의 구조화된 보관·검색
- **홈 화면 의존성**: 로그인 후 첫 화면(홈)에 공지 피드 카드 노출 예정 → 게시판이 선행 구축 필요

### 1.2 핵심 가치

1. **일관된 정보 허브**: 공지·자료·부서 게시판을 하나의 메뉴(`/board`)에서 통합 접근
2. **확장 가능한 분류 트리**: BoardCategory → Board → Post 구조로 부서별 게시판(2차) 추가 시 코드 변경 거의 없음
3. **저권한 발행 + 고권한 강조**: 모든 OPERATOR가 작성 가능하되 핀(공식화)은 ADMIN/작성자 한정
4. **자동 알림 연동**: 공지 발행 시 audience 매칭 사용자에게 개인 `Notification` 자동 생성
5. **위키형 영구 자료**: 자료실은 본문 markdown + 이미지 인라인 + 문서 첨부로 풍부한 콘텐츠 보관
6. **댓글 협업**: 공지·자료 모두 댓글 허용 — 피드백·질문 채널 일원화

### 1.3 관련 문서

- 종합 검토 보고서: `docs/04-operation/2026-04-24-종합검토보고서.md`
- 기존 알림 시스템: `services/attendance/prisma/schema.prisma:296` (`Notification` 모델)
- 후속 의존 기능: 홈 화면 (별도 PDCA, 게시판 완료 후 진행)

## 3. 핵심 도메인 모델

### 3.1 BoardCategory (대분류)

```
BoardCategory
├── id, code (notice/wiki/department), name
├── icon, sortOrder
└── isActive
```

1차 시드: `notice` (공지사항), `wiki` (자료실)
2차 시드: `department` (부서별 게시판) — 1차에는 DB에 존재하나 UI 숨김

### 3.2 Board (실제 글이 쌓이는 보드)

```
Board
├── id, categoryId, code, name, description
├── 권한: writeRoles[], readAudience (ALL/DEPARTMENT/ROLE), audienceTargetId
├── 옵션: allowComments, allowAttachments, postPinnable
└── sortOrder, isActive
```

1차 시드 보드 (Category × name):
| Category | Board name | code | writeRoles | readAudience |
|---|---|---|---|---|
| notice | 전사 공지 | notice-company | OPERATOR,MANAGER,ADMIN | ALL |
| notice | 부서 공지 | notice-dept | OPERATOR,MANAGER,ADMIN | DEPARTMENT (선택 시) |
| wiki | 디버깅·트러블슈팅 | wiki-debug | OPERATOR,MANAGER,ADMIN | ALL |
| wiki | 현장 업무 | wiki-field | OPERATOR,MANAGER,ADMIN | ALL |
| wiki | 기술 정보·매뉴얼 | wiki-tech | OPERATOR,MANAGER,ADMIN | ALL |
| wiki | 기타·자유 | wiki-misc | OPERATOR,MANAGER,ADMIN | ALL |

### 3.3 Post (글)

```
Post
├── id, boardId, authorId, title, content (markdown)
├── publishingDepartmentId, publishingDepartmentName (작성 시점 스냅샷)
├── 표시: isPinned, priority (NORMAL/IMPORTANT/URGENT)
├── 시간: publishedAt, expiresAt, createdAt, updatedAt
├── 통계: viewCount, isDeleted (soft)
└── 관계: board, author, reads[], comments[], attachments[]
```

### 3.4 PostRead (본인 읽음)

```
PostRead (postId, userId, readAt) — composite PK
```

1차는 본인 표시용. 관리자 통계는 2차.

### 3.5 Comment (댓글)

```
Comment
├── id, postId, authorId, content (text)
├── parentId (대댓글 1단계, 선택)
└── createdAt, updatedAt, isDeleted
```

### 3.6 Attachment (첨부파일)

```
Attachment
├── id, postId, fileName, fileSize, mimeType, storagePath
├── isInline (이미지 인라인이면 true), uploadedAt
```

저장 위치: 1차 로컬 볼륨 `/data/uploads/board/{postId}/`. 회사서버 이관 시 S3·MinIO 검토.

## 5. 범위 (Scope)

### 5.1 In Scope (1차)

- [x] BoardCategory ↔ Board ↔ Post ↔ Comment ↔ Attachment ↔ PostRead 모델
- [x] auth-service에 도메인 통합 (cross-service 호출 회피)
- [x] 1차 시드: notice 카테고리 2 보드 + wiki 카테고리 4 보드
- [x] CRUD API (글 작성·수정·삭제·핀·조회)
- [x] 댓글 CRUD (1단계 대댓글 포함)
- [x] 이미지 인라인 + 문서 첨부 (PDF/엑셀/워드/이미지)
- [x] Markdown 본문 (react-markdown 렌더링)
- [x] 본인 읽음 표시 (조회 시 PostRead 자동 기록)
- [x] 미읽음 카운트 API + 헤더 📢 아이콘 + 뱃지
- [x] 공지 작성 시 audience 매칭자에게 `Notification` 자동 발송
- [x] 보드 단위 글 목록 페이지네이션·검색
- [x] 발행 부서 필터 (`publishingDepartmentId` 쿼리)
- [x] 권한 매트릭스 적용 (writeRoles 검증, canPin 검증)
- [x] 2차 BoardCategory `department`는 DB seed에는 존재하나 UI 숨김 (FF: `featureBoardDept=false`)

### 5.2 Out of Scope (2차 이후)

- 부서별 게시판 자동 생성 (부서 추가 hook + UI 노출)
- 협업 편집 (다중 작성자, OT 동기화)
- 읽음 통계 대시보드 (관리자 — 누가 읽었나)
- 풀텍스트 검색 (PostgreSQL `tsvector` 인덱스)
- 작성자별 알림 옵트아웃
- 글 버전 이력 (위키 revision)
- 이모지 반응 (👍 ❤️)
- 즐겨찾기·구독
- 핀 동시 표시 개수 상한 (운영 중 남발 시 보드별 `maxPinned` 컬럼 추가)

### 9.2 핵심 아키텍처 결정

| 결정 | 선택 옵션 | 선택 | 근거 |
|---|---|---|---|
| 서비스 배치 | auth 서비스 확장 / 신규 notice 서비스 | **auth 서비스 확장** | author·dept 정보가 같은 서비스에 있어 cross-service 호출 회피, 1차 출시 속도 우선 |
| 본문 포맷 | Markdown / HTML / Tiptap | **Markdown (react-markdown + remark-gfm)** | 단순, 안전, 위키 문화에 적합 |
| 에디터 | textarea / EasyMDE / Tiptap | **EasyMDE 또는 textarea + 미리보기** | Tiptap은 2차 (협업 편집 도입 시) |
| 첨부 저장 | 로컬 볼륨 / S3 / DB BLOB | **로컬 볼륨** | 1차는 로컬 운영, 이관 시 S3 |
| 첨부 업로드 | multipart / presigned | **multipart** | BaaS 아닌 자체 서비스 |
| 알림 발송 | 동기 / 비동기 (queue) | **동기 (트랜잭션 분리)** | 1차 단순성, 폭증 시 RabbitMQ 전환 |
| 검색 | LIKE / FTS / 외부 검색엔진 | **LIKE (1차)** | 데이터 ~10K 이내 예상, 2차 FTS |
| 페이지네이션 | offset / cursor | **offset (1차)** | 단순성, 핀 정렬과 호환 |
| 권한 가드 | 미들웨어 / 라우트별 함수 | **라우트별 함수** | `canWrite/canPin/canEdit/canDelete` 명시적 헬퍼 |

### 10.1 기존 프로젝트 컨벤션

- [x] `CLAUDE.md` 코딩 컨벤션 정의됨 (kebab-case files, PascalCase classes, snake_case DB)
- [x] ESLint, Prettier, TypeScript strict 설정 존재
- [x] Prisma 스키마 분리 (`@@schema("auth")`)
- [x] API 라우트 `/api/v1/{resource}` 패턴

### 10.2 신규 정의 필요

| 카테고리 | 현재 상태 | 정의 사항 | 우선순위 |
|---|---|---|---|
| Markdown sanitize | 없음 | DOMPurify 또는 rehype-sanitize 표준 | High |
| 첨부 디렉터리 | 없음 | `/data/uploads/board/{postId}/` 컨벤션 | High |
| 첨부 확장자 화이트리스트 | 없음 | `.png .jpg .jpeg .gif .webp .pdf .xlsx .docx .pptx .txt .md .csv` | High |
| 첨부 크기 상한 | 없음 | 단일 파일 50MB | Medium |
| 권한 헬퍼 위치 | 없음 | `services/auth/src/application/board-permissions.ts` | High |

### 10.3 환경변수

| 변수 | 용도 | 범위 | 신규 |
|---|---|---|:---:|
| `BOARD_ATTACHMENT_DIR` | 첨부 저장 루트 (`/data/uploads/board`) | Server | ☑ |
| `BOARD_ATTACHMENT_MAX_SIZE` | 단일 파일 최대 크기 (bytes) | Server | ☑ |
| `BOARD_NOTICE_NOTIFY_AUDIENCE_THRESHOLD` | audience 사용자 N명 초과 시 비동기 (2차) | Server | ☐ (2차) |
