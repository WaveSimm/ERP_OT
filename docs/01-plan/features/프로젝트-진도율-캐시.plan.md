# [Plan] 프로젝트-진도율-캐시 (Project Aggregate Cache)

> **Phase**: Plan v1.0 / **Date**: 2026-04-28
> **Predecessor**: 홈 카드와 프로젝트 페이지 진도율 불일치 사례에서 발견

---

## 1. 개요

프로젝트/태스크의 derived 값(진도율·effective 날짜)을 매번 task+segment 조회·계산하던 방식에서 **DB 캐시 필드 + 갱신 hook** 패턴으로 전환. 홈 카드/프로젝트 페이지/대시보드 등 **모든 consumer가 같은 값**을 보고, 응답 빠르게.

### 1.1 발단

- 홈 "진행 프로젝트" Factory Pipeline 100% vs 프로젝트 페이지 75% 불일치
- 원인: 홈 = 본인 segment만 평균 / 프로젝트 페이지 = 전체 leaf segment 평균
- DB 필드가 없어서 화면별로 산정 기준이 다름

### 1.2 핵심 가치

1. **일관성**: 모든 화면이 동일 값
2. **성능**: 단일 컬럼 read (계산 부담 0)
3. **유지보수성**: 산정 로직 1곳 (`recomputeProjectAggregates`)
4. **확장성**: 향후 통계/대시보드 추가 시 그대로 활용

---

## 2. 캐시 대상 필드

| 모델 | 필드 | 산정식 |
|------|------|--------|
| Project | `overallProgress Float?` | leaf task(부모 X, 마일스톤 X)의 모든 segment `progressPercent` 평균 |
| Project | `effectiveStartDate DateTime?` | leaf task segments의 min(startDate) |
| Project | `effectiveEndDate DateTime?` | leaf task segments의 max(endDate) |
| Task | `effectiveStartDate DateTime?` | 해당 task segments의 min(startDate) |
| Task | `effectiveEndDate DateTime?` | 해당 task segments의 max(endDate) |

> **참고**: `Task.overallProgress`는 이미 DB 필드 — rollup 자동 갱신 hook도 이미 있음. 본 PDCA 범위 외.

---

## 3. 갱신 트리거

| 트리거 | 영향 |
|--------|------|
| TaskSegment 생성/수정/삭제 | 해당 task effectiveDates → project aggregates |
| Task 생성/삭제 | project aggregates |
| Task `overallProgress` 변경 | project aggregates (overallProgress 평균 재계산) |
| Task `parentId` / `isMilestone` 변경 | project aggregates (leaf 집합 변경) |

## 4. 범위

### 4.1 In Scope

- [ ] Prisma schema에 5개 필드 추가
- [ ] Migration SQL + 기존 데이터 backfill 1회
- [ ] `ProjectAggregateService.recomputeProject(projectId)` 신규 — 1곳에 산정 로직 집약
- [ ] `TaskAggregateService.recomputeTask(taskId)` 신규
- [ ] 변경 지점 모두에 hook 호출 (segment·task service들)
- [ ] Consumer 단순화: `project.service.ts` list/find, `me/projects` 라우트, 기타
- [ ] Sanity 테스트: 캐시값 == 매번계산값 일치 확인

### 4.2 Out of Scope

- Task `overallProgress` rollup (이미 동작)
- 부서/그룹 단위 weighted 진도율 (대시보드 자체 로직)
- 실시간 push (트리거 후 polling으로 충분)

## 6. 리스크

| 리스크 | 완화 |
|--------|------|
| Hook 누락 → stale 데이터 | 변경 위치 사전 grep + 검증 스크립트 |
| 동시성: 두 task 동시 변경 시 race | Prisma 트랜잭션 감싸기 |
| Migration 시 backfill 시간 | 데이터 적음 (~수십 프로젝트) → 1회성 |
| 기존 코드와 cache 값 다름 | sanity test로 1회 비교 |
