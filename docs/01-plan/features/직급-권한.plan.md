## 1. 배경 및 목표

### 현재 상태
- `UserRole`: `ADMIN` / `MANAGER` / `OPERATOR` / `VIEWER` — 4단계 존재
- 그러나 각 Role이 어떤 작업을 할 수 있는지 **명확한 정의와 서버 강제가 없음**
- 대부분의 라우트에 권한 검사 미적용 상태

### 목표
프로젝트 관리 시스템 내 **3단계 역할 레벨**을 명확히 정의하고, 각 라우트에 강제 적용한다.

> 업무상 직급(사원/대리/과장 등)은 `position` 텍스트 필드로 별도 관리하며 **시스템 권한과 무관**

---

## 2. 역할(Role) 레벨 정의

| 레벨 | Role | 한국명 | 권한 범위 |
|------|------|--------|-----------|
| 1 | `ADMIN` | 전체 관리자 | 시스템 내 모든 작업 가능 |
| 2 | `MANAGER` | 프로젝트 관리자 | 프로젝트 생성/수정, 자원 등록/관리, 태스크 전체 관리 |
| 3 | `OPERATOR` | 운영자 | 본인 태스크 수정/관리, 프로젝트 및 전체 현황 조회 |

> `VIEWER` role은 조회 전용으로 유지 (외부 참관자 등)

## 4. 기술 결정

### 구현 방식
- 기존 `requireRole()` 미들웨어 활용
- 각 라우트에 `preHandler: requireRole("ADMIN", "MANAGER")` 패턴으로 적용
- 본인/타인 구분이 필요한 태스크 수정은 라우트 핸들러 내 분기 처리

### 변경 없는 것
- DB 스키마 변경 없음 (기존 `UserRole` enum 유지)
- JWT 구조 변경 없음
- 업무상 직급(`position` 텍스트)은 그대로 유지, 권한 무관

---

## 5. 구현 순서

1. **Phase 1** — project-service: `requireRole` 미들웨어 정비
2. **Phase 2** — project-service: 프로젝트/태스크/자원 라우트 권한 적용
3. **Phase 3** — project-service: 대시보드/그룹/템플릿 라우트 권한 적용
4. **Phase 4** — 프론트엔드: Role별 버튼/메뉴 표시 제어

---

## 6. 사용자 스토리

- **US-01**: OPERATOR는 프로젝트 생성 버튼이 보이지 않는다
- **US-02**: OPERATOR는 본인 태스크 진행률을 수정할 수 있다
- **US-03**: OPERATOR는 타인 태스크를 수정할 수 없다
- **US-04**: MANAGER는 프로젝트를 생성하고 자원을 관리할 수 있다
- **US-05**: ADMIN은 사용자 관리를 포함한 모든 작업이 가능하다
- **US-06**: VIEWER는 조회만 가능하다
