## 1. 개요 (Overview)

구매(해외/국내) → 입고 → 회계처리 → 재고관리 → 출고/납품의 전 과정을 관리하는 모듈. 크게 두 영역으로 구분된다:

- **구매/회계 영역**: 해외 발주, 수입원가정산, 문서 발행 등 — 재무팀 + 영업팀 전용
- **재고관리 영역**: 재고 현황, 입출고, 창고 이동 등 — 전 부서(전 사원) 사용

국내 구매(전 사원 가능)와 해외 구매(영업팀) 모두 재고 시스템으로 연동되며, 회계 처리는 재무팀/영업팀만 접근한다.

### 배경 (Background)

현재 업무 흐름:

**해외 구매 (영업팀 + 재무팀)**:
1. **발주**: 해외 제조사에 발주 → 인보이스 수령 → 송금 (EUR/GBP/USD 다통화)
2. **선적/통관**: 선적 → 통관(관세·운반비·수수료) → 입고
3. **수입원가정산**: 관세업체에서 수입면장(수입신고필증) 수령 → 재무팀이 수입면장 기반으로 수입원가정산 작성 (송금액 + 관세/통관/운송비 = 수입원가)

**국내 구매 (전 사원)**:
4. **일반 구매**: 업무에 필요한 제품/자재를 국내에서 구매 → 입고 등록

**재고관리 (전 부서 공용)**:
5. **입고**: 해외/국내 구매 입고 시 재고번호(E01394_1 형태) 부여
6. **이동/출고**: 구매/창고이동/판매 이력 추적
7. **납품**: 고객사에 납품 시 재고에서 출고 처리

### 엑셀 분석 결과

**발주_인보이스.xlsx** (48행):
- 헤더: 계약#, 고객사, 제작사, 품목, 통화, 발주금액, 발주일, 선적예정, 선적일, 통관, 입고, 입고위치, 발주담당자, invoice#, due date, OA#, 통관담당, 비고
- 통화 그룹: EUR, GBP, USD 별 섹션
- 색상 코딩: 선적예정, 통관, 입고 상태별 색 구분
- 하단: 통화별 발주금액 합계, 통장잔고, 추가필요금액 계산

**수입원가정산.xlsx** (136행, 양식 반복):
- 구조: 1건당 1양식 (45행 반복)
- 항목: 신고번호, 공급자, 신고일
- 수입원가 = (1)송금액(외화×환율) + (2)관세+통관+운송료(관세, 국외운반비, 국내운반비, 통관수수료, 창고보관료, 취급수수료/송금수수료)
- 모델/규격별 수량·단가·금액 명세
- 공급가액 = 수입원가 / 1.1 (VAT 역산)

**재고합본.xlsx** (23행):
- 헤더: 분류, 구분, 재고번호, 일자-No., 품목명[규격명], 수량, 공급사, 제조사, 최초창고, 이동창고, 납품처, 시리얼/로트No., 단가, 공급가액, 합계, 프로젝트명, 담당자명, 원가번호, 기타정보
- 분류: 미착품, 상품, 원재료, 전기상품, 전기원재료
- 구분: 구매, 창고(이동), 판매
- 재고번호 형식: `E{5자리}_{순번}` (예: E01394_1)
- 각 분류별 소계

## 2. 문제 정의 (Problem Statement)

| 문제 | 영향 |
|------|------|
| 발주 현황이 엑셀에 분산, 실시간 추적 불가 | 선적/입고 지연 감지 어려움, 색상 코딩 수작업 |
| 수입원가정산이 수동 양식 복사/입력 | 원가 계산 오류 위험, 검색/조회 불편 |
| 재고가 엑셀 합본으로 관리 | 재고번호 중복 위험, 이동이력 추적 어려움 |
| 장비 마스터 등록 없이 자유 텍스트 입력 | 동일 장비 다른 이름으로 중복 등록 |
| 통화별 잔고/필요금액 수동 계산 | 송금 계획 수립 지연 |
| 회계시스템 별도 입력 필요 | 상법 회계 시스템은 연동 불가 — 이중 입력은 불가피하며, 체크리스트로 누락 방지 |

## 4. 핵심 도메인 개념

### 4.1 장비 마스터 (Product Master)

장비의 "종류"를 등록하는 카탈로그. 실제 개별 제품(시리얼)과 분리.

```
ProductMaster (장비 마스터)
├── id, name, modelName, manufacturer
├── category: 상품 | 원재료 | 미착품
├── defaultCurrency, referencePrice
└── specs (JSON): 규격 정보
```

**중복 방지**: `(manufacturer + modelName)` unique 제약

### 4.2 계약 (Contract)

사업 단위로 부여되는 계약. 하나의 계약 아래 여러 발주가 생성됨.

```
Contract (계약)
├── contractNumber: 계약번호
├── name: 계약명 (사업명)
├── customer: 고객사 (최종 납품 고객)
├── status: ACTIVE | COMPLETED | CANCELLED
├── startDate, endDate
├── notes
└── orders[]: 해외 발주 목록 (1:N)
```

### 4.3 발주 (Purchase Order - 해외)

하나의 계약에 속하는 개별 발주 건. 영업팀 담당자가 발주부터 입고까지 전 과정을 트래킹.

```
OverseasOrder (해외 발주)
├── contractId: 계약 FK
├── orderNumber: "#26-066" 형식 (연도-순번)
├── manufacturer: 제조사
├── currency: EUR | GBP | USD
├── items[]: 발주 품목 목록 (1:N → OverseasOrderItem)
├── status: 상태 FSM (아래 참조, 부분 입고 지원)
├── 날짜:
│   ├── orderDate: 발주일 (결재 승인 후 해외 발주한 날)
│   ├── estimatedProductionEnd: 제작 완료 예정일
│   ├── estimatedShipDate: 선적 예정일
│   ├── actualShipDate: 실제 선적일
│   ├── customsDate: 통관일
│   └── arrivalDate: 입고일
├── 제작 진행:
│   ├── productionProgress: 제작 진행율 (0~100%)
│   ├── productionNotes: 제작 현황 메모 (최신 상태 기록)
│   └── progressHistory[]: 진행율 변경 이력 (1:N → OrderProgressLog)
├── 위치: arrivalLocation
├── 담당: orderedBy (영업팀 담당자), customsHandler (통관 담당)
├── invoiceNo, dueDate, oaNumber
├── totalAmount: 발주 총액 (외화)
├── totalAmountKRW: 원화 환산 금액 (참고용)
└── notes
```

**OverseasOrderItem (발주 품목)**:
```
OverseasOrderItem
├── id
├── orderId: 발주 FK
├── productMasterId: 장비 마스터 FK
├── name: 품목명
├── spec: 규격/사양
├── quantity: 발주 수량
├── receivedQuantity: 입고 수량 (기본 0, 부분 입고 시 누적)
├── unitPrice: 단가 (외화)
├── amount: 금액 (quantity × unitPrice)
├── receiptStatus: PENDING | PARTIALLY_RECEIVED | FULLY_RECEIVED
└── notes
```

**상태 FSM**:
```
[DRAFT] → [PENDING_APPROVAL] → [APPROVED] → [ORDERED] → [IN_PRODUCTION] → [SHIPPED] → [CUSTOMS] → [PARTIALLY_RECEIVED] → [ARRIVED] → [CLOSED]
               │                                              │                                │
               └→ [REJECTED]                                  └→ 진행율 0~100% 트래킹          └→ 일부 품목만 입고 시
                   (수정 후 재상신)                                 100% 도달 시 선적 준비             전체 입고 완료 시 ARRIVED

부분 입고 규칙:
├── 품목별 입고 처리: OverseasOrderItem.receivedQuantity 누적
├── 일부 품목 입고 → 발주 상태: PARTIALLY_RECEIVED
├── 전체 품목 receivedQuantity == quantity → 발주 상태: ARRIVED
└── 입고 시마다 해당 품목의 InventoryItem 생성 (미착품 → 상품 자동 전환)

Phase 1 (결재 없음): DRAFT → ORDERED → IN_PRODUCTION → SHIPPED → CUSTOMS → [PARTIALLY_RECEIVED →] ARRIVED → CLOSED
Phase 2 (결재 연동): DRAFT → PENDING_APPROVAL → APPROVED → ORDERED → ...
```

**제작 진행 이력 (OrderProgressLog)**:
```
OrderProgressLog (진행 이력)
├── id
├── orderId: 발주 FK
├── progress: 진행율 (%)
├── note: 메모 (예: "PCB 조립 완료, 테스트 진행 중")
├── updatedBy: 기록자 (영업팀 담당자)
└── createdAt: 기록일시
```

> 영업팀 담당자가 제조사와 소통하며 진행율을 주기적으로 업데이트.
> 진행율 100% → 선적 준비 상태 → 선적일 입력 시 SHIPPED로 전환.

```
│ EUR     │  1   │  2   │  3   │  1   │  0   │  2   │  9  │
```

**발주 트래킹 목록** (담당자 기본 필터):

- 제작 진행율 바 표시 (IN_PRODUCTION 상태)
- 선적예정일 경과 시 경고 표시
- 통관 지연 경고
- 정렬: 상태순 / 날짜순 / 제조사순

**통화별 합계 요약** (엑셀 하단 합계 대체):
```
통화별 발주 합계:
├── EUR: €123,456 (발주 9건)
├── GBP: £45,000 (발주 2건)
├── USD: $67,890 (발주 2건)
├── 통장잔고: (수동 입력 또는 메모)
└── 추가필요금액: 합계 - 잔고 (자동 계산)
```

### 4.4 국내 구매 — 지출결의서 (Expense Request)

부서에서 지출결의서를 올리고, 결재라인을 통해 승인 후 재무팀으로 이관. 재무팀이 재고관리 대상 여부를 판정하고, 재고 대상이면 입고 시점에 재고번호를 부여.

지출결의서 = **ApprovalDocument (template: EXPENSE)** + **ExpenseFollowUp (재무 후속 처리)**.
결재 프로세스(작성/상신/승인/반려)는 범용 결재 프레임워크(ApprovalDocument)가 처리하고,
ExpenseFollowUp은 결재 **승인 후** 재무팀 고유 업무(접수/재고판정/입고)만 담당한다.

```
ApprovalDocument (EXPENSE 양식)          ExpenseFollowUp (후속 처리 데이터)
├── 결재 프로세스 (상신~승인)             ├── approvalDocumentId: FK
├── 동적 필드: 건명, 목적, 공급사,        ├── financeInfo:
│   품목 내역 (품목명/수량/단가/금액)      │   ├── receivedBy: 재무팀 접수자
├── 결재선, 참조자, 참고부서               │   ├── receivedAt: 접수일
└── postApprovalAction:                  │   ├── isInventoryTarget: 재고관리 대상 여부
    FINANCE_FORWARD                      │   ├── inventoryDecisionBy: 판정자
        │                                │   ├── inventoryDecisionAt: 판정일
        └→ 승인 시 ExpenseFollowUp        │   └── inventoryDecisionNote: 판정 사유
           자동 생성 → 재무팀 대시보드     ├── arrivalInfo:
                                         │   ├── arrivalDate: 입고일
                                         │   ├── arrivalLocation: 입고 위치
                                         │   └── confirmedBy: 입고 확인자
                                         ├── inventoryItemId?: 재고 연결 FK
                                         └── notes
```

**후속 처리 FSM** (ExpenseFollowUp.status — 결재 승인 이후만):
```
[FINANCE_RECEIVED] → [INVENTORY_DECIDED] → [ARRIVED] → [COMPLETED]
                           │
                           └→ isInventoryTarget = false: 재고 미대상 → 바로 COMPLETED
                           └→ isInventoryTarget = true: 입고 대기 → ARRIVED에서 재고번호 부여
```

### 4.5 수입원가정산 (Import Cost Settlement)

```
ImportCostSettlement (수입원가정산)
├── declarationNo: 신고번호
├── supplier: 공급자
├── declarationDate: 신고일
├── remittances[]: 송금 내역 (일자, 외화금액, 환율, 원화금액, invoiceNo)
├── duties[]: 관세/통관/운송 비용 (명시적 원가)
│   ├── type: 관세 | 국외운반비 | 국내운반비 | 통관수수료 | 창고보관료 | 취급수수료
│   ├── amount, vat, notes
│   └── awbNo (항공화물운송장)
├── extras[]: 비명시 부대비용 (특정 품목에 직접 귀속)
│   ├── name: 비용명 (예: 클리어런스, 검역비, 기타)
│   ├── amount: 금액
│   ├── targetItemId: 귀속 품목 FK (items 중 하나 지정)
│   └── notes
├── totalExtraCost: 비명시 부대비용 합계
├── totalImportCost: 수입원가합계 (자동 계산 = 송금액 + duties + extras)
├── supplyAmount: 공급가액 (= 원가 / 1.1)
├── vat: 부가가치세
└── items[]: 모델/규격별 명세 (수량, 단가, 금액)
    └→ adjustedUnitPrice: 원래 단가 + 귀속된 부대비용 반영 후 단가
```

**비명시 부대비용 귀속 규칙**:
- 각 부대비용 항목마다 귀속할 품목을 1개 지정 (전체 배분 아님)
- 한 품목에 여러 부대비용이 귀속될 수 있음
- 예: 클리어런스 100만원 → 품목A, 검역비 50만원 → 품목A (모두 품목A에 귀속)

### 4.6 SKU 모델 (ProductMaster + ProductVariant + InventoryItem)

**3계층 구조** (v1.6, 2026-05-13 도입):

```
ProductMaster (제품 마스터 — 추상 제품 카탈로그)
├── id
├── name: "Cable", "발전기"
├── model_name: "MCIL-6-FS", "EU22i"
├── manufacturer: "Mariada", "HONDA"
├── master_code: 짧은 코드 (3~6자, SKU code 생성용)  ← v1.6
│   예: "M6F" (MCIL-6-FS), "EU22i", "RBF" (Bandpass Filter)
├── key_attributes: ["length", "end_type"] — SKU code에 포함할 핵심 축 (jsonb)  ← v1.6
├── common_specs: 공통 사양 jsonb (모델 전체 공통)
├── unit_of_measure: "EA"|"m"|"kg"|"L"|"식" 등 (마스터 단위로 통일)  ← v1.6
├── default_currency, reference_price
└── UNIQUE(manufacturer, model_name) — 중복 마스터 방지

      │ 1:N
      ▼
ProductVariant (SKU — 옵션 조합별 단위)         ← v1.6 신규
├── id
├── productMasterId (FK)
├── sku_code: 의미 코드 (사람 친화)
│   규칙: {master_code}-{핵심옵션1}-{핵심옵션2}[-{핵심옵션3}]
│   예: "M6F-10M-FE", "EU22i-220V-WP"
│   - 최대 3개 옵션까지 코드에 포함 (master.key_attributes 따라)
│   - 길이 가이드: 20자 이하
│   - 옵션값은 회사 약어 표준 사용 (WP=Waterproof, FE=Free End, BLK=검정 등)
├── variant_specs: jsonb { length: "10m", voltage: "220V", uv_protect: true, ... }
│   - sku_code에 포함되지 않은 부가 옵션도 모두 저장
├── is_active
└── UNIQUE(productMasterId, variant_specs) — 같은 옵션 중복 방지

**가격 추적 정책 (v1.6 정정):**
- Variant에 표준가(`reference_price`) 필드 **없음** — 표준가 개념 자체 제거
- "현재 표시 가격" = 가장 최근 IN_STOCK InventoryItem의 unit_price (helper 메서드)
- "가격 변동 이력" = InventoryItem 시계열 (createdAt + unit_price) 자연 추출
- 발주·견적 line은 입력 시점 사용자가 직접 단가 입력 (스냅샷 컬럼 없음)
- 향후 판매 이력 모델에선 원가 + 판매가 둘 다 필드 (별도 영역)

      │ 1:N
      ▼
InventoryItem (실재고 인스턴스 — "이 입고건")
├── inventoryNo: "INV-2605-0001" (v1.6 새 규칙)
├── productMasterId: FK → ProductMaster
├── variantId: FK → ProductVariant                ← v1.6 신규
├── serialNumber: 시리얼 (INDIVIDUAL 만)
├── trackingMode: INDIVIDUAL | BULK
│   ├── INDIVIDUAL: 1재고번호 = 1개체 (시리얼 추적)
│   └── BULK: 1재고번호 = N수량 (동일 사양·동일 원가 머지)
├── totalQuantity: 전체 수량 (위치별 합산)  ← v1.6 정정 (이전: quantity)
├── category, currentStatus                    ← currentLocation 제거 (v1.6 정정)
├── unitPrice, supplyAmount, totalAmount
├── totalAdditionalCost, totalCostOfOwnership
├── supplierId: FK → Supplier
├── projectName, assigneeName
├── sourceType, sourceId, costSettlementId?
├── costEvents[]: 추가비용 이력 (1:N → AssetCostEvent)
└── locations[]: InventoryItemLocation (1:N — 위치별 수량 분산)

InventoryItemLocation (위치별 수량 — 1 inventory = N locations)  ← v1.6 신규
├── id
├── inventoryItemId (FK)
├── locationId (FK → storage_locations)
├── quantity
└── UNIQUE(inventoryItemId, locationId)
```

**원칙: 1 inventory_no = N 위치.** "이 입고건"은 inventoryNo로 고유 식별, 위치 분산은 자유. 회계·AS 추적은 inventoryNo 기반 (변동 없음).

#### 4.6.1 사용자 등록 흐름

```
1. 모델명/제조사 검색 → ProductMaster 매칭
   ├─ 있음 → 선택
   └─ 없음 → 마스터 신규 생성
              ├─ manufacturer + model_name 강제 UNIQUE
              ├─ master_code 입력 (3~6자, 자동 추천 가능: model_name 약자)
              └─ key_attributes 지정 (SKU code 핵심 옵션 2~3개)

2. Variant 선택
   ├─ 적합한 Variant 있음 → 선택
   └─ 없음 → 새 Variant 추가
              ├─ variant_specs 입력 (key_attributes 폼 + 자유 추가)
              └─ sku_code 자동 생성: {master_code}-{key_attr1}-{key_attr2}
                  - 약어 표준 자동 적용 (WP/FE/STD 등)
                  - 사용자가 수동 수정 가능 (충돌 검증)

3. Inventory 등록 (단가/공급사/수량/위치 등)
```

**약어 표준 (회사 운영 가이드):**
- 회사 내 옵션 약어 사전 관리 (예: 영문 풀 → 약어)
- 운영 매뉴얼에 약어표 별도 문서화
- 신규 옵션 등장 시 약어 추가 + 일관성 검토

#### 4.6.2 재고번호 규칙 (v1.6)

**형식**: `INV-{YY}{MM}-{NNNN}` (예: `INV-2605-0001`)

- **월별 시퀀스 reset** — 매월 1번부터
- **ecount legacy 데이터** (`E01125_1`, `49021`)는 그대로 유지, 신규만 새 규칙
- **AS관리 호환** — repair_orders.otInventoryNo 텍스트 참조, 기존 번호 영향 없음

#### 4.6.3 입고 처리 — 머지 제거 (v1.6 최종 결정)

**모든 입고 = 새 inventory_no 발급.** 머지 로직 도입 안 함.

- 같은 SKU·단가·공급사·오늘 입고분이라도 **별개 InventoryItem record**
- 같은 SKU 묶음 조회는 검색 UI 3단계 토글 (Variant 보기)에서 자동 합산 표시
- `InboundRequest = InventoryItem 1:1` 자연 매핑

**머지 제거 이유:**
- InboundRequest 입고 큐 + SKU 토글 도입으로 머지 필요성 줄어듦
- 각 입고 이벤트의 메타(시점·요청자·source) 보존 가능
- FIFO 추적이 더 정확 (record당 단일 단가)
- 머지 조건 7개 누락 위험·복잡도 제거
- 사용자 학습 부담 ↓

**대체:**
- 같은 SKU의 여러 inventory가 분산되어도 검색·통계 시 자동 합산
- "M6F-10M-FE 총 재고 80개 (3 inventory)" 같은 표시

#### 4.6.6 Bundle BOM (번들 판매 + AS 추적)  ← v1.6 신규

> 고객사에 시스템 패키지로 출고하고, 추후 AS 시 그 중 일부만 처리해야 하는 비즈니스 요구 대응.
> 1단계 Flat BOM (다단계·assembly 미도입).

**데이터 모델:**

```
BomDefinition (번들 템플릿 — 재사용 가능)
├── id, code: 마스터 코드 (예: KHOA-OCEAN-V1)
├── name: "해양관측 표준 세트"
├── items[]: BomItem
│   ├── variant_id (FK → ProductVariant)
│   ├── quantity
│   ├── slot_type: MAIN | OPTIONAL                ← 별도 옵션 슬롯
│   └── notes
├── reference_price: 사용자 직접 입력 (자동 합산 X)
│   - 일반 inventory의 unit_price 입력처럼 사용자가 정함
│   - 표준가 자동 계산 시도 안 함 (표준가 개념 제거됨)
└── is_active

BundleShipment (실제 출고된 번들 인스턴스 — AS 추적용)
├── id, code: "BD-2026-0042"                      ← YYYY 형식, AS-2026-001·PO-2026-0340과 일치
├── bom_definition_id? (어느 템플릿 기반, 옵션)
├── customer_id
├── project_id?
├── shipped_at
├── ship_to
├── warranty_until                                  ← 번들 단위 보증 만료 (공통)
├── total_price
└── notes

(기존 customer_assets 확장)
customer_assets
├── ...기존 필드
├── bundle_shipment_id? (FK)                       ← 신규 — 어느 번들의 일부인지
└── bundle_role                                     ← 신규 — "CTD 센서", "Battery 1번" 등
```

**워크플로우:**

```
1. BomDefinition 마스터 등록 (한 번)
   - items 정의: MAIN 슬롯 + OPTIONAL 슬롯
   - reference_price_auto 자동 계산, manual 입력 가능

2. 출고
   - 번들 선택 → 시스템이 각 Variant에서 FIFO IN_STOCK 추천
   - 사용자가 inventory_no 수정 가능 (하이브리드)
   - BundleShipment 생성, warranty_until 입력
   - 각 inventory_item → customer_assets 생성 + bundle_shipment_id 설정

3. AS 발생
   - repair_orders.customerAssetId → 단일 자산 수리
   - UI: 같은 BundleShipment의 형제 자산 표시
   - 보증 확인: BundleShipment.warranty_until 우선
   - 다른 자산도 같이 점검 권장

4. 자산 재배치 (특수 케이스)
   - 일반: bundle_shipment_id 유지 (역할만 수정 가능)
   - ADMIN 전용: 번들에서 분리 가능 (특수 상황)
```

**번들 ID 형식:** `BD-{YYYY}-{NNNN}` (예: `BD-2026-0042`)
- 결재(`OT-EXPENSE-...`)·수리(`AS-2026-001`)·발주(`PO-2026-0340`)와 일관

**OPTIONAL 슬롯 출고 정책:**
- **BundleShipment 생성 시점**에 OPTIONAL 슬롯 포함 여부 확정 (v1.6 범위)
- 출고 작업자는 확정된 items만 자동 차감 (재선택 X)
- BundleShipment.items 에 확정 포함된 OPTIONAL만 저장
- 향후 견적 모듈 도입 시: 견적이 BundleShipment 생성의 입력이 됨 (시점 동일)

**의도적 제외 사항:**
- 다단계 BOM (Hierarchical) — 운영 부담
- Manufacturing BOM (자체 조립용) — 비즈니스 비중 작음
- Sub-assembly — 1단계만

#### 4.6.11 InboundRequest — 입고 대기 큐 (v1.6 신규)

> 발주·지출결의서·경비정산·수동 등 **모든 입고 요청을 단일 큐로 통합**.
> 자재 담당자가 한 화면에서 처리. 정보 보강 + 머지 룰 + 위치 지정을 일관 흐름으로.

**모델:**

```
InboundRequest (입고 대기)
├── id, code: "IR-YYMM-NNNN"
├── status: PENDING | RECEIVED | CANCELED
├── source_type: OVERSEAS_ORDER | PURCHASE_ORDER | EXPENSE_FOLLOWUP | EXPENSE_SETTLEMENT | MANUAL
├── source_id: 원천 문서 ID
├── source_doc_number: 표시용 ("OT-EXPENSE-...", "PO-2026-0340")
├── requester_id, requested_at
├── notes
├── received_at?, received_by?
└── items[]: InboundRequestItem (1:N)

InboundRequestItem
├── id, inbound_request_id (FK)
├── product_master_id?       ← 발주 경유면 자동, 결재 경유면 null
├── variant_id?
├── item_name_raw            ← 원천에서 받은 자유 텍스트
├── description
├── quantity
├── unit_price?
├── supplier_id?
└── completeness_flag: AUTO_MATCHED | PARTIAL | MANUAL_NEEDED
```

**원천별 정보 완전성:**

| 원천 | Master/Variant | Supplier | 단가 | 입고자 작업 |
|------|:---:|:---:|:---:|---|
| 해외/국내 발주 | ✅ 자동 | ✅ 자동 | ✅ 자동 | **위치만** 지정 (한 클릭) |
| 지출결의서 결재 | ❌ | ❌ | ⚠ 금액만 | Master/Variant/Supplier/위치 입력 |
| 경비정산 | ❌ | ❌ | ⚠ | 모두 입력 |
| 수동 | 입력자 책임 | 입력자 책임 | 입력자 책임 | 직접 입력 |

`completeness_flag`로 입고 담당자가 즉시 인지.

**흐름:**

```
1. 원천 이벤트:
   - 발주 도착 → InboundRequest 자동 생성 (AUTO_MATCHED)
   - 재무 접수에서 "재고 체크" → InboundRequest 자동 생성 (PARTIAL/MANUAL_NEEDED)
   - 수동 입고 → 사용자가 InboundRequest 작성 또는 즉시 처리

2. 입고 대기 페이지 (자재 담당자):
   - PENDING 상태 목록 + completeness_flag 표시
   - 항목 클릭 → 입고 처리 폼

3. 입고 처리 폼:
   - 자동 매핑된 정보 표시 (수정 가능)
   - 누락 정보 입력 (Master/Variant/Supplier 검색·생성)
   - 위치 지정 (storage_locations 드롭다운)
   - "입고 확정" → InventoryItem 생성 (BULK 머지 룰 자동 적용)
   - InboundRequest.status = RECEIVED

4. 출처 추적:
   - InventoryItem.sourceType/sourceId = InboundRequest 정보
   - InboundRequest.source_type/id = 원천 문서 정보
   - → 2단 추적: inventory → 입고요청 → 원천 문서
```

**기존 흐름 통합:**
- `expense-followup.confirmArrival`: 즉시 placeholder inventory 생성 → InboundRequest 생성으로 변경
- `overseas-order.createFromReceipt`: 즉시 inventory 생성 → InboundRequest 생성으로 변경
- 두 경로 모두 동일한 입고 처리 페이지에서 처리

#### 4.6.10 회계·통계 정책 (v1.6 신규)

**가격 추적 — 표준가 개념 제거 (v1.6 정정):**

- Variant·Master 모두 표준가 필드 **없음**
- ProductVariantPriceHistory 테이블 **도입 안 함**
- 발주·견적 line에 스냅샷 컬럼 **불필요** — 사용자가 line 작성 시 직접 단가 입력

**대체:**
- "현재 표시 단가" 도출:
  ```
  SELECT unit_price FROM inventory_items 
  WHERE variant_id = X AND currentStatus = IN_STOCK 
  ORDER BY createdAt DESC LIMIT 1
  ```
  → 가장 최근 IN_STOCK 인벤토리의 단가가 자연스러운 "최근 시세"
- "가격 변동 이력" 도출:
  ```
  SELECT createdAt, unit_price FROM inventory_items 
  WHERE variant_id = X 
  ORDER BY createdAt
  ```
  → InventoryItem 시계열 자체가 가격 변동 그래프

**향후 (판매 이력 모듈에서):**
- SalesLineItem(가칭)에 원가(cost) + 판매가(sale_price) 둘 다 필드
- 마진 계산·통계 가능

**재고 회전율 — 도입 안 함 (백로그):**
- 운용 경험 누적 후 도입 검토
- 입출고 트랜잭션은 이미 기록 중 → 분석 후속 추가 가능

**다중 창고 (1 inventory = N location, InventoryItemLocation 분산):**

**원칙:** `inventory_no` 는 "이 입고건"의 고유 식별자. 한 입고건의 수량이 여러 위치에 분산 보관될 수 있음 — 별도 inventory 발급 X. 회계·AS 추적은 inventoryNo 기반.

```
M6F-10M-FE 50개 ₩90,000 입고 → INV-2605-0001
  ├─ InventoryItem
  │   ├─ inventoryNo: INV-2605-0001
  │   ├─ totalQuantity: 50
  │   └─ unitPrice: ₩90,000
  └─ InventoryItemLocation (입고 시 전체 OT_A1_2)
      └─ (INV-2605-0001, OT_A1_2, 50)

이후 20개를 OT_A2_3로 옮김 (TRANSFER):
  ├─ InventoryItem (그대로, totalQuantity 50)
  └─ InventoryItemLocation 갱신:
      ├─ (INV-2605-0001, OT_A1_2, 30)
      └─ (INV-2605-0001, OT_A2_3, 20)
```

**구현:**
- `inventory_items` 테이블의 `current_location` 컬럼 → InventoryItemLocation 테이블로 정규화
- 기존 single-location 데이터는 1 row InventoryItemLocation으로 자동 변환

**위치 간 이동 (TRANSFER):**
- `InventoryTransaction.type = TRANSFER` 트랜잭션 기록
- InventoryItemLocation 수량만 조정 (출고 위치 row 감소, 입고 위치 row 증가/신규)
- inventoryNo 변동 없음 → AS·회계 추적 일관성

**조회:**
- 창고별 재고: `SELECT inventory_no, quantity FROM inventory_item_location JOIN inventory_items WHERE location_id=X`
- 가용 합산: `SELECT SUM(totalQuantity) FROM inventory_items WHERE variant_id=Y AND currentStatus=IN_STOCK`
- 위치별 가용 분산: InventoryItemLocation 그룹핑

**출고 시 위치 선택:**
- 사용자가 위치 지정 → 그 위치의 InventoryItemLocation 차감
- 그 InventoryItem의 totalQuantity도 동시 감소

#### 4.6.9 시스템 연계 정책 (v1.6 신규)

**발주 ↔ Variant:**
- `PurchaseOrderItem`, `OverseasOrderItem`, `BomItem` 모두에 `variant_id` FK 추가
- 발주 시 Variant 정확히 명시 → 입고 시 자동 매칭

**수리 (AS관리) ↔ 재고:**
- 자동 부품 차감은 도입 안 함 (백로그)
- 수리 시 부품 사용은 별도 수동 출고 트랜잭션으로 처리
- 추후 Service BOM 도입 시 자동화 검토

**예약 재고 / 가용 재고:**
- "가용 재고" = 현재 IN_STOCK 수량 + 진행 중 발주 예정 수량
- 발주 상태가 CLOSED 아닌 한 가용 산입
- 영업 견적 시 가용 기준 표시
- 정확한 inventory 수량은 별도 표시

#### 4.6.8 운영 정책 (v1.6 신규)

**재고 평가 — FIFO (선입선출):**
- 같은 SKU 여러 batch 출고 시, 먼저 입고된 inventory_item부터 차감
- inventory_no별 추적이 이미 가능 → 정확한 원가 계산
- 출고 트랜잭션에 사용된 batch 명세 기록 (회계 추적)

**안전재고 — 도입 안 함 (백로그):**
- minimum_stock 컬럼 추가 안 함
- 운용 시작 후 필요성 확인되면 후속 Phase에서 도입

**단종 관리 — 도입 안 함 (백로그):**
- Variant.is_active 컬럼은 만들되 자동 처리 안 함
- 운용 도중 필요 시 활용

**Variant Merge — ADMIN 전용 UI:**
- 잘못 분리된 두 Variant를 하나로 통합
- 참조 inventory_items, BomItem 자동 재연결
- 통합 작업 audit log 보존
- 마이그레이션 직후 정리·운용 중 실수 보정용

#### 4.6.7 재고 검색·목록 UI 토글 (3단계 View)  ← v1.6

같은 데이터를 3가지 시점으로 전환:

```
│ [마스터] [Variant] [Inventory]      │
└─────────────────────────────────────┘

마스터 보기 (요약, 기본):

- 마스터 단위 행. 클릭 → Variant 보기 또는 펼침
```

### 4.7 자산 추가비용 이력 (Asset Cost History)

입고 이후 발생하는 모든 추가비용을 재고 단위로 추적. AS, 업그레이드, 부품/모듈 추가 등.

```
AssetCostEvent (자산 추가비용)
├── id
├── inventoryItemId: 재고 FK
├── type: AS | UPGRADE | PARTS_ADDITION | MODULE_ADDITION | REPAIR | OTHER
├── title: 비용 항목명 (예: "센서 모듈 추가", "펌웨어 업그레이드")
├── description: 상세 내역
├── vendor: 수행 업체/공급사
├── cost: 비용 금액 (KRW)
├── currency: 원화 외 결제 시 (EUR/USD/GBP)
├── foreignAmount: 외화 금액
├── exchangeRate: 적용 환율
├── eventDate: 발생일
├── performedBy: 처리 담당자
├── attachments[]: 증빙 첨부 (Attachment 연결)
├── relatedOrderId?: 관련 발주 FK (해외 부품 발주 시)
├── notes
├── createdBy: 등록자
└── createdAt
```

**재고 원가 연동 규칙**:
- 각 재고 아이템은 **초기 원가** (수입원가정산 또는 구매가) + **누적 추가비용**을 가짐
- `totalCostOfOwnership = initialCost + SUM(AssetCostEvent.cost)` (자동 계산)
- 재고 상세 페이지에서 초기 원가 + 추가비용 이력 + TCO 한눈에 확인

**추가비용 유형**:

| 유형 | 설명 | 예시 |
|------|------|------|
| AS | 사후 서비스, 수리 | 보증 수리, 유상 수리 |
| UPGRADE | 하드웨어/소프트웨어 업그레이드 | 펌웨어 업데이트, 사양 변경 |
| PARTS_ADDITION | 부품 추가 장착 | 센서 모듈, 확장 카드 |
| MODULE_ADDITION | 모듈 추가 | 통신 모듈, 전원 모듈 |
| REPAIR | 외부 수리 (수리관리와 별개) | 외주 수리, 교정 |
| OTHER | 기타 비용 | 운송비, 설치비 |

### 4.8 범용 전자결재 프레임워크 (Approval Framework)

모든 결재 업무를 하나의 프레임워크로 처리. ecount ERP 19종 양식을 포괄하며, 양식 추가가 자유로운 구조.

#### 4.8.1 결재 양식 (ApprovalTemplate)

결재 문서의 "종류"를 정의. 관리자가 등록/수정.

```
ApprovalTemplate (결재 양식)
├── id
├── code: 양식 코드 (unique, 예: "EXPENSE", "PURCHASE_ORDER")
├── name: 양식명 (예: "지출결의서", "기안서_사전결재")
├── category: 양식 분류
│   └→ GENERAL | PROCUREMENT | ATTENDANCE | HR | WELFARE
├── description: 양식 설명
├── fields[]: 동적 필드 정의 (JSON)
│   └→ [{ key, label, type(text|number|date|select|textarea), required, options? }]
├── defaultApprovalLineRule: 기본 결재선 규칙
│   └→ DEPARTMENT_LINE: 부서 결재라인 사용 (기존 ApprovalLine)
│   └→ FIXED: 고정 결재자 지정 (예: 관리팀장 고정)
│   └→ AMOUNT_BASED: 금액 기준 (100만원 이상 대표이사 추가)
├── postApprovalAction: 승인 후 자동 동작 (선택)
│   └→ null: 없음 (기안서, 경위서 등)
│   └→ FINANCE_FORWARD: 재무팀 전달 (지출결의서)
│   └→ ORDER_CONFIRM: 발주 확정 (발주서)
│   └→ INVENTORY_RELEASE: 출고 처리 (출고신청서)
│   └→ LEAVE_APPROVE: 휴가 승인 처리 (향후 근태 마이그레이션)
│   └→ OT_APPROVE: OT 승인 처리 (향후 근태 마이그레이션)
├── relatedService: 연동 서비스 (후처리 HTTP 콜백 대상)
│   └→ null | "equipment-service" | "attendance-service" | "project-service"
├── isActive: 사용 여부
├── sortOrder: 정렬 순서
└── createdAt, updatedAt
```

#### 4.8.2 결재 문서 (ApprovalDocument)

실제 기안/상신되는 개별 결재 건.

```
ApprovalDocument (결재 문서)
├── id
├── documentNumber: 문서번호 (자동 채번, 예: "EXP-2026-0142")
├── templateId: 결재 양식 FK
├── title: 결재 제목
├── requestedBy: 기안자 (userId)
├── department: 기안 부서
├── approvalStepCount: 결재 단계 수 (1 | 2 | 3)
│   └→ 기안 시 사용자가 선택 (기본값: ApprovalLine에서 자동 감지)
│   └→ 1단계: 기안자 → 1차 결재자 → 완료
│   └→ 2단계: 기안자 → 1차 → 2차 결재자 → 완료
│   └→ 3단계: 기안자 → 1차 → 2차 → 3차 결재자 → 완료
├── content: 동적 필드 데이터 (JSON — template.fields에 대응)
├── amount: 결재 금액 (해당 시)
├── referenceType: 연결 대상 유형 (선택)
│   └→ null | OVERSEAS_ORDER | EXPENSE_REQUEST | INVENTORY_ITEM | LEAVE | OVERTIME
├── referenceId: 연결 대상 ID (선택)
├── status: 결재 상태 FSM (아래 참조)
├── attachments[]: 첨부파일 목록 (1:N → Attachment)
├── notes: 기안 비고
├── referenceDepts[]: 참고부서 목록 (departmentId[])
│   └→ 결재 완료 후 해당 부서에 문서 전달 (열람용)
├── referencePersons[]: 참고담당자 목록 (userId[])
│   └→ 결재 완료 후 개인에게 문서 전달 (열람용)
├── submittedAt: 상신 일시
├── completedAt: 최종 처리 일시
├── forwardedAt: 참고부서/담당자 전달 일시
├── createdAt, updatedAt
└── steps[]: 결재 단계 목록 (1:N → ApprovalStep, 개수 = approvalStepCount)
```

#### 4.8.3 결재 단계 (ApprovalStep)

각 문서의 N단 결재 진행 상태.

```
ApprovalStep (결재 단계)
├── id
├── documentId: 결재 문서 FK
├── stepOrder: 단계 순번 (1, 2, 3)
├── roleName: 직책명 스냅샷 (예: "팀장", "이사", "대표이사") — PDF 결재란 표시용
├── approverId: 결재자 (userId)
├── approverName: 결재자명 (스냅샷)
├── status: PENDING | APPROVED | REJECTED | SKIPPED
├── comment: 결재 의견
├── actedAt: 처리 일시
└── createdAt

> **결재 단계 생성 규칙**:
> - 문서 생성 시 `approvalStepCount`만큼 ApprovalStep 레코드 생성
> - 결재자는 기안자의 ApprovalLine에서 순서대로 매핑:
>   - 1단계: approverId만
>   - 2단계: approverId + secondApproverId
>   - 3단계: approverId + secondApproverId + thirdApproverId
> - 수동 변경 가능 (결재선 검색으로 결재자 교체)
```

#### 4.8.4 결재 상태 FSM

```
[DRAFT] → [AGREEMENT_PENDING] → [SUBMITTED] → [STEP_1_PENDING] → [STEP_2_PENDING] → [STEP_3_PENDING] → [APPROVED]
               │                     │                │                    │                    │
               └→ RETURNED           │                └── REJECTED ────────┴── REJECTED ────────┘
               (합의자 부동의)         │
                                     └→ 합의자 없으면 바로 SUBMITTED
                                     └→ 결재 단계 수는 approvalStepCount에 따라 1~3단 가변
                                        (선택한 단계 이후는 자동 스킵)

승인 완료 후:
[APPROVED] → ① postApprovalAction 자동 실행 + ② 참고부서/담당자 전달
    │
    ├─ ① postApprovalAction (비즈니스 후처리):
    │   ├→ FINANCE_FORWARD: 재무팀 대시보드에 표시
    │   ├→ ORDER_CONFIRM: 발주 상태 → ORDERED
    │   ├→ INVENTORY_RELEASE: 재고 출고 트랜잭션 생성
    │   ├→ LEAVE_APPROVE: WorkScheduleEntry upsert (향후)
    │   └→ OT_APPROVE: WorkScheduleEntry upsert (향후)
    │
    └─ ② 참고 전달 (referenceDepts[] + referencePersons[]):
        ├→ 참고부서 전체 구성원에게 알림 + 참고함에 문서 표시
        ├→ 참고담당자 개인에게 알림 + 참고함에 문서 표시
        └→ 참고 수신자는 열람만 가능 (결재 권한 없음)
```

#### 4.8.5 첨부파일 (Attachment)

```
Attachment (첨부파일)
├── id
├── referenceType: APPROVAL_DOCUMENT | SETTLEMENT | ORDER | INVENTORY
├── referenceId: 대상 문서 ID
├── fileName: 원본 파일명
├── storagePath: 저장 경로
├── fileSize: 파일 크기
├── mimeType: image/jpeg, application/pdf 등
├── isScanned: 스캔 촬영 여부
├── uploadedBy: 업로드자
└── createdAt
```

#### 4.8.9 결재 문서 확장 필드

스크린샷에서 발견된 추가 필드 — ApprovalDocument에 반영:

```
ApprovalDocument (결재 문서) — 추가 필드
├── ccUsers[]: 참조자 목록 (userId[]) — 결재 진행 중 열람 가능 (의견 불요)
├── agreementUsers[]: 합의자 목록 (userId[]) — 결재 전 사전 합의 필요 (의견 필수)
├── richBody: 리치 텍스트 본문 (HTML)
├── itemsData: 세부 내역 테이블 (JSON[])
│   └→ [{ description, unitPrice, quantity, subtotal, vat }]
├── itemsTotal: 합계 금액 (자동 계산)
├── templateFooter: 양식 하단 문구 (양식에서 상속)
└── pdfGeneratedAt: PDF 생성 일시 (미리보기/출력 시)

> **참조자 vs 합의자 vs 참고부서/담당자 구분**:
> | 구분 | 시점 | 권한 | 용도 |
> |------|------|------|------|
> | **ccUsers** (참조자) | 결재 진행 중 | 열람만 가능 | 정보 공유 (FYI) |
> | **agreementUsers** (합의자) | 결재 시작 전 | 의견 필수 입력 (동의/부동의) | 타 부서 사전 합의 |
> | **referenceDepts/Persons** (참고부서/담당자) | 결재 완료 후 | 열람만 가능 | 후속 업무 전달 |
>
> **합의자 프로세스**: 상신 시 합의자에게 먼저 알림 → 전원 동의 후 결재 시작.
> 합의자 부동의 시 기안자에게 반환 (결재선 진입 전).
>
> **PDF 결재란 동적 렌더링**:
> - `approvalStepCount`에 따라 결재란 컬럼 수 변동
> - 1단계: | 기안 | 결재 |
> - 2단계: | 기안 | 결재 | 결재 |  (ecount 지출결의서 양식 참고)
> - 3단계: | 기안 | 결재 | 결재 | 결재 |  (ecount 기안서 양식 참고)
> - 각 컬럼에 roleName (직책), approverName (성명), actedAt (일자), 서명/도장 표시
> - 참고부서/담당자는 결재란 하단 별도 영역에 표시
```

  - 세부 내역 테이블 (행 추가/삭제, 단가×수량=소계, VAT 자동, 합계)

  - **PDF 미리보기**: 결재란이 선택한 단계 수에 맞춰 동적 렌더링
- **결재 상세**: 결재 단계 타임라인 + 승인/반려 버튼 + 의견 입력
- **상신함**: 내가 올린 문서의 진행 상태 추적

#### 4.8.11 근태 결재 마이그레이션 계획

현재 attendance-service 내부에 하드코딩된 휴가/OT 결재를 범용 프레임워크로 이관.

```

    └→ 지출결의서, 발주서, 기안서 등으로 검증

Phase B: 근태 결재 마이그레이션 (별도 Phase)
    └→ 휴가신청 → ApprovalDocument (template: LEAVE) 생성
    └→ OT신청 → ApprovalDocument (template: HOLIDAY_WORK) 생성
    └→ 승인 시 postApprovalAction → attendance-service 내부 API 호출
       └→ POST /internal/leave/{id}/approve
       └→ POST /internal/overtime/{id}/approve
    └→ 기존 LeaveService.approve(), OvertimeService.approve() 내부 로직은 유지
       (WorkScheduleEntry upsert 등)
    └→ 기존 leave.routes/overtime.routes의 approve/reject 엔드포인트는
       프레임워크 경유로 전환 (직접 호출 → deprecated)
    └→ 팀 근태 페이지(/me/team)의 승인 UI → 결재 대기함으로 통합
```

## 5. 역할 및 권한 (Roles)

| 역할 | 구매/회계 영역 | 재고 영역 |
|------|---------------|-----------|
| **전 사원** | 지출결의서 작성/상신 | 재고 조회, 입출고 등록, 창고 이동, 추가비용 등록 |

| **재무팀** | 송금, 수입원가정산, 지출결의서 접수/재고대상 판정, 문서 발행 | 전체 접근 |
| **대표이사** | 발주 결재 승인, 주요 계약 결재, 고액 지출결의서 결재 | 전체 접근 |

> **구매/회계**: 영업팀 + 재무팀 실무 협업, 대표이사 결재
> **국내 구매**: 전 사원 지출결의서 상신 → 부서 결재 → 재무팀 접수/재고판정
> **재고**: 전 부서 공용 — 구매(해외/국내) 입고 시 연동 (재무팀 판정 기반)

## 6. 업무 흐름 (Workflow)

### 6.1 해외 구매 흐름 (영업팀 + 재무팀)

```
[1] 장비 마스터 등록 (영업팀)
    └→ manufacturer + modelName unique 검증
    └→ 중복 시 기존 마스터 재사용

[2] 계약 등록 (영업팀)
    └→ 계약번호, 사업명, 고객사(납품 고객) 입력
    └→ 상태: ACTIVE

[3] 해외 발주 생성 (영업팀)
    └→ 계약 선택 → 제조사, 품목(마스터 참조), 통화, 금액
    └→ 상태: DRAFT
    └→ Phase 1: 직접 ORDERED로 전환
    └→ Phase 2+: 결재 상신 → PENDING_APPROVAL → 승인 시 APPROVED

[4] 발주 확정 → 해외 발주 + 송금 (영업팀 + 재무팀)
    └→ 해외 제조사에 발주 전달 (영업팀)
    └→ invoice#, due date, OA# 입력, 송금 처리 (재무팀)
    └→ 상태: ORDERED

[5] 제작 트래킹 (영업팀 담당자)
    └→ 제조사와 소통하며 제작 진행율 주기적 업데이트 (0~100%)
    └→ 진행 메모 기록 (예: "PCB 조립 완료, 테스트 진행 중")
    └→ 상태: IN_PRODUCTION
    └→ 담당자 대시보드에서 전체 발주 현황 트래킹

[6] 선적 (영업팀)
    └→ 진행율 100% 도달 → 선적 준비
    └→ 선적예정일 → 실제선적일 입력
    └→ 상태: SHIPPED

[7] 통관 (영업팀 + 재무팀)
    └→ 통관일, 통관담당자 입력
    └→ 관세업체로부터 수입면장(수입신고필증) 수령
    └→ 상태: CUSTOMS

[8] 입고 (영업팀 + 재무팀)
    └→ 입고일, 입고위치 입력 (영업팀)
    └→ 재무팀이 수입면장 기반으로 수입원가정산 작성 (송금액 + 관세/운송비 → 원가 산출)
    └→ 재고번호 자동 부여 (장비 마스터 + 시리얼 → 재고번호)
    └→ 재고 트랜잭션: "구매(입고)" 기록
    └→ 상태: ARRIVED

[9] 납품/출고 (영업팀)
    └→ 납품처, 프로젝트 입력
    └→ 재고 트랜잭션: "판매(출고)" 기록
    └→ 발주 상태: CLOSED
    └→ 계약 내 모든 발주 CLOSED 시 계약 상태: COMPLETED
```

### 6.2 국내 구매 흐름 — 지출결의서 (전 사원 → 부서 결재 → 재무팀)

```
[1] 지출결의서 작성 (전 사원)
    └→ 건명, 목적/사유, 품목(명/수량/단가), 공급사, 프로젝트(선택)
    └→ 견적서/영수증 등 증빙 첨부
    └→ 상태: DRAFT → 작성 완료 후 SUBMITTED (결재 상신)

[2] 부서 결재 (결재라인 — 부서장/대표이사)
    └→ 기존 결재라인(ApprovalLine)을 통해 승인/반려
    └→ 승인: 상태 → APPROVED
    └→ 반려: 상태 → REJECTED (사유 기재, 수정 후 재상신 가능)

[3] 재무팀 접수 (재무팀)
    └→ 결재 완료된 지출결의서가 재무팀 대시보드에 자동 표시
    └→ 재무팀 담당자가 접수 처리
    └→ 상태: FINANCE_RECEIVED

[4] 재고 대상 판정 (재무팀)
    └→ 재무팀이 지출 내역을 검토하여 재고관리 대상 여부 결정
    └→ 판정 기준: 금액, 자산 성격, 프로젝트 투입 여부 등
    └→ isInventoryTarget = true → 재고 대상 (입고 대기)
    └→ isInventoryTarget = false → 재고 미대상 (소모품 등) → 바로 COMPLETED
    └→ 상태: INVENTORY_DECIDED

[5] 입고 + 재고번호 부여 (재고 대상인 경우)
    └→ 제품 도착 시 입고일, 입고위치 입력
    └→ 재고번호 자동 부여 (장비 마스터 연결 또는 신규 등록)
    └→ InventoryItem 생성 + 지출결의서 연결 (sourceType: EXPENSE_REQUEST)
    └→ 재고 트랜잭션: "구매(입고)" 기록
    └→ 상태: ARRIVED → COMPLETED
```

### 6.3 재고 공통 흐름 (전 부서)

```
[1] 재고 조회 (전 사원)
    └→ 분류/위치/상태별 필터
    └→ 전체 재고 현황 확인

[2] 창고 이동 (전 사원)
    └→ 재고 트랜잭션: "창고(이동)" 기록
    └→ currentLocation 갱신

[3] 출고/반납 (전 사원)
    └→ 사용 목적, 프로젝트 입력
    └→ 재고 트랜잭션: "출고" 또는 "반납" 기록
```

### 6.4 추가비용 등록 흐름

```
[1] 재고 상세 → "비용 추가" (전 사원)
    └→ 유형 선택: AS | 업그레이드 | 부품추가 | 모듈추가 | 수리 | 기타
    └→ 비용 항목명, 금액, 업체, 일자, 증빙 첨부
    └→ 외화 결제 시: 통화, 외화금액, 환율 입력

[2] 저장 시 자동 처리
    └→ AssetCostEvent 생성
    └→ InventoryItem.totalAdditionalCost += cost
    └→ InventoryItem.totalCostOfOwnership 재계산

[3] 비용 이력 조회 (재고 상세 > 비용 이력 탭)
    └→ 초기 원가 (수입원가정산 / 구매가)
    └→ 추가비용 목록 (날짜순, 유형별 필터)
    └→ TCO = 초기원가 + 추가비용 합계
```

## 8. 기술 결정 (Technical Decisions)

### 8.1 서비스 배치

**결정: equipment-service 내 확장**

이유:
- 재고번호가 장비 관리(Equipment)와 직접 연결
- 기존 Part(부품), PurchaseOrder(발주) 모델과 구조 유사
- 수입원가는 수리관리의 RepairCost와 유사한 패턴
- 별도 서비스 불필요 (같은 DB 스키마)

### 8.2 UI 배치

**결정: 상단 탭 2개 분리 — 역할별 접근 제어**

```
AppLayout 상단 탭:
[프로젝트] [근태] [장비] [수리관리] [구매/회계] [재고] [결재] ← 신규 3개
                                    ↑ 재무+영업만   ↑ 전사원  ↑ 전사원
```

구매/회계 페이지 (재무팀 + 영업팀):

```
apps/web/src/app/procurement/
├── layout.tsx                 # 서브탭 레이아웃
├── page.tsx                   # 해외 발주 목록 (메인)
├── [id]/page.tsx              # 발주 상세
├── contracts/page.tsx         # 계약 목록
├── contracts/[id]/page.tsx    # 계약 상세 (하위 발주 목록)
├── expenses/page.tsx           # 지출결의서 목록 (재무팀: 접수/판정)
├── expenses/[id]/page.tsx     # 지출결의서 상세
├── masters/page.tsx           # 장비 마스터 목록

└── settlements/[id]/page.tsx  # 원가정산 상세
```

재고 페이지 (전 사원):
```
apps/web/src/app/inventory/
├── layout.tsx              # 서브탭 레이아웃
├── page.tsx                # 재고 목록 (메인)
├── [id]/page.tsx           # 재고 상세 (탭: 기본정보, 입출고이력, 비용이력, TCO)
└── transactions/page.tsx   # 입출고/이동 이력
```

**재고 상세 페이지 탭 구성**:
- **기본정보**: 재고번호, 품목, 시리얼, 위치, 상태
- **입출고 이력**: InventoryTransaction 목록
- **비용 이력**: 초기 원가 + AssetCostEvent 목록 (AS/업그레이드/부품추가 등)
- **TCO 요약**: 초기원가 + 추가비용 합계 = 총 보유비용, 비용 유형별 차트

### 8.3 재고번호 생성

- DB 시퀀스 `inventory_seq` 사용 (5자리, 0패딩)
- 형식: `E{seq}_{itemIndex}` (예: E02300_1)
- unique 제약: `inventoryNo` 컬럼

### 8.4 다통화 처리

- 발주: 통화(EUR/GBP/USD) + 외화금액 저장
- 원가정산: 외화금액 + 환율 → 원화금액 자동 계산
- 재고: 원화(KRW) 기준 단가/금액

### 8.5 문서 발행 시스템

**방식**: 서버사이드 PDF 생성 + 브라우저 인쇄/이메일 발송

**발행 가능 문서**:

| 문서 | 데이터 소스 | 용도 | 발송 대상 |
|------|-------------|------|-----------|
| 발주서 (Purchase Order) | OverseasOrder + items | 해외 제조사에 발주 요청 | 제조사 (이메일) |
| 수입원가정산서 | ImportCostSettlement | 사내 원가 기록/보관 | 재무팀 (인쇄/보관) |
| 거래명세서 | 납품 정보 + 재고 품목 | 고객사 납품 시 첨부 | 고객사 (이메일/인쇄) |

**기능**:
- PDF 미리보기 → 인쇄 또는 이메일 발송 선택
- 이메일 발송: 수신자 입력 → 첨부파일(PDF)로 발송
- 인쇄: 브라우저 인쇄 다이얼로그
- 발행 이력 기록 (언제, 누가, 어디로 발송했는지)

### 8.6 파일 스토리지 + 이미지 최적화

- **스토리지**: Docker 볼륨 마운트 (uploads/ 디렉토리)
- **최적화 라이브러리**: sharp (Node.js)
- **이미지 처리 규칙**:
  - 장변 최대 2000px 리사이징 (문서 판독 충분)
  - JPEG 품질 80% 압축
  - 원본 대비 약 70~90% 용량 절감
  - PDF 파일은 원본 유지
- **파일 제한**: 단일 파일 최대 10MB, 건당 첨부 최대 10개
- **스마트폰 스캔**: HTML5 `<input type="file" accept="image/*" capture="environment">`

### 8.7 범용 전자결재 프레임워크

- **서비스 배치**: **별도 approval-service (Port 3006)** 신규 생성
  - 이유: 결재 문서량(연 ~10,000건) + 파일 첨부 + PDF 생성 + 18종 양식이 auth-service 범위 초과
  - 결재선(ApprovalLine)은 auth-service에 유지, approval-service가 내부 API로 조회
  - FileService(업로드/이미지 최적화)도 approval-service에 포함
- **결재 FSM**: DRAFT → [합의] → SUBMITTED → STEP_N_PENDING → APPROVED | REJECTED (N단 가변)
- **양식 시스템**: ApprovalTemplate (동적 필드 정의 JSON) → 양식 추가 시 코드 변경 없음
- **후처리 훅**: postApprovalAction으로 업무별 자동 동작 (FINANCE_FORWARD, ORDER_CONFIRM 등)
- **결재선 연동**: 기존 ApprovalLine (3단+위임) 재사용, 양식별 규칙(고정/금액기준) 확장
- **근태 마이그레이션**: Phase 8에서 휴가/OT를 프레임워크 경유로 전환 (internal API 콜백)

### 8.8 알림 시스템

- **결재 알림**: 결재 요청/승인/반려/합의요청 시 대상자에게 알림
- **참고 전달 알림**: 결재 완료 후 참고부서/담당자에게 문서 전달 알림
- **발주 경고**: 선적예정일 경과, 통관 지연 시 담당자 알림
- **UI 표시**: AppLayout 헤더에 결재 대기 건수 배지 + 알림 드롭다운
- **구현 방식**: 기존 notifications 테이블 활용 (auth-service activity 이벤트 패턴 재사용)
  - Phase 1: DB 폴링 기반 (알림 테이블 + 주기적 조회)
  - 향후: WebSocket 실시간 알림 (고도화)

    - **1단계 Dry-run**: 자동 파싱 결과 CSV 리포트 생성

       - ✅ 명확 (대괄호 패턴 등) → 일괄 confirm
       - ⚠️ 추정 → 수동 확정
       - ❌ 사양 없음 → default variant (variant_specs = {})
    - **3단계 적용**: 
       - ProductVariant 생성 (default 포함)
       - inventory_items.variant_id 일괄 채움
       - tracking_mode 일관성 회복 (serial=NULL → BULK 등 보정)
    - **4단계 검증**: 같은 (master+variant) 그룹 재고 합산이 이전과 일치
    - **5단계 후속 (운영 중)**: 모델명별로 비슷한 variant 통합·세분화 점검
    
    **표준 키 사전**: `length`, `pin_count`, `core_count`, `freq`, `voltage`, `model_no`, `spec` (자유 텍스트 fallback). 신규 키 발견 시 사전 추가

## 12. 용어 정리

| 용어 | 설명 |
|------|------|
| 장비 마스터 (ProductMaster) | 장비 종류 카탈로그 (모델+제조사 기준) |
| 계약 (Contract) | 사업 단위 계약, 하나의 계약에 여러 발주 포함 (1:N) |
| 해외 발주 (OverseasOrder) | 계약 하위의 개별 해외 제조사 구매 건 |
| 수입면장 (수입신고필증) | 관세업체에서 수령하는 통관 서류 (신고번호, 관세 등 기재) |
| 수입원가정산 (ImportCostSettlement) | 재무팀이 수입면장 기반으로 작성하는 원가 계산서 (송금액+관세+운송비) |
| 재고번호 (inventoryNo) | E{5자리}_{순번} 형식의 유일 식별자 |
| 지출결의서 | ApprovalDocument(EXPENSE 양식) + ExpenseFollowUp 조합으로 구현 |
| ExpenseFollowUp | 결재 승인 후 재무팀 후속 처리 데이터 (접수/재고판정/입고) |
| 재고 대상 판정 | 재무팀이 지출결의서 검토 후 재고관리 대상(자산) vs 비대상(소모품) 결정 |
| 재고 트랜잭션 | 구매(입고) / 창고(이동) / 판매(출고) |
| 미착품 | 발주했으나 아직 입고되지 않은 품목 |
| OA# | Open Account 번호 (외상거래 번호) |
| AWB | Air Waybill (항공화물운송장) |
