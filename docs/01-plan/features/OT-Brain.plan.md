## 1. Overview

### 1.1 Purpose

오션테크의 모든 디지털 활동(NAS·ERP·미래 메일/Teams)을 통합하는 **회사 지식 hub**를 구축한다.

핵심 가치:
- **통합 검색**: "○○고객사 관련 모든 자료" = NAS 파일 + ERP 결재 + 거래 + AS + 작업비고 통합 한 화면
- **LLM 답변**: 회사 컨텍스트 기반 자동 응답 (RAG)
- **ERP 위젯**: 결재 작성·작업비고·고객사·프로젝트 페이지에서 관련 자료 자동 추천·자동 첨부

### 1.2 Background

2026-05-19 시작한 NAS 디스크 스캔(Tier 0~3) + 카탈로그 v5 분석에서 두 가지 결정적 통찰이 나왔다:

1. **다맥락 자료 = 핵심성이 아니라 반복 업무의 발자국** (`project_nas_repeat_pattern`).
   같은 hash가 800K+ 위치에 사본으로 존재. 이는 직원이 동일 업무마다 같은 서류를 반복 첨부한 흔적.
2. **도메인 정체 확인 — 선박 특별 검사 신청 업무** (`project_special_vessel_inspection`).
   만성호(13부서·229회)·일진호·씨로드호 등 임대 작업선의 선적증서·검사증서가 반복 첨부됨.

→ 이 반복 패턴을 **마스터 자산 + ERP 자동 첨부**로 환원하면 직접적 업무 자동화 가치가 발생한다.
이것이 OT-Brain의 첫 번째 구체적 수익 지점(PoC)이다.

### 1.3 Why a Separate System (별도 시스템 근거)

- ERP에 끼워넣으면 책임 비대·운영 부담·확장 제약
- OT-Brain은 ERP **외** 시스템(미래 메일·Teams·외부 자료)과도 연동 필요 → ERP 내부에 두면 모순
- ERP가 **source(데이터 제공) + consumer(위젯 소비)** 양방향이라는 정체성은 별도 시스템일 때만 성립
- 외부 노출·상품화 가능성도 열림

> **확정 (2026-05-21)**: 별도 독립 repo·docker·인프라. Production-quality 완성 후 ERP 연동.

## 3. 운영 모델 — C: 마스터 자산 + 사용처 매핑 (확정)

> **결정 (2026-06-01)**: 4개 후보(A 단일인덱스 / B 위치별독립 / **C 마스터+매핑** / D 원본/사본/변형) 중 **C 확정**.

### 3.1 구조

```
NAS (원본 보관소, 정보 축적 우선 — 건드리지 않음)
    │ 매주 정기 스캔 (G5)
    ▼
OT-Brain (가상 계층 — "정리"가 아니라 "구조화")
    ├─ 자산 라이브러리 (마스터) ── Platform 단독 보관 (G6)
    │   ├─ 선박 자산        (만성호·일진호·씨로드호 …)   ← PoC 1
    │   ├─ 행정 증명서      (무선국 검사 증명서 …)        ← PoC 3
    │   ├─ 계약서·보험 증권 (KHOA 사업 …)                ← PoC 2
    │   ├─ 부이 부품 사진   (대형부이-22/39/11 …)         (후순위)
    │   ├─ 장비 매뉴얼      (BNWAS, AADI TD268 …)         (후순위)
    │   └─ 장비 사양서·관측소 표지 …                       (후순위)
    │
    ├─ 사용처 매핑 (자산 ↔ 프로젝트·업무·부서, N:M)
    ├─ RAG 인덱스 (마스터 1회 임베딩 + 다중 라벨)
    └─ ERP 연동 API
        ├─ 프로젝트 시작 → 추천 첨부물 자동 제안
        ├─ 자산 갱신 → 사용처 알림
        └─ 새 자료 다맥락 등장 → 마스터화 후보 alert
```

### 3.2 핵심 원리

- 같은 hash → **마스터 자산 1개**로 수렴, **1회만 임베딩** (비용 절감)
- 자산은 **다중 태그**로 분류 (도메인·업무·유형·활용부서). 단일 분류 강제 금지 — 손실 큼.
  - 예: `선적증서(만성호).pdf` → [도메인: 선박자산][업무: 선박특별검사신청][유형: 증명서][부서: 13개 발주처]
- NAS 사본은 **건드리지 않고** "사본" 메타만 부여 (G2). 직원이 NAS 직접 사용해도 무방.

## 5. 거버넌스 정책 (G1~G6 확정)

> **결정 (2026-06-01)**: 6개 거버넌스 항목 모두 확정.

| # | 항목 | **확정 정책** | 비고 |
|---|------|--------------|------|
| **G1** | 마스터 식별 주체 | **혼합 — AI 후보 → 사람 승인** | AI가 다맥락(N회+) 자료를 마스터 후보로 자동 제안, 부서 책임자가 승인 |
| **G2** | NAS 사본 처리 | **사본 메타 부여** | NAS 원본 그대로 유지, Platform 가상 계층에서 "사본" 메타만 부여 (정보 축적 원칙) |
| **G3** | 갱신(증서 연차 등) | **혼합 — 자동 버전 + 책임자 검토** | 새 hash 감지 시 자동 버전 추가 → 책임자가 최신본 승인 |
| **G4** | 변형(이름 다른 유사) | **자동 클러스터링** | 이름+hash 근접도로 묶음 (예: "스피드호" vs "(스피드호)"), G1 혼합과 맞춰 사람 확인 |
| **G5** | 동기화 주기 | **매주 (PoC)** | PoC 단계 단순·안정. 검증 후 변경 감지(이벤트)로 전환 |
| **G6** | 자산 라이브러리 위치 | **Platform 단독** | OT-Brain 내 가상 계층, NAS 원본 불변. 별도 시스템 비전과 일치 |

### 5.1 마스터 식별 워크플로우 (G1+G3+G4 통합)

```
[매주 스캔 (G5)]
   ↓
[중복 hash 그룹화] ── G2: 사본 메타 부여
   ↓
[AI 마스터 후보 제안] ── 다맥락 N회+ / 이름·hash 클러스터링 (G4)
   ↓
[부서 책임자 승인] ── G1 혼합
   ↓
[마스터 자산 등록 + 다중 태그]
   ↓
[갱신 감지] ── 새 hash → 자동 버전 추가 → 책임자 검토 (G3)
```

## 6. 아키텍처 (초안 — Design 단계 확정)

### 6.1 시스템 경계

```
┌──────────────────────────────────────────────┐
│  OT-Brain (별도 repo / docker / 인프라)         │
│                                                │
│  ┌─ Ingestion ─────┐  ┌─ Knowledge Core ────┐ │
│  │ NAS 스캐너       │  │ 자산 라이브러리 DB    │ │
│  │ (현 nas-scan 이관)│→ │ 사용처 매핑 (N:M)    │ │
│  │ PDF/HWP 파서     │  │ RAG 인덱스 (pgvector)│ │
│  │ 해시·중복 탐지    │  │ 임베딩 (bge-m3)      │ │
│  └─────────────────┘  └──────────────────────┘ │
│           ↑                      ↓              │
│  ┌─ Search/LLM API ───────────────────────────┐│
│  │ 하이브리드 검색 (벡터+키워드) / LLM 답변      ││
│  └────────────────────────────────────────────┘│
└──────────────────┬─────────────────────────────┘
        source ↑   │   ↓ consumer (위젯·자동첨부)
┌──────────────────┴─────────────────────────────┐
│  erp-ot-platform (양방향 연동)                   │
└─────────────────────────────────────────────────┘
```

### 6.2 기술 스택 (ERP 자산 재사용 + 별도 인스턴스)

| 영역 | 선택 | 근거 |
|------|------|------|
| Vector DB | **pgvector 별도 인스턴스** | `project_nas_rag_vector_db` — Qdrant는 Phase 2 성능 평가 후 결정 |
| 임베딩 | **Ollama bge-m3** (로컬) | ERP 자연어검색에서 검증 (96.4%, HNSW) |
| OCR | **PaddleOCR + 텍스트 레이어** (비용 0) | `project_nas_rag_ocr_policy` — 유료 OCR은 태깅만, 추후 선별 |
| 검색 | 하이브리드 (벡터 + pg_trgm) | ERP 검색개선에서 검증 |
| 백엔드 | Node.js/TypeScript 또는 Python (Design에서 결정) | nas-scan은 현재 Python |
| 인프라 | Docker Compose (사내, GPU 불필요) | ERP와 동일 운영 방식 |

> **Design 단계 결정 필요**: 백엔드 언어(현 nas-scan Python vs ERP Node.js 통일), 인증 방식(자체 → ERP SSO), LLM(로컬 vs API).

## 7. 마일스톤 (6~9개월, 4-Phase)

| Phase | 기간 | 목표 | 산출물 |
|-------|------|------|--------|
| **Phase 0** | 1~2개월 | 비전·아키텍처 확정 + 인프라 + 별도 repo 구축 | 별도 repo, docker, pgvector 인스턴스, 데이터 모델 |
| **Phase 1** | 2~3개월 | NAS 흡수 + 단독 RAG 검색·답변 + **PoC 3종** | nas-scan 이관, 마스터 라이브러리, 검색 UI, 자동첨부 API |
| **Phase 2** | 2~3개월 | ERP 흡수 + 통합 검색 | ERP 도메인 흡수, 통합 검색 |
| **Phase 3** | 1~2개월 | ERP 연동 UX (위젯·자동첨부·SSO) | ERP 위젯, SSO |
| **Phase 4** | 지속 | 멀티모달·자동화·확장 (메일·Teams) | — |

### 7.1 Phase 1 PoC 우선 마일스톤 (이번 Plan 직후 진입)

1. M1: 데이터 모델 (자산 / 사용처 매핑 / 버전) + pgvector 스키마
2. M2: NAS 스캐너 이관 (현 `scripts/nas-scan/` → OT-Brain repo)
3. M3: 선박 자산 마스터 자동 도출 (파일명 패턴 `선적증서(*)` `어선검사증서(*)` 클러스터링 + 이름 정규화)
4. M4: 마스터 후보 → 사람 승인 UI (G1)
5. M5: ERP 자동 첨부 API + 위젯 (P1 선박 → P2 계약서 → P3 무선국)
