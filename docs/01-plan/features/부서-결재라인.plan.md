> **Feature**: 부서-결재라인
> **Phase**: Plan
> **Created**: 2026-03-21
> **Status**: Draft
> **Priority**: High
> **Estimated Scope**: Large (3~4주)

---

## 1. 배경 및 목적

### 현재 문제
- `UserProfile.departmentId` — 단순 문자열, 실체 없음
- MANAGER role이면 **전체 직원** 근태를 볼 수 있어 팀 경계 없음
- 휴가/OT 승인자가 role 기반으로만 결정 → 누가 승인할지 불명확
- 팀 관리 화면에 이름조차 표시 불가 (userId만 있음)

### 목표
1. 부서 계층 구조 (최대 3단계: 회사 → 본부 → 팀)를 관리하는 **Department 테이블** 신설
2. 사람별 1차·2차 결재자를 설정하는 **ApprovalLine 테이블** 신설
3. 기존 auth-service에 통합 (새 서비스 불필요)
4. attendance-service가 auth-service 내부 API로 결재자 조회
5. 팀 관리 화면에서 실제 이름·부서 표시

## 2. 핵심 요구사항

### FR-01: 부서 계층 관리 (Admin)
- 부서 생성/수정/삭제 (계층 구조, 최대 3단계)
- 부서장(headUserId) 지정
- 부서에 사용자 배정/이동
- 부서 비활성화 (소속 인원 이동 후)

### FR-02: 결재라인 설정 (Admin)
- 사용자별 1차 결재자 지정 (필수)
- 사용자별 2차 결재자 지정 (선택, 고액·장기 휴가 등에 사용)
- 결재라인 일괄 설정 (부서 단위)
- 결재자 부재 시 위임자(대리 결재자) 지정

### FR-03: 팀 경계 적용 (attendance-service)
- MANAGER는 자신이 결재자로 지정된 팀원 목록만 조회
- 팀원 근태 요약 = 자신의 결재 대상자만
- 승인 대기 목록 = 자신이 1차 결재자인 요청만

### FR-04: 사용자 이름 연동
- auth-service에서 내부 API로 사용자명 일괄 조회
- 팀 관리 화면, 근태 목록에 실제 이름 표시

### FR-05: 결재 흐름
- 휴가/OT 신청 → 1차 결재자에게 알림
- 1차 승인 → (2차 있으면) 2차 결재자에게 알림
- 모든 단계 완료 → 신청자에게 최종 승인 알림
- 반려 시 즉시 신청자에게 알림

### FR-06: 관리 UI
- 부서 트리 뷰 (Admin 메뉴)
- 결재라인 설정 화면 (사용자 목록 + 결재자 배정)
- 부서별 인원 현황

---

## 3. 비기능 요구사항

- 부서/결재라인 변경은 **auth-service 단일 진실 소스**
- attendance-service는 내부 API 호출만 (DB 직접 접근 금지)
- 결재자 정보는 Redis TTL 5분 캐시 (잦은 조회 최적화)
- 부서 삭제 전 소속 인원이 0이어야 함 (Soft delete 지원)

## 4. 도메인 모델

### 핵심 엔티티

```
Department (부서)
├── id, name, code (부서 코드, 예: "DEV-01")
├── parentId (상위 부서, null이면 최상위)
├── level (1=회사, 2=본부, 3=팀)
├── headUserId (부서장)
├── isActive
└── members: User[] (N:M 또는 User.departmentId FK)

ApprovalLine (결재라인)
├── id
├── userId (결재 대상자)
├── approverId (1차 결재자)
├── secondApproverId? (2차 결재자, optional)
├── delegateId? (위임자, 부재 시)
├── delegateUntil? (위임 만료일)
└── isActive
```

### 관계

```
User ──(N:1)──► Department  (소속 부서)
User ──(1:1)──► ApprovalLine (결재라인)
User(approver) ──(1:N)──► ApprovalLine (내가 결재자인 라인들)
Department ──(트리)──► Department (parent/child)
```

## 6. 구현 단계 (Phase)

### Phase 1 — 데이터 모델 + 기본 API (1주)
- [ ] auth-service: `Department`, `ApprovalLine` Prisma 모델
- [ ] DB 마이그레이션
- [ ] 부서 CRUD API (Admin)
- [ ] 결재라인 CRUD API (Admin)
- [ ] 내부 API: `GET /internal/users/:id/approver`
- [ ] 내부 API: `GET /internal/departments/:id/members`
- [ ] 내부 API: `GET /internal/users/bulk?ids=`

### Phase 2 — attendance-service 연동 (0.5주)
- [ ] 팀 근태 조회: 결재 대상자만 필터
- [ ] 휴가/OT 승인 대기: 내 결재 대상자만
- [ ] 결재자 알림 → auth-service 내부 API로 결재자 userId 조회

### Phase 3 — 관리 UI (1주)
- [ ] 부서 트리 뷰 + CRUD UI
- [ ] 결재라인 일괄 설정 UI
- [ ] 팀 관리 화면: 이름 표시 + 내 팀원만

### Phase 4 — 2차 결재 흐름 (0.5주)
- [ ] 2차 결재 상태 추가 (`PENDING_2ND`)
- [ ] 2차 결재자 알림 흐름
- [ ] 위임 기능 (대리 결재)

## 7. 주요 의사결정

| 결정 | 선택 | 이유 |
|------|------|------|
| Department 위치 | auth-service | 조직/인사 데이터, user와 동일 서비스 |
| ApprovalLine 위치 | auth-service | 결재자도 user 개념, 함께 관리 |
| attendance 연동 방식 | 내부 HTTP API | DB 직접 접근 금지 (서비스 경계) |
| 결재 계층 깊이 | 최대 3단계 | 회사→본부→팀, 그 이상은 YAGNI |
| 2차 결재 | Phase 4로 분리 | Phase 1~3으로 핵심 먼저 동작 |
| 부서별 vs 개인별 결재라인 | 개인별 (ApprovalLine) | 예외 케이스 처리 가능 |
