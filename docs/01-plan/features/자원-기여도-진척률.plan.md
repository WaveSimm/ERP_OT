## 1. Overview

### 1.1 배경

현재 자원 배정 모델(`SegmentAssignment`)은 **개인 점유율(`allocationPercent`)** 만 가진다.
이는 "이 사람이 자기 가용용량 중 몇 %를 이 구간에 쓰는가"이며, **"이 태스크(세그먼트) 안에서 이 자원의 몫이 얼마인가"(분담율)** 는 표현하지 못한다.

| 질문 | 현재 답할 수 있나 |
|------|------------------|
| "A는 자기 시간의 몇 %를 이 일에 쓰나?" | ✅ `allocationPercent` |
| "이 태스크에서 A의 몫은 전체의 몇 %인가?" | ❌ 불가능 |

또한 진척률은 `TaskSegment.progressPercent` 하나뿐이라 **누가 얼마나 했는지 자원별로 분해되지 않는다**. 태스크 진척률은 세그먼트 진척률의 일수 가중 평균(`TaskEntity.calculateAutoProgress`)으로만 산출된다.

### 1.2 목표

1. 배정(자원)마다 **분담율(contributionWeight)** 추가 — 세그먼트 단위
2. 배정(자원)마다 **본인 진척률(progressPercent)** 추가 — 각 자원이 자신의 진도율 바를 직접 관리
3. 세그먼트 진척률 = **Σ(분담율 × 자원 진척률) / Σ(분담율)** (정규화)
4. 태스크 진척률 = 기존 로직(세그먼트 일수 가중 평균) 유지

### 1.3 비목표 (Out of This PDCA)

- 공수(man-day)/비용 배분 산출 — 후속 PDCA로 분리 가능 (이번엔 진척률 가중만)
- 자원별 성과 리포트/대시보드 신규 화면 — 표시값은 derived 라 후속 확장
- `allocationPercent`(점유율) 기반 부하분석 로직 변경 — 그대로 유지

## 2. User Intent Discovery

### Q1. 핵심 문제
태스크에 여러 자원이 참여할 때 **각 자원이 그 태스크에 얼마나 기여(분담)하는지**를 배당·기록·집계할 수단이 없다. 점유율만으로는 "세 명 다 100% 바빴다"일 뿐 분담(합=100%)이 아니다.

### Q2. 주 사용자
내부 PM + 각 태스크 참여 자원(본인 진척 관리).

### Q3. 성공 기준
1. 배정마다 분담율·본인 진척률 저장
2. 세그먼트 진척률이 분담율 가중으로 자동 계산(수동 입력 폐기)
3. 각 자원이 본인 진도율 바를 직접 관리(본인/관리자 권한)
4. 기존 진척률 표시값 연속성 보장(마이그레이션 백필)

## 3. 확정 결정 (사용자 합의)

| # | 결정 | 내용 |
|---|------|------|
| D1 | 분담율 합 | 소프트 — 합≠100 허용, 계산 시 실제 합으로 정규화, UI 경고만 |
| D2 | 수동 진척 모드 | 폐기 (`isManualProgress` 경로 제거, 전부 자동 계산) |
| D3 | 100% 완료 시 작업일지 필수 | 유지 |
| D4 | 관리 단위 | 세그먼트마다 분담율·진척률 따로 |
| D5 | 본인 관리 권한 | `personUserId == 요청자` 또는 ADMIN/MANAGER. 외부인력/장비는 담당자/관리자 대리입력 |
| D6 | 자원 0명 세그먼트 | 진척 기록 주체 없음 → 세그먼트 진척 직접 입력 fallback. 자원 1명↑ 붙으면 derived 전환 |
