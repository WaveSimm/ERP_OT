## 1. 개요 (Overview)

전사원의 근태 현황을 한눈에 파악할 수 있는 **주간 캘린더 뷰** 페이지.
자원관리 메뉴에 "근태현황" 탭을 추가하여, 부서별로 그룹핑된 전 직원의 출근/외근/교육/출장/휴가 등 근태 상태를 시각적으로 표시한다.

### 배경 (Background)

- 현재 근태관리는 **개인별** 근태 신청 및 관리에 한정
- 관리자/팀장이 전사 근태를 한눈에 파악할 방법이 없음
- 각 직원이 개인 근태를 입력하면 **전체 공유**되어야 함
- 휴가/OT 등 기존 결재 시스템에서 **승인된 항목은 자동 반영**

### 핵심 목표

1. 전사원 근태 현황을 부서별 주간 캘린더로 한눈에 조회
2. 개인 근태 입력(출근 유형, 외근, 교육 등)이 모든 사용자에게 공유
3. 승인된 휴가/OT가 캘린더에 자동 반영
4. 모든 로그인 사용자가 조회 가능 (입력은 본인만)

## 2. 사용자 스토리 (User Stories)

### US-1: 전사 근태 캘린더 조회
- **As a** 로그인 사용자
- **I want to** 자원관리 > 근태현황 탭에서 전사원의 주간 근태를 볼 수 있다
- **So that** 팀원/타부서 일정을 파악하고 업무 조율이 가능하다
- **AC**: 부서별 섹션, 직원별 행, 요일별 컬럼으로 근태 블록 표시

### US-2: 주간 네비게이션
- **As a** 사용자
- **I want to** 이전 주 / 다음 주 / 오늘로 이동할 수 있다
- **So that** 과거 및 미래 일정도 확인할 수 있다
- **AC**: 좌우 화살표로 주 이동, "오늘" 버튼으로 현재 주 복귀

### US-3: 개인 근태 입력
- **As a** 로그인 사용자
- **I want to** 근태현황 캘린더에서 내 일정을 직접 추가할 수 있다
- **So that** 별도 메뉴 이동 없이 빠르게 근태를 등록한다
- **AC**: 본인 행에서만 추가 가능, 근무 유형(출근/외근/교육/출장) 선택

### US-4: 승인된 휴가/OT 자동 표시
- **As a** 사용자
- **I want to** 결재 승인된 휴가와 OT가 캘린더에 자동으로 나타난다
- **So that** 수동 입력 없이 결재 시스템과 연동된다
- **AC**: LeaveRequest(APPROVED/COMPLETED) → 해당 날짜에 휴가 블록 자동 표시
- **AC**: OvertimeRequest(APPROVED/COMPLETED) → 해당 날짜에 OT 블록 표시

### US-5: 부서별 접기/펼치기
- **As a** 사용자
- **I want to** 부서별 섹션을 접거나 펼칠 수 있다
- **So that** 관심 부서만 집중해서 볼 수 있다
- **AC**: 부서명 클릭 시 토글, 접힌 상태는 세션 유지

## 4. 데이터 모델 (Data Model)

### 4.1 신규 모델: WorkScheduleEntry (개인 근태 항목)

기존 AttendanceRecord는 출퇴근 기록(체크인/체크아웃)에 특화되어 있으므로, 
근태현황 캘린더용 **일정 항목**을 별도 모델로 관리한다.

```prisma
model WorkScheduleEntry {
  id          String   @id @default(cuid())
  userId      String
  date        DateTime @db.Date
  entryType   String   // WORK, FIELD, TRAINING, BUSINESS_TRIP, HALF_AM, HALF_PM, ANNUAL, SICK, SPECIAL, OT
  label       String?  // 사용자 메모 (예: "강남 현장", "안전교육")
  sourceType  String   @default("MANUAL")  // MANUAL | LEAVE_APPROVED | OT_APPROVED | ATTENDANCE
  sourceId    String?  // LeaveRequest.id or OvertimeRequest.id 등 원본 참조
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@unique([userId, date, entryType, sourceId])
  @@index([userId, date])
  @@index([date])
  @@map("work_schedule_entries")
  @@schema("attendance")
}
```

### 4.2 기존 모델 활용

| 모델 | 용도 |
|------|------|
| AttendanceRecord | 출퇴근 체크인/체크아웃 시 자동 WORK 엔트리 생성 |
| LeaveRequest | APPROVED 상태 변경 시 해당 날짜에 휴가 엔트리 자동 생성 |
| OvertimeRequest | APPROVED 상태 변경 시 해당 날짜에 OT 엔트리 자동 생성 |

### 4.3 entryType 코드

| Code | 표시명 | 색상 (Tailwind) |
|------|--------|-----------------|
| WORK | 출근 | bg-blue-100 text-blue-800 |
| FIELD | 외근 | bg-green-100 text-green-800 |
| TRAINING | 교육 | bg-purple-100 text-purple-800 |
| BUSINESS_TRIP | 출장 | bg-orange-100 text-orange-800 |
| HALF_AM | 오전반차 | bg-yellow-100 text-yellow-800 |
| HALF_PM | 오후반차 | bg-yellow-100 text-yellow-800 |
| ANNUAL | 연차 | bg-red-100 text-red-800 |
| SICK | 병가 | bg-pink-100 text-pink-800 |
| SPECIAL | 특별휴가 | bg-indigo-100 text-indigo-800 |
| OT | 야근/특근 | bg-gray-100 text-gray-800 |

## 5. API 설계 (API Design)

### 5.1 근태현황 조회 API

```
GET /api/v1/work-schedule?start=2026-04-07&end=2026-04-13
```

- 인증 필요 (JWT), 권한 제한 없음 (전체 공개)
- 응답: 부서별 그룹 + 직원별 엔트리 목록

```json
{
  "departments": [
    {
      "id": "dept-1",
      "name": "기술팀",
      "members": [
        {
          "userId": "user-1",
          "name": "홍길동",
          "entries": [
            { "id": "...", "date": "2026-04-07", "entryType": "WORK", "label": null },
            { "id": "...", "date": "2026-04-08", "entryType": "FIELD", "label": "강남 현장" }
          ]
        }
      ]
    }
  ]
}
```

### 5.2 개인 근태 추가 API

```
POST /api/v1/work-schedule
Body: { "date": "2026-04-07", "entryType": "FIELD", "label": "강남 현장" }
```

- 본인만 추가 가능 (req.userId 자동 적용)
- 같은 날 같은 유형 중복 방지

### 5.3 개인 근태 삭제 API

```
DELETE /api/v1/work-schedule/:id
```

- 본인이 수동 입력한 항목만 삭제 가능 (sourceType === "MANUAL")
- 자동 생성 항목(LEAVE_APPROVED, OT_APPROVED)은 삭제 불가

### 5.4 전사원 목록 API (기존)

```
GET /internal/users/all — auth-service 내부 API (이미 구현됨)
```

- 부서 정보 포함 필요 → 응답에 departmentId, departmentName 추가

## 7. 자동 반영 로직 (Auto-populate)

### 7.1 휴가 승인 시

LeaveRequest 상태가 APPROVED로 변경될 때:
1. startDate ~ endDate 범위의 각 날짜에 대해
2. WorkScheduleEntry 생성 (entryType = LeaveType 매핑)
3. sourceType = "LEAVE_APPROVED", sourceId = LeaveRequest.id

| LeaveType | entryType |
|-----------|-----------|
| ANNUAL | ANNUAL |
| HALF_AM | HALF_AM |
| HALF_PM | HALF_PM |
| SICK | SICK |
| SPECIAL | SPECIAL |

### 7.2 OT 승인 시

OvertimeRequest 상태가 APPROVED로 변경될 때:
1. 해당 date에 WorkScheduleEntry 생성
2. entryType = "OT", sourceType = "OT_APPROVED"
3. sourceId = OvertimeRequest.id

### 7.3 취소 시

LeaveRequest/OvertimeRequest가 CANCELLED로 변경될 때:
- 해당 sourceId의 WorkScheduleEntry를 삭제

## 9. 기술적 결정 (Technical Decisions)

| 항목 | 결정 | 이유 |
|------|------|------|
| 데이터 저장 | 별도 WorkScheduleEntry 테이블 | AttendanceRecord는 출퇴근 FSM 전용, 캘린더 일정은 다른 성격 |
| 조회 권한 | 전체 공개 (로그인만) | 사용자 요구: "각자 자기의 근태를 입력하면 이를 모든 이들이 공유" |
| 입력 권한 | 본인만 | 타인 근태 대리 입력 불가 |
| 부서 정보 | auth-service에서 조회 | 부서/사용자는 auth DB 소유 |
| 자동 반영 | 승인 시점에 동기적 생성 | 이벤트 기반(RabbitMQ)보다 단순, MVP 적합 |
| 뷰 단위 | 주간 (월~일) | 레퍼런스 스크린샷 기준, 가장 실용적 단위 |
