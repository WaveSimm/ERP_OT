## 2. 왜 이 시스템이 필요한가

### 2-1. 현재 우리 회사의 디지털 자산 현황

- **NAS**: 약 **45.4 TB** 사용 중 (총 78.5TB 용량)
- **파일 수**: 약 **790만 개**
- **주요 분포**:
  - 10.Project: **25.3 TB** (KHOA·KIOST·KMA·KWATER·KIGAM·KHNP·연구과제·OLD_PROJECTS 등)
  - 30.팀_개인: **4.9 TB** (사업팀·기술팀·연구팀·영업팀·관리팀·무인체계사업팀·임원진·공구관리)
  - 영상·사진(CCTV·교육 영상·실해역 자료): **5.4 TB+**
  - 그 외 자료: ~10TB
- **ERP-OT**: 결재·작업비고·게시판·고객사·거래·자산·견적·AS·경비정산 등 운영 데이터 누적

### 2-2. 풀어야 할 문제들

| 문제 | 현재 상황 | 영향 |
|---|---|---|
| **자료를 못 찾음** | 폴더 구조 알아야 도달 가능, 직원별 다른 분류 | 신규 직원 적응 어려움, 시간 낭비 |
| **누가 어떤 자료를 가졌는지 모름** | 개인 폴더에만 있는 자료, 부서 이동·퇴사 시 손실 | 회사 지식 자산 유실 |
| **같은 질문 반복** | "○○고객사 작년 견적이 얼마였지?" 매번 사람에게 물어봐야 함 | 업무 효율 저하 |
| **신입 교육 부담** | 회사 노하우가 사람·자료에 흩어져 있음 | 교육 기간 길어짐 |
| **자료 활용 못함** | 자료가 있어도 누가 어디에 보관했는지 모름 | 자료가 "묻혀" 있음 |

### 2-3. 시스템이 가져올 변화

## 3. 시스템 비전·방향

### 3-1. 핵심 정체성

**"오션테크 통합 지식 플랫폼"** (Enterprise Knowledge Platform)

- **별도 독립 시스템**: ERP-OT와 분리된 자체 시스템
- **양방향 ERP 연동**: ERP는 자료 제공 + 자료 활용 양쪽 모두
- **회사 전체 디지털 자산 흡수**: NAS + ERP + (장기) 메일·Teams·외부 자료
- **AI 기반 검색·답변**: 단순 키워드 검색을 넘어 의미 기반 검색 + LLM 답변 생성

### 3-2. 핵심 원칙

| 원칙 | 의미 |
|---|---|
| **정보 축적 우선** | 자료 삭제·정리는 부수적, 자료 발견·활용이 메인 |
| **중복은 분류의 표현** | 같은 자료가 여러 폴더에 있는 것 = 의도된 다중 분류. 보존 |
| **사용자 작업 흐름 보존** | 직원이 만들어둔 분류 체계 데이터화·학습 |
| **양방향 ERP 연동** | ERP가 source + consumer. ERP-OT와 독립적이지만 긴밀히 연동 |
| **단계적 확장** | NAS → ERP → 메일·Teams → 외부. 검증 후 확장 |
| **비용 최소화** | 무료 오픈소스 우선. 유료 API는 효과 확인 후 |

### 3-3. 사용 시나리오 (사용자별)

## 4-Z. 개발 방식 — Claude Code + CTO Lead Mode (AI Native)

본 시스템은 **AI Native 개발 방식**으로 진행됩니다. 사람이 모든 코드를 직접 작성하는 대신, AI specialist 에이전트들이 병렬로 작업합니다.

### 4-Z-1. Claude Code

- 자연어로 요청 → AI가 코드·설계·테스트 동시 작성
- 반복 작업 (12 페이지 동일 적용 등) 자동 일괄 처리
- 분석·디버깅·리팩토링 가속 (5~30배)
- 문서·주석·테스트 자동 생성

### 4-Z-2. CTO Lead Mode

CTO Lead 에이전트가 다음 전문 에이전트들을 **동시 병렬**로 가동:

| 에이전트 | 역할 |
|---|---|
| **product-manager** | 요구 분석, Plan 문서 작성 |
| **enterprise-expert** | 전략·기술 결정 (CTO급 의사결정) |
| **infra-architect** | 인프라·k8s·Docker·CI/CD 설계 |
| **bkend-expert** | Backend·DB·BaaS 설계·구현 |
| **frontend-architect** | UI·Component·Design System |
| **security-architect** | OWASP·인증·권한·감사 |
| **qa-strategist** | 테스트 전략·품질 지표 |
| **gap-detector** | 설계 ↔ 코드 일치도 분석 |
| **pdca-iterator** | 자동 개선 사이클 (90% Match Rate까지) |
| **report-generator** | 완료 보고서 자동 작성 |

→ 단일 트랙 대비 **2~3배 추가 가속**

### 4-Z-3. 가속 메커니즘 (오늘 실증)

| 작업 | Claude Code 시간 | 전통 1인 개발자 추정 | 가속 |
|---|:---:|:---:|:---:|
| 한글 collation 60컬럼 적용 | 1시간 | 1~2일 | 10~20배 |
| 정렬 기억 hook + 12페이지 | 2시간 | 1주 | 20배 |
| NAS 스캔 인프라 (24파일·KHOA 분할) | 반나절 | 2~3주 | 30~50배 |
| 카탈로그 리포트 재설계 | 30분 | 2~3일 | 20~40배 |
| 4 commit 분할·메시지·push | 20분 | 1시간 | 3배 |

### 4-Z-4. AI Native 개발의 의미 (직원 관점)

- **빠른 출시**: 6~9개월 → 약 1개월
- **잦은 갱신**: 의견 반영·기능 추가가 빠름 (며칠 단위)
- **품질 일관**: 자동 분석·검증으로 일관된 품질
- **문서 정확**: 코드와 문서 동기화 자동
- **사람 시간 절약**: 직원은 의사결정·검토에 집중

→ 우리 회사는 **AI Native 개발 회사**로 전환 가능. 이 프로젝트가 그 대표 사례.

### 5-3. 핵심 알고리즘·패턴

| 영역 | 기법 |
|---|---|
| **검색** | Hybrid Search (벡터 유사도 + 키워드 + 메타필터) |
| **임베딩** | bge-m3 (한국어·다국어 동시 우수, 1024 dim) |
| **분류 맥락** | Tier 3 hash 기반 다중 위치 그래프 |
| **OCR 정책** | 4-Tier 자동 분류 (텍스트레이어 우선 → PaddleOCR → CLOVA → Claude Vision) |
| **인덱싱** | Fire-and-forget 비동기 (현 ERP 패턴 재사용) |
| **권한** | RLS (Row Level Security) 기반 + 부서·역할 필터 |
| **변경 감지** | RabbitMQ subscribe + 일간 incremental scan |

## 12. 자주 묻는 질문 (FAQ)

**Q1. ERP-OT는 어떻게 되나요?**
A. 그대로 사용합니다. 지식 플랫폼은 ERP와 별도로 동작하며, ERP가 종료되더라도 영향 없습니다. 다만 ERP 안에 자료 검색·답변 위젯이 추가되어 더 편리해집니다.

**Q2. 내 개인 폴더 자료도 다 검색 가능한가요?**
A. 권한 정책에 따라 결정됩니다. 본인만 접근 가능한 폴더는 본인 검색 결과에만 나옵니다. 부서·역할 권한이 자동 적용됩니다.

**Q3. 자료가 삭제되나요?**
A. 아닙니다. 시스템은 NAS·ERP에서 자료를 **읽기만** 합니다. 삭제·변경 절대 안 함. 직원의 분류 체계도 그대로 보존됩니다.

**Q4. 영상 자료도 검색되나요?**
A. 영상 파일명·위치·시기는 검색 가능합니다. 영상 내용은 향후 (Phase 4) 자동 transcript로 검색 가능 예정.

**Q5. 영어·외국어 자료도 처리되나요?**
A. 가능합니다. bge-m3 임베딩 모델이 한국어·영어·중국어 등 다국어를 동시 처리합니다.

**Q6. 사용에 비용이 드나요?**
A. 직원은 무료. 회사 차원의 인프라 비용 (서버·LLM API)이 발생하나 자료 발견 시간 절감으로 ROI 충분.

**Q7. 잘못된 답변을 하면 어떻게 하나요?**
A. 시스템이 출처를 명시하므로 원본 확인 가능. 잘못된 답변 피드백 기능 제공. 점진적으로 정확도 개선.

**Q8. 언제 사용할 수 있나요?**
A. Claude Code + CTO Lead 모드 개발로 단계별 빠른 출시 예정.
- Phase 2 끝 (~2~3주 후, **2026년 6월 초·중**): NAS 자료 검색 시범 사용 가능
- Phase 3 끝 (~3~4주 후, **2026년 6월 중·하순**): NAS + ERP 통합 검색 정식 운영
- Phase 4 (지속): 메일·Teams·외부 자료 점진 확장

> **전통적 개발이라면 6~9개월** 걸릴 작업이
> **Claude Code 단독으로 1~2개월**,
> **Claude Code + CTO Lead (병렬 specialist)로 3~4주**에 가능합니다.
>
> 오늘 작업으로 실증됨: 한글 정렬 60컬럼 1시간·정렬 hook 12페이지 2시간·NAS 스캔 인프라 반나절

**Q9. 외부 회사도 사용 가능한가요?**
A. Phase 1~3은 사내 전용. Phase 4 이후 고객사·협력사용 제한 인터페이스 검토 가능.

**Q10. 보안 걱정은 없나요?**
A. 사내 자료가 외부 LLM(Claude API)으로 보내질지 결정이 보류되어 있습니다. 로컬 LLM(보안 최강) vs API(성능 우수) 사이에서 회사 정책에 따라 결정 예정.

## 부록 A. 기술 용어 사전 (Glossary)

> 본 문서에 등장하는 주요 기술 약어·기술명을 카테고리별로 정리합니다.
> 신규 직원이나 비개발 부서가 기술 영역을 이해할 때 참고용으로 활용하세요.

### A-1. 저장소 / 데이터 인프라

| 약어 | 풀이 | 의미 |
|---|---|---|
| **NAS** | Network Attached Storage | 사내 네트워크에 연결된 공용 파일 서버. 본 시스템에서 45TB 자료 보관 위치 |
| **SMB** | Server Message Block | 윈도우 기반 파일 공유 프로토콜. NAS를 드라이브로 마운트하는 방식 |
| **PST** | Personal Storage Table | Outlook의 메일 저장 파일 형식 (.pst). Phase 4 메일 흡수 시 처리 |
| **MinIO/S3** | — / Simple Storage Service | AWS S3 호환 객체 스토리지. 대용량 파일을 URL로 접근 |
| **pgvector** | PostgreSQL Vector Extension | PostgreSQL에서 벡터 검색을 가능하게 하는 확장 |
| **pg_trgm** | PostgreSQL Trigram Extension | 문자열을 3글자 단위로 쪼개 유사도 검색. 오타·부분 일치에 강함 |

### A-2. AI / 검색 핵심

| 약어 | 풀이 | 의미 |
|---|---|---|
| **RAG** | Retrieval-Augmented Generation | 검색 + LLM 답변 생성. 질문 → 관련 자료 검색 → LLM이 자료 기반 답변 |
| **LLM** | Large Language Model | 대규모 언어 모델 (Claude, GPT, Qwen 등) |
| **OCR** | Optical Character Recognition | 이미지·스캔본에서 글자 추출 |
| **BM25** | Best Matching 25 | 키워드 검색의 표준 알고리즘. 단어 빈도·문서 길이를 고려한 점수 산출 |
| **Embedding** | 임베딩 | 텍스트를 의미 기반 숫자 벡터로 변환. 유사도 계산의 기초 |
| **bge-m3** | BAAI General Embedding M3 | BAAI 연구소의 다국어 임베딩 모델. 한국어 우수, 1024차원 |
| **Chunk** | 청크 | 큰 문서를 검색·임베딩하기 좋은 크기로 자른 조각 |
| **Token** | 토큰 | LLM·임베딩 모델이 텍스트를 다루는 최소 단위. 한국어는 영어보다 1.5~2배 소비 |
| **dim** | dimension | 벡터의 차원 수. "1024 dim" = 1024개 숫자로 구성된 벡터 |
| **Hybrid Search** | 하이브리드 검색 | 벡터(의미) + 키워드(BM25/pg_trgm) + 메타필터 결합 검색 |
| **Top-K** | — | 검색 결과 상위 K개. 보통 5~20개 |
| **RLS** | Row Level Security | PostgreSQL의 행 단위 권한 제어. DB 레벨에서 권한 강제 |

### A-3. 인증 / 보안

| 약어 | 풀이 | 의미 |
|---|---|---|
| **SSO** | Single Sign-On | 한 번 로그인 → 여러 시스템 자동 인증. ERP 로그인 시 지식 플랫폼도 자동 접속 |
| **VPN** | Virtual Private Network | 외부에서 사내망 접속용 암호화 터널 |
| **OWASP** | Open Web Application Security Project | 웹 보안 표준 단체. "OWASP Top 10" 취약점 가이드 유명 |
| **Auth** | Authentication / Authorization | 인증(누구) + 인가(권한) |

### A-4. 개발 / 아키텍처

| 약어 | 풀이 | 의미 |
|---|---|---|
| **REST API** | Representational State Transfer | HTTP 기반 표준 API 설계 방식 (GET/POST/PUT/DELETE) |
| **CI/CD** | Continuous Integration / Continuous Deployment | 코드 변경 시 자동 빌드·테스트·배포 파이프라인 |
| **k8s** | Kubernetes | 컨테이너 오케스트레이션 플랫폼. Docker 컨테이너 대량 관리 |
| **BaaS** | Backend as a Service | 백엔드를 서비스로 사용 (Supabase, Firebase 등) |
| **LTS** | Long-Term Support | 장기 지원 버전. 안정성·보안 패치 보장 |
| **ORM** | Object-Relational Mapping | DB 테이블을 코드 객체로 다루는 도구 (Prisma) |
| **cron** | (Chronos) | 유닉스 스케줄러. "야간 cron" = 매일 정해진 시간 자동 실행 |
| **I/O** | Input/Output | 디스크·네트워크 데이터 흐름 |
| **PoC** | Proof of Concept | 개념 증명. 본격 개발 전 실현 가능성 검증용 시제품 |

### A-6. 메시징 / 캐시

### A-10. 한 문장으로 다시 읽는 시스템 아키텍처

> **NAS · ERP**의 자료를 **OCR**로 텍스트화하고, **bge-m3** 임베딩으로 **pgvector**에 저장.
> 사용자가 질문하면 **Hybrid Search**(벡터 + **BM25** / **pg_trgm**)로 **Top-K** 자료를 찾고,
> **RLS**로 권한 필터링한 뒤 **LLM**에게 컨텍스트와 함께 전달해 답변 생성.
> 변경 감지는 **RabbitMQ**, 캐시는 **Redis**, 인프라는 **Docker**(향후 **k8s**),
> 인증은 ERP **SSO** 연동.
