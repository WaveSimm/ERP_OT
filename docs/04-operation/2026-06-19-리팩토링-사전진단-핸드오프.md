## 0. 핵심 결론

두 프로젝트 모두 **핵심 엔진은 견고하나 "경계·정리·버전관리"가 무너진 상태**.
코드 품질보다 **리팩토링 착수 전에 차단해야 할 안전 문제(미커밋·평문 비밀번호·무인증 엔드포인트)가 먼저**다.
→ **"리팩토링보다 안전망(Phase 0) 선행"이 명확.**

### 작업 환경 결정: **서버 주(主) + dev 보조**
- **이번 리팩토링 스프린트는 서버에서 수행**(주말 = 서비스 중지 가능, 블로커 해제).
- 이유:
  1. **검증 가능성** — 전체 스택을 실데이터·실연동(사진첩·검색·청킹)과 함께 돌릴 수 있는 곳은 서버뿐. dev는 Docker 중지 + stale 가능 → "진짜 안 깨졌는지" 확인 불가.
  2. **드리프트 근원이 서버에 있음** — 실제 돌고 있는 코드는 서버 워킹트리. 서버에서 git 커밋·push 해야 dev/서버/master 3중 분기 치료.
  3. **ot-brain 운영 코드는 Linux 네이티브**(`_chunk_*.sh`·tunnel·청킹 워커·docker compose build) — Windows dev에서 검증 불가.
- **dev PC 용도**: ERP_OT 순수 TS 앱·서비스 로직 작성/리뷰만. 단 **반드시 git 경유**(직접 파일 복사 금지 — 드리프트의 근원).

## 2. ERP_OT — 진단 요약

### 구조·유지보수
- **Clean Arch는 "폴더 이름만"**: `domain/repositories/*` 인터페이스는 **죽은 추상화**(auth만 실제 구현). 9개 서비스 중 8개가 Prisma 직접 주입, **17개 라우트가 `prisma.*` 직접 호출**(API→Infra 누수). 예: `services/project/src/api/routes/task.routes.ts:59`, `me.routes.ts`(7×), `internal.routes.ts` auth(9×).
- **프론트 god-file**: `apps/web/src/lib/api.ts` **2191줄·`any` 97개**, 자체 규칙("api-client 경유") 위반. 1000줄 넘는 페이지 10개(`projects/[id]/page.tsx` 2257, `repair/[id]/page.tsx` 1897, `TaskDrawer.tsx` 1538, `resources/page.tsx` 1519, `me/dashboard/page.tsx` 1504).
- **equipment = god-service**: 36개 application 서비스가 6개 바운디드 컨텍스트(장비·수리·재고·발주·계약·통관) 혼재. ⚠️ **AS관리는 production** → 분리 리스크 높음(drop&rebuild 자유 적용 안 됨).
- **중복**: `approval/new/page.tsx`(923) ↔ `approval/[id]/edit/page.tsx`(722) 사실상 쌍둥이(공유 상태·드롭다운 로직 5–49행 거의 동일). `errorResponse` 헬퍼 2종이 **응답 형태 상이**: `services/shared/src/utils/response.ts:15`(`{success,error}`) vs `services/shared/src/errors/error-format.ts:14`(`{error}`).
- **고아 패키지**: `@erp-ot/ui`(빈 스텁, `packages/ui/src/index.ts` 주석만), `@erp-ot/api-client`(import 0건).
- **내부인증 불일치**: equipment는 인라인 `x-internal-token` 체크(`procurement.routes.ts:9-12`), project는 글로벌 `requireInternal` 의존.

### 운영·보안·성능·빌드
- **미커밋 드리프트(확정)**: `git diff HEAD` 기준 소스 변경은 **딱 2개** — `services/project/src/api/routes/internal.routes.ts`(+16, `GET /internal/projects`), `services/equipment/src/api/routes/procurement.routes.ts`(+21, `GET /internal/contracts`). 나머지는 bkit/PDCA 기록 + lockfile. 추가로 `apps/field/`(~90 미추적), `docker-compose.dev-lan.yml`(미추적).
- **C2 무인증**: equipment `buildApp()`에 `requireInternal` 미등록 → `GET /internal/contracts` 토큰 없이 도달 가능 추정. → equipment에 `requireInternal` 등록 또는 인라인 토큰 체크 추가.
- **내부인증 약점**: approval(`src/index.ts:105-112`)·equipment `internalOrderRoutes`는 인라인 `token !== INTERNAL_API_TOKEN`(403, 길이검증 없음, 비상수시간 비교) / project·attendance는 shared `requireInternal`(401 + 길이≥16). → shared로 일원화.
- **빌드**: compose 5종(`yml`/`.local`/`.dev-lan`/`.prod-test`/`.gpu`) + `--profile build` 베이스이미지(`erp-ot-build-base/runtime-base`) **먼저 빌드 안 하면 전 서비스 빌드 실패** → 신규 서버에서 잊기 쉬움. web은 `node:20-alpine`에서 `npm install && npm run dev`(dev 모드 의도적), expense/approval은 dist 빌드(코드 변경 시 `docker compose build` 필수).
- **양호**: 9개 서비스 `GET /health`, 비root `USER app` + HEALTHCHECK, 시크릿 외부화(`${VAR:?}`), `.env.example`만 추적, 127.0.0.1 바인딩, helmet, Zod env 검증, raw SQL 바인드 파라미터. `apps/field/`는 cookie-jar BFF 인증 + CSRF로 잘 설계됨.

## 3. ot-brain — 진단 요약

### 판정: **견고한 코어 + 실험적 스프롤**
데이터 흐름(Python 수집→pgvector→TS knowledge-api→nas-file)은 단일 소스로 깔끔, 9-모델 Prisma 스키마 일관, loopback-only 바인딩 양호. 단 **루트가 작업 쓰레기장**.

### 구조·유지보수
- **루트 `_*` 약 30개**(로그·런처·진단·csv·txt). `.gitignore`는 `*.log`만 커버 → `.py/.sh/.csv/.txt`가 다 새어나감.
- **중복 스크립트 쌍**: 루트 `_chunk_worker.sh`·`_tunnel_keeper.sh`(=실제 live)가 `scripts/chunking/`의 정돈된 사본과 분기. `_night_chunk.cmd`→`_chunk_run8.sh`가 루트 사본을 참조.
- **server.ts 940줄**: 문서검색 핸들러 단일 ~400줄(`services/knowledge-api/src/server.ts:424-830`), `/ask`(:830-910). 임베딩·LLM 클라이언트가 server.ts에 인라인(`services/embedding/`는 README 스텁뿐).
- **언어/런타임 혼재**: TS 서비스 + 루트 Python 진단 + shell + cmd.

### 운영·보안·성능·빌드
- **기동 footgun**: `npm run serve`(`services/knowledge-api/package.json:15`)가 `--env-file` 없음 → `.env` 미로드 → **전부 401**.
  - **정상 기동**: `cd services/knowledge-api && node --env-file=../../.env --import tsx src/server.ts`
  - 운영(docker compose)은 `environment:`로 주입되어 401 footgun은 **dev 전용**. API/PG 모두 127.0.0.1 바인딩.
- **성능**: 임베딩이 **청크당 Ollama 1콜**(배치 아님, 각 180s 타임아웃) → 배치화가 최대 처리량 개선. DB 커넥션 풀 없음(워커당 단일 `psycopg2`, 재연결 없음) → 14h 장시간 런 취약.
- **양호(보존)**: 청킹 idempotent·restartable(`FOR UPDATE SKIP LOCKED` 클레임, 기동 시 고아 `PROCESSING→PENDING` 리셋, per-doc commit, EmbedError→PENDING 재시도 / Exception→FAILED). nas-file 경로 confinement(`..` 차단·`ROOT_KEY` prefix·file-only·RFC5987·SVG/HTML inline 제외). 임베딩 재시도 지수백오프(4xx no-retry / 429+5xx retry).
- **잔여**: `_chunk_status.txt` 기준 196 PENDING(pdf 25/docx 162/pptx 9) — 파이프라인 미완 배수. CORS `*`(loopback 한정이라 위험 낮음).
