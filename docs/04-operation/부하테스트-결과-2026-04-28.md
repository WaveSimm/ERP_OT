**핵심 결론**:
- **출근 burst (시나리오 A)**: prod 모드 30~34% 개선, **90 VU에서 timeout 해소** ⭐
- **평상 mix (시나리오 B)**: prod p99가 562ms → 173ms (**69% 개선**) — Next.js dev 모드 컴파일 부담이 tail latency 주범이었음
- **검색 burst (시나리오 C)**: 거의 차이 없음 (Ollama가 병목)

\* 32~34% 실패율은 worklog 작성 endpoint 데이터 부재 노이즈로 실 사용 무관 (v1.0 §1 주석 참조)

\* 실패율 32~34%는 worklog 작성 endpoint 데이터 부재(`/me/tasks` 반환 비어있음)·일부 미존재 endpoint 호출에 의한 것으로, 실 사용 시 일반 사용자가 만나는 상황 아님.

\*\* Ollama CPU > 100%는 멀티 코어 활용 (NVIDIA GPU 가속이 추가로 동작).

## 2. 시나리오별 상세 결과

### Round 1: A 출근 burst (dev 60 VU)
- 로그인 + 홈 5 API 동시 호출 시뮬레이션
- **응답 시간 한계**: login p95 1.53s, home p95 974ms — 목표(800ms/500ms) 초과
- 4832 reqs, 58 fail (1.20%)
- 결론: **dev 모드 60 VU burst는 임계점**

### Round 2: B 평상 mix (dev 60 VU 10분)
- 평상시 ERP 사용 패턴 (게시판/검색/비고/결재/프로젝트/대시보드)
- **응답 시간 매우 양호**: p95 116ms, p99 491ms ✅
- 10361 reqs, 3407 fail (32.88%) — fail의 대부분은 부하 사용자 본인 task 없음으로 인한 worklog 작성 404
- 결론: **평상 부하는 dev 모드도 충분**

### Round 3: C 검색 burst (dev 30 VU 5분)
- 자연어 검색 60% + posts 20% + worklogs 20%
- **응답 시간 양호**: search_all p95 230ms, search_posts 203ms, search_worklogs 211ms ✅
- 3849 reqs, 153 fail (3.97%)
- Ollama CPU 86.6% — 거의 단일 코어 한계 도달
- 결론: **30 VU 검색은 Ollama가 안정 처리**

### Round 4: A 출근 burst STRESS (dev 90 VU)
- 90 VU 동시 burst → **i/o timeout 다발 발생**
- login p95 2.23s, home p95 1.35s
- 5485 reqs, 102 fail (1.85%) — Next.js dev 컴파일 부하 + 동시 connection 한계
- 결론: **dev 모드 90 VU burst는 처리 불가** (prod 빌드 시 개선 예상)

### Round 5: B 평상 mix STRESS (dev 90 VU 10분)
- **응답 시간 stable**: p95 123ms, p99 562ms ✅
- 13115 reqs, 4499 fail (34.30%) — Round 2와 동일 원인
- 결론: **평상 부하는 90 VU도 dev 모드에서 처리 가능**

### Round 6: C 검색 burst STRESS (dev 60 VU 5분)
- **응답 시간 양호**: search p95 469~493ms (목표 800ms 통과) ✅
- 7345 reqs, 230 fail (3.13%)
- **Ollama CPU 183.4%** — 멀티 코어 활용, 임계점 근접
- 결론: **60 VU 검색은 Ollama 임계점에 근접**, 80 VU 이상은 별도 검증 필요

## 3. 임계점 분석 (가장 중요)

### dev 모드의 한계

1. **출근 burst (시나리오 A)** — Web (Next.js dev) 컴파일 부담
   - 60 VU: 응답 시간 임계값 초과 (login 1.5s, home 1s)
   - 90 VU: i/o timeout 발생 → **처리 불가**
   - **prod 빌드 전환이 가장 시급**

2. **검색 burst (시나리오 C)** — Ollama 임베딩
   - 30 VU: CPU 87% (안정)
   - 60 VU: CPU 183% (멀티 코어 활용 중) — 단일 노드 한계 근접
   - **80 VU 이상부터 임베딩 throughput 병목 예상**

### 안정 동작 영역

- **평상 mix**: 60~90 VU 모두 p95 < 130ms로 매우 양호 (dev 모드에서도)
- **검색**: 60 VU까지 p95 < 500ms (Ollama 임계점 근접하나 아직 통과)

### 시스템 리소스 사용

| 컨테이너 | 평균 CPU | 비고 |
|---------|:---:|------|
| auth-service | 2~42% | 출근 burst에서 JWT verify 부담 |
| project-service | 1~6% | 매우 가벼움 |
| web (Next.js dev) | 12~25% | dev 컴파일 부담 (prod에서 개선 예상) |
| postgres | 1~3% | 충분 |
| ollama | 0~183% | **검색 시 핵심 부하** |

호스트 메모리 사용은 모든 라운드에서 < 4GB (15.2GB의 26%)로 여유.

## 9. dev vs prod 결론 (v1.1)

### 결정적 발견

1. **Next.js dev 모드의 컴파일 부담**이 burst 시나리오의 응답 시간 주범
   - dev p99 562ms → prod p99 173ms (mix scenario)
   - 출근 burst 90 VU에서 dev는 timeout 다발, prod는 timeout 0
2. **검색은 Ollama가 100% 병목** — web 모드 영향 없음
3. **이관 시점 권장 운영 모드: prod (Next.js standalone)**
   - 메모리 사용량도 dev 대비 작음
   - 첫 요청 응답 시간 즉시 (dev는 컴파일 1~3초)

### 회사 서버 이관 GO/NO-GO 기준

| 시나리오 | 60명 prod | 90명 prod | GO/NO-GO |
|---------|:---:|:---:|:---:|
| 평상시 사용 (B) | p95 84ms | p95 84ms | **GO** ✅ |
| 출근 burst (A) | login 1.02s | login 1.55s, timeout 0 | **GO** ⚠️ (목표 800ms 초과지만 처리 가능) |
| 동시 검색 (C) | p95 ~500ms | (90 VU 미측정) | **GO** ✅ (60 VU 통과, 80+는 별도 측정) |

**종합**: 회사 서버 이관 후 동일 시나리오 재실행 추천 (호스트 H/W 향상 + 네트워크 환경 차이 검증).
