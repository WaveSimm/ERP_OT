# 운영서버 교체(R740) 이관 계획 — 2026-07-22

## 1. 배경

- 금주 발주 완료, **다음주 입고 예정**. 구 운영서버(.41)는 보상판매로 이관 완료 후 반납.
- 당초 계획(GPU만 추가, T4→ 문서상 근거: `2026-06-18-NAS검색-품질속도개선.md`)에서 대리점 제안으로 서버 전체 교체로 변경.
- 백업/DR 서버(.42)는 그대로 유지.

### 신규 서버 사양 (Dell R740, 보상판매)

| 항목 | 사양 |
|---|---|
| CPU | Xeon Gold 6138 2.0GHz 20코어(40스레드) |
| RAM | 128GB |
| 디스크 | 2.4TB 10k SAS ×2, RAID1 (실용량 2.4TB) |
| GPU | RTX Pro 5000 Blackwell 48GB |
| 전원 | 1500W 듀얼 |

### 현 서버 실태 (2026-07-22 실측)

- 디스크: PERC H730 뒤 HDD 2장 — sda2 274G(`/`, 40G 사용) + sdb2 3.3T(`/data`, 162G 사용). **총 사용량 ≈ 200GB** → 2.4TB 충분.
- 신구 모두 HDD(PERC)라 10k SAS 전환에 따른 성능 하락 없음.
- RAM 30G+스왑 7G 상시 사용 → 128GB로 해소.

## 2. 확정 결정사항 (2026-07-22)

1. **컷오버: 평일 야간** (퇴근 후 19시~, 다운타임 30분~1시간)
2. **OS: Ubuntu 26.04 LTS** — ~~당초 24.04(Blackwell 드라이버 570+/CUDA 12.8+ 요구 사유)~~ **7/23 재검토로 변경**. 근거: ① 26.04는 CUDA·드라이버가 Ubuntu 공식 아카이브에 네이티브 탑재(드라이버 595 계열, 570+ 요건 충족), ② nvidia-container-toolkit 공식 지원 플랫폼에 26.04 등재, ③ 현 운영서버(.41)가 26.04라 호스트 cron·스크립트 이관 시 버전 차이 변수 제거, ④ LTS 지원 2년 연장. 유의: NVIDIA 자체 apt repo는 아직 ubuntu2604 미제공(호스트에 NVIDIA 배포판 CUDA 툴킷 직접 설치 불가 — 전 컨테이너 구조라 불필요), Blackwell은 **open 커널 모듈 필수**. **폴백**: 입고+1~2일 GPU 검증 실패 시 24.04로 재설치(무중단 구간, 반나절)
3. **파티션: LVM** — `/` 300G, `/data` 1.5T, **미할당 예비 600G**(폭주 격리 + lvextend 온라인 확장)
4. **구 서버 대기: 컷오버 후 1주**(롤백 보험) → 소거 → 반납. ※ 대리점 반납 기한 확인 필요
5. **GPU: 검색 스택만 먼저** — 리랭커+RAG qwen+임베딩을 서버 GPU로 이관(개발 PC 수동노드 의존 제거). 영상 캡션 2차(VLM)는 안정화 후 별도 착수

## 4. 프로비저닝 (운영 무중단, 새 서버 임시 IP)

기존 `사내서버-이관-가이드-Ubuntu-2026-06-12.md` + .42 DR 구축 절차 재사용. 현 서버와 동일한 26.04라 호스트 레벨 차이 없음.

- [ ] Ubuntu 26.04 설치, LVM(/ 300G, /data 1.5T, 예비 600G 미할당)
- [ ] netplan **단일 파일**로 고정 IP(임시) — 파일 중복+NM DHCP fallback 함정(`netplan-static-ip-nm-fallback` 사건) 재발 금지
- [ ] NVIDIA 드라이버(26.04 아카이브 595 계열, **-open 모듈 확인** — Blackwell은 proprietary 불가) + nvidia-container-toolkit → `nvidia-smi`, `docker run --rm --gpus all ubuntu nvidia-smi` 확인. 호스트 CUDA 툴킷 불필요(전 컨테이너, NVIDIA repo에 ubuntu2604 아직 없음)
- [ ] docker/compose 설치 — **기본 경로 사용** (구 서버의 /data/docker·containerd bind mount 응급처치는 청산, docker root만 필요 시 /data로 daemon.json 설정)
- [ ] CIFS 마운트: fstab **soft 옵션 필수**(6/15 서버 동결 사건)
- [ ] SSH 계정 재생성: oceantech, dev, kimny, wltnchoi (+ 키, `SSH-개발자계정-온보딩` 문서 참조)
- [ ] git 리포 클론(ERP_OT, ot-brain, photo-album) + **`.git/hooks/post-commit` 수동 복사**(비추적)
- [ ] .env 파일들 복사 (git 비추적 — 누락 시 web 빌드가 localhost를 구움: `web-proxy-rewrites-localhost-bake`)
- [ ] pg_dump 복원 리허설: **반드시 `pg_dump --clean`**(seed가 데이터 덮어씀 — .42 구축 때 확인된 함정)
- [ ] web 프로덕션 빌드: compose command `--include=dev` 유지(devDep 함정)
- [ ] 전체 스택 기동 + 헬스체크, GPU override(`docker-compose.gpu.yml`)로 리랭커·ollama 기동 확인 — 이미지들의 CUDA 12.8/Blackwell(sm_120) 호환 선확인
- [ ] 호스트 cron 10종 이식(주석 상태로 준비, 컷오버 시 활성화): exif-start/stop, erp-nas-backup(daily/weekly), nas-poll, nas-reconcile, nas-extract, disk-monitor, **ops-notifier(호스트 메일 전담)**, dr-sync-42
- [ ] **디스크 사용량 알람 신설**(단일 볼륨이라 필수 — ops.monitor에 스크립트+행 추가, 6/30 인시던트 잔여 항목 해소)
- [ ] /home/oceantech/backups 복사

## 5. 컷오버 런북 (평일 야간)

1. 공지 후 구 서버 ERP 스택 정지 (cron도 비활성화)
2. 최종 `pg_dump --clean` + 볼륨 tar → 새 서버 복원
3. 구 서버 IP 해제 → 새 서버 **.41 승계** (netplan 단일 파일)
4. 새 서버 스택 기동(GPU override 포함) + 헬스체크·로그인·대시보드·NAS검색 스모크 테스트
5. 새 서버 cron 활성화 (dr-sync-42 → .42 동기화 재개 확인, RPO 10분 복원)
6. 구 서버는 네트워크 분리 상태로 1주 보관 (전원은 켜둘 필요 없음)
7. 개발 PC 리랭커/RAG 정지 — 검색 스택은 이제 서버 GPU

**롤백**: 새 서버에서 치명 문제 시 → 새 서버 IP 해제 → 구 서버 재기동(.41) → 원상복구. 1주 내 언제든 가능.
