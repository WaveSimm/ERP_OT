# 리팩토링 CTO 재검토 — 서버 실태 기준 (2026-06-19)

> **목적**: dev PC 핸드오프 진단(`2026-06-19-리팩토링-사전진단-핸드오프.md`)이 서버 실태와 크게 달라, **서버(192.168.0.41)의 실제 구동 코드 기준으로 4팀 병렬 재진단**(읽기 전용)한 결과를 종합. 이 문서가 **서버 리팩토링의 단일 기준**이다.
> **방식**: bkit CTO 모드, 4팀 병렬(ERP_OT 구조 / ERP_OT 운영·보안·빌드 / ot-brain / 횡단 시퀀싱), 코드 변경 없음.
> **결론 한 줄**: dev의 "즉시 위험 Critical 4"는 서버에서 **사실상 전부 무효**. 진짜 리팩토링 블로커는 **양 repo 자동 테스트 0개 + ERP_OT의 死 CI**다.

## 1. 🔴 진짜 블로커 — 자동 안전망 0

| 항목 | ERP_OT | ot-brain |
|---|---|---|
| 테스트 파일(`*.test.ts`/`*.spec.ts`/`test_*.py`) | **0** | **0** |
| CI 워크플로 | `ci.yml` 1개 | 없음 |

**ERP_OT `ci.yml`은 死 워크플로**: 트리거가 `on: [main, develop]`인데 **기본 브랜치는 `master`** → 어떤 push/PR에서도 실행 안 됨. 게다가 `pnpm test`가 호출돼도 테스트 0개라 회귀 보장 0.

**시퀀싱 함의**: 모든 리팩토링의 "안 깨졌는지" 검증이 **서버 실연동 수동확인(docker build/up + 화면·검색·사진첩 호출)에 100% 의존**. 유일한 즉시 정적 안전망은 `pnpm build`(타입체크). → 이것이 "서버 주(主)" 결정의 진짜 근거.

## 2. ERP_OT 구조 부채 (요약)

- **Clean Architecture는 외형뿐**: 살아있는 repository 패턴은 **auth 서비스 하나**(`UserPrismaRepository implements IUserRepository`). **project의 도메인 인터페이스 7개는 100% 死추상화**(`implements` 0건, import는 필터타입뿐). 17개 라우트가 `prisma.*` 직접 호출(API→Infra 누수). project 2곳(`notification.routes.ts`·`my-tasks.routes.ts`)은 서비스 완전 우회.
- **equipment = God-service**: 약 7개 바운디드 컨텍스트(장비·AS수리·재고·조달·CRM·카탈로그·센서), prisma 콜 431, **`infrastructure/` 계층 부재**, god-route `procurement.routes.ts`(321줄/35핸들러). ⚠️ **AS관리 production** → 분리는 고리스크.
- **프론트 god-file**: `apps/web/src/lib/api.ts` **2199줄·`any` 493·export 객체 60개**. 1000줄+ 페이지/컴포넌트 10개(`projects/[id]/page.tsx` 2257, `repair/[id]/page.tsx` 1897(production AS), `TaskDrawer.tsx` 1538 …).
- **중복/고아**: `errorResponse` 2종이 **응답 형태 불일치**(`{error}` vs `{success,error}`), `event-publisher.ts`·`auth.middleware.ts` 바이트동일 twin, 빈 패키지 `@erp-ot/ui`·import 0 `@erp-ot/api-client`, `@erp-ot/config` dangling export, `services/user` health-only 스텁.
- **FSM 우회**: `procurement.routes.ts`의 내부승인 콜백이 `order.fsm` 우회해 `overseasOrder` status 직접변경(정합성 갭).

## 3. ERP_OT 운영·보안 부채 (요약)

- **내부인증 4중 분기**: shared `requireInternal`은 attendance/project/equipment만. **auth·approval·expense·ocr는 인라인**(길이검증·상수시간비교 없음, 401/403 불일치). 특히 **ocr `auth.middleware.ts:24-28`은 폐기된 `userRole="ADMIN"` 자동승격 잔존**(권한상승 표면).
- **ocr 부트스트랩 이탈**: helmet/rate-limit/cookie 미등록, **CORS 기본값 `*`**(`config/env.ts:8`). OCR은 무거운 CPU 작업 → rate-limit 부재 DoS 표적.
- **JWT 시크릿 최소길이 불일치**: auth/expense/user=32, 나머지 5개=16. 동일 시크릿 공유라 16자 검증 서비스가 약한 키를 통과시킴 → 전체 하향.
- **빌드 함정**: `build-base`/`runtime-base` 선빌드를 강제하는 `depends_on` 없음 → clean/이관 빌드 시 실패 가능. web은 운영에서도 dev 모드 구동.
- **로드테스트 계정 추적**: `scripts/load-test/users.json`(90계정 정적비번) git 추적 → gitignore 처리 + 운영 DB 활성여부 점검 권장.
- **양호**: 하드코딩 시크릿 없음, `.env` 미추적, helmet/127.0.0.1 바인딩/Zod env 검증 대체로 적용.

## 4. ot-brain — 견고한 코어 + 좁은 부채

- **보존 1순위 강점**: 인제스천 idempotent·restartable(`extract_documents.py:505` `FOR UPDATE SKIP LOCKED`, EmbedError→PENDING 복원, 해시 dedup 1회 임베딩), 검색 graceful degradation(reranker 세마포어+회로차단기+폴백), nas-file 경로 confinement, 127.0.0.1 바인딩. **성능 주석이 SQL마다 근거 기록** — 도메인 지식 보존 가치 최상.
- **진짜 부채(단 하나)**: `server.ts` 968줄 중 **`/documents/search` 단일 핸들러 470줄** + **테스트 0개** = "건드리기 무서운 핵심 자산". 무상태 클라이언트(ollama/reranker/auth)가 server.ts에 인라인.
- **소부채**: 임베딩 클라이언트 3중 중복(TS 2곳 재시도 없음 vs Python 백오프), `npm run serve` `--env-file` 누락(단 **운영 docker엔 무영향** — dev 함정만), hash IN절 문자열보간(현재 hex필터로 안전하나 `=ANY($1::text[])` 권장), knowledge-api/nas-file CORS 와일드카드 잔존, Prisma 풀 미설정.
- dev가 우려한 루트 `_*` cruft·평문비번·런북 미커밋은 **서버에 없음**.

## 5. 리팩토링 로드맵 — 서버 실태로 재정의

### Phase 0 — 안전망 (리팩토링 아님, 무조건 선행)
- [ ] `git pull`(ERP_OT origin 대비 behind, 핸드오프 문서 미수신) — **단 미커밋 드리프트·off-limits 보존**(stash→pull→pop 권장)
- [ ] DB 백업 신선화·검증(자동 백업 동작 확인 + 직전 수동 스냅샷)
- [ ] 청킹 워커 미가동 확인(SKIP LOCKED 고아 방지)
- [ ] **ERP_OT 드리프트 2건 커밋→PR**(보안 아닌 위생, 둘 다 인증 확인됨). 서버 배포본과 바이트 정합 확인. `.claude/settings.local.json`은 분리/무시
- [ ] **死 CI 교정**: `ci.yml` 트리거 `main/develop`→`master`
- [ ] **최소 스모크 테스트 도입**: 9서비스 `/health` + `/internal/*` 401/통과 계약 + 기존 k6 scenario-b/c 1회 실행해 에러율 baseline
- [ ] dev 전용 항목 **제외 확정**: C1·C2·C3·ot-brain 루트정리·런북이전(이미 완료)
- ⛔ off-limits: `otbrain://` 폐기 미커밋 변경은 건드리지/커밋하지 말 것

### Phase 1 — Quick Wins (저위험·고가치)
- 고아 청소: `@erp-ot/ui`·`@erp-ot/api-client` 삭제(import 0), project 死인터페이스 7개 삭제(또는 구현 결정), `@erp-ot/config` dangling export 정리
- 응답계약 단일화: legacy `errorResponse` 제거 → `{error:{code,message}}` 수렴, `event-publisher`·공통 인증 미들웨어 `@erp-ot/shared` 승격
- **내부인증 일원화**: auth/approval/expense/ocr → shared `requireInternal`(특히 **ocr 자동ADMIN 제거**, approval 수동호출 5곳→글로벌 훅)
- ocr 부트스트랩 정렬(helmet/rate-limit/cookie) + **CORS `*` 차단**, JWT 시크릿 `min(32)` 통일
- ot-brain: server.ts **무상태 클라이언트 3종 추출**(`lib/ollama.ts`·`lib/reranker.ts`·`lib/auth.ts` — 순수 이동, 968→~600줄)
- **게이트**: 변경마다 `pnpm build` 타입체크 + 영향 서비스 rebuild/health

### Phase 2 — 구조 개선 (중위험, 계획 필요)
- ERP_OT `lib/api.ts` 도메인 분할(**barrel re-export로 import 비파괴**) + `any` 점진 타이핑, approval 폼 공통화, 라우트 prisma 누수 흡수(**repository 정책 결정 선행** — 권장: 死인터페이스 삭제 후 "application→Prisma 직접" 공식화)
- ot-brain `server.ts` 검색 핸들러 4모듈 분해(parser/sql/scoring/grouping), 임베딩 배치화+재시도 통합, Prisma 풀 명시
- **강제 선행조건**: Phase 1 스모크 테스트 + **검색 골든 회귀 스냅샷**(질의→결과 id순서). 없으면 god-file 분해 금지

### Phase 3 — 대수술 (고위험, 게이트)
- equipment God-service 컨텍스트 분할. **AS production** → 배포단위 분할 금지. 1차는 `application/{repair,inventory,procurement,contract}/` 폴더 모듈 경계 + infra 계층 신설(런타임 무변경)만. 선행: Prisma 스키마 seam 분석 + 마이그 플레이북 + DB 백업 + Phase 2 안전망 누적

## 7. 상위 권장 5선 (리스크-가치)

1. **[Phase 0] 안전망 먼저** — 死 CI 교정 + 스모크/계약 테스트 + 검색 골든 스냅샷. 테스트 0개가 모든 후속의 실질 차단요인.
2. **[Phase 0~1] 무위험 청소** — 드리프트 커밋(위생) + 고아 패키지·死인터페이스 삭제. 인지부하를 가장 싸게 제거.
3. **[Phase 1] 내부인증 일원화** — ocr 자동ADMIN 제거 포함, shared로 수렴(보안+일관성).
4. **[Phase 1] api.ts·server.ts 무파괴 분할** — barrel/순수이동으로 호출부 불변, 후속 타이핑·분해의 발판.
5. **[Phase 3·게이트] equipment 경계화** — 폴더 모듈 경계부터, 서비스 물리분리는 별도 PDCA. production AS는 한 번에 쪼개지 말 것.
