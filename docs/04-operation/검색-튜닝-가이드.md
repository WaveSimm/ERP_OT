# 검색 튜닝 가이드 (Hybrid Search)

> 검색개선 PDCA 결과로 도입된 환경변수 8개로 검색 동작을 재배포 없이 조정하는 방법.

---

## 1. 점수 합산 식

```
final_score = embed_score · W_e
            + GREATEST(exact_bonus, trgm_score) · W_k
            + (ILIKE_match ? KEYWORD_MATCH_BONUS : 0)

WHERE final_score ≥ MIN_SCORE
```

| 변수 | 기본값 | 설명 |
|------|:---:|------|
| `SEARCH_EMBED_WEIGHT` (W_e) | 0.6 | 임베딩 점수 가중치 |
| `SEARCH_KEYWORD_WEIGHT` (W_k) | 0.4 | 키워드 점수 가중치 (W_e + W_k = 1.0 강제) |
| `SEARCH_MIN_SCORE` | 0.35 | 최종 점수 컷오프 |
| `SEARCH_TITLE_EXACT_BONUS` | 0.30 | title ILIKE 매칭 시 가산 (자동) |
| `SEARCH_CONTENT_EXACT_BONUS` | 0.15 | content ILIKE 매칭 시 가산 (자동) |
| `SEARCH_TITLE_TRGM_WEIGHT` | 0.20 | title `similarity()` 가중치 |
| `SEARCH_CONTENT_TRGM_WEIGHT` | 0.10 | content `similarity()` 가중치 |
| `SEARCH_KEYWORD_MATCH_BONUS` | 0.20 | ILIKE 매칭 시 절대 가산 (가중치 미적용) |

---

## 2. 시나리오별 조정

| 증상 | 권장 조정 | 비고 |
|------|----------|------|
| 정확 매칭 글이 결과에 없음 | `SEARCH_MIN_SCORE` ↓ (예: 0.30) | 컷오프 완화 |
| 무관한 글이 자꾸 나옴 | `SEARCH_MIN_SCORE` ↑ (예: 0.40) | 컷오프 강화 |
| 영문 약어(DAAU 등) 매칭이 약함 | `SEARCH_TITLE_TRGM_WEIGHT` ↑ (0.25) | trigram 강화 |
| 정확 매칭 글이 의미 매칭 글에 밀림 | `SEARCH_KEYWORD_MATCH_BONUS` ↑ (0.30) | 절대 보너스 강화 |
| 의미 검색 약화 | `SEARCH_EMBED_WEIGHT` ↑ (0.7) + `SEARCH_KEYWORD_WEIGHT` ↓ (0.3) | 합 1.0 유지 |
| 짧은 데이터셋(<100건) | `SEARCH_MIN_SCORE` 0.30~0.35 | 임베딩 점수 좁아짐 |
| 큰 데이터셋(>1000건) | `SEARCH_MIN_SCORE` 0.45+ | 분포 넓어짐 |

## 3. 적용 절차

```bash
# 1. docker-compose.yml의 environment에서 값 수정
# 2. 두 서비스 재기동
docker compose up -d auth-service project-service

# 3. 헬스체크
curl http://localhost:3001/health
curl http://localhost:3003/health

# 4. 결과 변화 확인 (브라우저: /board/search?q=...)
```

---

## 4. 디버그 도구

특정 쿼리에 대한 점수 분포 확인:

```bash
node scripts/debug-search.mjs 메인보드
```

출력: 모든 게시글의 embed/exact/trgm/final 점수 (테이블 형태). 조정 필요한 항목 즉시 파악 가능.

---

## 5. 가중치 합 검증

서비스 시작 시 `SEARCH_EMBED_WEIGHT + SEARCH_KEYWORD_WEIGHT = 1.0`을 자동 검증.
합이 1.0이 아니면 startup error로 즉시 실패 → 잘못된 값 운영 차단.

---

## 6. 주의사항

- **점수의 의미가 변경됨**: 자연어검색(0.0~1.0 코사인) → 검색개선(0.0~1.0 합산 점수). UI에 표시되는 "유사도 %"는 이제 절대 코사인이 아님.
- **trigram 인덱스는 GIN**: INSERT/UPDATE 빈도가 높은 테이블에서는 부하 가능. 게시판/비고는 빈도 낮아 무관.
- **한국어 형태소 분석 미적용**: "디버깅"(쿼리) ≠ "debugging"(데이터)는 매칭 안 됨. 임베딩이 cross-lingual로 어느 정도 보완.
- **작은 데이터셋의 한계**: 데이터 < 100건 환경에서는 컷오프 효과 약함. 정확 매칭 신호에 의존하는 게 나음.
