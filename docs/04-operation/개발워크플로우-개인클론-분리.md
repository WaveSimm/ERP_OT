| 권한 공유 | 레포 내 `.claude/settings.local.json`이 공유되어 한 명이 Claude에 준 권한이 전원에게 적용 |

## 3. 개발자 워크플로우

### 3.1 최초 1회 — 개인 클론 만들기

```bash
# 자기 계정으로 SSH 접속 후
cd ~
git clone git@github.com:<org>/ERP_OT.git   # 또는 로컬: git clone /home/oceantech/ERP_OT
cd ERP_OT
git config user.name  "<본인 이름>"
git config user.email "<본인 이메일>"
pnpm install
```

Claude CLI는 **자기 클론 안에서** 실행한다. 계정별 `~/.claude`라 메모리·권한·세션이 자동 분리된다.

### 3.2 일상 작업 — feature 브랜치 + PR

```bash
cd ~/ERP_OT
git checkout master && git pull
git checkout -b feat/무언가-요약
# ... 수정 (Claude CLI 포함) ...
git commit
git push -u origin feat/무언가-요약
# GitHub에서 PR 생성 → 리뷰 → master 머지
```

- master 직접 push 금지 (GitHub branch protection ON)
- 커밋 단위는 작게, PR 제목은 한국어 요약 (기존 컨벤션 유지)

### 3.3 수정한 코드 확인(테스트) 방법

떠 있는 컨테이너는 배포 트리 기준이므로 개인 클론의 수정은 **자동 반영되지 않는다** (의도된 안전장치).

| 대상 | 방법 |
|---|---|
| 프론트(apps/web) | 자기 클론에서 `npm run dev -- -p <개인포트>` → 핫리로드로 확인. API는 공용 백엔드를 그대로 사용 |
| 백엔드(services/*) | 해당 서비스 1개만 자기 클론에서 대체 포트로 기동해 확인, 또는 소규모 변경은 PR 리뷰 → 머지 → 배포 후 확인 |
| DB 스키마 변경 | 운영 DB 직접 실험 금지. 변경 SQL을 PR에 포함하고 배포 절차에서 적용 (project 스키마는 migrate deploy 금지 — 직접 SQL 관행 유지) |

### 3.4 주의 — 공용 자원이므로 지킬 것

1. **운영 DB에 테스트 쓰기 주의**: 개인 dev 서버도 공용 백엔드/DB를 바라보므로, 쓰기 동작 테스트는 실데이터에 들어간다. 실험성 데이터는 만들었으면 지우고, 대량 실험은 사전 공지.
2. **프로덕션 컨테이너 재시작 금지**(배포 담당 외): 전원 docker 그룹이라 기술적으로는 가능하지만, `docker compose restart web` 한 방이 곧 배포다.
3. **배포 트리 직접 편집 금지**: `/home/oceantech/ERP_OT`는 읽기만.

## 4. 역할별 전체 흐름 (개발자 ↔ 메인 개발자)

```
[개발자]                          [메인 개발자(배포 담당)]

1. 자기 클론에서 브랜치 생성
2. 수정 (자기 Claude CLI 활용)
3. 커밋 → GitHub에 push
4. PR 생성
                          5. PR 리뷰·검증   ← 검증은 여기서
                          6. master 머지
                          7. 배포 트리에서 git pull
                          8. 빌드 + 재기동  ← 배포는 여기서
```

- **검증의 주 무대는 PR 단계(5번)다.** 개발자가 커밋했다고 바로 빌드하는 게 아니라,
  메인 개발자가 PR을 리뷰하고 머지 여부를 결정한다. 이때 Claude CLI를 검증 도구로
  활용할 수 있다 (예: `/review <PR번호>`로 코드리뷰, 변경 영향 분석 요청).
- **빌드·배포(7~8번)는 검증이 끝난 뒤의 기계적 작업**이다. master에 머지된 것만
  배포 트리로 pull하여 빌드·재기동한다.
- **커밋은 아무것도 배포하지 않는다.** 개발자가 아무리 커밋·push해도 메인 개발자가
  머지·배포하기 전에는 서비스에 닿지 않는다.
- 책임 분담: 개발자 = 개발→커밋→push→PR까지 / 메인 개발자 = PR 검증→머지→pull→빌드·배포
- (향후) PR 리뷰 부담이 커지면 GitHub Actions로 자동 검사(타입체크·빌드)를 PR에 붙여
  메인 개발자의 검증 부담을 줄일 수 있다 — 필요해질 때 도입.

## 5. 배포 절차 (배포 담당)

```bash
cd /home/oceantech/ERP_OT
git status          # dirty면 중단 — 원인 파악 먼저
git pull            # PR 머지분 반영
# 백엔드: docker compose build <service> && docker compose up -d <service>
# 프론트: docker compose exec web sh -c "npm run build" && docker compose restart web
```

- **dirty-guard**: 배포 트리에 미커밋 변경이 있으면 배포 중단이 원칙 (스크립트화 예정)
- 배포는 지정 담당(현재 oceantech 계정)만 수행

## 7. FAQ

**Q. 개발환경을 새로 세팅해야 하나?**
아니다. 서버·도커·DB 전부 지금 그대로 공유한다. 바뀌는 건 "코드를 어디서 고치느냐"뿐이다.

**Q. 클론하면 용량은?**
레포 1개당 수백 MB 수준. `/` 파티션 여유(225GB+) 충분.

**Q. 내 수정이 서비스에 언제 반영되나?**
PR이 master에 머지되고, 배포 담당이 배포 트리에서 pull + 리빌드한 시점. 그 전에는 절대 반영되지 않는다 — 그게 이 구조의 목적이다.
