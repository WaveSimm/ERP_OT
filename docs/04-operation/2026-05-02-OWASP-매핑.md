## 3. Kill-chain (상호 연쇄 위험)

### Chain A — 세션 탈취 영구화 (가장 위험)
```
XSS (1건만 있어도)         ← C1 (localStorage 노출)
  → Access Token 탈취
  → /api/v1/auth/refresh 호출            ← C8 쿠키 path 불일치 무력화 시 refresh 실패하더라도
                                            8h Access TTL이 매우 김 → 실용적 영구 세션
  → DB dump 시 Refresh 모두 활성         ← C5 평문 저장
  → audit/IP 추적 불가                    ← C6 헤더 drop
영향: 완전한 세션 하이재킹 + 사후 추적 불가
```

### Chain B — 권한 승격
```
INTERNAL_API_TOKEN 노출 (env, 로그, 컨테이너 metadata 어디서나)
  → approval-service의 모든 라우트에 X-Internal-Token만 붙이면 userRole=ADMIN 자동 부여
  → 결재 임의 승인/반려 → 휴가/OT 자동 동기화 → 장기적 데이터 오염
완화: token min(16) 검증은 됐으나, 토큰 자체가 평문 env로 모든 컨테이너에 공유
```

### Chain C — JWT Secret 우회
```
공격자가 .env에 JWT_ACCESS_SECRET 누락된 환경에서 auth-service 기동 (C2 fallback)
  → 공격자가 동일 fallback secret으로 ADMIN 토큰 자체 서명
  → 모든 서비스 jwtVerify 통과 (project/equipment/attendance/approval은 env Zod로 fallback 차단됐으나 auth가 발급한 토큰은 신뢰)
완화: 모든 서비스가 동일 secret 강제 검증 + auth도 Zod 적용 필요
```

### Chain D — Brute Force + Email Pollution
```
H1 (rate-limit 없음) + H2 (실패 로그 userId=email)
  → 무한 시도 + activityLog 테이블이 임의 email 문자열로 오염
  → 추후 audit 분석 시 노이즈 + 사용자 ID 인덱스 손상 가능
```

## 4. 추가 발견 (NEW-#) — 코드 spot 분석

### NEW-1. Folder API 인가 부재 (Critical, A01)
**위치**: `services/project/src/api/routes/folder.routes.ts:6-9`
```ts
fastify.get("/", async (request) => {
  const userId = (request as any).userId || (request as any).user?.id || "unknown";
  return fastify.folderService.list(userId);
});
```
- `userId` fallback을 `"unknown"` 문자열로 → service에서 어떻게 사용되는지 무관하게 의미 미정
- preHandler 없음 (인증은 글로벌이라 통과하나, 서비스 내부에서 userId-기반 ACL이 있다면 우회 가능)

### NEW-2. Comment IDOR 잠재 (High, A01)
**위치**: `services/auth/src/api/routes/comment.routes.ts:50-72`
- `PATCH /comments/:id`, `DELETE /comments/:id`가 라우트 단계에서 `postId` 컨텍스트나 author 검증 없이 service에 위임
- service 내부에서 author 체크가 빠지면 즉시 IDOR (현재 service 코드는 확인 못함 → 검증 권고)

### NEW-3. Board target_department 권한 우회 잠재 (High, A01)
**위치**: `services/auth/src/application/board-permissions.ts:15-25`
- `canRead`는 `board.audienceTargetId`만 검사하고 **post 단위 `targetDepartmentId`를 무시**
- `Post.targetDepartmentId`가 설정된 부서 공지를 다른 부서가 읽을 수 있음
- 검색 라우트(`search.service.ts`)도 동일 함수 사용 → 검색 결과로도 노출

### NEW-4. JWT Algorithm Confusion 가능 (High, A02)
**위치**: `services/auth/src/application/auth.service.ts:99,103,107`
```ts
jsonwebtoken.sign(..., this.accessSecret, { expiresIn: "8h" })   // alg 미지정 → HS256 default
jsonwebtoken.verify(token, this.accessSecret)                     // algorithms 옵션 없음
```
- `verify`에 `algorithms: ["HS256"]` 미지정 → `none` 또는 `RS256` 변조 토큰 수용 위험 (jsonwebtoken 9.x 이후 default 차단됐으나 구버전 호환성 위험)
- Access TTL이 **8시간** (CLAUDE.md는 1h라 명시했으나 실제 코드는 8h) — 권한 회수 지연

### NEW-5. Search Raw SQL 안전성 (Low, A03)
**위치**: `services/auth/src/application/search.service.ts:104-145`
- `Prisma.sql\`...\`` 템플릿 + `${q}`로 파라미터 바인딩 → SQL injection 차단 OK
- `${cfg.titleExactBonus}::float` 등 cfg 값은 환경변수 출처 → operator misconfig시 cast 에러만 발생

### NEW-6. AuthUserContext 부서 미설정 시 canRead 우회 (Medium, A01)
**위치**: `services/auth/src/api/routes/post.routes.ts:18-20, 64-67`
- `extractUser(req)`는 `departmentId`가 없는 컨텍스트 반환 → `getFeed` 호출 시 `canRead`에서 `DEPARTMENT` 보드는 항상 false. 단 `getFeed`는 service 내부에서 audience 체크 없이 boardWhere로만 필터 → 부서 공지가 모든 사용자에 노출되는지 확인 필요

### NEW-7. 단일 디바이스 강제 (Low, A07)
**위치**: `auth.service.ts:35`
```ts
await this.prisma.refreshToken.deleteMany({ where: { userId: user.id } });
```
- 로그인 시 기존 refresh 모두 삭제 → 다른 디바이스 자동 로그아웃. 의도라면 OK이나 UX/보안 trade-off 명시 필요

### NEW-8. attendance INTERNAL_TOKEN 신뢰 자동 ADMIN (Critical, A01+A02)
**위치**: `services/approval/src/api/middleware/auth.middleware.ts:21-25`
```ts
if (internalToken && internalToken === process.env.INTERNAL_API_TOKEN) {
  request.userId = "system";
  request.userRole = "ADMIN";
  return;
}
```
- 토큰 1개 누설 시 모든 approval 라우트가 ADMIN으로 동작
- attendance도 동일 패턴인지 차후 확인 권고 (현재 attendance middleware는 `/internal/*` skip만 있음 → approval보다는 안전)

### NEW-9. /users/members 라우트 권한 (Medium, A01)
**위치**: `services/auth/src/api/routes/user.routes.ts:24-31`
- `GET /api/v1/users/members?all=true` — 인증만 있으면 누구나 전 직원 ID/이름/부서/직급 dump 가능. 공격자가 social engineering 자료로 활용

### NEW-10. attachment download MIME 신뢰 (Low, A04)
**위치**: `services/auth/src/api/routes/attachment.routes.ts:63-65`
- 업로드 시 저장한 mimeType을 그대로 응답 헤더에 → 업로더가 mime 위장 시 브라우저 sniff 위험. `X-Content-Type-Options: nosniff` 필수.

### NEW-11. Mass Assignment 잠재 (Medium, A04)
**위치**: 다수 routes에서 `req.body as any` (H8, H13)
- 예: `PATCH /api/v1/auth/me`는 body에서 `name`만 추출하지만, user.routes의 일부는 Zod로 화이트리스트 → 일관성 부재
- `userService.update`의 `dto` 자체는 Zod 검증되나, 향후 새 필드 추가 시 의도치 않은 필드 노출 위험

### NEW-12. publishActivity가 password reset 등 민감 액션 RabbitMQ로 fanout (Medium, A09)
**위치**: `services/auth/src/api/routes/user.routes.ts:100-106`
- `password_reset` 액션이 description+metadata로 RabbitMQ exchange `erp.activity`에 publish
- 컨슈머 보안 + 메시지 영속성 (`persistent: true`) → 큐/DLQ 노출 시 누가 누구의 비번을 리셋했는지 평문 보관

### NEW-13. Error stack trace 노출 가능 (Low, A09)
**위치**: `services/project/src/index.ts:178-184`
- `app.log.error(error)` 후 일반 사용자에 `INTERNAL_ERROR`만 응답 — OK
- 단 auth-service는 `setErrorHandler`가 명시적으로 안 보임 → Fastify 기본 동작이 NODE_ENV 기준이므로 prod 빌드에서만 안전. dev/stage 노출 가능

### NEW-14. CORS origin env 미검증 (Medium, A05)
**위치**: `services/auth/src/index.ts:43-46`
- `origin: process.env.CORS_ORIGIN || "http://localhost:3000"` — 단일 문자열만 허용. 배열·정규식 지원 없으면 multi-tenant 환경 confusion 가능. 또한 `*`로 설정 시 `credentials: true`와 함께 동작하지 않음 (Fastify가 차단) — operator 실수 시 CORS 자동 OFF로 디버깅 어려움.

## 5. 패치 의존성 그래프 (롤아웃 순서)

```
Layer 1: 환경/시크릿 강화 (선행 필수, 다른 패치의 토대)
  ├─ C2  (auth-service Zod env 검증) ─────────┐
  ├─ C3  (`?? ""` fallback 코드 정리)         │
  ├─ INF-1, INF-2, INF-3 (시크릿 회전)        │── 동시 적용
  └─ NEW-14 (CORS env 검증)                   │
                                              ▼
Layer 2: 인증 미들웨어 통합 (C7) — Layer 1 없으면 통합 후에도 같은 버그 잔존
  ├─ shared 패키지 `requireAuth`, `requireInternal` 신설
  ├─ 4개 서비스 마이그레이션
  ├─ NEW-8 자동 ADMIN 승격 제거 (system 사용자도 명시적 권한 부여)
  └─ project/attendance `/internal/*` skip 패턴 표준화
                                              ▼
Layer 3: 토큰/세션 모델 변경 (C1, C5, C8, NEW-4, NEW-7)
  ├─ Refresh httpOnly cookie 통합
  ├─ Refresh 토큰 hash 저장 (DB 마이그레이션 + 기존 세션 invalidate)
  ├─ 쿠키 path 통일 `/api/v1/auth`
  ├─ JWT verify에 algorithms 명시
  └─ Access TTL 1h로 단축 (CLAUDE.md 명세 일치)
                                              ▼
Layer 4: 인가 강화 (C4, NEW-1, NEW-2, NEW-3, NEW-9)
  ├─ /tasks/mine, /me/* userEmail → userId 기반으로 변환
  ├─ Folder/Comment 라우트 preHandler + service 양쪽 검증
  ├─ canRead에 post.targetDepartmentId 반영
  └─ /users/members 권한 정책 결정 (전직원 노출 vs MANAGER+)
                                              ▼
Layer 5: 프록시/관측 (C6, H1, H2, H11)
  ├─ Web 프록시가 cookie/x-forwarded-for/user-agent forward
  ├─ rate-limit + helmet 전 서비스 일괄
  ├─ 로그인 실패 시 userId=null + email은 metadata로
  └─ console.* → fastify.log 일괄 치환
                                              ▼
Layer 6: 데이터 무결성 (H3~H6, H8, H12, H13)
  └─ Clean Architecture 정합성 + DTO 표준화는 별도 PDCA로 분리 권고
```

**원칙**: Layer 2를 Layer 1 없이 진행하면 통합 미들웨어가 같은 fallback 버그를 재배포함. Layer 3은 Refresh DB 스키마 변경으로 운영 중 다운타임 1회 필요 (모든 사용자 재로그인). 메모리에 등록한 "수정 우선순위 — 세세한 기능 먼저, 보안 Critical은 일괄 패치"와 정합.

## 6. Plan 단계 입력 — 다음 PDCA에서 결정 필요한 항목

### 6-1. 토큰 모델 결정
| 항목 | 옵션 | 권장 |
|---|---|---|
| Access Token 저장소 | (a) localStorage 유지 (b) 메모리(Zustand) + sessionStorage refresh trigger (c) httpOnly cookie + CSRF token | **(c)** — XSS 면역. CSRF 비용은 SameSite=strict + 더블 서브밋 토큰으로 |
| Access TTL | 1h (CLAUDE.md) vs 8h (현재 코드) | **1h** + sliding refresh — 권한 회수 응답성 |
| Refresh 저장 방식 | 평문 vs hash(SHA-256) vs hash + JWT-ID | **hash(SHA-256)** — DB 유출 시 활성 세션 제로 |
| 다중 디바이스 | 강제 단일 vs 디바이스별 다중 | **다중 + 사용자 메뉴에 "다른 기기 모두 로그아웃" 액션** |
| JWT 알고리즘 | HS256 vs RS256 | **HS256 유지** (모노레포 + 단일 발급자) but `verify`에 명시. 향후 외부 IdP 통합 시 RS256 |
| Refresh rotation | 매 refresh마다 새 토큰 (현재 OK) vs reuse detection | reuse detection 추가 (탈취 감지) |

### 6-2. 에러 응답 포맷 표준
- 현재 2종: `{error: "msg"}` (auth) vs `{code, message}` (project)
- **결정안**: `{error: {code, message, details?}}` 통일. auth-service의 모든 응답 변경 (frontend api.ts 동시 수정)

### 6-3. DTO 표준화
- shared 패키지에 Zod schema 단일 출처
- `req.body as any` 285건 → 모두 `safeParse` 의무화
- ESLint rule `no-explicit-any` 점진 적용 (현재 `as any` 512건)

### 6-4. Internal API 통신 모델
- **현재**: 평문 X-Internal-Token + approval은 자동 ADMIN 승격
- **옵션 A**: mTLS (Docker 내부망에서 인증서 발급)
- **옵션 B**: JWT 서비스 토큰 (auth-service가 짧은 TTL의 service-to-service JWT 발급)
- **권장 (이관 전)**: 옵션 B + INTERNAL_API_TOKEN은 부트스트랩 전용으로 격리

### 6-5. 보안 헤더 정책
| 헤더 | 값 | 적용 위치 |
|---|---|---|
| Strict-Transport-Security | `max-age=63072000; includeSubDomains; preload` | next.config.js + 리버스 프록시 |
| X-Content-Type-Options | `nosniff` | 모든 응답 (특히 attachment) |
| X-Frame-Options | `DENY` | 전체 |
| Content-Security-Policy | `default-src 'self'` 시작점 | next.config.js (보고서 별첨) |
| Referrer-Policy | `strict-origin-when-cross-origin` | 전체 |

### 6-6. 감사 로그 보강
- 로그인 실패 시 `userId=null, metadata={attemptedEmail, ip, userAgent, attemptCount}`
- C6 패치로 IP/UA 헤더 forward 후 audit consumer가 분리 컬럼으로 저장
- `password_reset` 등 민감 액션은 RabbitMQ payload에서 본문 metadata 마스킹

### 6-7. Rate Limit 정책 (H1)
| 라우트 | 제한 | 근거 |
|---|---|---|
| `POST /api/v1/auth/login` | 5/min/IP, 10/15min/email | brute-force |
| `POST /api/v1/auth/refresh` | 10/min/IP | replay 방지 |
| `GET /api/v1/search` | 30/min/user | 임베딩 비용 |
| 일반 API | 100/min/IP | 기본 |
