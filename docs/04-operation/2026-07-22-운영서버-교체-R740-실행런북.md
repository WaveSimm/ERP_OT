# 운영서버 교체(R740) 실행 런북 — 명령어 따라하기

> 대상: 초보 개발자. OS 설치 후 이 문서를 브라우저로 열어놓고 위에서부터 순서대로 실행하면 됩니다.
> 계획 문서: `2026-07-22-운영서버-교체-R740-이관계획.md` (결정사항·타임라인은 그쪽 참조)
>
> **표기 규칙**
> - `새서버$` = 새 R740에서 실행, `구서버$` = 현 운영서버(192.168.0.41)에서 실행
> - 명령 블록은 한 줄씩 복사-붙여넣기. **✅ 확인** 줄의 결과가 다르면 멈추고 보고할 것
> - 비밀번호·자격증명은 이 문서에 없음 — 기존 서버에서 파일째 복사하는 방식이라 따로 입력할 일 없음

> ⚠️ **OS 버전 정정**: 계획서엔 24.04로 적었으나 **현 운영서버가 Ubuntu 26.04 LTS**로 확인됨.
> **동일 버전(26.04) 설치를 권장** — 이관 차이가 없어지고 Blackwell 드라이버 지원도 더 좋음. 아래 명령은 24.04/26.04 공통.

## Phase 1 — OS 설치 (설치 마법사 화면 따라하기)

> ⚠️ **이 Phase만은 명령어가 아니라 "설치 마법사 화면"입니다.** 마우스 없이
> **방향키(↑↓) / Tab / Enter / Space**로 조작하는 텍스트 메뉴입니다.
> 여기서 LVM 볼륨(디스크 분할)까지 전부 만들고 나오며, Phase 2부터는 전부 복사-붙여넣기 명령어입니다.

### 1-1. 설치 USB 만들기 (사무실 PC에서)

1. https://ubuntu.com/download/server 에서 **Ubuntu Server 26.04 LTS** ISO 다운로드 (Desktop 아님!)
2. https://rufus.ie 에서 Rufus 다운로드 → 실행
3. USB(8GB 이상) 꽂고 → 장치: 해당 USB → 부트 선택: 다운받은 ISO → [시작] → 경고는 모두 확인
4. 완료되면 USB를 뽑아 새 서버 앞면 USB 포트에 꽂는다

### 1-2. 서버 부팅 → USB로 부팅

1. 서버 전원 켜고 Dell 로고에서 **F11** 연타 → Boot Manager
2. **One-shot UEFI Boot Menu** → USB 항목(Disk connected to front USB…) 선택
3. GRUB 메뉴에서 "Try or Install Ubuntu Server" → Enter

> 💡 만약 부팅 전 화면에서 디스크가 2개(2.4TB ×2)로 보인다면 RAID1이 미구성인 것.
> 이 경우 **F2**(System Setup) → Device Settings → PERC → Virtual Disk Management →
> Create Virtual Disk → RAID1, 디스크 2개 선택 → 생성 후 다시 부팅. (대리점이 보통 구성해서 옴)

### 1-3. 설치 마법사 — 스토리지 전까지

| 화면 | 선택 |
|---|---|
| Language | English (한국어 없음, 이후 서버 로케일과 무관) |
| Keyboard | English (US) → Done |
| Type of install | Ubuntu Server → Done |
| Network | 자동 DHCP로 IP가 하나 잡히면 그대로 Done (임시 IP는 나중에 명령어로 설정) |
| Proxy | 빈칸 그대로 Done |
| Mirror | 그대로 Done |

### 1-4. 스토리지 화면 — ★가장 중요, 천천히★

1. "Guided storage configuration" 화면에서 **"Custom storage layout"** 선택 → Done
2. AVAILABLE DEVICES에 **약 2.2T짜리 디스크 1개**(PERC 가상디스크)가 보인다
   - 2개가 보이면 중단 → 위 1-2의 💡 RAID 구성부터
3. 그 디스크에 커서 → Enter → **"Use As Boot Device"** 선택 (부팅 파티션이 자동 생성됨)
4. 같은 디스크의 **"free space"** 에 커서 → Enter → **"Add GPT Partition"**
   - Size: 비워둠(최대) / Format: **"Leave unformatted"** ← 중요! → Create
5. 화면 하단 **"Create volume group (LVM)"** 선택
   - name: `ubuntu-vg` / 방금 만든 unformatted 파티션에 **Space로 체크** → Create
6. 생성된 `ubuntu-vg` 아래 "free space" → Enter → **"Create Logical Volume"** — 3번 반복:

| 순서 | name | Size | Format | Mount |
|---|---|---|---|---|
| ① | `lv-root` | `300G` | ext4 | `/` |
| ② | `lv-swap` | `16G` | **swap** | (자동) |
| ③ | `lv-data` | `1.5T` | ext4 | `/data` (Other → 직접입력) |

7. **ubuntu-vg에 free space가 ~380G 남아있는지 확인** — 이게 의도된 예비 공간(더 만들지 말 것!)
8. 최종 FILE SYSTEM SUMMARY에 `/ 300G`, `/data 1.5T`, `swap 16G`, `/boot/efi` 가 보이면 → Done
9. "Confirm destructive action" → **Continue** (새 디스크라 지울 데이터 없음)

### 1-5. 프로필·SSH

| 화면 | 입력 |
|---|---|
| Your name | OceanTech |
| Server name | `erp-server` (또는 원하는 이름) |
| Username | **`oceantech`** ← 반드시 이 철자 (모든 스크립트가 /home/oceantech 고정) |
| Password | 관리자에게 받은 것 (잊으면 안 됨) |
| Upgrade to Ubuntu Pro | Skip for now |
| SSH | **"Install OpenSSH server" Space로 체크** → Done |
| Featured snaps | 아무것도 체크하지 말고 Done |

설치 진행 → "Install complete!" 나오면 **Reboot Now** → USB 뽑기 → 재부팅 후 로그인(oceantech).

### 1-6. 설치 결과 확인 (여기부터 명령어 복사-붙여넣기)

```bash
df -h /  /data
```
✅ 확인: `/` 약 300G, `/data` 약 1.5T 두 줄이 보여야 함

```bash
sudo vgdisplay ubuntu-vg | grep Free
free -h | head -2
```
✅ 확인: `Free PE / Size` 에 약 380G 남아있고, Swap 16G가 보여야 함

## Phase 2 — 기본 설정 (새서버, 콘솔에서)

### 2-1. 임시 고정 IP (.43)

먼저 네트워크 카드 이름 확인 (구서버는 `eno4`였지만 새 서버는 다를 수 있음):

```bash
ip link
```
✅ 확인: `lo` 말고 `eno1` 또는 `enp…` 같은 이름 확인. 아래에서 `eno1` 자리에 그 이름을 넣는다.

netplan 파일은 **반드시 1개만** (과거 파일 2개 공존 → 재부팅마다 DHCP로 되돌아가는 사고 있었음):

```bash
ls /etc/netplan/
sudo rm /etc/netplan/*.yaml
sudo nano /etc/netplan/01-static.yaml
```

아래 내용 입력(들여쓰기 그대로, `eno1`은 위에서 확인한 이름으로):

```yaml
network:
  version: 2
  ethernets:
    eno1:
      dhcp4: false
      addresses: [192.168.0.43/16]
      routes:
        - to: default
          via: 192.168.0.1
      nameservers:
        addresses: [192.168.0.1, 8.8.8.8]
```

```bash
sudo netplan apply
ip -4 addr show | grep 192.168
ping -c 2 8.8.8.8
```
✅ 확인: `192.168.0.43`이 보이고 ping 응답이 와야 함. 이후부터는 사무실 PC에서 `ssh oceantech@192.168.0.43`으로 원격 작업 가능.

### 2-2. 기본 패키지·시간대

```bash
sudo apt update && sudo apt -y upgrade
sudo apt -y install curl git rsync smbclient cifs-utils htop net-tools
sudo timedatectl set-timezone Asia/Seoul
timedatectl
```
✅ 확인: `Time zone: Asia/Seoul`, `NTP service: active`

## Phase 3 — 도커 + NVIDIA 드라이버 (새서버)

### 3-1. 도커 (공식 저장소)

```bash
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker oceantech
```

도커 데이터를 `/data`로 (구서버와 동일 정책 — 컨테이너·이미지·볼륨 전부 /data에 저장):

```bash
sudo mkdir -p /data/docker
sudo tee /etc/docker/daemon.json > /dev/null <<'EOF'
{
  "data-root": "/data/docker"
}
EOF
sudo systemctl restart docker
sudo systemctl enable docker
```

**한 번 로그아웃했다가 다시 로그인** (docker 그룹 반영) 후:

```bash
docker info | grep "Docker Root"
docker run --rm hello-world
```
✅ 확인: `Docker Root Dir: /data/docker`, hello-world 성공 메시지

### 3-2. NVIDIA 드라이버 (RTX Pro 5000 Blackwell)

```bash
sudo ubuntu-drivers install
sudo reboot
```

재부팅 후:

```bash
nvidia-smi
```
✅ 확인: 표에 `RTX PRO 5000` + 드라이버 버전 570 이상. 안 보이면 `sudo ubuntu-drivers list` 결과를 보고할 것

컨테이너에서 GPU 쓰게 하는 툴킷:

```bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt update && sudo apt -y install nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
docker run --rm --gpus all ubuntu nvidia-smi
```
✅ 확인: 컨테이너 안에서도 같은 GPU 표가 나와야 함

## Phase 4 — 구서버에서 데이터 가져오기 (운영 무중단)

> 여기부터는 **구서버(.41)** 와 **새서버(.43)** 를 오가며 작업. SSH 접속 두 개 띄워놓으면 편함.

### 4-1. 구서버 → 새서버 SSH 키 연결

```bash
# 구서버$
ssh-keygen -t ed25519 -N "" -f ~/.ssh/id_ed25519 2>/dev/null; ssh-copy-id oceantech@192.168.0.43
ssh oceantech@192.168.0.43 hostname
```
✅ 확인: 비밀번호 없이 새서버 hostname이 출력

### 4-2. 리포·스크립트·자격증명 통째 복사 (구서버에서 실행)

git clone이 아니라 **rsync 통복사**를 쓴다 — `.env`, `.git/hooks`(비추적 훅), 로컬 상태까지 한 번에 가므로 누락이 없음:

```bash
# 구서버$  (약 3~4GB, 수 분 소요)
rsync -a --info=progress2 /home/oceantech/ERP_OT/    oceantech@192.168.0.43:/home/oceantech/ERP_OT/
rsync -a --info=progress2 /home/oceantech/ot-brain/  oceantech@192.168.0.43:/home/oceantech/ot-brain/
rsync -a --info=progress2 /home/oceantech/pims/      oceantech@192.168.0.43:/home/oceantech/pims/
rsync -a /home/oceantech/photo-album/ oceantech@192.168.0.43:/home/oceantech/photo-album/
rsync -a /home/oceantech/photo-album.git/ oceantech@192.168.0.43:/home/oceantech/photo-album.git/
rsync -a /home/oceantech/*.sh /home/oceantech/backups oceantech@192.168.0.43:/home/oceantech/
rsync -a /home/oceantech/.nas-backup-cred /home/oceantech/.ssh/ oceantech@192.168.0.43:/home/oceantech/.sshtmp/
sudo rsync -a /etc/nas-cred oceantech@192.168.0.43:/home/oceantech/nas-cred.tmp
```

새서버에서 제자리에 놓기:

```bash
# 새서버$
cat ~/.sshtmp/id_* ~/.sshtmp/known_hosts >/dev/null 2>&1 && cp -n ~/.sshtmp/id_* ~/.sshtmp/known_hosts ~/.ssh/ 2>/dev/null
chmod 700 ~/.ssh && chmod 600 ~/.ssh/id_* 2>/dev/null; rm -rf ~/.sshtmp
sudo mv ~/nas-cred.tmp /etc/nas-cred && sudo chmod 600 /etc/nas-cred
chmod 600 ~/.nas-backup-cred
ls /home/oceantech/ERP_OT/.env /home/oceantech/ot-brain/.env /home/oceantech/pims/.env
```
✅ 확인: .env 3개가 전부 보여야 함 (**하나라도 없으면 중단** — 없으면 web 빌드가 잘못된 주소를 구워버림)

### 4-3. NAS 마운트 (soft 옵션 필수 — 과거 서버 전체 동결 사고 원인)

```bash
# 새서버$
sudo mkdir -p /mnt/nas/oceantech
echo '//192.168.0.220/oceantech /mnt/nas/oceantech cifs credentials=/etc/nas-cred,ro,iocharset=utf8,vers=3.0,uid=1000,gid=1000,_netdev,nofail,soft,echo_interval=10,retrans=2,actimeo=1 0 0' | sudo tee -a /etc/fstab
sudo systemctl daemon-reload && sudo mount -a
ls /mnt/nas/oceantech | head -3
```
✅ 확인: NAS 폴더 목록이 보여야 함

## Phase 5 — 사전 리허설 기동 (새서버, 임시 IP로 — 운영 계속 무중단)

> 목적: 컷오버 날 밤에 처음 해보는 게 아니라, **미리 한 번 전부 돌려보고** 문제를 낮에 잡아두는 것.

### 5-1. 연습용 데이터 주입 (최신 백업 사용)

```bash
# 구서버$  (운영 그대로 둔 채 덤프만 뜬다)
docker exec erp-ot-postgres pg_dump -U erp_user -d erp_ot --clean --if-exists | gzip > /tmp/erp_ot_rehearsal.sql.gz
rsync /tmp/erp_ot_rehearsal.sql.gz oceantech@192.168.0.43:/tmp/
```

```bash
# 새서버$  — DB 컨테이너만 먼저 띄우고 복원
cd /home/oceantech/ERP_OT
docker compose up -d postgres redis rabbitmq
sleep 20
zcat /tmp/erp_ot_rehearsal.sql.gz | docker exec -i erp-ot-postgres psql -U erp_user -d erp_ot -q
docker exec erp-ot-postgres psql -U erp_user -d erp_ot -tAc "SELECT count(*) FROM public.auth_users"
```
✅ 확인: 직원 수(수십 명)가 나와야 함. **주의: 복원은 반드시 `--clean` 덤프로** — 아니면 seed 데이터가 실데이터를 덮는 사고(.42 구축 때 확인된 함정)

### 5-2. 전체 스택 빌드·기동 (첫 빌드는 30분 이상 걸릴 수 있음)

```bash
# 새서버$
cd /home/oceantech/ERP_OT && docker compose build && docker compose up -d
cd /home/oceantech/ot-brain && docker compose -f docker-compose.yml -f docker-compose.gpu.yml up -d --build
cd /home/oceantech/pims && docker compose -f docker-compose.prod.yml up -d --build
docker ps --format 'table {{.Names}}\t{{.Status}}' | sort
```
✅ 확인: 컨테이너 20여 개가 `Up` (일부 `healthy`까지 1~2분). `Restarting` 반복이 있으면 `docker logs <이름> --tail 50` 결과를 보고

### 5-3. 브라우저 스모크 테스트

사무실 PC 브라우저에서:
- `http://192.168.0.43:3000` → ERP 로그인 → 대시보드·프로젝트 목록·근태 화면
- NAS 검색 한 건 실행 (GPU 리랭커 동작 확인)

```bash
# 새서버$  — GPU가 실제로 쓰였는지
nvidia-smi | grep -E 'MiB|%'
```
✅ 확인: 검색 직후 GPU 메모리 사용량이 0이 아니어야 함

### 5-4. cron 미리 심어두기 (전부 주석 상태로!)

```bash
# 구서버$
crontab -l | ssh oceantech@192.168.0.43 "sed 's/^/#MIGRATE# /' | crontab -"
```

```bash
# 새서버$
crontab -l | head -3
```
✅ 확인: 모든 줄이 `#MIGRATE# `로 시작해야 함 (컷오버 전에 이중 실행되면 안 되므로)

## Phase 6 — 컷오버 (평일 야간, 다운타임 시작)

> 소요 목표 30분~1시간. 시작 전 사내 공지. **아래 순서를 바꾸지 말 것.**

### 6-1. 구서버 정지

```bash
# 구서버$  — cron 먼저 끈다 (백업·동기화가 중간에 돌면 안 됨)
crontab -l > ~/crontab.backup.txt && crontab -r
cd /home/oceantech/ERP_OT && docker compose stop
cd /home/oceantech/ot-brain && docker compose stop
cd /home/oceantech/pims && docker compose -f docker-compose.prod.yml stop
docker ps
```
✅ 확인: `docker ps` 결과에 **postgres 3개만** 남기고 없어야 함 → postgres도 앱이 다 죽은 뒤 덤프 뜨고 정지할 것이므로 일단 둔다

### 6-2. 최종 데이터 이관

```bash
# 구서버$  ① DB 3개 최종 덤프 → 새서버로
```
