# 사내 전산실 서버 이관 가이드 (Ubuntu)

> **목적**: 현재 Windows 개발 PC에서 운영 중인 ERP-OT 플랫폼 + OT-Brain(지식플랫폼)을
> **사내 전산실의 Linux(Ubuntu) 서버**로 이관합니다.
> 대상 서버는 **완전히 포맷 후 OS부터 새로 설치**합니다.
>
> **작성일**: 2026-06-12
> **난이도**: 초보자도 따라 할 수 있도록 모든 명령어를 그대로 복사–붙여넣기 형태로 제공합니다.

---

## 0. 이번 이관의 조건 (먼저 확인)

| 항목 | 결정 내용 |
|------|-----------|
| **GPU** | ❌ 없음 (CPU 전용) → 검색/RAG는 동작하지만 느릴 수 있음. GPU 설정 제거 필요 |
| **이관 범위** | ✅ ERP_OT + ot-brain **둘 다** |
| **데이터** | ✅ 기존 데이터 이전 (DB + 업로드 파일 모두) — production AS관리 데이터 포함 |
| **접속 방식** | ✅ 사내 LAN 공유 (고정 IP) → 여러 직원이 `http://서버IP:3000` 으로 접속 |

> 💡 **NAS는 이관 대상이 아닙니다.** 사내 NAS(Synology, 192.168.0.220)는 별도 물리 장비로
> 그대로 둡니다. 새 우분투 서버는 네트워크로 NAS에 **접근만** 합니다.

## 0-1. 대상 서버 하드웨어 (Dell PowerEdge R730)

| 항목 | 사양 | 이관 관점 메모 |
|------|------|----------------|
| 모델 | Dell PowerEdge R730 | 서버급 — 적합 |
| 현재 OS | **Windows Server 2012 R2** | ⚠️ 포맷 대상 — 아래 경고 필독 |
| CPU | Xeon E5-2670 v3 (12C/24T) | 24스레드 — 컨테이너 다수 + CPU 검색에 충분 |
| RAM | 32GB | ERP 16컨테이너 + ot-brain 충분. CPU LLM은 느림(아래) |
| BIOS | **Legacy 모드** | Ubuntu도 **Legacy(BIOS) 모드로 설치** (UEFI 전환 불필요) |
| 디스크 0 | 278.88GB (MBR) | → **Ubuntu OS(`/`)** 설치용 |
| 디스크 1 | 3.27TB (GPT) | → **데이터 디스크(`/data`)** — DB·볼륨·백업·코드 |
| RAID | Dell PERC H730 Mini | Ubuntu 24.04 기본 드라이버(megaraid_sas) 지원 ✅ |

### ⛔ 포맷 전 반드시 확인 — 기존 Windows Server에 보존할 데이터가 없는지

이 서버는 **현재 Windows Server 2012 R2가 운영 중**입니다. 포맷하면 그 안의 모든 것이 사라집니다.
포맷을 시작하기 전에 **전산 담당자와 함께** 아래를 확인하세요:

- [ ] 이 서버가 현재 다른 사내 서비스(파일공유·DB·인증·업무앱 등)를 돌리고 있지 않은가?
- [ ] 디스크 0/1 안에 보존해야 할 회사 데이터가 없는가? (있으면 먼저 백업)
- [ ] 이 서버를 포맷해도 된다는 **명시적 승인**을 받았는가?

> 위 3개가 모두 ✓ 가 아니면 **절대 포맷하지 마세요.** 운영 중인 서버일 수 있습니다.

### 디스크 배치 전략 (권장)

```
디스크 0 (278GB, MBR)   →  Ubuntu OS  ( /         )   ← 부팅·시스템
디스크 1 (3.27TB, GPT)  →  데이터     ( /data     )   ← Docker 볼륨(DB·모델·업로드) + 백업 + 코드
```

이렇게 나누는 이유: **DB·검색모델·업로드·백업은 시간이 지나며 커집니다.** 278GB OS 디스크에 두면
금방 가득 차므로, **무거운 데이터는 전부 3.27TB 디스크로** 보냅니다(3부에서 Docker 데이터 경로 이전).

### PERC H730 RAID 메모

- 위 "디스크 0/1"은 이미 PERC가 만들어 둔 **가상 디스크(Virtual Disk)** 로 보입니다.
  기존 구성을 **그대로 사용**하는 것이 가장 간단합니다(설치 시 Ubuntu가 두 디스크로 인식).
- 만약 RAID 구성을 새로 하려면(예: 데이터 디스크 이중화), 부팅 시 **iDRAC** 또는
  `Ctrl+R`(PERC 설정 화면)에서 진행해야 하며, 이는 별도 검토가 필요합니다. 잘 모르면 기존 구성 유지.

### 💡 iDRAC 원격 콘솔 활용

R730에는 **iDRAC**(원격 관리 카드)가 있습니다. 모니터·키보드를 서버에 직접 연결하기 어렵다면
iDRAC 웹 콘솔(Virtual Console)로 **원격에서 OS 설치 화면까지** 조작할 수 있습니다.
iDRAC IP는 부팅 시 화면 또는 전산 담당자에게 확인하세요.

## 전체 흐름 한눈에 보기

```
[1] 현재 PC (포맷 전!)          [2] 새 우분투 서버              [3] 데이터 복원 & 가동
─────────────────────         ─────────────────────────      ─────────────────────────
1. 코드 GitHub push 확인   →   4. Ubuntu Server 설치       →   9.  DB 복원 (ERP + ot-brain)
2. DB 백업 (2개)               5. 고정 IP / 사용자 / 방화벽       10. 업로드 파일 복원
3. 업로드파일·.env 백업        6. Docker 설치                    11. 동작 확인 체크리스트
   → USB/외장하드로 복사       7. 코드 clone (2개 repo)          12. 자동시작·백업 설정
                               8. CPU 설정 + .env 작성           13. (별도) NAS 검색 연동
```

> ⛔ **가장 중요한 원칙**: **현재 PC를 포맷하기 전에 1~3단계 백업을 반드시 끝내고,
> 백업 파일을 USB/외장하드 등 "포맷되지 않는 다른 곳"에 복사**해야 합니다.
> 포맷하면 모든 게 사라집니다.

# 1부 — 현재 PC에서 (포맷 전 백업)

> 이 작업은 **지금 쓰던 Windows PC**에서 합니다. 모든 명령어는 **PowerShell**에서 실행합니다.
> 시작 메뉴 → "PowerShell" 검색 → 실행.

## 1-1. 컨테이너가 켜져 있는지 확인

```powershell
cd E:\claude\ERP_OT
docker compose ps
```

`erp-ot-postgres`, `ot-postgres` 등이 `running` 상태여야 백업할 수 있습니다.
꺼져 있다면 먼저 켭니다:

```powershell
cd E:\claude\ERP_OT
docker compose up -d
cd E:\claude\ot-brain
docker compose up -d
```

## 1-2. 코드가 GitHub에 모두 올라가 있는지 확인

```powershell
cd E:\claude\ERP_OT
git status
git push

cd E:\claude\ot-brain
git status
git push
```

> ⚠️ **중요 — 커밋 안 된 변경사항 주의**
> 메모리 기록상 **ot-brain에는 아직 커밋하지 않은 작업 변경분**이 있을 수 있습니다
> (otbrain:// 프로토콜 폐기 관련, "커밋 금지" 상태로 대기 중).
> `git status` 에 빨간색 파일이 보이면 **그 파일들이 포맷과 함께 사라집니다.**
> 다음 중 하나로 보존하세요:
> 1. (권장) 작업 트리 통째로 백업 폴더에 복사 — 아래 1-5에서 repo 폴더 전체를 백업하면 자동 포함
> 2. 또는 패치로 저장: `git diff > E:\이관백업\otbrain-uncommitted.patch`
>
> **판단이 필요하면 멈추고 사용자(본인)에게 확인하세요.** 임의로 commit 하지 마세요.

## 1-3. 데이터베이스 백업 (2개 — ERP, ot-brain)

백업을 모아둘 폴더를 먼저 만듭니다:

```powershell
mkdir E:\이관백업
mkdir E:\이관백업\db
mkdir E:\이관백업\uploads
```

**(A) ERP 데이터베이스** (`erp_ot`):

```powershell
docker exec erp-ot-postgres pg_dump -U erp_user -d erp_ot --no-owner --no-privileges | Out-File -Encoding utf8 E:\이관백업\db\erp_ot.sql
```

**(B) ot-brain 데이터베이스** (`otbrain`):

```powershell
docker exec ot-postgres pg_dump -U otbrain -d otbrain --no-owner --no-privileges | Out-File -Encoding utf8 E:\이관백업\db\otbrain.sql
```

> ℹ️ `Out-File -Encoding utf8` 은 PowerShell에서 한글 깨짐·인코딩 문제를 막기 위한 것입니다.

백업이 제대로 됐는지 파일 크기를 확인합니다 (0KB면 실패):

```powershell
dir E:\이관백업\db
```

## 1-4. 업로드 파일(첨부) 백업

ERP는 첨부파일을 Docker "볼륨"에 저장합니다. 볼륨은 폴더처럼 직접 복사가 안 되므로,
임시 도우미 컨테이너로 압축해서 꺼냅니다. **아래 5줄을 그대로** 실행하세요:

```powershell
docker run --rm -v erp_ot_auth_uploads:/data -v E:\이관백업\uploads:/backup alpine tar czf /backup/auth_uploads.tar.gz -C /data .
docker run --rm -v erp_ot_approval_uploads:/data -v E:\이관백업\uploads:/backup alpine tar czf /backup/approval_uploads.tar.gz -C /data .
docker run --rm -v erp_ot_expense_uploads:/data -v E:\이관백업\uploads:/backup alpine tar czf /backup/expense_uploads.tar.gz -C /data .
docker run --rm -v erp_ot_ocr_uploads:/data -v E:\이관백업\uploads:/backup alpine tar czf /backup/ocr_uploads.tar.gz -C /data .
docker run --rm -v erp_ot_file_storage:/data -v E:\이관백업\uploads:/backup alpine tar czf /backup/file_storage.tar.gz -C /data .
```

> ℹ️ 볼륨 이름은 `erp_ot_` 로 시작합니다(compose 프로젝트명). 이름이 다르면
> `docker volume ls` 로 실제 이름을 확인하세요.
> `ollama_data`(검색 모델), `ocr_models`(OCR 모델)는 **백업하지 않아도 됩니다** —
> 새 서버에서 자동으로 다시 내려받습니다.

## 1-5. .env 파일과 코드 폴더 백업 (시크릿 보존)

`.env` 파일에는 비밀번호·JWT 시크릿이 들어 있습니다. **git에는 올라가지 않으므로 따로 백업**해야
기존 데이터(암호화된 토큰 등)와 호환됩니다.

```powershell
copy E:\claude\ERP_OT\.env E:\이관백업\erp_ot.env
copy E:\claude\ot-brain\.env E:\이관백업\otbrain.env
```

또한 안전을 위해 **두 repo 폴더 전체**도 백업해 두면(커밋 안 된 변경분까지 포함) 가장 안전합니다:

```powershell
robocopy E:\claude\ERP_OT E:\이관백업\code\ERP_OT /E /XD node_modules .next .git\lfs
robocopy E:\claude\ot-brain E:\이관백업\code\ot-brain /E /XD node_modules .next .git\lfs
```

- [ ] `git push` 완료 (양쪽 repo)

- [ ] `erp_ot.env`, `otbrain.env` 백업됨

# 2부 — 새 우분투 서버 설치

## 2-1. Ubuntu Server 설치

1. **Ubuntu Server 24.04 LTS** 설치 USB를 만듭니다.
   - 다운로드: https://ubuntu.com/download/server
   - USB 만들기 도구: [Rufus](https://rufus.ie) (Windows에서)
2. 서버를 USB로 부팅 → 설치 진행.
   - ⚠️ **부팅은 Legacy(BIOS) 모드로** 합니다. R730이 이미 Legacy 모드이므로 BIOS 설정은
     건드리지 말고, 부팅 메뉴(F11)에서 USB를 **"BIOS"/"Legacy"** 항목으로 선택해 부팅하세요
     (UEFI 항목 아님). 디스크 0이 MBR이라 Legacy로 설치해야 깔끔합니다.
3. 설치 중 화면 안내:
   - 언어: English (권장 — 터미널 한글 깨짐 예방)
   - **디스크(Storage) — "Custom storage layout" 선택** (디스크 2개를 역할대로 나누기 위해):
     - **디스크 0 (278GB)** → 여기에 OS 설치.
       - 파티션 테이블이 남아 있으면 reformat. 전체를 `ext4`, 마운트 `/` 로 지정.
       - 이 디스크가 **부팅 장치(boot device)** 가 되도록 합니다(설치 마법사가 자동 처리).
     - **디스크 1 (3.27TB)** → 데이터용.
       - 기존 파티션 삭제 후 전체를 `ext4`, **마운트 `/data`** 로 지정.
     - swap은 만들지 않아도 됩니다(RAM 32GB로 충분). 필요 시 설치 후 swapfile로 추가 가능.
   - Profile 설정: 이름/서버이름/사용자ID/비밀번호 입력
     - 예) name: `oceantech`, server name: `erp-server`, username: `oceantech`
   - **"Install OpenSSH server" 체크** ✅ (원격 접속에 필요 — 매우 중요)
   - Featured server snaps: 아무것도 선택 안 함 (Docker는 뒤에서 직접 설치)
4. 설치 완료 후 재부팅 → USB 제거.
   - 재부팅 시 **부팅 순서 1번이 디스크 0(278GB)** 인지 확인하세요(BIOS Boot 메뉴). 아니면
     Windows 부팅 잔재나 데이터 디스크로 부팅을 시도할 수 있습니다.

설치 후 `/data` 가 제대로 잡혔는지 확인:
```bash
df -h /data
```
3.27TB 용량이 보이면 정상입니다. 안 보이면 12부 문제해결을 참고하세요.

설치가 끝나면 서버 화면에 로그인 프롬프트가 뜹니다. 위에서 만든 ID/비밀번호로 로그인합니다.

## 2-2. (편하게) 다른 PC에서 원격 접속

서버 앞에서 직접 타이핑해도 되지만, **본인 PC에서 SSH로 접속하면 복사–붙여넣기가 쉬워** 권장합니다.

서버에서 IP 확인:
```bash
ip addr show
```
`192.168.0.xxx` 같은 주소를 찾습니다 (이게 임시 IP — 2-4에서 고정으로 바꿉니다).

본인 Windows PC의 PowerShell에서:
```powershell
ssh oceantech@192.168.0.xxx
```
(비밀번호 입력 후 접속) — 이제부터 명령어는 이 SSH 창에 붙여넣으면 됩니다.

## 2-4. 고정 IP 설정 (사내 LAN 공유의 핵심)

여러 직원이 항상 같은 주소로 접속하려면 서버 IP가 **고정**되어야 합니다.

먼저 네트워크 인터페이스 이름을 확인합니다 (보통 `enp3s0`, `eth0` 등):
```bash
ip addr show
```

설정 파일을 엽니다:
```bash
sudo nano /etc/netplan/50-cloud-init.yaml
```
> 파일명이 다를 수 있습니다: `ls /etc/netplan/` 로 확인 후 그 파일을 여세요.

아래처럼 작성합니다 (★ 표시는 사내망에 맞게 수정):

```yaml
network:
  version: 2
  ethernets:
    enp3s0:                      # ★ 위에서 확인한 인터페이스 이름
      dhcp4: no
      addresses:
        - 192.168.0.50/24        # ★ 서버에 줄 고정 IP (사내 IP 대역에 맞게)
      routes:
        - to: default
          via: 192.168.0.1       # ★ 공유기/게이트웨이 주소
      nameservers:
        addresses: [8.8.8.8, 1.1.1.1]
```

> ⚠️ 들여쓰기(띄어쓰기)가 매우 중요합니다. **Tab 금지, 스페이스만** 사용하세요.
> 저장: `Ctrl+O` → `Enter`, 종료: `Ctrl+X`.

적용:
```bash
sudo netplan apply
```

> 고정 IP는 **공유기 관리자나 전산 담당자와 협의**해 이미 다른 장비가 쓰지 않는 주소로 정하세요.
> 이 가이드에서는 예시로 **`192.168.0.50`** 을 서버 IP로 사용합니다. 본인 값으로 바꿔 읽으세요.

다시 SSH로 접속할 때는 고정 IP로 접속합니다:
```powershell
ssh oceantech@192.168.0.50
```

## 2-5. 방화벽 설정 (직원 접속 허용)

```bash
sudo ufw allow 22/tcp      # SSH (원격 관리)
sudo ufw allow 3000/tcp    # ERP 웹 (직원 접속 — 이 포트만 외부 공개)
sudo ufw enable
sudo ufw status
```

> 💡 백엔드 서비스(3001~3008)와 ot-brain(3100/5433 등)은 **외부에 열지 않습니다.**
> 직원 브라우저는 오직 **3000번 포트(웹)** 으로만 접속하고, 웹 서버가 내부에서 나머지를 중계합니다.
> (보안상 최소 노출 원칙)

## 3-1. Docker 설치

**그 다음 한 번 로그아웃 후 다시 SSH 접속**해야 권한이 적용됩니다. 재접속 후 확인:

```bash
docker --version
docker compose version
```

## 3-1b. ⭐ Docker 데이터 경로를 /data(3.27TB)로 이전 (필수)

Docker는 기본적으로 모든 데이터(DB·검색모델·업로드 볼륨)를 OS 디스크의 `/var/lib/docker` 에
저장합니다. 그대로 두면 **278GB OS 디스크가 가득 차서 서버가 멈출 수 있습니다.**
**아직 컨테이너를 하나도 띄우기 전**인 지금, 저장 위치를 3.27TB 디스크로 옮깁니다.

```bash
# 1) Docker 잠시 중지
sudo systemctl stop docker docker.socket

# 2) 데이터 폴더 생성
sudo mkdir -p /data/docker

# 3) Docker 설정에 데이터 경로 지정
sudo nano /etc/docker/daemon.json
```
파일에 아래 내용을 그대로 입력하고 저장(`Ctrl+O`→`Enter`, `Ctrl+X`):
```json
{
  "data-root": "/data/docker"
}
```
```bash
# 4) Docker 다시 시작
sudo systemctl start docker

# 5) 경로가 바뀌었는지 확인
docker info | grep "Docker Root Dir"
```
출력이 `Docker Root Dir: /data/docker` 이면 성공입니다.

> 💡 이렇게 하면 PostgreSQL 데이터, ollama/OCR 모델, 모든 업로드 볼륨이 전부 3.27TB 디스크에
> 저장됩니다. 코드(repo)는 작아서 어디 둬도 되지만, 아래 3-3에서 **코드와 백업도 /data**에 둡니다.

## 3-2. Node.js 설치 (호스트용 — nas-file 서버에만 사용)

## 3-3. 코드 가져오기 (git clone)

홈 폴더에 두 repo를 내려받습니다.

> ⚠️ **폴더 이름 주의**: ot-brain이 ERP의 Docker 네트워크(`erp_ot_erp-network`)를 찾으려면

# 4부 — CPU 전용 설정 (GPU 제거)

현재 설정은 NVIDIA GPU를 사용하도록 되어 있어, **GPU가 없는 서버에서는 그대로 두면 기동에 실패**합니다.
GPU 예약 부분을 제거합니다.

## 4-1. ERP의 ollama GPU 설정 제거

```bash
nano ~/ERP_OT/docker-compose.yml
```
`ollama:` 서비스 안에서 아래 블록(7줄)을 찾아 **삭제하거나 맨 앞에 `#`을 붙여 주석 처리**합니다:

```yaml
    deploy:                          # ← 이 줄부터
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]    # ← 이 줄까지 삭제/주석
```
저장: `Ctrl+O` → `Enter`, 종료: `Ctrl+X`.

> 검색 임베딩 모델(bge-m3)은 CPU로도 잘 돌아갑니다. 다만 검색 첫 호출이 조금 느릴 수 있습니다.

## 4-2. ot-brain의 reranker GPU 설정 제거

```bash
nano ~/ot-brain/docker-compose.yml
```
`ot-reranker:` 서비스 안의 동일한 `deploy:` ~ `capabilities: [gpu]` 블록을 삭제/주석 처리합니다.

저장 후 종료.

> ⚠️ **CPU에서 reranker는 느립니다.** 검색 재정렬(reranking)이 타임아웃(5초)을 넘으면
> 시스템이 자동으로 재정렬을 건너뛰고 기본 검색 결과를 보여줍니다(회로차단기). 즉 **검색은 동작하되
> 정렬 품질이 약간 낮아질 수 있습니다.** 사용량을 보고 추후 GPU 도입 시 되돌리면 됩니다.

## 4-3. CPU/GPU 환경 분리 — base는 CPU 기본, GPU는 override 파일

> 📌 **환경별 차이는 `docker compose` override 파일로 분리합니다.** 그래야 사내 CPU 서버와
> GPU 개발 PC가 같은 git 저장소를 써도 서로의 설정을 덮어쓰지 않습니다.

- **base** (`docker-compose.yml`) = **GPU 없이도 뜨는 CPU-safe 기본값**. 사내 서버는 그냥:
  ```bash
  cd ~/ERP_OT   && docker compose up -d
  cd ~/ot-brain && docker compose up -d
  ```
- **GPU 머신**(드라이버+툴킷 있는 개발 PC 등)은 **`docker-compose.gpu.yml` override를 추가**해서 기동:
  ```bash
  cd ~/ERP_OT   && docker compose -f docker-compose.yml -f docker-compose.gpu.yml up -d
  cd ~/ot-brain && docker compose -f docker-compose.yml -f docker-compose.gpu.yml up -d
  ```
  > `docker-compose.gpu.yml`은 ollama/reranker에 GPU 예약을 다시 얹고, ot-brain의
  > `RERANK_CAP`을 60(CPU 기본 12)으로 되돌립니다. 파일 주석에 설명이 있습니다.

**나중에 이 사내 서버에 GPU를 장착한다면** — 드라이버/툴킷부터 설치한 뒤 위 GPU 기동 명령을 쓰면 됩니다:
```bash
nvidia-smi                                       # GPU 목록이 보여야 함
sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
docker run --rm --gpus all ubuntu nvidia-smi     # 컨테이너에서 GPU 인식 확인
```
설치 후엔 base에 GPU 블록을 손으로 넣지 말고 **`-f docker-compose.gpu.yml`만 추가**하세요.
> 기동 로그에 더 이상 `NVIDIA Driver was not detected` 경고가 없으면 GPU 모드로 전환된 것입니다.

# 5부 — 환경변수(.env) 작성

## 5-1. ERP의 .env

가장 안전한 방법은 **1-5에서 백업한 기존 `.env`를 그대로 사용**하는 것입니다
(JWT 시크릿이 동일해야 기존 토큰/암호화 데이터와 호환).

백업한 `erp_ot.env`를 서버로 옮긴 뒤:
```bash
cp ~/erp_ot.env ~/ERP_OT/.env     # 백업 파일을 서버 홈에 올려둔 경우
nano ~/ERP_OT/.env
```

> 📌 **파일을 서버로 옮기는 방법** (본인 PC PowerShell에서):
> ```powershell
> scp E:\이관백업\erp_ot.env oceantech@192.168.0.50:~/erp_ot.env
> scp E:\이관백업\otbrain.env oceantech@192.168.0.50:~/otbrain.env
> ```

`.env` 에서 **LAN 공유에 맞게 아래 항목을 수정/추가**합니다:

```env
# compose 프로젝트명 고정 (볼륨/네트워크 이름을 erp_ot_* 로 통일 — ot-brain 연동에 필요)
COMPOSE_PROJECT_NAME=erp_ot

# 직원들이 접속할 서버 주소로 변경 (★ 본인 고정 IP)
CORS_ORIGIN=http://192.168.0.50:3000
```

> 그 외 비밀번호·시크릿 항목은 **기존 백업값 그대로 두세요.** 새로 바꾸면 기존 암호화 데이터와
> 어긋날 수 있습니다.

## 5-2. ot-brain의 .env

```bash
cp ~/otbrain.env ~/ot-brain/.env
nano ~/ot-brain/.env
```
기존 값(특히 `POSTGRES_PASSWORD`, `OTBRAIN_AUTH_SECRET`)을 그대로 유지합니다.

> ⚠️ **CPU 전용 환경의 RAG(답변 생성) 주의**
> ot-brain은 검색 결과로 답변을 생성할 때 로컬 LLM(`qwen2.5:7b`)을 쓸 수 있습니다.
> 이 모델은 **GPU 없이 CPU로 돌리면 매우 느립니다(한 답변에 수십 초~수 분).**
> 따라서 이관 초기에는 `.env` 에서 아래처럼 **검색 전용 모드**를 권장합니다:
> ```env
> LLM_MODE=search-only
> ```
> 이러면 "AI 답변 생성"은 끄고 **검색·문서 찾기 기능만** 빠르게 동작합니다.
> 나중에 GPU를 도입하면 `local`/`hybrid` 로 바꾸면 됩니다. (`search-only` 면 qwen 모델을
> 내려받을 필요도 없습니다.)

# 6부 — ERP 가동 + 데이터 복원

> 순서가 중요합니다: **① DB만 먼저 켜기 → ② DB 복원 → ③ 나머지 전체 켜기 → ④ 업로드 복원**.
> 이렇게 해야 기존 데이터 위에 깨끗하게 복원됩니다.

## 6-1. DB 컨테이너만 먼저 기동

```bash
cd ~/ERP_OT
docker compose up -d postgres
```
30초쯤 기다린 뒤 준비 상태 확인:
```bash
docker compose logs postgres | tail -n 20
```
`database system is ready to accept connections` 가 보이면 준비 완료입니다.

## 6-2. ERP DB 복원

먼저 1-3에서 만든 `erp_ot.sql`을 서버로 옮깁니다 (본인 PC PowerShell):
```powershell
scp E:\이관백업\db\erp_ot.sql oceantech@192.168.0.50:~/erp_ot.sql
```

서버에서 복원:
```bash
cat ~/erp_ot.sql | docker exec -i erp-ot-postgres psql -U erp_user -d erp_ot
```
> 진행 중 `CREATE TABLE`, `COPY` 등이 주르륵 출력됩니다.
> `already exists` 같은 경고가 일부 나와도 대부분 정상입니다(확장/기본 객체 중복).


상태 확인:
```bash
docker compose ps
```
모든 서비스가 `running`(또는 ocr-engine은 `healthy`)이면 정상입니다.

## 6-4. 업로드 파일(첨부) 복원

1-4의 tar.gz 5개를 서버로 옮깁니다 (본인 PC PowerShell):


서버에서 각 볼륨으로 복원합니다 (도우미 컨테이너 사용):
```bash
docker run --rm -v erp_ot_auth_uploads:/data -v ~/:/backup alpine sh -c "cd /data && tar xzf /backup/auth_uploads.tar.gz"
docker run --rm -v erp_ot_approval_uploads:/data -v ~/:/backup alpine sh -c "cd /data && tar xzf /backup/approval_uploads.tar.gz"
docker run --rm -v erp_ot_expense_uploads:/data -v ~/:/backup alpine sh -c "cd /data && tar xzf /backup/expense_uploads.tar.gz"
```

> 볼륨은 컨테이너가 한 번 떠야 생성됩니다.

# 8부 — 자동 시작 설정 (재부팅 대비)

서버는 정전·재부팅 후에도 **자동으로 다시 떠야** 합니다.

## 8-1. Docker 컨테이너 자동 시작

좋은 소식: docker-compose에 이미 `restart: unless-stopped` 가 설정되어 있어,
**Docker 데몬이 부팅 시 자동 시작되면 컨테이너도 자동으로 복구**됩니다.
Docker 자체의 부팅 자동시작만 켭니다:

```bash
sudo systemctl enable docker
```

확인 (서버 재부팅 테스트):
```bash
sudo reboot
```
재접속 후:
```bash
docker ps
```
컨테이너들이 저절로 떠 있으면 성공입니다.

| 재부팅 복구 | `sudo reboot` 후 `docker ps` | 자동 기동 |

> 서버 안의 백업도 함께 사라집니다.

# 11부 — 문제 해결 (자주 묻는 것)

**Q. `docker compose up` 에서 `could not select device driver "nvidia"` 오류**
→ 4부(GPU 제거)를 안 했거나 빠뜨린 것입니다. ollama / ot-reranker의 `deploy:` 블록을 다시 확인하세요.

**Q. 직원 PC에서 `http://192.168.0.50:3000` 접속이 안 됨**
→ ① 방화벽: `sudo ufw status` 에 `3000/tcp ALLOW` 있는지 확인.
→ ② 서버 IP가 맞는지: 서버에서 `ip addr show`.
→ ③ 같은 사내망(같은 공유기)에 있는지 확인.

**Q. 로그인은 되는데 데이터가 비어 있음**
→ DB 복원(6-2)이 빠졌거나 실패한 것입니다. `docker exec -it erp-ot-postgres psql -U erp_user -d erp_ot -c "\dt"` 로 테이블이 있는지 확인 후 다시 복원하세요.

**Q. 첨부파일이 안 열림 (404)**
→ 업로드 볼륨 복원(6-4)이 빠진 것입니다. tar 복원 명령을 다시 실행하세요.

**Q. 서비스가 계속 재시작됨 (`Restarting`)**
→ `docker compose logs -f <서비스명>` 으로 원인 확인. 보통 DB가 아직 준비 안 됐거나 `.env`
값 누락입니다. 1~2분 기다려도 안 되면 로그를 보세요.

**Q. 한글 파일명/데이터가 깨짐**
→ DB 백업 시 인코딩 문제일 수 있습니다. 1-3을 `Out-File -Encoding utf8` 로 다시 받으세요.

**Q. `df -h /data` 에 3.27TB가 안 보임 (설치 후)**
→ 설치 때 디스크 1을 `/data` 로 마운트 지정하지 않은 것입니다. 수동 마운트:
```bash
lsblk                              # 3.27TB 디스크 이름 확인 (예: sdb)
sudo mkfs.ext4 /dev/sdb1           # ⚠️ 비어 있는 데이터 디스크에만! (OS 디스크 절대 금지)
sudo mkdir -p /data
echo "/dev/sdb1 /data ext4 defaults 0 2" | sudo tee -a /etc/fstab
sudo mount -a
df -h /data
```
→ 이 작업은 **반드시 Docker를 띄우기 전(3-1b 전)** 에 해야 합니다. 이미 컨테이너를 띄운 뒤라면
순서가 꼬이므로, 헷갈리면 멈추고 확인을 요청하세요.

**Q. 부팅이 안 되거나 Windows로 부팅됨**
→ BIOS 부팅 순서에서 **디스크 0(278GB, Ubuntu 설치한 디스크)** 를 1순위로 올리세요(부팅 시 F2/F11).
Legacy 모드인지도 확인합니다.

**Q. 디스크 용량이 빠르게 참 (`df -h /` 가 가득)**
→ Docker 데이터 경로 이전(3-1b)이 빠진 것입니다. `docker info | grep "Docker Root Dir"` 가
`/data/docker` 인지 확인하세요. 아니면 3-1b를 다시 수행합니다(컨테이너를 내린 뒤).

# 13부 — ⚠️ NAS 파일 열기(검색결과) 연동 — **추가 검토 필요**

> **이 부분은 위 1~12부와 분리해 마지막에 진행하길 권장합니다.**
> ERP 통합검색에서 NAS 파일을 "열기/미리보기" 하는 `nas-file` 서버(:3105)는
> **현재 Windows 전용으로 작성**되어 있어, 우분투에서는 그대로 동작하지 않습니다.

## 무엇이 문제인가

`services/nas-file/server.mjs` 는:
1. NAS 경로를 **Windows UNC 형식**(`\\192.168.0.220\oceantech`)으로 다루고, 경로를 **백슬래시(`\`)로 정규화**합니다. → 리눅스 경로(`/mnt/nas/...`)와 맞지 않습니다.
2. NAS SMB 인증을 **"Windows 사용자 세션"** 에 의존합니다(코드 주석에 명시). → 리눅스에선 별도 **CIFS 마운트 + 자격증명**이 필요합니다.
3. ERP 검색 인덱스(DB)에 저장된 파일 경로도 **Windows UNC 형식**일 가능성이 큽니다. → 리눅스 마운트 경로로의 **변환 규칙**이 필요합니다.

## 권장 진행 방식 (택1)

**방식 A — 우분투로 완전 이전 (코드 수정 필요)**
1. NAS를 CIFS로 마운트:
   ```bash
   sudo apt install -y cifs-utils
   sudo mkdir -p /mnt/nas/oceantech
   # 자격증명 파일 생성 (NAS 계정)
   sudo nano /etc/nas-credentials      # username=... / password=... 두 줄
   sudo chmod 600 /etc/nas-credentials
   # /etc/fstab 에 추가 (부팅 시 자동 마운트)
   #   //192.168.0.220/oceantech /mnt/nas/oceantech cifs credentials=/etc/nas-credentials,uid=1000,iocharset=utf8 0 0
   sudo mount -a
   ```
2. `server.mjs` 의 경로 처리(백슬래시 정규화·`NAS_ROOT`)를 **리눅스 경로 기준으로 수정**.
3. ERP 검색결과의 저장 경로(UNC) → 마운트 경로 변환 로직 점검.
4. systemd 서비스로 상시 구동:
   ```bash
   sudo nano /etc/systemd/system/nas-file.service
   ```
   ```ini
   [Unit]
   Description=OT-Brain NAS file server
   After=network-online.target mnt-nas-oceantech.mount

   [Service]
   Environment=NAS_FILE_PORT=3105
   Environment=NAS_ROOT=/mnt/nas/oceantech
   ExecStart=/usr/bin/node /home/oceantech/ot-brain/services/nas-file/server.mjs
   Restart=always
   User=oceantech

   [Install]
   WantedBy=multi-user.target
   ```
   ```bash
   sudo systemctl daemon-reload
   sudo systemctl enable --now nas-file
   ```

**방식 B — 과도기적으로 Windows 1대 유지 (코드 수정 0)**
- nas-file 서버만 기존 Windows PC(또는 사내 Windows 1대)에서 계속 돌리고,
  ERP는 그 PC를 가리키게 둡니다. NAS 검색-열기 기능만 Windows에 남기고 나머지는 우분투로 이전.

> 💡 **방식 A는 코드 수정과 테스트가 필요하므로, 변경 전 검토안(영향·경로변환·테스트 범위)을
> 먼저 정리해 확인받은 뒤 진행하길 권장합니다.** 이관 1~12부(ERP 본체)가 안정화된 후 별도로
> 다루는 것이 안전합니다.
