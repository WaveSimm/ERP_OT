## 2. 조치 내역

### 2-1. 사진 폴더(촬영세션)별 묶음 + 결과수 50

- **문제**: 동일 날짜 한 폴더의 사진 수십 장이 결과 칸을 잠식 → 여러 현장을 작업해도 한 곳만 보임.
- **조치**: 이미지 그룹키를 파일명 → **폴더(촬영세션)**로 변경. 대표 1장 + `folderFiles`(nas_folder 전체 사진수)/`photosMatched` 노출. UI에 "📷 이 폴더 N장". 프론트 topK 20→50.
- **검증**: "축산항 사진" → 20개가 전부 다른 세션, 각 폴더 실제 사진수 정확.

### 2-2. 문서 본문 작업일 추출 → 연도검색 문서 확장

- **문제**: "감천항 2024년 작업일지"가 2016·2017 문서를 상위 노출. 문서엔 사진 `taken_at` 같은 날짜 필드가 없어 연도가 무시됨. 파일명/폴더 연도는 불안정(개인 백업폴더 "24년 바탕화면" 안에 실제론 2020/2023 작업).
- **조치**: 본문 청크에서 작업일 추출 → `knowledge.nas_document.doc_date`. 작업일지 "일자 및 시간" 라벨 우선 → 본문 첫 날짜 폴백, 잘못된 날짜는 NULL(`safe_make_date`). **164,895건** 부여. 검색의 후보(datematch)·점수(date_hit)·하드필터·감점을 `taken_at OR doc_date`로 확장, 출력에 `docDate`.
- **재현 스크립트**: `ot-brain/scripts/doc-date-extract.sql` (reconcile 후 재실행, idempotent, 본문 스캔 ~60분).
- **검증**: 감천항 2020/2017/2024년 작업일지 각각 해당 연도만 정확 노출. 사진·일반검색 회귀 없음.

### 2-3. 검색 속도 튜닝 (품질 무관 — 결과 동일)

- **① df(IDF 용어빈도)**: `term_freq`(188만) FILTER 풀스캔(515ms) → ≥3글자 토큰은 `WHERE ILIKE`로 trgm GIN(`idx_term_freq_term_trgm`) 개별조회(~수ms), ≤2글자 한국어("포항"·"관측")만 1회 풀스캔. df 합산값 동일.
  - 한계: 2글자 토큰은 trgm(3글자 단위) 인덱싱 불가 → 구조적으로 풀스캔 잔존.
- **② work_mem 4MB→32MB** (`ot-brain/scripts/db-tuning.sql`, `ALTER SYSTEM`): 날짜범위 검색 시 해당 연도 수십만 건 bitmap이 work_mem 초과로 lossy 전환되며 heap recheck 폭증. 32MB로 해소(16MB면 충분, 여유 부여). 서버 RAM 30GB·저동시성이라 OOM 위험 없음.
- **측정**:

  | 검색어 | 이전 | 이후 |
  |--------|-----:|-----:|
  | 용역계약서 pdf (df) | 543ms | 2ms |
  | 감천항 2024년 작업일지 (df) | 998ms | 6ms |
  | 포항 2021년 (mainsql) | 1647ms | 445ms |
  | 흥해 2022년 사진 | 3057ms | 2046ms |
  | 감천항 2024년 작업일지 | 4082ms | 2834ms |

### 2-4. dev GPU 리랭커 장애 진단·복구 (운영)

- **증상**: 검색이 느리고(`rerank` 콜드 ~4s 타임아웃) 랭킹 품질 저하. 진단 결과 **dev PC(192.168.182.176)의 리랭커 도커 컨테이너 중지**가 원인.
  - 네트워크 비대칭(dev→서버 OK, 서버→dev 무응답)으로 처음엔 방화벽 의심 → 실제론 컨테이너 중지(ICMP 무응답은 Windows 기본값).
- **조치**: dev PC에서 컨테이너 재기동 → 포트 3102/11434 OPEN, 검색 리랭킹 정상(웜 ~380ms) 복구.
- **재발 방지**: 컨테이너 `restart: unless-stopped`(이미 설정), Docker Desktop "로그인 시 자동시작" + Windows 자동로그인 또는 서비스 등록 필요. 근본적으론 GPU를 항상 켜진 서버로 이관(T4 예정).

### 2-5. 시기 0건 매칭 안내

- **문제**: "sm050 2026년"처럼 해당 시기 데이터가 없을 때, 날짜 없는 파일(이미지/바이너리)만 남아 **전 기간 결과를 그 연도 결과인 것처럼** 오인.
- **조치**: 날짜 질의인데 결과 중 그 시기(`taken_at`/`doc_date`) 매칭이 0건이면 서버가 `dateMatched=false` + `dateLabel` 반환 → 프론트가 "🗓 {연도}에 해당하는 결과가 없어 전 기간 결과를 표시합니다" 배너.
- **검증**: sm050 2026년→false(배너), 감천항 2020년 작업일지→true, 용역계약서 pdf(날짜없음)→null.

## 3. 검토(검증) 요청 처리

- **감천항 2024년 작업일지**: 연도필터 정상 동작 확인(2-2).
- **sm050 2026년**: **2026 데이터 없음** 확인. SM050 파랑계 유지보수 프로젝트 기간 ~2007–2020(사진 EXIF 최신 2020, 유지보수 폴더 2012–2017). 경로 속 "2026" 31건은 전부 `YYYYMMDDHHMMSS` 타임스탬프의 시·분·초 우연 일치(오탐) → 연도를 경로매칭에서 제외하는 설계가 옳음을 재확인.
