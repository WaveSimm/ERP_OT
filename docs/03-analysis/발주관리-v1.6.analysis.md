## 3. Gap 목록

### 🔴 High 우선순위 (3건) — Design 경로 정정 필수

#### H1. API Endpoint prefix 오기재

| 항목 | 값 |
|---|---|
| Design §5.1~5.3 명시 | `/api/v1/equipment/orders/:id/invoice` 등 9개 엔드포인트 |
| 실제 코드 | `services/equipment/src/index.ts:285` — `app.register(orderSettlementRoutes, { prefix: "/api/v1/procurement" })` |
| 실제 경로 | `/api/v1/procurement/orders/:id/invoice` |
| 영향 | Design 문서 보고 API 호출 시 404 |
| 조치 | Design §5 전체 prefix를 `/api/v1/procurement/...` 로 정정 |

#### H2. PO PDF 페이지 경로 오기재

| 항목 | 값 |
|---|---|
| Design §9.1 | `apps/web/src/app/procurement/orders/[id]/print/page.tsx` |
| 실제 파일 | `apps/web/src/app/procurement/orders/[id]/**po-print**/page.tsx` |
| 호출 코드 | `[id]/page.tsx:395` — `window.open(.../po-print)` |
| 조치 | Design §9.1 경로를 `po-print`로 정정 |

#### H3. 재무 접수 sub-tab 페이지 경로 오기재

| 항목 | 값 |
|---|---|
| Design §6.4 | `/procurement/settlements` (sub-tab 결재송금/발주송금) |
| 실제 구현 | `apps/web/src/app/procurement/**expenses**/page.tsx:26-47` 에 sub-tab 구현 |
| 추가 발견 | `/procurement/settlements` 는 별개 메뉴 (수입원가정산 ImportCostSettlement, 부대비용/공급가액 처리) |
| 영향 | Design 다이어그램이 두 별개 메뉴를 동일 경로로 표기 |
| 조치 | Design §6.4 경로를 `/procurement/expenses`로 정정 + `/procurement/settlements`(수입원가)와의 관계 명시 |

### 🟡 Medium 우선순위 (4건)

#### M1. OverseasOrder approverName 컬럼 — 스키마에 없음 (의도 vs 구현 불일치)

| 항목 | 값 |
|---|---|
| Design §3.1 | "v1.6 신규 컬럼 **6종**" — approverId/Name × 3 |
| 실제 스키마 | `prisma/schema.prisma:1143-1145` — ID 3개만 (`approverName` 없음) |
| 실제 동작 | 프론트엔드 form state로만 존재. 백엔드 저장은 ID만. 표시는 `userMap[approverId]` lookup |
| **의도 불일치** | Design에 명시한 "사용자 삭제 후 결재 히스토리 보존" 의도가 **달성되지 않음** |
| 조치 옵션 | (A) Design을 "ID 3 컬럼만"으로 정정 — 의도 포기 / (B) 코드에 Name × 3 컬럼 추가 마이그레이션 — 의도 보존 |

#### M2. Permissions 일괄 완화 미적용 (procurement.routes.ts)

| 항목 | 값 |
|---|---|
| Design §10 | "**모든 액션** OPERATOR 까지 허용" |
| 실제 코드 | OPERATOR 허용된 라우트 3개만: `POST /orders`, `POST /:id/cancel-submission`, `POST /:id/progress` |
| 미적용 라우트 12개 | PATCH /:id, transition, items 추가/수정, receive, link-inventory 등 — ADMIN/MANAGER만 |
| 영향 | OPERATOR가 발주 등록은 가능하지만 수정·상태전환·품목추가는 불가 |
| 조치 옵션 | (A) Design을 "신규 액션과 일부 핵심만 OPERATOR"로 정정 / (B) procurement.routes.ts 권한 일괄 완화 |

#### M3. 발주 목록 페이지 경로

| 항목 | 값 |
|---|---|
| Design §6.3 | `/procurement/orders` |
| 실제 파일 | `apps/web/src/app/procurement/orders/page.tsx` **존재하지 않음** |
| 실제 경로 | `apps/web/src/app/procurement/page.tsx` (procurement 루트) |
| 조치 | Design §6.3 경로를 `/procurement`로 정정 |

#### M4. overpaid 처리 정책 불일치

| 항목 | 값 |
|---|---|
| Design §5.3.1 | "actualAmount > payment.amount → 400 (실제 송금이 요청보다 많을 수 없음)" |
| 실제 코드 | `order-settlement.service.ts:193-195` — `// 더 보낸 케이스는 별도 처리 필요 — 일단 허용 (overpaid)` |
| 영향 | Design은 400 에러 명시, 실제는 허용. 정책 불일치 |
| 조치 옵션 | (A) Design을 "overpaid 허용"으로 정정 / (B) 코드에 400 throw 추가 |
