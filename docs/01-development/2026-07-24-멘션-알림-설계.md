# @멘션 알림 (게시글·덧글·이슈·작업일지) — 설계 문서

작성일: 2026-07-24 · 상태: 설계(구현 전)

## 배경 / 목적
게시글·덧글·이슈·작업일지 등에서 `@직원`으로 멘션하면 해당 직원에게 **알림**이 가도록 한다.

현황을 조사한 결과, **멘션 뼈대는 이미 존재하지만 사실상 휴면 상태**다:
- `Mention` 모델(project)·실시간 소켓(`mention:created`)·알림 벨 API(`/notifications`)·벨 UI(`AppLayout`, 30초 폴링)까지 이미 있다.
- 그러나 **task 댓글 한 곳에만** 연결돼 있고, **프론트가 멘션을 전혀 안 보낸다**(`commentApi.create`가 `{content}`만 전송, `@` 입력 UI 없음).
- 즉 백엔드는 준비됐는데 **@를 만드는 화면이 없어 아무도 못 쓴다.**

## 범위 / 결정 (확정)
- **A. 알림 통합**: 모든 멘션을 **하나의 알림 벨**로 모은다(사용자 경험 우선).
- **B. 우선순위**: **태스크계(댓글·작업일지·이슈) 먼저 → 그 다음 게시판(게시글·덧글)까지 전부.** 게시판은 제외가 아니라 **후속 단계**다.
- **C. 채널**: 우선 **앱 내 벨(+실시간 소켓)만**. 이메일/푸시는 추후(§8).
- **게시판을 뒤로 두는 이유**: 게시글·덧글은 **auth 서비스**, 멘션·벨은 **project 서비스** — 서비스 간 알림 라우팅이 별도 설계라 태스크계로 패턴을 검증한 뒤 착수한다(4단계).

## 데이터 모델 — `Mention` 폴리모픽 전환
지금은 `commentId`에 하드 고정이라 다른 엔티티에 못 붙는다. 아래로 일반화한다.

```
// AS-IS
model Mention { id, commentId, userId, isRead }

// TO-BE
model Mention {
  id         String
  sourceType String   // "COMMENT" | "WORKLOG" | "ISSUE" | "POST" | "BOARD_COMMENT"
  sourceId   String   // 해당 엔티티 id
  taskId     String?  // 딥링크·가시성 판단용(태스크계일 때)
  userId     String   // 멘션 대상
  actorId    String   // 멘션한 사람
  isRead     Boolean  @default(false)
  createdAt  DateTime @default(now())
  @@unique([sourceType, sourceId, userId])
  @@index([userId, isRead])
}
```
**이관**: 기존 rows는 `sourceType="COMMENT", sourceId=commentId, taskId=comment.taskId`로 백필 후 `commentId` 컬럼 제거. (data-migration 스크립트 — `services/project/prisma/data-migrations/`)

> ⚠️ 이 스키마 확정이 이 문서의 **핵심**. 되돌리기 어려운 지점이므로 여기서 합의.

## API 변경
- **댓글**(이미 수신): 프론트만 `mentionedUserIds` 전달하면 됨.
- **작업일지·이슈**: 생성/수정 DTO에 `mentionedUserIds?` 추가 → 서비스에서 `Mention` 생성 + 소켓 emit(댓글과 동일 패턴 복제).
- **`/notifications`**: `sourceType`별로 **미리보기 텍스트 + 딥링크 정보**(taskId 등)를 include/resolve해서 반환. 벨에서 "누가·어디서" 보이고 클릭 시 이동.
- **자기 멘션 제외**: `mentionedUserIds`에서 actor 본인 제거(서버측 방어).

## 프론트 설계
- **공용 `MentionInput`**(재사용): 텍스트에어리어에서 `@` 감지 → 직원 드롭다운(`usersApi.members`, 이름/부서 필터, 키보드 탐색) → 선택 시 `@이름` 삽입 + 내부적으로 `mentionedUserIds` 수집. 저장 시 함께 전송.
- **렌더링**: 저장된 본문에서 `@이름`을 하이라이트 표시(멘션 토큰 규칙 확정 필요 — 예: `@[이름](userId)` 저장 후 렌더 시 치환).
- **적용처(1차)**: `TaskDrawer` 댓글 → 작업일지 입력 → 이슈 입력.
- **벨/알림 페이지**: `sourceType`별 아이콘·문구 + 해당 태스크/작업일지/이슈로 **딥링크 이동**, 클릭 시 읽음 처리(기존 API 재사용).

## 단계 계획 & 완료 기준
1. **공용 `MentionInput` + task 댓글 연결** — 백엔드 무수정. `@` → 저장 → 벨 알림까지 end-to-end 1개. (완료: 다른 직원 계정에서 벨에 뜸)
2. **`Mention` 폴리모픽 전환 + 이관** — 기존 댓글 알림 동작 유지 확인. `/notifications` 맥락·딥링크 반환.
3. **작업일지·이슈 확대** — 두 입력에 `MentionInput` 재사용 + 백엔드 멘션 생성. (완료: 3종 모두 벨 도달)
4. **게시판(게시글·덧글, auth) 확대** — 프론트 `MentionInput` 그대로 재사용. 관건은 **auth의 멘션을 project 알림 벨로 모으는 서비스 간 경로**:
   - 방식 후보 (택1, 이 단계 진입 전 확정):
     (a) auth가 project 내부 API(`X-Internal-Token`)로 멘션 알림 생성 호출,
     (b) RabbitMQ 이벤트(`mention.created`) 발행 → project가 구독해 `Mention` 적재,
     (c) 알림을 별도 **공용 알림 서비스**로 분리(가장 크고, 이메일/푸시 확장과 함께 갈 때 유리).
   - 게시글·덧글은 `sourceType="POST"|"BOARD_COMMENT"`, 딥링크는 게시판 경로.
   - (완료: 게시판에서 @ → 동일한 벨에 도달, 클릭 시 게시글로 이동)

## 엣지 / 리스크
- **이관 안전성**: 폴리모픽 전환 시 기존 알림 유실 없이 백필.
- **이름 해석**: `Mention`은 userId만 저장 → 표시용 이름은 프론트에서 `usersApi.members`로 해석(또는 알림 응답에 이름 포함).
- **삭제·비활성 직원**: 멘션 대상이 비활성이면 후보에서 제외/무시.
- **권한·가시성**: 태스크 접근권 없는 직원 멘션 시 알림 노출 정책 결정(우선 "멘션되면 알림은 감, 클릭 시 권한 체크").
- **중복 알림**: 같은 대상 재멘션/수정 시 `@@unique`로 1건 유지.
- **본문 수정 시**: 멘션 추가/삭제 반영(댓글 update 경로 이미 존재, 동일 패턴).

## 8. 추후 확장 — 푸시 + 이메일 (설계 시 미리 열어둘 것)
1차는 앱 내 벨만이지만, **나중에 이메일·브라우저 푸시를 무리 없이 붙일 수 있도록** 지금부터 아래를 지킨다.

**핵심 설계 원칙 — 알림 생성을 단일 진입점으로**
- 멘션이 발생하는 모든 경로가 **하나의 함수**(예: `notify(userId, event)`)를 통해 알림을 만들게 한다.
- 이 지점이 **채널 분기점**이 되어, 이후 벨 외에 이메일·푸시를 **팬아웃**만 추가하면 되게 한다. (엔티티마다 알림 로직이 흩어지면 확장이 지옥이 됨)

**추가 시 필요한 것 (추후)**
- **사용자 알림 설정**: 채널별 on/off(벨/이메일/푸시), 멘션 종류별 설정.
- **이메일**: 발송 인프라 결정(현재 RabbitMQ 존재 → 큐 기반 비동기 발송 권장). 템플릿·수신거부·발송 실패 재시도.
- **브라우저 푸시(Web Push)**: 서비스워커 + VAPID 키 + 구독 토큰 저장. 브라우저 닫혀도 도달. 권한 동의 UX 필요.
- **서비스 간 통합(게시판 포함)**: 이때 auth·project 멘션을 **공용 알림 서비스/이벤트**로 모으는 구조를 함께 확정(§B 보류분과 연동).

**단계**: 벨(1차) → 이메일(요청 시) → 웹푸시(요청 시). 각 단계는 위 단일 진입점에 채널 하나 추가하는 형태.

## 테스트 체크리스트
- `@` 입력 → 직원 드롭다운·키보드 선택 동작
- 저장 → `Mention` 생성 → 대상 계정 벨 미읽음 +1(실시간 소켓 & 폴링)
- 알림 클릭 → 해당 태스크/작업일지/이슈로 이동 + 읽음 처리
- 본문 수정으로 멘션 추가/삭제 반영
- 자기 자신 멘션은 알림 없음
- 폴리모픽 이관 후 **기존 댓글 알림 유실 없음**
