## 6. 별도 PDCA로 분리된 Gap (4건)

### G1 — Internal JWT Big Bang (별도 PDCA `보안-internal-jwt-bigbang`)

**FR-30 미구현 이유**: `/internal/service-token` 핸들러가 복잡하고 기존 X-Internal-Token과 병행 필요. Layer 5.5로 분리.

| 항목 | 내용 |
|---|---|
| **영향도** | High |
| **보안 영향** | INTERNAL_API_TOKEN 누출 시 5분 한정 → 무제한으로 변경 |
| **작업량** | 4-6h (auth-service: /internal/service-token 엔드포인트 + sign/verify + TTL 5분) |
| **권장 시기** | 이관 직전 또는 이관 2주 후 (검증 기간) |

**설계 개요**:
1. auth-service: `POST /internal/service-token` (X-Internal-Token 검증 후 5분 TTL JWT 발급)
2. 모든 서비스: `requireInternal` 미들웨어가 `Authorization: Bearer <service-token>` 수용
3. 운영 1주: 두 방식 병행
4. 운영 2주: X-Internal-Token 제거

### G4 — 활성 세션 UI (별도 PDCA `세션-관리-UI`)

**FR-14 부분 구현**: 백엔드 API는 완성 (RefreshToken 다중 저장, GET /api/v1/auth/sessions), UI는 미구현.

| 항목 | 내용 |
|---|---|
| **영향도** | Medium (UX, 선택적) |
| **작업량** | 2-3h (settings/sessions/page.tsx 신규 + API 호출) |
| **권장 시기** | 이관 후 1개월 (사용자 피드백 수집 후) |

**기능**:
- 현재 기기 (파란 뱃지)
- 다른 기기 목록 (시간/위치/UA)
- "다른 기기 모두 로그아웃" 버튼

### G7 — 보안 알람 인프라 (별도 PDCA `보안-알람-인프라`)

**FR-15 부분 구현**: Pino warn 기록만, RabbitMQ `security.alert` 채널 미발행.

| 항목 | 내용 |
|---|---|
| **영향도** | Low (운영 모니터링) |
| **작업량** | — (구조 설계 필요) |
| **권장 시기** | 이관 후 (24/7 모니터링 체계 구축 후) |

**기능**:
- Refresh reuse 감지 → security.alert 발행
- DLQ/audit 컨슈머 구독

### G8 — Resource ID 통합 (별도 PDCA `auth-resource-id-통합`)

**FR-17 이연**: `/me/*` 라우트의 `req.userEmail` → `req.userId` 매핑 필요하나, Resource.userId가 email 저장 schema 제약.

| 항목 | 내용 |
|---|---|
| **영향도** | Out-of-scope (아키텍처 리팩터) |
| **작업량** | — (schema 변경 스코프 미정) |
| **권장 시기** | Phase D 또는 별도 Clean Architecture PDCA |

**이유**: 기존 email → userId 매핑이 1:1 보장 안 됨. 이메일 변경/충돌 시 권한 누수 가능. 별도 매핑 테이블 필요.
