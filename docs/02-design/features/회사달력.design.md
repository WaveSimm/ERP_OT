## 1. 개요

### 1.1 설계 목표

1. **단일 모델**로 4 타입 (PUBLIC_HOLIDAY/COMPANY_HOLIDAY/EVENT/WORKDAY) 통합 관리
2. **2차 확장 컬럼 미리 추가** — recurrence/targetDepartmentId/startTime/endTime은 더미로 두고 향후 활성화
3. **자체 캘린더 UI** — 외부 라이브러리(FullCalendar 등) 의존 0, Tailwind grid 7×6
4. **read-only 소비** — attendance/project가 internal HTTP로 조회 (향후)
5. **권한 가드** — 모든 변경 API에 ADMIN 가드, 조회는 인증자 전체 허용
6. **(v1.2) 한국 공휴일 자동 갱신** — KASI 특일 정보 API로 부팅 시 자동 + 관리자 수동 트리거. `source` 컬럼으로 수동/자동 분리
7. **(v1.3) 휴일 시각 통합** — 자원관리·프로젝트·장비/센서 화면의 7개 timeline 컴포넌트가 회사달력 휴일을 일관 인식·표시. 공통 hook `useHolidaysMap`으로 페이지 단위 1회 fetch + props 전파. **계산 로직 적용은 별개 (백로그)**

### 1.2 설계 원칙

- **Single Model**: CompanyCalendarEntry 1개 테이블만 추가
- **Snapshot 없음**: 단순 등록 정보, 변경 시 즉시 반영
- **Soft Delete 미사용**: 1차 단순 hard delete (작은 데이터·관리 부담↓)
- **Forward-Compatible**: 2차 컬럼이 nullable → 마이그레이션 추가 비용 0
- **(v1.2) Idempotent Sync**: KASI sync는 `(source, externalId)` unique 키로 upsert. 반복 호출해도 결과 동일
- **(v1.2) Source Isolation**: 자동 sync는 `source=KASI` row만 건드림 — 수동 등록 row는 절대 수정/삭제 안 함
- **(v1.3) Visual-Only Holiday**: 시각 표시(색상)에만 휴일 반영. 부하율·peak·overload 계산은 휴일 무시 (사용자 "일단 이대로" 결정). 옵션 B (peak 계산만 휴일 제외) 백로그
- **(v1.3) Priority Rule**: 공휴일 > 주말 > 평일 — 색상 함수에서 휴일 분기 우선
- **(v1.3) Single Fetch per Page**: 페이지에서 `useHolidaysMap()` 1회 호출 → 자식 컴포넌트로 props 전파. 자식 컴포넌트의 자체 fetch 금지 (중복 호출 방지)

### 2.2 데이터 흐름 — 등록 시

```
ADMIN → POST /api/v1/calendar
  ├─ 1. authenticate hook + requireRole("ADMIN")
  ├─ 2. Zod 검증 (type, startDate ≤ endDate, 길이)
  ├─ 3. Prisma create (source=MANUAL 자동 세팅)
  └─ 4. 201 → 객체 반환
```

### 2.2.1 데이터 흐름 — 한국 공휴일 자동 갱신 (v1.2)

```
[부팅 시] auth-service index.ts ready hook
  └─ setImmediate(() => holidaySyncService.syncYear(now), syncYear(now+1))   # fire-and-forget

[수동 트리거] ADMIN → POST /api/v1/calendar/sync-holidays?year=2026
  ├─ 1. authenticate + requireRole("ADMIN") + rate-limit
  ├─ 2. holidaySyncService.syncYear(2026)
  │     ├─ a. KASI 12회 호출 (1~12월) → DTO 평탄화 → [{date, dateName, isHoliday}, ...]
  │     ├─ b. existing = WHERE source=KASI AND year(start_date)=2026
  │     ├─ c. diff: create / update / delete (externalId 비교)
  │     └─ d. Prisma transaction (createMany + updateMany + deleteMany)
  └─ 3. 200 → { year, fetched, created, updated, deleted, durationMs }

* MANUAL row는 어떤 단계에서도 SELECT/WRITE 대상이 아님 (source=KASI 필터 고정)
```

## 3. 데이터 모델

### 3.1 Prisma 스키마 (services/auth/prisma/schema.prisma 추가분)

```prisma
model CompanyCalendarEntry {
  id          String              @id @default(cuid())
  type        CalendarEntryType
  title       String
  description String?
  startDate   DateTime            @db.Date @map("start_date")
  endDate     DateTime            @db.Date @map("end_date")
  isAllDay    Boolean             @default(true) @map("is_all_day")
  color       String?                                  // hex 또는 null (타입별 기본)

  // 2차 활성화 컬럼 (nullable, 1차 미사용)
  recurrence         String?                            // "YEARLY" 등
  targetDepartmentId String?  @map("target_department_id")
  startTime          String?  @map("start_time")        // "HH:mm"
  endTime            String?  @map("end_time")

  // v1.2 — 한국 공휴일 자동 갱신
  source     CalendarEntrySource  @default(MANUAL)               // MANUAL | KASI
  externalId String?              @map("external_id")            // KASI: locdate(YYYYMMDD)+dateName 해시 또는 합성키

  createdBy   String              @map("created_by")
  createdAt   DateTime            @default(now()) @map("created_at")
  updatedAt   DateTime            @updatedAt @map("updated_at")

  @@index([startDate, endDate])    // 기간 조회 (홈·관리 UI 핫패스)
  @@index([type])                   // 타입 필터
  @@index([source])                 // v1.2 — sync 대상 필터
  @@unique([source, externalId])    // v1.2 — KASI upsert 키 (NULL 허용 시 MANUAL은 제약 없음)
  @@map("company_calendar_entries")
}

enum CalendarEntryType {
  PUBLIC_HOLIDAY
  COMPANY_HOLIDAY
  EVENT
  WORKDAY
}

enum CalendarEntrySource {
  MANUAL
  KASI
}
```

**externalId 합성 규칙 (v1.2)**: `${locdate}-${dateName}` (예: `20260101-신정`)
이유: 같은 날 다른 공휴일이 있을 수 있고 (대체공휴일 같은 날 다른 이름), KASI 응답에는 안정 ID가 없음.

### 3.2 핵심 인덱스

| 인덱스 | 용도 |
|---|---|
| `(startDate, endDate)` | 월간 조회 — `WHERE NOT (endDate < $from OR startDate > $to)` |
| `(type)` | 타입 필터 (휴일만, 행사만 등) |
| `(source)` (v1.2) | sync 시 KASI row만 빠르게 조회 |
| `unique(source, externalId)` (v1.2) | KASI 멱등 upsert 키 |

### 3.3 검증 규칙 (Zod)

- `type ∈ enum 4종`
- `title.length 1~200`
- `description.length ≤ 1000`
- `startDate, endDate` YYYY-MM-DD 형식
- `endDate >= startDate` (refine)
- `color` `^#[0-9a-fA-F]{6}$` 또는 null

**v1.2 sync 입력 검증**:
- `year` 정수, 범위 `[현재 연도 - 1, 현재 연도 + 5]` (오·남용 방지)

### 4.3 주요 엔드포인트 상세

#### `GET /api/v1/calendar`

**Query Params:**
| 파라미터 | 타입 | 기본 | 설명 |
|---|---|---|---|
| `from` | YYYY-MM-DD | 당월 1일 | 조회 시작 |
| `to` | YYYY-MM-DD | 당월 말일 | 조회 끝 |
| `type` | enum | — | 타입 필터 (옵션) |

**Response (200):**
```json
[
  {
    "id": "ckxx...",
    "type": "COMPANY_HOLIDAY",
    "title": "창립기념일",
    "description": "전사 휴무",
    "startDate": "2026-05-15",
    "endDate": "2026-05-15",
    "isAllDay": true,
    "color": "#fb923c",
    "recurrence": null,
    "targetDepartmentId": null,
    "startTime": null,
    "endTime": null,
    "createdBy": "user_id",
    "createdAt": "2026-04-27T...",
    "updatedAt": "2026-04-27T..."
  }
]
```

정렬: `startDate asc, id asc`

#### `POST /api/v1/calendar`

**Request:**
```json
{
  "type": "COMPANY_HOLIDAY",
  "title": "창립기념일",
  "description": "전사 휴무",
  "startDate": "2026-05-15",
  "endDate": "2026-05-15",
  "color": "#fb923c"
}
```

**Response (201):** Calendar entry 객체

**Errors:**
- `400 INVALID_INPUT` — Zod 위반
- `400 INVALID_DATE_RANGE` — endDate < startDate
- `403 FORBIDDEN` — ADMIN 아님

#### `GET /api/v1/calendar/upcoming?days=14`

**Response (200):** 오늘 ~ N일 후의 항목 (정렬: `startDate asc`)

홈 카드용 — 휴일·행사 다가오는 5건 정도 표시.

#### `GET /internal/calendar/holidays?from&to`

**Headers:** `X-Internal-Token: ...`

**Response (200):**
```json
[
  { "date": "2026-01-01", "title": "신정", "type": "PUBLIC_HOLIDAY" },
  { "date": "2026-05-15", "title": "창립기념일", "type": "COMPANY_HOLIDAY" }
]
```

기간 내 휴일을 일자별로 정규화(date range를 일별 분해)하여 반환 — attendance/project가 그대로 일자 비교 사용.

#### `POST /api/v1/calendar/sync-holidays?year=YYYY` (v1.2)

**Headers**: `Authorization: Bearer ...` (ADMIN)

**Query Params**:
| 파라미터 | 타입 | 기본 | 설명 |
|---|---|---|---|
| `year` | integer | 현재 연도 | 동기화할 연도 |

**동작**:
1. KASI `getRestDeInfo` 12회 호출 (1~12월) → `{locdate, dateName, isHoliday}` 배열 합산
2. 기존 `WHERE source='KASI' AND EXTRACT(YEAR FROM start_date)=year` SELECT
3. diff:
   - 신규(KASI에는 있고 DB에 없음): `INSERT (source=KASI, externalId)`
   - 갱신(externalId 동일, 다른 필드 변경): `UPDATE`
   - 삭제(DB에 있지만 KASI에 없음, 예: 임시공휴일 취소): `DELETE`
4. 단일 트랜잭션

**Response (200)**:
```json
{
  "year": 2026,
  "fetched": 18,
  "created": 15,
  "updated": 0,
  "deleted": 0,
  "durationMs": 1240
}
```

**Errors**:
- `400 INVALID_INPUT` — year 범위 위반
- `403 FORBIDDEN` — ADMIN 아님
- `429 TOO_MANY_REQUESTS` — rate limit 초과 (분당 6회)
- `502 KASI_UNAVAILABLE` — KASI API 호출 실패 (재시도 후에도)
- `500 INTERNAL_ERROR` — DB 트랜잭션 실패

**Rate Limit**: 분당 6회 (10초당 1회) — 1라인 글로벌 카운터, 사용자 무관

| HTTP | Code | 메시지 | 원인 |

| 401 | UNAUTHENTICATED | 인증 필요 | 토큰 없음 |

| 409 | KASI_ROW_READONLY (v1.2) | 자동 갱신 항목은 직접 수정/삭제할 수 없습니다 | source=KASI PATCH/DELETE |
| 429 | TOO_MANY_REQUESTS (v1.2) | 잠시 후 다시 시도하세요 | sync rate-limit |
| 502 | KASI_UNAVAILABLE (v1.2) | 공휴일 정보 서버에 접근할 수 없습니다 | 외부 API 실패 |
