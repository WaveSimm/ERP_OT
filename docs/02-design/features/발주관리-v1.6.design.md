## 1. Overview

### 1.1 Goals

1. **발주별 커스텀 결재라인**: 발주서마다 결재선을 다르게 지정 가능 (공통 마스터 강제 ❌)
2. **회계정산 워크플로우 도입**: Invoice 발행 → 네고/추가비용 Amendment 이력 → 송금 처리 분리 추적
3. **다회차 송금 지원**: 1건의 송금 요청을 실제 송금 시 자동 분할 (해외무역 다회차 송금 관행 반영)
4. **재무팀 단일 처리 큐**: 결재송금(EXPENSE) + 발주송금(OrderPayment) 한 메뉴 + sub-tab 구분
5. **발주 흐름 명확화**: ORDERED → PURCHASING(발주완료) → SHIPPED 강제, IN_PRODUCTION 단계 폐기
6. **발주번호 자리수 효율화**: PO-YYYY-NNNN(연간 시퀀스, 4자리 부족) → PO-YYMM-NNNN(월별 시퀀스 + 월별 식별)
7. **PO PDF 자체 출력**: 인쇄 페이지 + 사용자 다운로드 후 이메일 직접 발송 (자동 발송 안 함)
8. **권한 일괄 완화**: 제조사·고객사·결재·송금 액션을 OPERATOR 까지 허용 (삭제만 ADMIN 유지)

### 1.2 사용자 의도 (2026-05-15 질의응답)

| # | 결정 | 의도 |
|---|---|---|
| 1 | OverseasOrder approver IDs 3종 (approverId/secondApproverId/thirdApproverId) | **각 발주서마다 결재선을 다르게**. 공통 결재선 마스터가 아닌 발주별 커스텀. 3단은 자유 지정 |
| 2 | OrderPayment 부분 결제 자동 분할 | **다회차 송금 케이스**. 1 송금 요청을 여러 차례 나눠 송금 (해외무역 관행) |
| 3 | 재무 접수 sub-tab (결재송금/발주송금 통합 페이지, 별도 탭) | **재무팀 동일 처리 부서**이지만 **세부 내용은 다름** → 메뉴 통합, sub-tab 구분 |
| 4 | PO-YYYY-NNNN → PO-YYMM-NNNN | **자리수 효율화 + 월별 구분 가능**. 연간 666건 수준이라 4자리 시퀀스 부족 + 월별 그룹화 |

### 1.3 Non-Goals

- PO PDF 자동 이메일 발송 (사용자가 다운로드 후 직접 발송 — 발송 통제 + 첨부 자유)
- 발주번호 체계 마이그레이션 기존 데이터 보존 (666건 일괄 새 형식으로 변환, 23개 결재 문서 title/content 동기화)
- SETTLEMENT 상태 전환 가능 유지 (legacy enum만 보존, 신규 진입 차단)

## 3. Data Model

### 3.1 OverseasOrder 변경 (v1.6 신규 컬럼 6종)

```prisma
model OverseasOrder {
  // ... 기존 필드 ...

  // v1.6 신규 (2026-05-14)
  customer            String?                                    // 발주별 고객사 (계약 client 기본값, 사용자 덮어쓰기 가능)
  paymentTerms        String?  @map("payment_terms")             // T/T·L/C·무역금융 등 (oa_number와 분리)

  // v1.6 신규 — 발주별 결재라인 (3단, 자유 지정)
  approverId          String?  @map("approver_id")               // 1차 결재자
  approverName        String?  @map("approver_name")
  secondApproverId    String?  @map("second_approver_id")        // 2차 결재자
  secondApproverName  String?  @map("second_approver_name")
  thirdApproverId     String?  @map("third_approver_id")         // 3차 결재자
  thirdApproverName   String?  @map("third_approver_name")

  // 신규 관계 (3.2, 3.3, 3.4)
  invoice             OrderInvoice?
  payments            OrderPayment[]
}
```

> **의도**: 결재라인은 발주서마다 다름 (공통 마스터 강제 X). 단계 의미는 부서장/임원/대표 같은 고정 의미가 아니라 **자유 3단** (사용자가 매번 선택). `approverName` 도 함께 저장하여 사용자 삭제 후에도 결재 히스토리 보존.

### 3.2 OrderInvoice (신규, 1:1 with OverseasOrder)

```prisma
model OrderInvoice {
  id              String        @id @default(cuid())
  orderId         String        @unique @map("order_id")
  invoiceNumber   String        @map("invoice_number")
  invoiceDate     DateTime      @map("invoice_date") @db.Date
  amount          Decimal       @db.Decimal(12, 2)              // 현재 금액 (네고·추가비용 반영)
  initialAmount   Decimal       @map("initial_amount")          // 최초 invoice 금액
  currency        OrderCurrency
  amountKRW       Decimal?      @map("amount_krw")
  dueDate         DateTime?     @map("due_date") @db.Date
  paymentTerms    String?       @map("payment_terms")           // T/T, L/C, 무역금융 등
  notes           String?
  createdAt       DateTime      @default(now())
  updatedAt       DateTime      @updatedAt

  order           OverseasOrder           @relation(fields: [orderId], references: [id], onDelete: Cascade)
  amendments      OrderInvoiceAmendment[]

  @@map("order_invoices")
  @@schema("equipment")
}
```

- **`initialAmount` vs `amount`**: 최초 invoice 금액과 현재 금액을 분리. Amendment 이력과 함께 추적.

### 3.3 OrderInvoiceAmendment (신규, N:1 with Invoice)

```prisma
model OrderInvoiceAmendment {
  id              String   @id @default(cuid())
  invoiceId       String   @map("invoice_id")
  previousAmount  Decimal  @map("previous_amount") @db.Decimal(12, 2)
  newAmount       Decimal  @map("new_amount") @db.Decimal(12, 2)
  reason          String                                          // "네고" / "추가비용" / "기타"
  description     String?
  amendedBy       String?  @map("amended_by")
  createdAt       DateTime @default(now())

  invoice         OrderInvoice @relation(fields: [invoiceId], references: [id], onDelete: Cascade)

  @@index([invoiceId])
  @@map("order_invoice_amendments")
  @@schema("equipment")
}
```

- Invoice 금액 변경 시마다 row 추가. `previousAmount → newAmount` + 사유(네고/추가비용/기타).
- Invoice 본체는 항상 현재 금액만 보유. 히스토리는 amendments 만으로 구성.

### 3.4 OrderPayment (신규, N:1 with OverseasOrder)

```prisma
model OrderPayment {
  id              String              @id @default(cuid())
  orderId         String              @map("order_id")
  paymentDate     DateTime?           @map("payment_date") @db.Date    // REQUESTED 상태는 null 허용
  amount          Decimal             @db.Decimal(12, 2)
  currency        OrderCurrency
  amountKRW       Decimal?            @map("amount_krw")
  exchangeRate    Decimal?            @map("exchange_rate")
  paymentMethod   String?             @map("payment_method")           // T/T, L/C, 무역금융 등
  bankReference   String?             @map("bank_reference")
  notes           String?

  // v1.6 — 송금 요청 워크플로우
  status          OrderPaymentStatus  @default(COMPLETED)              // 기존 데이터 호환: COMPLETED 기본
  rejectReason    String?             @map("reject_reason")
  requestedBy     String?             @map("requested_by")             // 발주 담당자
  requestedAt     DateTime?           @map("requested_at")
  completedBy     String?             @map("completed_by")             // 재무팀 사용자
  completedAt     DateTime?           @map("completed_at")
  createdBy       String?             @map("created_by")
  createdAt       DateTime            @default(now())
  updatedAt       DateTime            @updatedAt

  order           OverseasOrder @relation(fields: [orderId], references: [id], onDelete: Cascade)

  @@index([orderId])
  @@index([paymentDate])
  @@map("order_payments")
  @@schema("equipment")
}
```

- 발주 1건당 N개 Payment (다회차 송금).
- `status` 추가로 송금 요청 → 처리 워크플로우 도입.
- `default(COMPLETED)` 는 기존 데이터 호환 (v1.6 이전 Payment는 모두 완료 상태로 간주).

## 4. State Machines

### 4.1 Order FSM (v1.6 변경)

```
   DRAFT ────────── (submit) ──────► PENDING_APPROVAL
                                     │
   ┌────── (취소 v1.6 신규) ──────────┘
   │
   ▼                       (approve) ┌──► APPROVED ──► ORDERED ──► PURCHASING (v1.6 신규)
   DRAFT                              │                                  │
                                      │ (reject)                         ▼
                                      └──► REJECTED ──► DRAFT       SHIPPED ──► CUSTOMS
                                                                                 │
                                                                                 ├──► PARTIALLY_RECEIVED ──► ARRIVED
                                                                                 └──► ARRIVED ──► CLOSED

   IN_PRODUCTION  (legacy 호환만, 신규 진입 차단) ──► SHIPPED
   SETTLEMENT     (legacy 호환만, 신규 진입 차단) ──► CLOSED
```

#### 4.1.1 전이 규칙 (`order.fsm.ts`)

```typescript
const TRANSITIONS: Record<OrderStatus, OrderStatus[]> = {
  DRAFT:              ["PENDING_APPROVAL"],
  PENDING_APPROVAL:   ["APPROVED", "REJECTED", "DRAFT"],  // v1.6: 상신 취소 허용
  APPROVED:           ["ORDERED"],
  REJECTED:           ["DRAFT"],
  ORDERED:            ["PURCHASING"],                      // v1.6: IN_PRODUCTION 폐기 — 강제 PURCHASING
  IN_PRODUCTION:      ["SHIPPED"],                         // legacy 호환
  PURCHASING:         ["SHIPPED"],                         // v1.6 신규
  SHIPPED:            ["CUSTOMS"],
  CUSTOMS:            ["PARTIALLY_RECEIVED", "ARRIVED"],
  PARTIALLY_RECEIVED: ["ARRIVED"],
  ARRIVED:            ["CLOSED"],                          // v1.6: SETTLEMENT 진입 차단
  SETTLEMENT:         ["CLOSED"],                          // legacy 호환만
  CLOSED:             [],
};
```

#### 4.1.2 v1.6 변경 사항 요약

- **PURCHASING 신규**: ORDERED → PURCHASING(발주완료, PO 발송) → SHIPPED. 제작중(IN_PRODUCTION) 단계 사용 안 함 (실제 운영에서 의미 없는 단계로 판단)
- **상신 취소**: PENDING_APPROVAL → DRAFT 허용 (실수 상신 회수)
- **SETTLEMENT 차단**: 송금은 OrderPayment로 별도 관리 — 발주 상태 변경 없음 (legacy enum만 유지)
- **입고완료 자동 결정**: 부분/전체 입고 시 자동 ARRIVED 전환 — 사용자 수동 전환 버튼 제거

### 4.2 OrderPayment FSM (신규)

```
                     ┌─ (재무팀 송금 완료) ─► COMPLETED
                     │
    (없음) ─(요청)─► REQUESTED ─┤
                     │
                     └─ (재무팀 반려) ─► REJECTED
                                          │
                                          └─ (재요청 시 새 row 생성)

    부분 결제 자동 분할:
    REQUESTED (amount=1000) ──(actual=600 송금)──► COMPLETED (amount=600, completed)
                                                   + 자동 새 row: REQUESTED (amount=400, 잔여)
```

#### 4.2.1 전이 규칙

| from | to | 트리거 | 권한 | 부수효과 |
|---|---|---|---|---|
| (없음) | REQUESTED | 발주 담당자 "💸 송금 요청" | 발주 담당자/OPERATOR | requestedBy/At 기록 |
| REQUESTED | COMPLETED | 재무팀 "완료" + 실제 송금액 = 요청액 | 재무팀/ADMIN/MANAGER/OPERATOR | paymentDate, completedBy/At 기록 |
| REQUESTED | COMPLETED (분할) | 재무팀 "완료" + 실제 송금액 < 요청액 | 재무팀 | **원본 amount=실제송금액, COMPLETED 처리** + **새 REQUESTED row 자동 생성 (amount=잔여)** |
| REQUESTED | REJECTED | 재무팀 "반려" | 재무팀 | rejectReason 기록 |
| COMPLETED | (변경 없음) | - | - | 종착 |
| REJECTED | (없음) | - | - | 종착. 신청자가 새 송금 요청 시 새 row |

#### 4.2.2 부분 결제 자동 분할의 의도

**왜 자동 분할인가**: **다회차 송금** (해외무역의 일반 관행). 환차익·은행 송금 한도·계약 조건으로 1건의 송금 요청이 여러 차례 나누어 송금되는 경우. 기존엔 잔여를 사용자가 다시 등록해야 했음 → 자동화로 처리 부담 감소.

**처리 흐름**:
1. 발주 담당자: 송금 요청 (amount=$10,000 REQUESTED)
2. 재무팀: 실제 송금 ($6,000) → "완료" 클릭 + 실제송금액 입력
3. 시스템 자동:
   - 원본 row: amount=$6,000, status=COMPLETED, completedAt 기록
   - 신규 row: amount=$4,000, status=REQUESTED (잔여)
4. 발주 담당자 추가 조치 불필요 — 새 REQUESTED를 재무팀이 다음 송금 시 처리

### 5.3 Payment Request 워크플로우 (v1.6 신규)


#### 5.3.1 `PATCH /payments/:paymentId/complete` 동작

```typescript
// body: { paymentDate: ISO, actualAmount: number, exchangeRate?: number, bankReference?: string }

//   if (actualAmount === payment.amount) → status=COMPLETED, 단순 완료

//   if (actualAmount > payment.amount) → 허용 (overpaid). 실제 송금이 더 많은 케이스 수용
```

## 7. 발주별 결재라인 (Approval 연동)

### 7.1 결재라인 데이터 흐름

```
1. 발주 등록: OverseasOrder에 approverId/secondApproverId/thirdApproverId 저장
                         ↓
2. 상신: POST /api/v1/approval/documents (or /internal/documents)
   body: {
     templateCode: "OVERSEAS_ORDER",
     title: "발주서 PO-2605-0001",
     approvalLine: [                         ◀── 🔑 'approvalLine' (NOT 'steps')
       { stepOrder: 1, approverId: "...", approverName: "심윤송" },
       { stepOrder: 2, approverId: "...", approverName: "김임원" },
       { stepOrder: 3, approverId: "...", approverName: "대표" },
     ],
     referenceType: "OVERSEAS_ORDER",
     referenceId: order.id
   }
                         ↓
3. approval-service: 결재 문서 생성 + 결재 진행
                         ↓
4. 결재 완료: webhook → POST /internal/expenses/follow-up (2026-05-12)
   → expense-followup record 생성 (재무 후속 처리)
```

### 7.2 핵심 구현 주의 사항

> ⚠️ **`body.approvalLine` 필수** ([memory/project_approval_create_api_contract.md](../../../))
> - approval document POST body는 **`approvalLine`** 필드.
> - `body.steps` 로 보내면 silent fallback 으로 본인 기본 결재선 자동 로드됨.
> - v1.6 초기 구현에서 이 버그로 결재선이 본인 단독으로 자동 변경되는 회귀 발생 → 수정 완료.

### 7.3 본인 기본 결재선 prefill

발주 등록 시 발주자(currentUser)의 기본 결재선 자동 로드:

```typescript
// auth-service
GET /internal/users/:userId/approver
→ { approverId, approverName, secondApproverId, secondApproverName, thirdApproverId, thirdApproverName }
```

사용자는 prefill된 결재선을 발주별로 수정 가능.

## 8. 발주번호 체계 (PO-YYMM-NNNN)

### 8.1 형식 변경

| 항목 | v1.5 이전 | v1.6 |
|---|---|---|
| 형식 | `PO-YYYY-NNNN` | `PO-YYMM-NNNN` |
| 예시 | PO-2026-0001 | PO-2605-0001 |
| 시퀀스 리셋 | 연 단위 | 월 단위 |
| 자리수 | 9자리 | 9자리 (자리수 동일하나 YYMM이 더 정보 풍부) |

### 8.2 의도

- **자리수 효율화**: 연간 666건 수준 → 4자리 시퀀스 부족할 수 있음. 월 시퀀스로 리셋하면 100건 미만 유지 가능.
- **월별 구분 가능**: PO 번호만 봐도 발주 월 식별. 회계 마감·정산 시 유리.

### 8.3 생성 로직 (`overseas-order.service.ts`)

```typescript
async generateOrderNumber(): Promise<string> {
  const yymm = formatYYMM(new Date());  // "2605"
  const prefix = `PO-${yymm}-`;
  const last = await prisma.overseasOrder.findFirst({
    where: { orderNumber: { startsWith: prefix } },
    orderBy: { orderNumber: "desc" },
  });
  const nextSeq = last ? parseInt(last.orderNumber.split("-")[2]) + 1 : 1;
  return `${prefix}${String(nextSeq).padStart(4, "0")}`;
}
```

### 8.4 마이그레이션 (666건)

- 기존 666건 PO-YYYY-NNNN → PO-YYMM-NNNN 일괄 변환
- 결재 문서 23건의 title/content 동기화 (orderNumber 참조 부분)
- 마이그레이션 스크립트: `scripts/` (commit 213a021 에 포함)
