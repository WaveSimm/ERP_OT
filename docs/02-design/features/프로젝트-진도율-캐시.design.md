# [Design] 프로젝트-진도율-캐시 (Project Aggregate Cache)

> **Phase**: Design v1.0 / **Date**: 2026-04-28
> **Plan**: [프로젝트-진도율-캐시.plan.md](../../01-plan/features/프로젝트-진도율-캐시.plan.md)

## 1. 설계 개요

5개 캐시 필드 + 1개 service(`AggregateService`) + 6개 hook 지점.

```
┌─ TaskService.{create/update/delete}Task ──┐
├─ TaskService.{create/update/delete}Segment┤
└─ TemplateService 적용 시                   ┘
        │
        ▼
   AggregateService.recomputeTask(taskId)
        │
        ▼ (task의 effectiveDates 갱신)
   AggregateService.recomputeProject(projectId)
        │
        ▼ (project overallProgress + effectiveDates 갱신)
   DB Write (트랜잭션 내)
```

## 2. Schema 변경

```prisma
model Project {
  // ... 기존 필드
  overallProgress     Float?     @map("overall_progress")
  effectiveStartDate  DateTime?  @map("effective_start_date")
  effectiveEndDate    DateTime?  @map("effective_end_date")
}

model Task {
  // ... 기존 필드 (overallProgress는 이미 있음)
  effectiveStartDate  DateTime?  @map("effective_start_date")
  effectiveEndDate    DateTime?  @map("effective_end_date")
}
```

> Task의 effective dates는 **자기 segments의 min/max**, Project의 effective dates는 **leaf task segments 전체의 min/max**로 의미가 다름.

## 3. Migration

### 3.1 Schema 변경 + backfill

```sql
-- Project: 신규 필드
ALTER TABLE project.projects
  ADD COLUMN overall_progress double precision,
  ADD COLUMN effective_start_date timestamp(3),
  ADD COLUMN effective_end_date timestamp(3);

-- Task: 신규 필드
ALTER TABLE project.tasks
  ADD COLUMN effective_start_date timestamp(3),
  ADD COLUMN effective_end_date timestamp(3);

-- Backfill (1회): 모든 task의 effective dates
UPDATE project.tasks t SET
  effective_start_date = sub.s,
  effective_end_date = sub.e
FROM (
  SELECT s."taskId", MIN(s."startDate") AS s, MAX(s."endDate") AS e
  FROM project.task_segments s
  GROUP BY s."taskId"
) sub
WHERE t.id = sub."taskId";

-- Backfill: 모든 project의 aggregates (leaf task only, isMilestone=false)
WITH parents AS (
  SELECT DISTINCT "parentId" FROM project.tasks WHERE "parentId" IS NOT NULL
),
leaf_segments AS (
  SELECT t."projectId", s."progressPercent", s."startDate", s."endDate"
  FROM project.tasks t
  JOIN project.task_segments s ON s."taskId" = t.id
  WHERE t.id NOT IN (SELECT "parentId" FROM parents)
    AND t."isMilestone" = false
)
UPDATE project.projects p SET
  overall_progress = sub.avg_p,
  effective_start_date = sub.s,
  effective_end_date = sub.e
FROM (
  SELECT "projectId",
         AVG("progressPercent") AS avg_p,
         MIN("startDate") AS s,
         MAX("endDate") AS e
  FROM leaf_segments
  GROUP BY "projectId"
) sub
WHERE p.id = sub."projectId";
```

## 4. AggregateService 설계

```typescript
// services/project/src/application/aggregate.service.ts (신규)
export class AggregateService {
  constructor(private prisma: PrismaClient | Prisma.TransactionClient) {}

  /** 단일 task의 effectiveDates 재계산 (DB 갱신) */
  async recomputeTask(taskId: string): Promise<void> {
    const segs = await this.prisma.taskSegment.findMany({
      where: { taskId },
      select: { startDate: true, endDate: true },
    });
    const start = segs.length > 0 ? new Date(Math.min(...segs.map(s => s.startDate.getTime()))) : null;
    const end = segs.length > 0 ? new Date(Math.max(...segs.map(s => s.endDate.getTime()))) : null;
    await this.prisma.task.update({
      where: { id: taskId },
      data: { effectiveStartDate: start, effectiveEndDate: end },
    });
  }

  /** project의 overallProgress + effectiveDates 재계산 */
  async recomputeProject(projectId: string): Promise<void> {
    // leaf task = parent가 아닌 + 마일스톤이 아닌
    const allTasks = await this.prisma.task.findMany({
      where: { projectId },
      select: { id: true, parentId: true, isMilestone: true },
    });
    const parentIds = new Set(allTasks.filter(t => t.parentId).map(t => t.parentId!));
    const leafIds = allTasks.filter(t => !parentIds.has(t.id) && !t.isMilestone).map(t => t.id);

    if (leafIds.length === 0) {
      await this.prisma.project.update({
        where: { id: projectId },
        data: { overallProgress: null, effectiveStartDate: null, effectiveEndDate: null },
      });
      return;
    }

    const segs = await this.prisma.taskSegment.findMany({
      where: { taskId: { in: leafIds } },
      select: { progressPercent: true, startDate: true, endDate: true },
    });
    if (segs.length === 0) {
      await this.prisma.project.update({
        where: { id: projectId },
        data: { overallProgress: null, effectiveStartDate: null, effectiveEndDate: null },
      });
      return;
    }

    const avg = segs.reduce((s, x) => s + x.progressPercent, 0) / segs.length;
    const start = new Date(Math.min(...segs.map(s => s.startDate.getTime())));
    const end = new Date(Math.max(...segs.map(s => s.endDate.getTime())));

    await this.prisma.project.update({
      where: { id: projectId },
      data: { overallProgress: avg, effectiveStartDate: start, effectiveEndDate: end },
    });
  }

  /** Task 변경 시 단축 호출 — task + project 일괄 */
  async recomputeTaskAndProject(taskId: string, projectId: string): Promise<void> {
    await this.recomputeTask(taskId);
    await this.recomputeProject(projectId);
  }
}
```

## 5. Hook 등록 위치

| 파일 | 메서드 | hook |
|------|--------|------|
| `task.service.ts` | `createTask` | `recomputeProject(projectId)` |
| `task.service.ts` | `updateTask` (status, parentId, isMilestone, progress) | `recomputeProject(projectId)` |
| `task.service.ts` | `deleteTask` | `recomputeProject(projectId)` |
| `task.service.ts` | `createSegment` | `recomputeTaskAndProject(taskId, projectId)` |
| `task.service.ts` | `updateSegment` | `recomputeTaskAndProject(taskId, projectId)` |
| `task.service.ts` | `deleteSegment` | `recomputeTaskAndProject(taskId, projectId)` |
| `template.service.ts` | apply (다수 task/segment 생성) | `recomputeProject(projectId)` 1회 |
| (이미 있는 rollup) | `_rollupParentProgress` | rollup 후 `recomputeProject` 추가 |

> 모두 트랜잭션(`prisma.$transaction`) 안에서 실행하거나, 변경 직후 즉시 호출.

## 9. 보안·안정성

- 트랜잭션: `prisma.$transaction` 사용 시 hook도 같은 tx에 포함
- 무한 재귀 방지: `recomputeProject`는 task만 read + project update — task 변경 hook을 다시 트리거하지 않음
- 동시성: 두 task 동시 변경 시 마지막 작업이 final, 약간의 stale은 허용 (다음 변경 시 재갱신)
