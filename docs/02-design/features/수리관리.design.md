## 1. 아키텍처 결정

### 1.1 서비스 배치

**결정: equipment-service 내 확장** (Plan 기술 결정 유지)

이유:
- 수리관리는 장비/센서와 밀접하게 연결 (equipmentId, sensorId FK)
- 기존 MaintenanceRecord와 상호 연동 (AS 완료 → 정비이력 생성)
- AssetSchedule에 수리 기간 반영 (IN_MAINTENANCE)
- 별도 서비스 시 cross-service 트랜잭션 문제 발생

```
equipment-service (port 3005)
├── 기존: Equipment, Sensor, Category, Maintenance, Schedule, Deployment, ...
└── 신규: Customer, CustomerAsset, RepairOrder, InspectionReport, RepairCost,
          RepairQuote, QuoteItem, Part, PartTransaction, PurchaseOrder, Shipment
```

### 1.2 DB 스키마

- `equipment` 스키마 내 신규 테이블 추가
- 기존 테이블과 FK 관계 유지

### 1.3 API 경로

```
/api/v1/customers          — 고객 CRUD
/api/v1/customer-assets    — 고객 자산 CRUD
/api/v1/repair-orders      — AS 접수/관리
/api/v1/inspection-reports — 점검보고서
/api/v1/repair-costs       — 비용 관리
/api/v1/repair-quotes      — 견적 관리
/api/v1/parts              — 부품 관리
/api/v1/part-transactions  — 부품 입출고
/api/v1/purchase-orders    — 발주 관리
/api/v1/shipments          — 제조사 발송/입고
/api/v1/repair-stats       — 통계
```

async generateOrderNumber(): Promise<string> {

  const result = await this.prisma.$queryRaw<[{nextval: bigint}]>`

## 3. 상태 전이 (FSM)

### 3.1 RepairOrder 상태 전이 규칙

```
RECEIVED → INSPECTING_1ST         (1차 점검 시작)
INSPECTING_1ST → QUOTED           (견적 필요 시)
INSPECTING_1ST → REPAIRING        (무상 or 자사 장비: 견적 스킵)
INSPECTING_1ST → SHIPPED_TO_MFG   (제조사 수리 필요)
INSPECTING_1ST → COMPLETED        (정상동작: 수리 불필요)
QUOTED → APPROVED                 (고객 견적 승인)
QUOTED → CANCELLED                (고객 거부/취소)
APPROVED → REPAIRING              (자체 수리 시작)
APPROVED → SHIPPED_TO_MFG         (제조사 수리)
REPAIRING → COMPLETED             (자체 수리 완료)
SHIPPED_TO_MFG → RECEIVED_FROM_MFG (제조사 수리 완료 입고)
RECEIVED_FROM_MFG → INSPECTING_2ND (2차 점검 시작)
INSPECTING_2ND → COMPLETED        (2차 점검 완료)
COMPLETED → CLOSED                (완결 처리)
* → CANCELLED                     (어디서든 취소 가능)
```

### 3.2 FSM 구현

```typescript
const REPAIR_ORDER_TRANSITIONS: Record<RepairOrderStatus, RepairOrderStatus[]> = {
  RECEIVED:          ['INSPECTING_1ST', 'CANCELLED'],
  INSPECTING_1ST:    ['QUOTED', 'REPAIRING', 'SHIPPED_TO_MFG', 'COMPLETED', 'CANCELLED'],
  QUOTED:            ['APPROVED', 'CANCELLED'],
  APPROVED:          ['REPAIRING', 'SHIPPED_TO_MFG', 'CANCELLED'],
  REPAIRING:         ['COMPLETED', 'CANCELLED'],
  SHIPPED_TO_MFG:    ['RECEIVED_FROM_MFG', 'CANCELLED'],
  RECEIVED_FROM_MFG: ['INSPECTING_2ND', 'COMPLETED', 'CANCELLED'],
  INSPECTING_2ND:    ['COMPLETED', 'CANCELLED'],
  COMPLETED:         ['CLOSED'],
  CLOSED:            [],
  CANCELLED:         [],
};
```

### 3.3 상태 변경 시 자동 동작

| 상태 전이 | 자동 동작 |
|-----------|----------|
| → INSPECTING_1ST | inspector1stId 자동 배정 |
| → SHIPPED_TO_MFG | Shipment(OUTBOUND) 자동 생성, 장비 상태 → IN_MAINTENANCE |
| → RECEIVED_FROM_MFG | Shipment(INBOUND) 업데이트 |
| → INSPECTING_2ND | inspector2ndId 배정 |
| → COMPLETED | completedAt 기록, MaintenanceRecord 자동 생성 |
| → CLOSED | closedAt 기록, 장비 상태 → AVAILABLE |

## 4. API 설계

### 4.1 고객 API

```
GET    /api/v1/customers              — 목록 (검색, 페이지네이션)
POST   /api/v1/customers              — 생성
GET    /api/v1/customers/:id          — 상세 (자산 포함)
PATCH  /api/v1/customers/:id          — 수정
DELETE /api/v1/customers/:id          — 삭제

GET    /api/v1/customer-assets?customerId=xxx  — 고객별 자산 목록
POST   /api/v1/customer-assets        — 자산 등록
GET    /api/v1/customer-assets/:id    — 자산 상세 (AS 이력 포함)
PATCH  /api/v1/customer-assets/:id    — 자산 수정
DELETE /api/v1/customer-assets/:id    — 자산 삭제
```

### 4.2 수리 접수 API

```
GET    /api/v1/repair-orders          — 목록 (상태 필터, 검색, 페이지네이션)
POST   /api/v1/repair-orders          — 접수 생성
GET    /api/v1/repair-orders/:id      — 상세 (모든 하위 데이터 포함)
PATCH  /api/v1/repair-orders/:id      — 수정
PATCH  /api/v1/repair-orders/:id/status — 상태 변경 (FSM)
PATCH  /api/v1/repair-orders/:id/tech-status   — 점검진행상황 업데이트
PATCH  /api/v1/repair-orders/:id/sales-status  — 영업부진행상황 업데이트
DELETE /api/v1/repair-orders/:id      — 삭제 (RECEIVED 상태만)
```

### 4.3 점검보고서 API

```
GET    /api/v1/inspection-reports?repairOrderId=xxx — 보고서 조회
POST   /api/v1/inspection-reports     — 보고서 생성
PATCH  /api/v1/inspection-reports/:id — 보고서 수정
```

### 4.4 비용 API

```

// repair-order.service.ts
async onStatusChange(order: RepairOrder, newStatus: RepairOrderStatus) {
  // AS 접수 시 장비 상태 변경
  if (newStatus === 'INSPECTING_1ST' && order.equipmentId) {
    await this.prisma.equipment.update({
      where: { id: order.equipmentId },
      data: { status: 'IN_MAINTENANCE' },
    });
  }
  // AS 완료 시 장비 상태 복원
  if (newStatus === 'CLOSED' && order.equipmentId) {
    await this.prisma.equipment.update({
      where: { id: order.equipmentId },
      data: { status: 'AVAILABLE' },
    });
  }
}
```

## 13. 점검보고서 첨부 + 이미지 리사이즈 (v1.2, 2026-05-05)

### 13.1 데이터 모델 — JSON 확장 (마이그레이션 0)

`InspectionReport.inspectionSteps`는 JSON. 기존:
```ts
{ step: 1, content: "...", result: "..." }
```
확장 (선택 필드):
```ts
{
  step: 1,
  content: "...",
  result: "...",
  attachments?: Array<{
    id: string;             // attachment row id
    fileName: string;
    fileSize: number;
    mimeType: string;
    storageKey: string;     // LocalFsStorage 키
    isImage: boolean;       // mimeType.startsWith("image/")
    uploadedAt: string;
  }>
}
```

기존 데이터 호환: `attachments` 필드 없는 row는 빈 배열로 표시.

### 13.2 공통 helper — `apps/web/src/lib/image-compress.ts`

### 13.3 점검보고서 첨부 API

기존 `attachmentApi.upload(file, isInline)` (auth-service 라우트)를 재활용:
- POST `/api/v1/attachments/upload` — multipart/form-data, postId/inline 옵션
- 응답: `{ id, storageKey, fileName, fileSize, mimeType, ... }`

점검보고서 저장 시 inspectionSteps JSON에 attachment 메타데이터 임베드 (FK 관계 X — 단순 JSON 보관).

### 13.4 UI 변경

**`apps/web/src/app/repair/[id]/page.tsx` ReportTab**
- 단계별 영역에 "📎 첨부" 버튼 + 드래그 드롭 영역
- 첨부된 파일 목록 (이미지는 thumb, 비이미지는 아이콘)
- 업로드 시 `compressImage` 자동 호출 (이미지만)
- 임시 보관: 사용자가 "저장" 누르기 전까지는 클라이언트 메모리 (form.inspectionSteps[i].attachments)
- 저장 시 inspectionSteps JSON에 포함되어 InspectionReport에 저장됨

**인쇄 페이지 `/repair/[id]/report/print`**
- 점검 내용 단계 끝에 이미지 그리드 (2~3 컬럼, max 4장 권장)
- 비이미지 첨부는 인쇄에 미포함 (또는 파일명만 텍스트로)
- 인쇄 시에만 표시 — `@media print` 영향 없음
