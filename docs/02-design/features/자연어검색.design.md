## 1. 개요

### 1.1 설계 목표

1. **로컬 임베딩 모델**(bge-m3) + **pgvector**로 외부 API 의존 0
2. **비동기 fire-and-forget 인덱싱** — 글 작성 응답 지연 없음
3. **권한 필터 절대 우회 불가** — 검색 결과에 audience/멤버십 위반 노출 0
4. **단일 검색 API** — 게시판 Post + 작업비고 WorkLog 모두 한 번 호출로 통합 결과
5. **관측 가능성** — 임베딩 실패는 로그만, 글 작성·수정에 영향 없음

### 1.2 설계 원칙

- **Service Boundary 유지**: auth는 Post + 임베딩, project는 WorkLog + 임베딩, 검색 API는 auth가 합성 (project를 internal HTTP)
- **Pure Embedding Service**: Ollama 호출 wrapper (인터페이스화 → 향후 모델 교체 용이)
- **Soft Index**: 임베딩 컬럼이 NULL인 글은 검색 결과에 안 나옴 — 정확성 우선
- **Snapshot 검색 결과**: 권한 검증을 검색 시점에 적용 (저장된 권한 캐시 안 함)

### 2.2 데이터 흐름 — 글 작성 시 임베딩

```
사용자 → POST /api/v1/boards/:code/posts
  → Post 즉시 저장 (트랜잭션 내)
  → 응답 201 Created (사용자 즉시 응답 받음)
  → fire-and-forget:
       1. embedding.service.embedText(`${title}\n${content}`)
       2. Ollama POST /api/embed → vector 1024d
       3. UPDATE board_posts SET embedding=$1, embedded_at=NOW() WHERE id=$2
       4. 실패 시 logger.error만, 사용자 영향 없음
```

### 2.3 검색 시퀀스

```
사용자 → GET /api/v1/search?q=디버깅 메모리&scope=all&limit=20
  → search.service.ts:
       1. q를 embedText() → queryVec
       2. (병렬)
          a. SELECT post FROM board_posts WHERE embedding IS NOT NULL
             ORDER BY embedding <=> $1 LIMIT topK*2
          b. POST {project}/internal/work-logs/semantic-search
             body: { queryVec, userId, userEmail, role, limit: topK*2 }
             → project가 권한 필터 후 ranked 결과 반환
       3. 양쪽 결과 합쳐 top K (cosine 거리 기준)
       4. (Post 측) 권한 헬퍼 통과한 결과만 (audience)
       5. snippet 추출, score 계산
       6. 결과 200 반환
```

### 2.4 의존성

| 컴포넌트 | 의존 | 목적 |
|---|---|---|
| embedding.service | Ollama HTTP | 임베딩 호출 |
| search.service (auth) | embedding.service, prisma, project internal HTTP | 통합 검색 |
| post.service (확장) | embedding.service, fire-and-forget | 인덱싱 hook |
| work-log.service (확장) | embedding.service | 인덱싱 hook |
| pgvector | PostgreSQL 16 | 벡터 저장·검색 |

```prisma
model WorkLog {
  // ... 기존
```


### 3.4 임베딩 입력 텍스트 포맷

| 도메인 | 입력 텍스트 |
|---|---|
| Post | `${title}\n\n${content}` (제목·본문) |
| WorkLog | `[${taskName}] ${segmentName ?? ""}: ${content}` (작업명·세그먼트·본문) |

**길이 상한**: bge-m3는 8192 token 지원, 안전을 위해 client에서 5000자 truncate.

### 4.2 Internal API (project-service)

#### `POST /internal/work-logs/semantic-search`

**Headers:** `X-Internal-Token: ...`

**Request:**
```json
{
  "queryVec": [0.123, -0.456, ...],   // 1024 차원
  "userId": "...",
  "userEmail": "...",
  "userRole": "ADMIN|MANAGER|OPERATOR|VIEWER",
  "limit": 40
}
```

**Response (200):**
```json
[
  {
    "id": "ckxx",
    "taskId": "...",
    "taskName": "납품",
    "segmentName": "납품 2",
    "projectId": "...",
    "projectName": "KHOA 주요해역부이 신설",
    "authorId": "...",
    "authorName": "심윤송",
    "content": "...",
    "workedAt": "2026-04-27",
    "score": 0.821
  }

```

내부 처리:
1. `<=>` 거리 연산자로 top K 후보 (limit*2)
2. 권한 필터: `isProjectMember(userEmail, projectId)` 또는 ADMIN
3. score = `1 - distance` (cosine similarity)
4. 정렬: score desc

### 4.3 권한 필터링 의사코드

```ts
// auth-service/search.service.ts
async function search(q: string, scope: string, limit: number, user: AuthUser) {
  const queryVec = await embeddingService.embedText(q);

  const tasks: Promise<Result[]>[] = [];

  if (scope === "all" || scope === "posts") {
    tasks.push(searchPosts(queryVec, limit * 2, user));
  }
  if (scope === "all" || scope === "worklogs") {
    tasks.push(searchWorkLogsRemote(queryVec, limit * 2, user));
  }

  const results = (await Promise.all(tasks)).flat();
  results.sort((a, b) => b.score - a.score);
  return results.slice(0, limit);
}

async function searchPosts(queryVec: number[], k: number, user: AuthUser): Promise<Result[]> {
  const rows = await prisma.$queryRaw<any[]>`
    SELECT p.id, p.title, p.content, p.published_at, p.board_id, p.author_id,
           u.name AS author_name, b.code AS board_code, b.name AS board_name,
           c.code AS cat_code,
           1 - (p.embedding <=> ${queryVec}::vector) AS score
    FROM public.board_posts p
    JOIN public.boards b ON b.id = p.board_id
    JOIN public.board_categories c ON c.id = b.category_id
    JOIN public.auth_users u ON u.id = p.author_id
    WHERE p.embedding IS NOT NULL AND p.is_deleted = false
    ORDER BY p.embedding <=> ${queryVec}::vector
    LIMIT ${k}
  `;

  // 권한 필터: 각 row에 대해 canRead(board, post, user) 적용
  // (board.readAudience와 post.targetDepartmentId 검증)
  const filtered = await filterByPermission(rows, user);
  return filtered.map(toPostResult);
}
```

## 6. Embedding 흐름

### 6.1 글 작성 시 (post.service.ts 수정)

```ts
async create(boardCode: string, data: ..., user: ...) {
  const post = await prisma.$transaction(async (tx) => {
    const created = await tx.post.create({ data: {...} });
    if (data.attachmentIds) await tx.attachment.updateMany({...});
    return created;
  });

  // fire-and-forget: 임베딩
  void embedAndStore(post.id, post.title, post.content);

  return post.id;
}

async function embedAndStore(postId: string, title: string, content: string) {
  try {
    const text = `${title}\n\n${content}`.slice(0, 5000);
    const vec = await embeddingService.embedText(text);
    await prisma.$executeRaw`
      UPDATE public.board_posts
      SET embedding = ${vec}::vector, embedded_at = NOW()
      WHERE id = ${postId}
    `;
  } catch (err) {
    logger.error({ err, postId }, "[embed] failed");
  }
}
```

### 6.2 글 수정 시

`update()` 끝에서 동일 fire-and-forget. 본문 변경 시에만 재임베딩 (`if (data.title !== undefined || data.content !== undefined)`).

### 6.3 글 삭제 시 (소프트)

`isDeleted=true` 설정만, 임베딩 컬럼은 그대로. 검색 쿼리 WHERE에 `is_deleted = false` 항상 포함.

### 6.4 백필 (1회성 cli)

```bash
pnpm --filter @erp-ot/auth-service exec tsx scripts/backfill-embeddings.ts
pnpm --filter @erp-ot/project-service exec tsx scripts/backfill-embeddings.ts
```

흐름:
1. `WHERE embedding IS NULL` 글 100개씩 조회
2. 임베딩 → UPDATE
3. progress log (예: 250/1000 완료)
4. 실패는 skip + 재실행 가능 (idempotent)

## 7. 에러 처리

### 7.1 에러 코드

| HTTP | Code | 메시지 | 원인 |
|---|---|---|---|
| 400 | QUERY_TOO_SHORT | 검색어가 너무 짧습니다 | q 길이 < 2 |
| 401 | UNAUTHENTICATED | 인증 필요 | 토큰 없음 |
| 503 | SEARCH_UNAVAILABLE | 검색 서비스를 사용할 수 없습니다 | Ollama 다운/timeout |
| 500 | INTERNAL_ERROR | 서버 오류 | 기타 예외 |

### 7.2 Graceful Degradation

| 상황 | 동작 |
|---|---|
| Ollama 다운 | 검색은 503, 글 작성은 정상 (임베딩만 실패 — 추후 백필) |
| pgvector 누락 | 검색 503 + 명확한 로그 |
| project-service 응답 없음 | 검색은 posts만 반환 (worklog 부분 실패 허용), warning 로그 |

## 12. 마이그레이션 / 롤백

### 12.1 마이그레이션

```sql
-- public schema (auth)
CREATE EXTENSION IF NOT EXISTS vector;
ALTER TABLE public.board_posts ADD COLUMN embedding vector(1024), ADD COLUMN embedded_at TIMESTAMP;
CREATE INDEX board_posts_embedding_hnsw_idx ON public.board_posts USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);

-- project schema
ALTER TABLE project.work_logs ADD COLUMN embedding vector(1024), ADD COLUMN embedded_at TIMESTAMP;
CREATE INDEX work_logs_embedding_hnsw_idx ON project.work_logs USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
```

### 12.2 롤백

```sql
DROP INDEX board_posts_embedding_hnsw_idx;
DROP INDEX work_logs_embedding_hnsw_idx;
ALTER TABLE public.board_posts DROP COLUMN embedding, DROP COLUMN embedded_at;
ALTER TABLE project.work_logs DROP COLUMN embedding, DROP COLUMN embedded_at;
-- (vector 확장은 DROP 안 함)
```

### 12.3 디스크 사용량

- Embedding 1024 차원 × float4 × 글 1만 건 = ~40MB (board) + 40MB (worklog) = **약 80MB**
- HNSW 인덱스: ~3배 = ~240MB
- 총 약 320MB (글 1만+1만 기준)
