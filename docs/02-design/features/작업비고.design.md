## 1. 개요

### 1.1 설계 목표

1. **단일 모델 도메인** — Task에 종속된 시계열 기록만 처리, 부수 모델 추가 없이 단순화
2. **services/project 통합** — Task·TaskSegment·SegmentAssignment·Resource 모두 같은 schema에 있어 cross-service 호출 0건
3. **권한 헬퍼 캐싱** — `canCreateWorkLog`는 SegmentAssignment 조회를 1회/요청으로 제한
4. **UI 재활용** — 게시판의 PostMarkdownView·sanitize 라이브러리 그대로 활용
5. **별도 페이지 자동 집계** — `/work-logs` 페이지 = WorkLog 테이블의 read-only 뷰, Post 모델 변환 없음

### 1.2 설계 원칙

- **Single Model**: WorkLog 테이블 1개로 끝, Comment/Attachment 등 추가 모델 없음
- **No Cross-Service**: services/project 내 완결, attendance/auth 호출 없음
- **Pure Permission Helpers**: `canCreate/canEdit/canDelete` 외부 의존 0
- **Soft Delete**: 모든 변경은 `isDeleted` 플래그 (감사 추적)
- **Snapshot Author**: `authorId` 작성 시점 사용자 ID 그대로 (이름 변경 영향 없음)

### 2.2 데이터 흐름 — WorkLog 생성

```
사용자 → POST /api/v1/tasks/:taskId/work-logs
  ├─ 1. authenticate hook (req.userId, req.userRole)
  ├─ 2. canCreateWorkLog(taskId, user, prisma):
  │     ├─ ADMIN/MANAGER → 항상 true
  │     └─ OPERATOR → isAssignedToTask(userId, taskId)
  │         └─ Resource.findFirst(userId=req.userEmail) → resourceId
  │             └─ SegmentAssignment.findFirst(resourceId, segment.taskId=taskId) → 존재?
  ├─ 3. WorkLog.create({ taskId, segmentId?, authorId, content, workedAt })
  └─ 4. 201 → WorkLog 객체 반환
```

> 주의: `Resource.userId`는 user-service의 email을 보관 (실제 데이터: `"yssim@oceant.onmicrosoft.com"`). 내부 호출에 `req.userEmail`을 사용해야 함 (project-service 기존 my-tasks.routes.ts와 동일 패턴).

## 3. 데이터 모델

### 3.1 ER 관계

```
Task 1 ─── N WorkLog
TaskSegment 0..1 ─── N WorkLog
User (authorId, snapshot)
```

### 3.2 Prisma 스키마 (services/project/prisma/schema.prisma 추가분)

```prisma
model WorkLog {
  id          String    @id @default(cuid())
  taskId      String
  segmentId   String?
  authorId    String                                  // user-service userId (cuid)
  authorName  String                                  // 작성 시점 스냅샷 (감사 추적·표시용)
  content     String    @db.Text                      // markdown
  workedAt    DateTime  @db.Date                      // 실제 작업 일자 (소급 입력)
  isDeleted   Boolean   @default(false)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  task    Task         @relation(fields: [taskId], references: [id], onDelete: Cascade)
  segment TaskSegment? @relation(fields: [segmentId], references: [id], onDelete: SetNull)

  @@index([taskId, isDeleted, workedAt(sort: Desc)])  // 작업별 시계열 (핫패스)
  @@index([authorId, workedAt(sort: Desc)])           // 본인 작업일지
  @@index([segmentId])                                 // 세그먼트별 필터
  @@map("work_logs")
  @@schema("project")
}
```

> **Task / TaskSegment 모델에 역참조 추가**:
> ```prisma
> model Task {
>   ...
>   workLogs WorkLog[]
> }
> model TaskSegment {
>   ...
>   workLogs WorkLog[]
> }
> ```

### 3.3 핵심 인덱스 설계

| 인덱스 | 용도 | 비용 |
|---|---|---|
| `WorkLog(taskId, isDeleted, workedAt desc)` | 작업별 시계열 (TaskDrawer 비고 탭) | 핵심 핫패스 |
| `WorkLog(authorId, workedAt desc)` | 본인 작업일지 (`/me/work-logs`) | 보조 |
| `WorkLog(segmentId)` | 세그먼트 필터 | 보조 |

> **추가 검토**: 프로젝트 전체 시계열은 Task → workLogs join으로 처리. 별도 `(projectId, workedAt desc)` 인덱스는 추후 데이터 양 보고 결정.

## 4. API 명세

### 4.1 엔드포인트 목록 (7개)

| Method | Path | 설명 | 권한 |
|---|---|---|---|
| GET | `/api/v1/tasks/:taskId/work-logs` | 작업별 시계열 | 인증 |
| POST | `/api/v1/tasks/:taskId/work-logs` | 작성 | canCreate |
| PATCH | `/api/v1/work-logs/:id` | 수정 | canEdit |
| DELETE | `/api/v1/work-logs/:id` | 소프트 삭제 | canDelete |
| GET | `/api/v1/me/work-logs?from&to&projectId&limit` | 본인 시계열 | 인증 |
| GET | `/api/v1/projects/:projectId/work-logs?from&to&authorId` | 프로젝트 통합 시계열 | 인증 |
| GET | `/api/v1/me/work-log-projects` | 본인이 비고 작성한/참여한 프로젝트 목록 (사이드바용) | 인증 |

### 4.2 주요 엔드포인트 상세

#### `GET /api/v1/tasks/:taskId/work-logs`

**Query Params:**
| 파라미터 | 타입 | 기본 | 설명 |
|---|---|---|---|
| `segmentId` | string | — | 특정 세그먼트로 필터 |
| `limit` | number | 100 | 최대 |

**Response (200):**
```json
[
  {
    "id": "ckxx",
    "taskId": "tsk_1",
    "segmentId": "seg_2",
    "segmentName": "납품 2",
    "authorId": "u1",
    "authorName": "심윤송",
    "content": "## 오늘 한 일\n- A\n- B",
    "workedAt": "2026-04-27",
    "isDeleted": false,
    "createdAt": "2026-04-27T10:00:00Z",
    "updatedAt": "2026-04-27T10:00:00Z"
  }
]
```

> 최대 100건 (TaskDrawer 1개 작업 기준), 정렬: `workedAt desc, createdAt desc`. 삭제된 항목은 ADMIN만 포함.

#### `POST /api/v1/tasks/:taskId/work-logs`

**Request:**
```json
{
  "content": "오늘 작업 내용...",
  "workedAt": "2026-04-27",
  "segmentId": "seg_2"
}
```

**Response (201):** WorkLog 객체

**Errors:**
- `400 INVALID_INPUT` — Zod 위반 (content 1~50KB, workedAt YYYY-MM-DD)
- `403 FORBIDDEN_WORK_LOG_CREATE` — 비배정자 (OPERATOR가 본인 배정 외 작업에 작성)
- `404 TASK_NOT_FOUND` — 작업 없음
- `404 SEGMENT_NOT_FOUND` — segmentId가 해당 task에 없음

#### `PATCH /api/v1/work-logs/:id`

**Request:** `{ content?, workedAt? }`
**Response (200):** WorkLog 객체
**Errors:**
- `403 FORBIDDEN_WORK_LOG_EDIT` — 본인 아님 + ADMIN 아님
- `404 WORK_LOG_NOT_FOUND` — 글 없음 또는 이미 삭제됨

#### `DELETE /api/v1/work-logs/:id`

**Response (204):** 본문 없음 (소프트 삭제)
**Errors:**
- `403 FORBIDDEN_WORK_LOG_EDIT`
- `404 WORK_LOG_NOT_FOUND`

#### `GET /api/v1/projects/:projectId/work-logs`

**Query Params:**
| 파라미터 | 타입 | 기본 | 설명 |
|---|---|---|---|
| `from` | YYYY-MM-DD | -30d | 조회 시작 |
| `to` | YYYY-MM-DD | today | 조회 끝 |
| `authorId` | string | — | 작성자 필터 |
| `taskId` | string | — | 작업 필터 |
| `limit` | number | 200 | 최대 |
| `cursor` | string | — | 페이지네이션 (lastWorkedAt:lastId) |

**Response (200):**
```json
{
  "items": [
    {
      "id": "...",
      "taskId": "...",
      "taskName": "납품",
      "segmentId": "...",
      "segmentName": "납품 2",
      "authorId": "...",
      "authorName": "심윤송",
      "content": "...",
      "workedAt": "2026-04-27",
      "createdAt": "...",
      "updatedAt": "..."
    }
  ],
  "nextCursor": "2026-04-26:ckxx" | null
}
```

> 정렬: `workedAt desc, createdAt desc, id desc`. 권한: 프로젝트 멤버(SegmentAssignment 보유자) + ADMIN/MANAGER. 비-멤버는 403 FORBIDDEN_READ.

#### `GET /api/v1/me/work-log-projects`

본인이 WorkLog를 작성한 적 있거나, 본인이 자원으로 배정된 프로젝트 목록 — **프로젝트 게시판 사이드바용**.

**Response (200):**
```json
[
  { "projectId": "p1", "projectName": "...", "status": "ACTIVE", "logCount": 12, "lastLogAt": "2026-04-27" },
  ...
]
```

> ADMIN은 전체 프로젝트 + 각각 logCount/lastLogAt.

### 4.3 권한 헬퍼 의사코드

```ts
// services/project/src/application/work-log-permissions.ts

export interface WorkLogContext {
  authorId: string;
  isDeleted: boolean;
}

export interface AuthUser {
  id: string;       // userId (cuid)
  email: string;    // user.email
  role: string;     // ADMIN/MANAGER/OPERATOR/VIEWER
}

// 단순 권한 (DB 조회 없음)
export function canEditWorkLog(log: WorkLogContext, user: AuthUser): boolean {
  if (log.isDeleted && user.role !== "ADMIN") return false;
  return user.role === "ADMIN" || log.authorId === user.id;
}
export const canDeleteWorkLog = canEditWorkLog;

// 복합 권한 (Resource + SegmentAssignment 조회 필요)
export async function canCreateWorkLog(
  prisma: PrismaClient,
  taskId: string,
  user: AuthUser,
): Promise<boolean> {
  if (user.role === "ADMIN" || user.role === "MANAGER") return true;
  if (user.role === "VIEWER") return false;
  return await isAssignedToTask(prisma, user.email, taskId);
}

export async function isAssignedToTask(
  prisma: PrismaClient,
  userEmail: string,
  taskId: string,
): Promise<boolean> {
  const resource = await prisma.resource.findFirst({
    where: { userId: userEmail },
    select: { id: true },
  });
  if (!resource) return false;
  const assignment = await prisma.segmentAssignment.findFirst({
    where: {
      resourceId: resource.id,
      segment: { taskId },
    },
    select: { id: true },
  });
  return assignment !== null;
}

// 프로젝트 멤버 검증 (project read 권한)
export async function isProjectMember(
  prisma: PrismaClient,
  userEmail: string,
  projectId: string,
): Promise<boolean> {
  const resource = await prisma.resource.findFirst({
    where: { userId: userEmail },
    select: { id: true },
  });
  if (!resource) return false;
  const assignment = await prisma.segmentAssignment.findFirst({
    where: {
      resourceId: resource.id,
      segment: { task: { projectId } },
    },
    select: { id: true },
  });
  return assignment !== null;
}
```

> 비용: `isAssignedToTask` = 2 쿼리 (Resource + SegmentAssignment). 한 요청에서 1회 호출이라 캐싱 불필요. 다중 검증 필요 시 service 메서드 안에서 result 재사용.

## 6. 에러 처리

### 6.1 에러 코드 정의

| HTTP | Code | 메시지 | 원인 | 처리 |
|---|---|---|---|---|
| 400 | INVALID_INPUT | 입력값 오류 | Zod 위반 | 입력 재요청 |
| 401 | UNAUTHENTICATED | 인증 필요 | 토큰 없음/만료 | /login 리다이렉트 |
| 403 | FORBIDDEN_WORK_LOG_CREATE | 비고 작성 권한 없음 | 비배정자 OPERATOR | "권한 없음" 표시 |
| 403 | FORBIDDEN_WORK_LOG_EDIT | 비고 수정/삭제 권한 없음 | 본인 아님 | "권한 없음" 표시 |
| 403 | FORBIDDEN_READ | 프로젝트 조회 권한 없음 | 비-멤버 | 빈 결과 또는 메시지 |
| 404 | TASK_NOT_FOUND | 작업 없음 | id 오류 | 토스트 |
| 404 | SEGMENT_NOT_FOUND | 세그먼트 없음 또는 task 불일치 | id 오류 | 토스트 |
| 404 | WORK_LOG_NOT_FOUND | 비고 없음 또는 삭제됨 | id 오류 | 토스트 후 새로고침 |
| 500 | INTERNAL_ERROR | 서버 오류 | 예외 | 로그 + 토스트 |

### 6.2 에러 응답 포맷 (게시판과 동일 SSOT)

```json
{
  "error": {
    "code": "FORBIDDEN_WORK_LOG_CREATE",
    "message": "이 작업에 비고를 작성할 권한이 없습니다.",
    "details": { "taskId": "tsk_1" }
  }
}
```

### 8.1 테스트 범위

| 유형 | 대상 | 도구 |
|---|---|---|

| Manual | TaskDrawer 비고 탭, 프로젝트 게시판 페이지 | 브라우저 |

### 8.2 핵심 테스트 케이스

| FR | 케이스 | 기대 결과 |
|---|---|---|
| FR-03 | 배정자 OPERATOR가 본인 배정 작업에 작성 | 201 |
| FR-03 | 비배정 OPERATOR가 작성 시도 | 403 FORBIDDEN_WORK_LOG_CREATE |
| FR-03 | ADMIN/MANAGER가 임의 작업에 작성 | 201 |
| FR-04 | 본인 비고 수정 | 200 |
| FR-04 | 타인 비고 수정 (non-ADMIN) | 403 |
| FR-04 | ADMIN이 타인 비고 수정 | 200 |
| FR-05 | 본인 비고 삭제 (소프트) | 204, isDeleted=true |
| FR-07 | 프로젝트 멤버가 통합 조회 | 200 |
| FR-07 | 비-멤버 (non-ADMIN) | 403 FORBIDDEN_READ |
| FR-13 | 사이드바: 본인 참여 프로젝트만 (ADMIN은 전체) | 권한별 다른 결과 |
| FR-19 | XSS payload (`<script>`) | sanitize 적용 |
| 추가 | segmentId가 다른 task 소속 → POST | 404 SEGMENT_NOT_FOUND |

### 8.3 Happy / Edge / Error

- **Happy**: 배정자 → TaskDrawer 비고 탭 → 작성 → 다른 날 소급 입력 → 프로젝트 게시판에서 확인
- **Edge**: 작업 삭제 시 WorkLog 동반 삭제 (CASCADE) — Task.onDelete: Cascade
- **Edge**: 세그먼트 삭제 시 segmentId만 NULL로 보존 (WorkLog는 유지) — SetNull
- **Error**: 인덱스 사용량 모니터링 — 데이터 누적 후 (projectId 인덱스) 추가 결정

## 12. 마이그레이션 / 롤백

### 12.1 마이그레이션 (게시판과 동일 패턴)

```bash
# project schema는 multi-schema이므로 schema-prefix 명시 필요할 수 있음
cd services/project
DATABASE_URL="postgresql://erp_user:erp_password@localhost:5432/erp_ot" \
  pnpm exec prisma migrate diff --from-schema-datasource prisma/schema.prisma \
  --to-schema-datamodel prisma/schema.prisma --script \
  > prisma/migrations/$(date +%Y%m%d%H%M%S)_add_work_log/migration.sql

cat prisma/migrations/.../migration.sql | docker exec -i erp-ot-postgres psql -U erp_user -d erp_ot
```
