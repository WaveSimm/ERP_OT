### v1.6 변경 내역 (2026-05-04 — 휴가 유형 정리)

**배경**: v1.5에서 시간 단위 휴가 도입 후, 사용자 운영 검토 결과 단순화 필요.
- 반차 시작 시간으로 오전/오후 자동 구분 가능 → HALF_AM/HALF_PM 분리 무의미
- 가정의날 1H/2H를 enum 분리 → endTime 사용자 입력 로직 + FAMILY_MAX_MINUTES 검증 폐기 가능

**Schema 변경** (data 손실 OK, 사용자 동의):
- `LeaveType` enum: HALF_AM/HALF_PM 폐기 → **HALF** 단일, **FAMILY_DAY_2H** 추가
- `EntryType` enum: 동일 (HALF, FAMILY_DAY_2H)

**최종 휴가 유형**:
| 라벨 | enum | 시간 입력 | duration |
|------|------|:--------:|:--------:|
| 연차 | ANNUAL | ❌ | 평일 |
| 반차 | HALF | ✅ startTime | 4h (0.5일) |
| 1/4연차 | QUARTER | ✅ startTime | 2h (0.25일) |
| 가정의날(1H) | FAMILY_DAY | ✅ startTime | 1h (0.125일) |
| 가정의날(2H) | FAMILY_DAY_2H | ✅ startTime | 2h (0.25일) |
| 병가 | SICK | ❌ | 평일 |
| 경조사 | BEREAVEMENT | ❌ | 평일 |
| 공가 | SPECIAL | ❌ | 평일 |

**Backend 단순화** (`leave.service.ts`):
- `KOR_TO_LEAVE_TYPE`: 새 enum 매핑 (legacy "반차(오전)/(오후)"는 HALF로 alias)
- `TIME_BASED_TYPES = ["HALF", "QUARTER", "FAMILY_DAY", "FAMILY_DAY_2H"]`
- `TYPE_DEFAULT_MINUTES`: 모두 type별 고정 (HALF=240, QUARTER=120, FAMILY_DAY=60, FAMILY_DAY_2H=120)
- **`resolveTimeRange` 단순화** — 모든 type이 startTime + 고정 duration → endTime 자동
- **`FAMILY_MAX_MINUTES` 검증 폐기** — enum으로 자연 보장
- `calcDays`: type별 고정값 (시간 단위는 enum별 고정)

**Frontend 단순화** (`approval/new + edit`):
- `isFamilyDay()` helper 폐기 — endTime 사용자 입력 케이스 사라짐
- 모든 시간 단위 휴가의 endTime input은 항상 disabled (자동)
- `getTimeBasedDuration()` 4종 매핑 (반차, 1/4연차, 가정의날(1H), 가정의날(2H))
- `isTimeBasedLeave()` 4종 판정

**seed.ts**:
- LEAVE template options: `["연차", "반차", "1/4연차", "가정의날(1H)", "가정의날(2H)", "병가", "경조사", "공가"]`

**라이브 검증**:
- 반차 09:30 → 자동 13:30 (HALF, 0.5일) ✅
- 가정의날(1H) 17:00 → 자동 18:00 (FAMILY_DAY, 0.125일) ✅
- 가정의날(2H) 15:00 → 자동 17:00 (FAMILY_DAY_2H, 0.25일) ✅

### v1.5 변경 내역 (2026-05-04 — LEAVE 시간 단위 휴가 + enum 분리)

**배경**: v1.4에서 LEAVE 전자결재 통합 후, 회사 실제 운영에서 1/4연차(2h)·가정의날(1h, 최대 2h)·반차 시간대 운영이 필요 — 시간 단위 입력 추가. 동시에 기존 enum `FAMILY` 가 가정의날·경조사 두 의미 혼용 → 분리.

**Schema 변경**:
- `LeaveType` enum: `FAMILY` 폐기 → `FAMILY_DAY` (가정의날, 1h default) + `BEREAVEMENT` (경조사, 일 단위) 추가
- `EntryType` enum: 동일 (`FAMILY` → `FAMILY_DAY` + `BEREAVEMENT`)
- 기존 LeaveRequest 7건 손실 (사용자 동의, 배포 전)

**시간 단위 휴가 로직** (`services/attendance/src/application/leave.service.ts`):
- `TIME_BASED_TYPES = ["HALF_AM", "HALF_PM", "QUARTER", "FAMILY_DAY"]`
- `TYPE_DEFAULT_MINUTES`: HALF_AM/PM=240, QUARTER=120, FAMILY_DAY=60
- 시작 시간만 받고 type 기반 자동 종료 시간 (FAMILY_DAY만 사용자가 1h or 2h 명시 가능)
- `FAMILY_MAX_MINUTES = 120` (가정의날 최대 2시간 검증, `FAMILY_DAY_OVER_LIMIT` 에러)
- days 계산: 시간 단위면 `minutesToDays` (1h = 0.125일), 일 단위면 기존 `calcDays`
- WorkScheduleEntry에 startTime/endTime 같이 저장 → 자원현황·캘린더 등에서 활용

**KOR_TO_LEAVE_TYPE 매핑 갱신**:
- "1/4연차" → QUARTER
- "가정의날" → FAMILY_DAY (신규)
- "경조사" → BEREAVEMENT (신규, 이전 FAMILY로 잘못 매핑됨)

**Frontend** (`apps/web/src/app/approval/new/page.tsx` + `[id]/edit/page.tsx`):
- LEAVE 템플릿 fields: startTime + endTime 추가 (옵션)
- 시간 input 항상 표시 (옵션이라는 의도), leaveType이 시간 단위가 아니면 disabled
- leaveType 변경 시 시간 input 자동 활성/비활성·재계산
- startTime onChange → type 기반 endTime 자동 (FAMILY_DAY 제외, 사용자 입력 보존)
- 가정의날 endTime은 사용자 입력 가능 (1h or 2h)
- 자동 제목에 시간 정보 (`[1/4연차] 5/19 14:00~16:00 - 개인사정`)
- LEAVE 12-col 레이아웃: row1 leaveType(4)+startDate(4)+endDate(4), row2 startTime(4)+endTime(4)+빈(4), row3 reason(12)

**도구 분류**:
- 시간 입력 활성화 휴가: 반차(오전), 반차(오후), 1/4연차, 가정의날
- 시간 입력 비활성화 휴가: 연차, 병가, 경조사, 공가/특별휴가

**테스트 통과**:
- 반차 09:30 → 자동 13:30 (4h, 0.5일)
- 1/4연차 14:00 → 자동 16:00 (2h, 0.25일)
- 가정의날 17:00~18:00 (1h, 0.125일)
- 가정의날 16:00~18:00 (2h, 0.25일)
- 가정의날 15:00~18:00 거부 (FAMILY_DAY_OVER_LIMIT)
- 경조사 5/25~5/26 (시간 무관, 2일)

**자원현황 (Resource utilization)** — backend WorkScheduleEntry에 시간 데이터 저장됨. UI 시각화는 후속 PDCA `인력현황-계산-개선`에서 활용.

### v1.4 변경 내역 (2026-05-04 — LEAVE 전자결재 통합)

**배경**: v1.3에서 OT(휴일근무)를 전자결재로 통합한 패턴을 LEAVE에도 동일 적용. 기존 LEAVE는 두 갈래(대시보드 모달 직접 신청 + 전자결재 LEAVE 템플릿 half-broken) — 사용자 혼란 + 전자결재 referenceId 미설정으로 deadcode.

**적용 (OT v1.3 패턴 재사용)**:
1. **대시보드 모달 폐기 → 전자결재 redirect**: `apps/web/src/components/AttendanceView.tsx:LeaveRequestForm` 73줄 모달 → 9줄 redirect 버튼 (`/approval/new?template=LEAVE`)
2. **전자결재 LEAVE 템플릿**: 기존 fields 그대로 (leaveType select / startDate / endDate / reason)
3. **postAction 재작성**: `services/approval/src/application/document.service.ts:LEAVE_APPROVE`
   - Before: `referenceId` 기반 (attendance pre-create 필요 — 실제론 안 됨, broken)
   - After: `doc.fields`에서 leaveType/startDate/endDate/reason 추출 → `/internal/leave/from-approval` 직접 호출
4. **신규 internal endpoint**: `services/attendance/src/api/routes/internal.routes.ts:POST /internal/leave/from-approval`
5. **신규 service 메서드**: `services/attendance/src/application/leave.service.ts:createFromApproval`
   - 한국어 → enum 매핑 (KOR_TO_LEAVE_TYPE: "연차"→ANNUAL 등)
   - 반차/QUARTER/FAMILY: endDate=startDate 강제
   - balance 사전 검증 (`INSUFFICIENT_BALANCE` 에러 — pendingDays 거치지 않고 usedDays 즉시 증가)
   - WorkScheduleEntry 자동 반영 (entryType 매핑)
6. **자동 제목 생성** (UI): `[연차] 5/9~5/12 - 가족여행`
7. **UI 레이아웃 정합**: `/approval/new` + `/approval/[id]/edit` + `/approval/[id]` 모두 LEAVE 12-col grid (leaveType+startDate+endDate 한 줄, reason 전체 폭)

**제거된 것**: 대시보드 LeaveRequestForm 모달 (반차 시간 입력, 잔여 잔여 미리보기 등) — 전자결재 폼으로 통합

**효과**:
- 사용자 진입점 통일 (모든 결재 신청은 `/approval/new`)
- 결재 워크플로 일관 (휴가/OT/구매/지출 모두 동일 UX)
- 코드 중복 감소 (LeaveRequestForm 73줄 → 9줄)
- 전자결재 LEAVE 템플릿 정상 동작 (이전엔 dead code)

**LEAVE-OT 차이 인정**:
- LEAVE는 start~end range (OT는 multi-date) — 사용자 의도와 정합
- LEAVE는 leaveType 한국어 select (OT는 type 없음) — 매핑 backend
- LEAVE는 LeaveBalance 차감 (OT는 balance 무관) — 사전 검증 로직 보존

### v1.3 변경 내역 (2026-05-04 — OT-단순화 브레인스토밍 결과)

**배경**: 기존 `OvertimeRequest` 모델이 너무 복잡 (다단계 결재·시간/임금 디테일·평일 OT 포함)해서 결재 부담이 큼. 후속 PDCA `인력현황-계산-개선`에서 휴일근무 데이터를 활용해야 하는데, 현재 모델로는 실데이터가 빈약. 단순화 후 데이터 활용성 ↑.

**5가지 결정 (브레인스토밍 Q1~Q5)**:

| Q | 결정 | 사유 |
|---|------|------|
| Q1 | **평일 야근 신청 폐기** | 출퇴근 시스템으로 자동 인식, 신청 자체 불필요 |
| Q2 | **전일 8h 단일 옵션** | 임금 연계는 별도 시스템, 시간 디테일 불요 |
| Q3 | **결재라인 시스템 활용 (1단계 default, 2~3단계 확장 가능)** | 휴가/출장과 동일 인프라, ApprovalLine 변경만으로 단계 변경 |
| Q4 | **OvertimeRequest 폐기 + HolidayWorkRequest 신규 (clean slate)** | 배포 전이므로 마이그레이션 부담 0 |
| Q5 | **projectId + taskId 옵션 (segment 자동 추론)** | 인력현황 PDCA Phase 2에서 segment-level capacity 계산에 활용 |

**신규 모델 명세**:
```prisma
model HolidayWorkRequest {
  id               String          @id @default(cuid())
  userId           String
  date             DateTime        @db.Date  // 휴일/주말이어야 (검증)
  reason           String
  status           ApprovalStatus
  approverId       String?
  secondApproverId String?         // 다단계 옵션 (default null = 1단계)
  thirdApproverId  String?         // 다단계 옵션 (default null)
  approvedAt       DateTime?
  rejectReason     String?
  projectId        String?         // 옵션 — 어느 프로젝트 위해 휴일근무
  taskId           String?         // 옵션 — 어느 task 위해 (segment 자동 추론)
  createdAt        DateTime        @default(now())
  updatedAt        DateTime        @updatedAt

  @@index([userId, status])
  @@index([approverId, status])
  @@index([secondApproverId, status])
  @@index([thirdApproverId, status])
  @@index([date])
  @@map("holiday_work_requests")
  @@schema("attendance")
}
```

**제거된 컬럼 (vs 기존 OvertimeRequest)**: plannedHours, actualHours, segmentId, completedAt, overtimeRate

**변경 영향 범위**:
- **Backend**: `services/attendance/` — overtime.routes/service 폐기, holiday-work.routes/service 신규
- **Frontend**: `apps/web/src/components/AttendanceView.tsx`, `MyAttendanceCard.tsx`, `AttendanceOverview.tsx` — "OT" 라벨 → "휴일근무" 변경
- **WorkScheduleEntry.entryType.OT**: 의미 재해석 — 휴일근무로 사용 (enum 값 변경 없음, 호환 유지)
- **자동 동기화 hook**: `OvertimeService.approve` → `HolidayWorkService.approve` (기존 패턴 재사용)
- **신청 가능 날짜 검증**: 회사달력의 PUBLIC_HOLIDAY/COMPANY_HOLIDAY + 토/일 + WORKDAY 제외

**후속 PDCA 의존성**: 본 변경 완료 후 `인력현황-계산-개선` PDCA가 HolidayWorkRequest 데이터로 segment-level capacity 계산.

**Design 디테일 (Do 단계에서 확정)**:
1. WorkScheduleEntry.OT enum 값 처리 (유지 + 의미 재해석 vs HOLIDAY_WORK 신규)
2. 신청 가능 날짜 검증 로직 (회사달력 internal API 호출)
3. UI 라벨 변경 범위 전수 (`summary.totalOtHours` → `summary.totalHolidayWorkHours` 호환)
4. project/task 입력 UI (드롭다운 + 자동 필터)
5. ApprovalLine 미설정 사용자 처리 (default 부서장 자동 지정 vs 신청 거부)

## 2. 데이터 모델

*(Plan v1.1과 동일 — 이미 구현 완료)*

```prisma
enum EntryType { WORK FIELD TRAINING BUSINESS_TRIP HALF_AM HALF_PM ANNUAL SICK SPECIAL OT }
enum EntrySource { MANUAL LEAVE_APPROVED OT_APPROVED }

model WorkScheduleEntry {
  id          String      @id @default(cuid())
  userId      String
  date        DateTime    @db.Date
  entryType   EntryType
  label       String?
  sourceType  EntrySource @default(MANUAL)
  sourceId    String?
  createdAt   DateTime    @default(now())
  updatedAt   DateTime    @updatedAt

  @@unique([userId, date, entryType, sourceId])
  @@index([userId, date])
  @@index([date])
  @@index([sourceId])
  @@map("work_schedule_entries")
  @@schema("attendance")
}
```
