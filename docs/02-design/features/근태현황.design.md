## 2. 데이터 모델 (Data Model)

### 2.1 신규 Prisma 모델

**파일**: `services/attendance/prisma/schema.prisma`

```prisma
enum EntryType {
  WORK
  FIELD
  TRAINING
  BUSINESS_TRIP
  HALF_AM
  HALF_PM
  ANNUAL
  SICK
  SPECIAL
  OT

  @@schema("attendance")
}

enum EntrySource {
  MANUAL
  LEAVE_APPROVED
  OT_APPROVED
  ATTENDANCE

  @@schema("attendance")
}

model WorkScheduleEntry {
  id          String      @id @default(cuid())
  userId      String
  date        DateTime    @db.Date
  entryType   EntryType
  label       String?
  sourceType  EntrySource @default(MANUAL)
  sourceId    String?
  createdAt   DateTime    @default(now())
  updatedAt   DateTime    @updatedAt

  @@unique([userId, date, entryType, sourceId])
  @@index([userId, date])
  @@index([date])
  @@index([sourceId])
  @@map("work_schedule_entries")
  @@schema("attendance")
}
```

### 2.2 DB 마이그레이션 (init-db.sql 추가)

```sql
-- attendance 스키마에 추가
CREATE TYPE attendance.entry_type AS ENUM (
  'WORK', 'FIELD', 'TRAINING', 'BUSINESS_TRIP',
  'HALF_AM', 'HALF_PM', 'ANNUAL', 'SICK', 'SPECIAL', 'OT'
);

CREATE TYPE attendance.entry_source AS ENUM (
  'MANUAL', 'LEAVE_APPROVED', 'OT_APPROVED', 'ATTENDANCE'
);

CREATE TABLE attendance.work_schedule_entries (
  id          TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
  user_id     TEXT NOT NULL,
  date        DATE NOT NULL,
  entry_type  attendance.entry_type NOT NULL,
  label       TEXT,
  source_type attendance.entry_source NOT NULL DEFAULT 'MANUAL',
  source_id   TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(user_id, date, entry_type, source_id)
);

CREATE INDEX idx_wse_user_date ON attendance.work_schedule_entries(user_id, date);
CREATE INDEX idx_wse_date ON attendance.work_schedule_entries(date);
CREATE INDEX idx_wse_source ON attendance.work_schedule_entries(source_id);
```

## 3. API 설계 (API Design)

### 3.1 근태현황 주간 조회

```
GET /api/v1/work-schedule?start=2026-04-07&end=2026-04-13
Authorization: Bearer <JWT>
```

**처리 흐름**:
1. auth-service `/internal/users/all-with-departments` 호출 → 전체 사용자+부서 목록
2. `WorkScheduleEntry` 테이블에서 date 범위 조회
3. 부서별 그룹핑 + 사용자별 엔트리 매핑

**응답**:
```json
{
  "weekStart": "2026-04-07",
  "weekEnd": "2026-04-13",
  "departments": [
    {
      "id": "dept-1",
      "name": "경영진",
      "sortOrder": 0,
      "members": [
        {
          "userId": "user-1",
          "name": "김대표",
          "entries": [
            {
              "id": "entry-1",
              "date": "2026-04-07",
              "entryType": "WORK",
              "label": null,
              "sourceType": "MANUAL",
              "sourceId": null
            }
          ]
        }
      ]
    }
  ],
  "unassigned": [
    {
      "userId": "user-x",
      "name": "미배정직원",
      "entries": []
    }
  ]
}
```

### 3.2 개인 근태 추가

```
POST /api/v1/work-schedule
Authorization: Bearer <JWT>
Content-Type: application/json

{
  "date": "2026-04-07",
  "entryType": "FIELD",
  "label": "강남 현장"
}
```

**비즈니스 규칙**:
- `req.userId` 자동 적용 (본인만 추가)
- 같은 날짜 + 같은 entryType + sourceId=null 중복 시 에러
- 유효 entryType만 허용 (Zod 검증)

**응답**: `201 Created` + 생성된 엔트리

### 3.3 개인 근태 삭제

```
DELETE /api/v1/work-schedule/:id
Authorization: Bearer <JWT>
```

**비즈니스 규칙**:
- 본인의 항목만 삭제 가능 (`userId === req.userId`)
- `sourceType === "MANUAL"`인 항목만 삭제 가능
- 자동 생성(LEAVE_APPROVED, OT_APPROVED) 항목은 삭제 불가 → 400 에러

**응답**: `204 No Content`

### 3.4 auth-service 내부 API 확장

```
GET /internal/users/all-with-departments
X-Internal-Token: <token>
```

**응답**:
```json
[
  {
    "id": "user-1",
    "name": "김대표",
    "email": "kim@example.com",
    "departmentId": "dept-1",
    "departmentName": "경영진",
    "departmentSortOrder": 0
  }
]
```

기존 `/internal/users/all`은 부서 정보가 없으므로 신규 엔드포인트 추가.

## 4. 자동 반영 로직 (Auto-populate)

### 4.1 휴가 승인 시 (LeaveService.approve 수정)

`leave.service.ts`의 `approve()` 메서드에서 `isFinalApproval === true` 분기에 추가:

```typescript
// 최종 승인 시 WorkScheduleEntry 자동 생성
if (isFinalApproval) {
  const entryType = this.mapLeaveTypeToEntryType(req.type);
  const dates = this.getDateRange(req.startDate, req.endDate);
  for (const date of dates) {
    await tx.workScheduleEntry.upsert({
      where: {
        userId_date_entryType_sourceId: {
          userId: req.userId, date, entryType, sourceId: id,
        },
      },
      create: {
        userId: req.userId, date, entryType,
        sourceType: "LEAVE_APPROVED", sourceId: id,
      },
      update: {},  // 이미 있으면 무시
    });
  }
}
```

**LeaveType → EntryType 매핑**:
| LeaveType | EntryType |
|-----------|-----------|
| ANNUAL | ANNUAL |
| HALF_AM | HALF_AM |
| HALF_PM | HALF_PM |
| SICK | SICK |
| SPECIAL | SPECIAL |

### 4.2 OT 승인 시 (OvertimeService.approve 수정)

```typescript
if (isFinalApproval) {
  await tx.workScheduleEntry.upsert({
    where: {
      userId_date_entryType_sourceId: {
        userId: req.userId, date: req.date, entryType: "OT", sourceId: id,
      },
    },
    create: {
      userId: req.userId, date: req.date, entryType: "OT",
      sourceType: "OT_APPROVED", sourceId: id,
      label: `${req.plannedHours}h`,
    },
    update: {},
  });
}
```

### 4.3 휴가/OT 취소 시

`LeaveService.cancelRequest()` 및 `OvertimeService.cancel()`에서:

```typescript
// 연동 엔트리 삭제
await this.prisma.workScheduleEntry.deleteMany({
  where: { sourceId: id },

```

## 8. auth-service 내부 API 확장

**파일**: `services/auth/src/api/routes/internal.routes.ts`

```typescript
// GET /internal/users/all-with-departments
fastify.get("/users/all-with-departments", async (req, reply) => {
  const profiles = await prisma.userProfile.findMany({
    include: {
      user: { select: { id: true, name: true, email: true } },
      department: { select: { id: true, name: true, sortOrder: true } },
    },
  });
  // 부서 없는 사용자도 포함
  const usersWithDept = profiles.map(p => ({
    id: p.user.id,
    name: p.user.name,
    email: p.user.email,
    departmentId: p.department?.id ?? null,
    departmentName: p.department?.name ?? null,
    departmentSortOrder: p.department?.sortOrder ?? 999,
  }));
  // UserProfile 없는 사용자 추가
  const profileUserIds = new Set(profiles.map(p => p.userId));
  const allUsers = await prisma.user.findMany({
    where: { id: { notIn: Array.from(profileUserIds) } },
    select: { id: true, name: true, email: true },
  });
  for (const u of allUsers) {
    usersWithDept.push({
      id: u.id, name: u.name, email: u.email,
      departmentId: null, departmentName: null, departmentSortOrder: 999,
    });
  }
  return reply.send(usersWithDept);
});
```

## 11. 기술적 결정 사항

| 결정 | 선택 | 대안 | 이유 |
|------|------|------|------|
| 데이터 모델 | 별도 WorkScheduleEntry | AttendanceRecord 확장 | AttendanceRecord는 FSM(출퇴근 상태) 전용, 캘린더 일정은 다른 도메인 |
| 부서 정보 | auth-service 내부 API | attendance DB에 복제 | DB 복제 시 동기화 이슈, 내부 API 호출이 Clean Arch 적합 |
| 자동 반영 | 승인 트랜잭션 내 동기 생성 | RabbitMQ 비동기 이벤트 | MVP 단순성, 트랜잭션 일관성 보장 |
| 프론트 구조 | 단일 컴포넌트 | 별도 페이지 | 자원관리 탭 내 서브 기능, 기존 패턴 유지 |
| 수동 추가 유형 | WORK/FIELD/TRAINING/BUSINESS_TRIP만 | 전체 유형 | 휴가/OT는 결재 시스템 통해서만 반영 (데이터 무결성) |
| 캐싱 | Redis 5분 (부서+사용자 목록) | 캐시 없음 | 부서 정보는 자주 변경되지 않음 |
