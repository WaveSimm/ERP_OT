## 1. 개요 (Overview)

### 1.1 설계 목표

1. **2-level 분류 트리** 구조로 부서별 게시판(2차) 확장 시 코드 변경 최소화
2. **auth-service 통합 설계**로 author/department 정보 cross-service join 회피
3. **권한 가드 헬퍼 (`canWrite/canPin/canEdit/canDelete`)** 명시화로 라우트마다 일관 검증
4. **첨부 저장 추상화** (1차 로컬 볼륨 → 2차 S3 전환 무코드 수정)
5. **알림 발송 hook 분리** (트랜잭션 외부) — 알림 실패가 글 작성을 막지 않도록

### 1.2 설계 원칙

- **Single Responsibility**: Service 1개당 도메인 1개 (Board / Post / Comment / Attachment / NoticeNotification)
- **Pure Domain**: 권한 헬퍼는 외부 의존 없는 순수 함수
- **Fail-Safe Notification**: 알림 발송 실패 → 로그만 남기고 글 작성은 성공
- **Soft Delete**: 모든 컨텐츠는 `isDeleted` 플래그 (감사·복구 대비)
- **Snapshot**: 발행 부서명은 작성 시점 스냅샷 (사용자 부서 이동에 영향 없음)

### 2.2 데이터 흐름 — 글 작성 시퀀스

```
사용자 → POST /api/v1/boards/:code/posts
  ├─ 1. authenticate hook
  ├─ 2. board 조회 + canWrite(board, user) 검증
  ├─ 3. publishingDepartment = user.profile.department (스냅샷)
  ├─ 4. Prisma TX:
  │     ├─ Post 생성
  │     └─ Attachment 생성 (이미 업로드된 파일 ID 연결)
  ├─ 5. TX 외부 (fire-and-forget):
  │     └─ if (board.category === "notice") → noticeNotifyHook(post, audience)
  └─ 6. 201 Created (post detail) 반환
```

### 3.2 Prisma 스키마 (services/auth/prisma/schema.prisma 추가분)

```prisma
// ─── 게시판 도메인 ───────────────────────────────────────────────

model BoardCategory {
  id        String   @id @default(cuid())
  code      String   @unique                              // "notice" | "wiki" | "department"
  name      String                                        // 표시명
  icon      String?                                       // 이모지 또는 아이콘 키
  sortOrder Int      @default(0) @map("sort_order")
  isActive  Boolean  @default(true) @map("is_active")
  isVisible Boolean  @default(true) @map("is_visible")    // FF: 2차에 활성화될 카테고리
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  boards    Board[]

  @@index([sortOrder])
  @@map("board_categories")
}

model Board {
  id              String   @id @default(cuid())
  categoryId      String   @map("category_id")
  code            String   @unique                         // "notice-company", "wiki-debug"
  name            String
  description     String?

  // 권한
  writeRoles      String[]                                 // ["OPERATOR","MANAGER","ADMIN"]
  readAudience    BoardAudience @default(ALL) @map("read_audience")
  audienceTargetId String? @map("audience_target_id")     // deptId or roleName

  // 옵션
  allowComments   Boolean  @default(true)  @map("allow_comments")
  allowAttachments Boolean @default(true)  @map("allow_attachments")
  postPinnable    Boolean  @default(true)  @map("post_pinnable")

  sortOrder       Int      @default(0) @map("sort_order")
  isActive        Boolean  @default(true) @map("is_active")
  createdAt       DateTime @default(now()) @map("created_at")
  updatedAt       DateTime @updatedAt @map("updated_at")

  category        BoardCategory @relation(fields: [categoryId], references: [id])
  posts           Post[]

  @@index([categoryId, sortOrder])
  @@index([isActive])
  @@map("boards")
}

model Post {
  id              String   @id @default(cuid())
  boardId         String   @map("board_id")
  authorId        String   @map("author_id")

  // 발행 부서 스냅샷
  publishingDepartmentId   String? @map("publishing_department_id")
  publishingDepartmentName String? @map("publishing_department_name")

  title           String
  content         String   @db.Text                        // markdown 원문

  // 표시 옵션
  isPinned        Boolean  @default(false) @map("is_pinned")
  priority        Int      @default(0)                     // 0=NORMAL,1=IMPORTANT,2=URGENT

  // 시간
  publishedAt     DateTime @default(now()) @map("published_at")
  expiresAt       DateTime? @map("expires_at")
  isDeleted       Boolean  @default(false) @map("is_deleted")

  // 통계
  viewCount       Int      @default(0) @map("view_count")

  createdAt       DateTime @default(now()) @map("created_at")
  updatedAt       DateTime @updatedAt @map("updated_at")

  board           Board       @relation(fields: [boardId], references: [id])
  author          User        @relation("PostAuthor", fields: [authorId], references: [id])
  reads           PostRead[]
  comments        Comment[]
  attachments     Attachment[]

  @@index([boardId, isDeleted, isPinned, publishedAt(sort: Desc)])  // 목록 조회 최적화
  @@index([authorId])
  @@index([publishingDepartmentId])                                 // 발행부서 필터
  @@map("board_posts")
}

model PostRead {
  postId   String   @map("post_id")
  userId   String   @map("user_id")
  readAt   DateTime @default(now()) @map("read_at")

  post     Post @relation(fields: [postId], references: [id], onDelete: Cascade)

  @@id([postId, userId])
  @@index([userId])
  @@map("board_post_reads")
}

model Comment {
  id        String   @id @default(cuid())
  postId    String   @map("post_id")
  authorId  String   @map("author_id")
  parentId  String?  @map("parent_id")                     // 1단계 대댓글
  content   String   @db.Text
  isDeleted Boolean  @default(false) @map("is_deleted")
  createdAt DateTime @default(now()) @map("created_at")
  updatedAt DateTime @updatedAt @map("updated_at")

  post      Post     @relation(fields: [postId], references: [id], onDelete: Cascade)
  parent    Comment? @relation("CommentTree", fields: [parentId], references: [id])
  children  Comment[] @relation("CommentTree")

  @@index([postId, createdAt(sort: Asc)])
  @@map("board_comments")
}

model Attachment {
  id          String   @id @default(cuid())
  postId      String?  @map("post_id")                     // 임시 업로드는 null, 글 발행 시 연결
  uploadedBy  String   @map("uploaded_by")
  fileName    String   @map("file_name")
  fileSize    Int      @map("file_size")                   // bytes
  mimeType    String   @map("mime_type")
  storageKey  String   @unique @map("storage_key")          // 로컬: 상대경로, S3: object key
  isInline    Boolean  @default(false) @map("is_inline")    // markdown 본문 내 이미지
  uploadedAt  DateTime @default(now()) @map("uploaded_at")

  post        Post? @relation(fields: [postId], references: [id], onDelete: SetNull)

  @@index([postId])
  @@map("board_attachments")
}

enum BoardAudience {
  ALL
  DEPARTMENT
  ROLE
}
```

**Request:** `{ "isPinned": true }`

### 4.3 권한 헬퍼 의사코드

```ts
// services/auth/src/application/board-permissions.ts

export function canRead(board: Board, post: Post, user: AuthUser, userDeptId?: string): boolean {
  if (post.isDeleted) return user.role === "ADMIN";
  switch (board.readAudience) {
    case "ALL":        return true;
    case "DEPARTMENT": return user.role === "ADMIN" || userDeptId === board.audienceTargetId;
    case "ROLE":       return user.role === "ADMIN" || user.role === board.audienceTargetId;
  }
}

export function canWrite(board: Board, user: AuthUser): boolean {
  return board.writeRoles.includes(user.role);
}

export function canEdit(post: Post, user: AuthUser): boolean {
  return user.role === "ADMIN" || post.authorId === user.id;
}

export function canDelete(post: Post, user: AuthUser): boolean {
  return canEdit(post, user);
}

export function canPin(post: Post, user: AuthUser): boolean {
  return user.role === "ADMIN" || post.authorId === user.id;
}

export function canComment(board: Board, user: AuthUser): boolean {
  return board.allowComments && user.role !== "VIEWER";
}
```

각 라우트 핸들러에서 명시적으로 호출 (미들웨어 wrapping은 인증/Role check까지만, 도메인 권한은 함수 호출).

## 6. 알림 발송 시퀀스 (Notice → Notification)

### 6.1 시퀀스 다이어그램

```
사용자          auth-service                attendance-service
  │                  │                              │
  │ POST /posts      │                              │
  ├─────────────────▶│                              │
  │                  │ TX: Post.create              │
  │                  │ TX: Attachment.update        │
  │                  │ TX commit                    │
  │                  │                              │
  │ 201 Created      │                              │
  │◀─────────────────┤                              │
  │                  │ async (fire-and-forget)      │
  │                  │ if (board.category=='notice')│
  │                  │   audience 사용자 ID 수집     │
  │                  │   POST /internal/notif/bulk  │
  │                  ├─────────────────────────────▶│
  │                  │                              │ Notification.createMany
  │                  │ 200 OK (or fail → log)       │ (transaction)
  │                  │◀─────────────────────────────┤
```

### 6.2 audience 사용자 수집 로직

```ts
async function collectAudienceUserIds(post: Post, board: Board): Promise<string[]> {
  switch (board.readAudience) {
    case "ALL":
      return prisma.user.findMany({ where: { isActive: true }, select: { id: true } });
    case "DEPARTMENT":
      return prisma.userProfile.findMany({
        where: { departmentId: post.publishingDepartmentId },
        select: { userId: true }
      });
    case "ROLE":
      return prisma.user.findMany({
        where: { role: board.audienceTargetId as UserRole, isActive: true },
        select: { id: true }
      });
  }
}
```

### 6.3 Internal API (attendance-service)

```
POST /internal/notifications/bulk
Headers: X-Internal-Token: $INTERNAL_API_TOKEN
Body: {
  "userIds": ["u1","u2",...],
  "type": "notice.published",
  "source": "board",
  "title": "[전사 공지] 4월 안전점검 일정",
  "body": "본문 발췌 100자",
  "priority": 2,
  "linkUrl": "/board/notice/notice-company/{postId}",
```

기존 `Notification` 모델 (`services/attendance/prisma/schema.prisma:296`) 활용. 새 endpoint만 추가.

### 6.4 실패 처리

- 알림 hook 실패 → `pino.error()` 기록만, 사용자 응답에 영향 없음
- 폭증 우려 (audience=ALL × 100명) → 1차는 동기 처리 (PostgreSQL `createMany` 100건 ~50ms)
- 2차에 RabbitMQ로 비동기 전환

## 8. 에러 처리 (Error Handling)

### 8.1 에러 코드 정의

| HTTP | Code | 메시지 | 원인 | 처리 |
|---|---|---|---|---|
| 400 | INVALID_INPUT | 입력값 오류 | Zod 위반 | 입력 재요청 |
| 401 | UNAUTHENTICATED | 인증 필요 | 토큰 없음/만료 | /login 리다이렉트 |
| 403 | FORBIDDEN_READ | 읽기 권한 없음 | audience 불일치 | "권한 없음" 표시 |
| 403 | FORBIDDEN_WRITE | 작성 권한 없음 | writeRoles 불일치 | "권한 없음" 표시 |
| 403 | FORBIDDEN_EDIT | 수정 권한 없음 | 본인 아님 | "권한 없음" 표시 |
| 403 | FORBIDDEN_PIN | 핀 권한 없음 | ADMIN/작성자 아님 | "권한 없음" 표시 |
| 404 | BOARD_NOT_FOUND | 보드 없음 | code 오류 | 404 페이지 |
| 404 | POST_NOT_FOUND | 글 없음 | id 오류·삭제됨 | 404 페이지 |
| 404 | COMMENT_NOT_FOUND | 댓글 없음 | id 오류 | 토스트 후 새로고침 |
| 413 | FILE_TOO_LARGE | 파일 크기 초과 | > 50MB | 메시지 표시 |
| 415 | UNSUPPORTED_TYPE | 지원하지 않는 파일 | 화이트리스트 외 | 메시지 표시 |
| 500 | INTERNAL_ERROR | 서버 오류 | 예외 | 로그 + 사용자 토스트 |

## 9. 보안 (Security Considerations)

- [x] **XSS 방지**: react-markdown + remark-gfm + rehype-sanitize (`<script>`, `<iframe>`, `on*` 차단)
- [x] **인증·인가**: 모든 엔드포인트 `authenticate` 미들웨어 + 권한 헬퍼
- [x] **첨부 확장자 화이트리스트**: 확장자 + MIME 두 단계 검증
- [x] **첨부 크기 제한**: 50MB
- [x] **SQL Injection**: Prisma ORM (Parameterized)
- [x] **CSRF**: SameSite cookie + Origin 검증 (인증 시스템 기존 정책)
- [x] **Rate Limiting**: 1차 글 작성 분당 10건/사용자 (helmet/@fastify/rate-limit)
- [x] **첨부 다운로드 권한**: audience 검증 후 스트림 응답
- [x] **소프트 삭제**: 데이터 복구 가능, ADMIN 감사 추적
- [x] **HTTPS 강제**: 회사 서버 이관 시 Traefik + Let's Encrypt

추가 검토:
- [ ] 첨부 파일명 sanitize (`path traversal` 방지) — `path.basename()` + 화이트리스트 정규식
- [ ] 글 본문 길이 상한 (예: 100KB) — 초과 시 400

**Permission Matrix (19건 — FR ID 단위):**
- [ ] FR-03: OPERATOR가 전사 공지 작성 → 200
- [ ] FR-03: VIEWER가 작성 시도 → 403 FORBIDDEN_WRITE
- [ ] FR-04: 본인 글 수정 → 200
- [ ] FR-04: 타인 글 수정 (non-ADMIN) → 403 FORBIDDEN_EDIT

- [ ] FR-05: 작성자 본인 핀 → 200
- [ ] FR-05: ADMIN이 타인 글 핀 → 200
- [ ] FR-05: MANAGER가 타인 글 핀 → 403 FORBIDDEN_PIN
- [ ] FR-06: 페이지네이션 핀 우선 정렬 검증
- [ ] FR-07: 글 상세 조회 → viewCount++, PostRead upsert 1건
- [ ] FR-08: 댓글·대댓글 (parentId 1단계) 작성·삭제
- [ ] FR-09: 50MB 초과 파일 → 413
- [ ] FR-09: .exe 업로드 → 415
- [ ] FR-10: markdown XSS payload (`<script>alert(1)</script>`) → sanitize 됨
- [ ] FR-11: 검색 LIKE 매칭
- [ ] FR-12: publishingDeptId 필터
- [ ] FR-13: 미읽음 카운트 카테고리별
- [ ] FR-14: 공지 작성 → audience 사용자 N명에게 Notification N건 생성
- [ ] FR-19: feed 핀 우선 5건

### 10.3 Happy Path / Edge / Error

- **Happy**: OPERATOR 가입 → 글 작성 → 첨부 → 댓글 → 핀 → 읽음 → 알림 도달
- **Edge**: 작성자가 본인 글을 작성하면서 ADMIN으로 승급 → 핀 권한 그대로
- **Edge**: audience=DEPARTMENT인 글에 audience 외 사용자가 직접 URL 접근 → 403
- **Error**: 첨부 업로드 후 글 작성 실패 → 첨부 orphan (24h 후 GC, 1차는 수동)
- **Error**: 알림 발송 실패 → 글 작성 성공 (fire-and-forget)

### 16.3 Post 모델 확장 (schema.prisma)

```prisma
enum FeatureRequestStatus {
  SUBMITTED       // 접수
  UNDER_REVIEW    // 검토 중
  APPROVED        // 승인 (개발 대기)
  IN_PROGRESS     // 진행 중
  COMPLETED       // 완료
  REJECTED        // 거부
  ON_HOLD         // 보류

  @@schema("public")
}

enum FeatureRequestType {
  BUG             // 버그
  NEW_FEATURE     // 신규 기능
  IMPROVEMENT     // 개선
  UI_UX           // UI/UX
  DOCS            // 문서·매뉴얼
  OTHER

  @@schema("public")
}

model Post {
  // ... 기존 필드 ...

  // 기능 요구 카테고리 전용 (board의 category가 'feature-request'일 때만 사용)
  requestStatus    FeatureRequestStatus?  @map("request_status")
  requestType      FeatureRequestType?    @map("request_type")
  assigneeId       String?                @map("assignee_id")
  moduleArea       String?                @map("module_area")     // 'procurement', 'expense', 'project', ...
  releaseVersion   String?                @map("release_version") // 적용된 릴리즈 (예: "v1.7.0")
  resolvedAt       DateTime?              @map("resolved_at")

  assignee         User?                  @relation("PostAssignee", fields: [assigneeId], references: [id])

  @@index([requestStatus])
  @@index([requestType])
  @@index([assigneeId])
}

model User {
  // 기존
  authoredPosts    Post[] @relation("PostAuthor")
  assignedPosts    Post[] @relation("PostAssignee")  // 신규
}
```

### 16.4 워크플로우

```
[직원] 요구 등록 (Post.create)
   ↓ requestStatus = SUBMITTED, 알림 → 관리자
[관리자] 검토
   ↓ requestStatus = UNDER_REVIEW
[관리자] 결정
   ├─ APPROVED → assigneeId 지정 → IN_PROGRESS
   ├─ REJECTED → 거부 사유 (댓글), 알림 → 요청자
   └─ ON_HOLD → 보류 사유 (댓글)
   ↓
[담당자] 개발 진행
   ↓ requestStatus = IN_PROGRESS, 진행 상황을 댓글로 공유
[담당자] 완료
   ↓ requestStatus = COMPLETED, releaseVersion 기록, resolvedAt 설정, 알림 → 요청자
```
