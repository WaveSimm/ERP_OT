# 자원 기여도 기반 진척률 설계

> 작성일: 2026-06-23
> 대상 서비스: `services/project` (포트 3003) + `apps/web`
> 관련 기능: 자원-모델-분리(Phase 5 완료), 프로젝트-진도율-캐시

## 1. 배경 / 문제

현재 자원 배정 모델(`SegmentAssignment`)은 **개인 점유율(`allocationPercent`)** 만 가진다.
이는 "이 사람이 자기 가용용량 중 몇 %를 이 구간에 쓰는가"이며, **"이 태스크(세그먼트) 안에서 이 자원의 몫이 얼마인가"(분담율)** 는 표현하지 못한다.

또한 진척률은 `TaskSegment.progressPercent` 하나뿐이라 **누가 얼마나 했는지 자원별로 분해되지 않는다**.
태스크 진척률은 세그먼트 진척률의 기간(일수) 가중 평균(`TaskEntity.calculateAutoProgress`)으로만 산출된다.

## 2. 목표

1. 배정(자원)마다 **분담율(contributionWeight)** 추가 — 세그먼트 단위로 관리
2. 배정(자원)마다 **본인 진척률(progressPercent)** 추가 — 각 자원이 자신의 진도율 바를 직접 관리
3. **세그먼트 진척률 = Σ(분담율 × 자원 진척률) / Σ(분담율)** (분담율 정규화)
4. 태스크 진척률 = 기존 로직(세그먼트 일수 가중 평균) 유지 — 세그먼트 진척률만 derived 로 전환

## 3. 확정된 결정 (사용자 합의)

| # | 결정 | 내용 |
|---|------|------|
| D1 | 분담율 합 | **소프트**. 합이 100이 아니어도 차단하지 않고 계산 시 실제 합으로 정규화. UI는 합≠100 경고만 표시 |
| D2 | 수동 진척 모드 | **폐기**. `isManualProgress`/수동 입력 경로 제거. 진척률은 전부 자동 가중 계산 |
| D3 | 작업일지 필수 (개정) | **본인 진척률을 100%로 완료할 때 본인(`authorId==요청자`) 작업일지 1건 이상 필수**. 각자 자기 작업 내역을 기록하게 강제(이 기능의 목적). 태스크/세그먼트 진척률은 derived라 100% 직접입력 불가 → 거기엔 검증 없음 (`WORK_LOG_REQUIRED_FOR_COMPLETION`) |
| D4 | 관리 단위 | **세그먼트마다** 분담율·진척률 따로 관리 |
| D5 | 본인 관리 권한 | 자원 본인(`personUserId == 요청자`) 또는 ADMIN/MANAGER가 진척률 수정. 외부인력/장비는 로그인이 없으므로 태스크 담당자/관리자가 대신 입력 |
| D7 | 투입률 UI | `allocationMode` HOURS(h/일) 토글 제거 — **"내 투입률"은 % 단일 표시/입력**(라벨: 점유율→내 투입률). 백엔드 데이터모델(AllocationMode/allocationHoursPerDay)은 유지, UI만 PERCENT 고정 |
| D8 | 내 대시보드 작업목록 | 컬럼 재구성: 배당율 → **내 투입 · 분담 · 내 진척률** 순. 진행률 편집은 세그먼트 직접→**내 배정 진척**(`updateAssignmentProgress`)으로 위임. 백엔드 `my-tasks.routes` 에 `myProgressPercent`·`contributionWeight` 추가. `/me/segments/:id/progress`(칸반)도 `taskService.updateAssignmentProgress`로 위임(직접 세그먼트 쓰기는 derived 캐시라 덮어써짐) |

## 4. 데이터 모델 변경

### 4.1 `SegmentAssignment` (필드 2개 추가)

```prisma
model SegmentAssignment {
  // ... 기존 필드 ...
  contributionWeight Float @default(0)   // 분담율 0~100 (세그먼트 내 참여자 합 권장=100)
  progressPercent    Float @default(0)   // 이 자원의 본인 진척률 0~100
  // ...
}
```

### 4.2 `TaskSegment.progressPercent` — derived(캐시) 로 의미 변경

- 더 이상 직접 입력값이 아니라 **배정들로부터 재계산되는 캐시**.
- 배정이 1개 이상이면: `Σ(weight × progress) / Σ(weight)` (Σweight=0 이면 0).
- **배정이 0개인 세그먼트(자원 미배정)**: 진척률을 기록할 주체가 없음 → **fallback 으로 세그먼트 진척률 직접 입력 허용**(기존 동작 유지). 자원이 1명 이상 붙는 순간 derived 로 전환. (D6 — 본 설계의 보강 결정)

### 4.3 `Task.isManualProgress` — 폐기 (D2)

- 컬럼은 마이그레이션에서 즉시 drop 하지 않고 **항상 false 로 강제 + 코드 경로 제거** 후, 후속 정리 마이그레이션에서 drop (자원-모델-분리 패턴과 동일하게 점진적).

## 5. 데이터 마이그레이션 (백필)

기존 데이터 정합성 유지:

1. 세그먼트별로 기존 배정들에 분담율 균등 분배: `contributionWeight = 100 / (해당 세그먼트 배정 수)`.
2. 각 배정의 `progressPercent = 소속 세그먼트의 현재 progressPercent`.
   → 마이그레이션 직후 derived 세그먼트 진척률 = 기존 값과 동일(연속성 보장).
3. `isManualProgress=true` 였던 태스크: 세그먼트 진척률을 그대로 백필하므로 표시값 변화 없음.

## 6. 백엔드 변경

### 6.1 도메인 — `task.entity.ts`
- `TaskSegmentData`/`calculateAutoProgress` 는 그대로(세그먼트 진척률 입력만 derived).
- 신규 순수함수: `calculateSegmentProgress(assignments): number` = Σ(w×p)/Σw.

### 6.2 application — `task.service.ts`
- **신규** `recalculateSegmentProgress(segmentId)`: 배정 집계 → `taskSegment.progressPercent` 갱신 → `recalculateTaskProgress(taskId)` 연쇄.
- `upsertAssignment`: `contributionWeight` 수용. 저장 후 `recalculateSegmentProgress`.
- `removeAssignment`: 삭제 후 `recalculateSegmentProgress`.
- **신규** `updateAssignmentProgress(segmentId, resourceId, progressPercent, requesterId)`:
  - 권한: `personUserId === requesterId` 또는 ADMIN/MANAGER (라우트에서 검증).
  - 100% 완료-작업일지 검증(D3)을 **세그먼트 derived 진척률이 100 도달 시** 로 이전.
  - `TaskScheduleHistory`(PROGRESS_UPDATED) + ActivityLog 기록.
- `updateSegment`: `progressPercent` 직접 입력은 **배정 0개일 때만** 허용(D6), 그 외 무시/400.

### 6.3 API — `task.routes.ts`
- `upsertAssignmentSchema` 에 `contributionWeight: z.number().min(0).max(100).optional()` 추가.
- **신규** `PATCH /:taskId/segments/:segmentId/assignments/:resourceId/progress`
  - body: `{ progressPercent: number(0..100), changeReason?: string }`
  - preHandler 없음(라우트 본문에서 본인/관리자 판정). 본인 판정: 해당 배정의 `personUserId === req.userId`.

## 7. 프론트엔드 변경 (`apps/web`)

### 7.1 API 클라이언트 — `lib/api/core-domains.ts`
- `upsertAssignment` 데이터에 `contributionWeight?: number` 추가.
- **신규** `updateAssignmentProgress(projectId, taskId, segmentId, resourceId, { progressPercent })`.
- `types.ts` `SegmentAssignment` 에 `contributionWeight`, `progressPercent` 추가.

### 7.2 TaskDrawer.tsx
- 세그먼트 단일 진척 슬라이더(현 276–290행)를 **배정별 진척률 바 리스트**로 교체:
  - 각 배정 행에: 자원명 · 분담율 입력 · **본인 진척률 바/입력** · 점유율(allocationPercent).
  - 세그먼트 헤더에 **derived 진척률**(읽기전용) + "분담율 합 NN%" 경고(≠100).
  - 배정 0개 세그먼트: 기존 단일 진척 슬라이더 fallback 유지.
- 본인 진척률 바는 `personUserId === 현재 사용자` 또는 관리자만 편집 가능(그 외 읽기전용).

### 7.3 ResourcePickerPopover.tsx
- 신규 배정 시 기본 `contributionWeight` = `100 / (배정 후 인원수)` 자동 제안(균등). 아바타 tooltip 에 분담율 병기.

## 8. 검증 / 엣지케이스

- Σweight=0 (전원 0 분담): 세그먼트 진척률 0 처리(0 나눗셈 방지).
- 자원 1명: 분담율 무관하게 그 자원 진척률 = 세그먼트 진척률.
- 배정 추가/삭제 시 분담율 재정규화는 자동 계산이 흡수(저장값은 보존, 표시·계산만 정규화).
- 외부인력/장비 진척률: 관리자/담당자 입력 경로로만 변경.

## 9. 영향 범위 / 비고

- `recalculateTaskProgress` 호출 지점(생성·세그먼트 변경)에 세그먼트 재계산이 선행되어야 함.
- Gantt/대시보드/내태스크 등 `progressPercent` 표시처는 derived 값을 그대로 읽으므로 표시 로직 변경 불필요.
- WebSocket `segment:updated` 이벤트는 진척률 변경 시에도 emit.
