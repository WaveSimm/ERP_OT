## 1. 개요

### 1.1 설계 목표

1. **격리된 부하 시뮬레이션**: 부하 사용자(`@erp-ot.load`)가 일반 화면에 노출되지 않으며 테스트 후 즉시 일괄 정리 가능
2. **실 ERP 사용 패턴 반영**: 출근 burst → 평상 mix → 검색 burst 3 단계 시나리오
3. **dev 모드 60 vs 90명 비교**: dev 모드 6 round로 부하 임계점 식별 (v1.1: prod 비교는 별도 PDCA)
4. **재현 가능한 매트릭스**: 회사 서버 이관 후 동일 절차로 환경 차이 비교
5. **임계점 식별**: 90명 스트레스에서 어떤 컴포넌트(Ollama·DB·web·JWT)가 먼저 한계 도달

### 1.2 설계 원칙

- **Repository-level 필터**: 사용자 조회 필터를 repository 메서드에 적용 → 모든 consumer 자동 격리
- **Internal API 격리**: 다른 서비스가 호출하는 `/internal/users/*`도 필터 적용 → 결재/멤버 select에서 자동 제외
- **검색 결과 격리**: 자연어검색·검색개선의 SQL `WHERE`에 작성자 필터 추가
- **CASCADE FK 활용**: 사용자 삭제 시 종속 데이터 자동 cleanup
- **시나리오 idempotent**: 시드/cleanup 모두 반복 실행 가능

## 2. 격리 정책 상세

### 2.1 도메인 + 환경변수

```
사용자 식별:
  email LIKE '%@erp-ot.load'
  id LIKE 'loadtest-%'

환경변수:
  HIDE_LOAD_TEST=true  (default, 일반 조회 시 자동 제외)
  HIDE_LOAD_TEST=false (디버그 시 명시적 노출)
```

### 2.2 필터 적용 위치 매트릭스 (사전 grep 결과)

| 위치 | 파일 | 함수/라우트 | 필터 적용 방식 |
|------|------|------------|----------|
| **L1. UserRepository (auth)** | `services/auth/src/infrastructure/repositories/user.prisma.repository.ts` | `findAll()`, `findMany(...)` | `where`에 `email NOT LIKE '%@erp-ot.load'` 자동 추가 (HIDE_LOAD_TEST=true 시) |
| **L2. /api/v1/users** | `services/auth/src/api/routes/user.routes.ts:18` | `GET /` | L1로 자동 적용 |
| **L3. /api/v1/users/members** | `user.routes.ts:24` | `GET /members` (결재자 select) | L1로 자동 적용 |
| **L4. /internal/users/bulk** | `internal.routes.ts:27` | `GET /users/bulk?ids=...` | 명시적 ID 요청 — **필터 적용 안 함** (load test 자체에서는 정상 fetch 필요) |
| **L5. /internal/users/all** | `internal.routes.ts:79` | `GET /users/all` (project-service가 호출) | L1로 자동 적용 |
| **L6. /internal/users/all-with-departments** | `internal.routes.ts:135` | `GET /...` (resource list) | L1로 자동 적용 |
| **L7. /internal/users/under/:id** | `internal.routes.ts:87` | 팀장/총괄/대표 하위 멤버 | 부서 멤버 select 시 사용 — L1로 자동 적용 |
| **L8. 자연어검색 결과** | `search.service.ts` searchPosts SQL | scored CTE 내부 | `WHERE u.email NOT LIKE '%@erp-ot.load'` 추가 |
| **L9. 검색개선 worklog 결과** | `work-log.service.ts` semanticSearch SQL | scored CTE 내부 | `WHERE w.author_id NOT LIKE 'loadtest-%'` 추가 |
| **L10. 게시판 작성자 노출** | `post.service.ts` listPosts | 게시글 목록 — 일반 사용자가 부하 글을 보지 않도록 | post author 기준 필터 (옵션, 1차는 L1 의존) |
| **L11. 댓글 작성자** | `comment.service.ts` listComments | 부하 댓글 노출 방지 | 1차 미적용 (작성자 join 필터 복잡) — Out of Scope |

**핵심 결정**:
- L4 (`/internal/users/bulk`)만 필터 미적용 (load test가 본인 정보 조회 시 필요)
- L11 댓글은 1차 미적용 — 작성자 join 비용 vs 노출 위험 trade-off

### 2.3 환경변수 처리 규칙

```typescript
// services/auth/src/infrastructure/repositories/user.prisma.repository.ts (예시)
const HIDE_LOAD_TEST = process.env.HIDE_LOAD_TEST !== "false";
const LOAD_TEST_DOMAIN = process.env.LOAD_TEST_DOMAIN ?? "@erp-ot.load";

private buildWhere(extra: Prisma.AuthUserWhereInput = {}): Prisma.AuthUserWhereInput {
  return HIDE_LOAD_TEST
    ? { ...extra, NOT: { email: { contains: LOAD_TEST_DOMAIN } } }
    : extra;
}
```

검색 SQL 측 (Prisma `$queryRaw` 인터폴레이션):
```typescript
const hideLoadTest = process.env.HIDE_LOAD_TEST !== "false";
const loadTestDomain = process.env.LOAD_TEST_DOMAIN ?? "@erp-ot.load";

// 검색 SQL의 author JOIN 절에 조건부 추가
${hideLoadTest ? Prisma.sql`AND u.email NOT LIKE '%' || ${loadTestDomain}` : Prisma.empty}
```

## 3. 시드 데이터 설계

### 3.1 시드 스크립트 구조

```typescript
// services/auth/prisma/seed-load-test.ts
const LOAD_TEST_USER_COUNT = parseInt(process.env.LOAD_TEST_USER_COUNT ?? "90", 10);
const PASSWORD = process.env.LOAD_TEST_PASSWORD ?? "loadtest123!";
const HASH = bcrypt.hashSync(PASSWORD, 10);

const departmentIds = await getDeptIds(); // 기존 부서 N개 round-robin 배정

for (let i = 1; i <= LOAD_TEST_USER_COUNT; i++) {
  const idx = String(i).padStart(3, "0");
  const id = `loadtest-${idx}`;
  await prisma.authUser.upsert({
    where: { id },
    create: {
      id,
      email: `loadtest-${idx}@erp-ot.load`,
      name: `[LOAD] 테스트${i}`,
      passwordHash: HASH,
      role: "OPERATOR",
      isActive: true,
    },
    update: {},
  });

  // 프로필 + 부서 배정
  await prisma.authUserProfile.upsert({
    where: { userId: id },
    create: {
      userId: id,
      departmentId: departmentIds[i % departmentIds.length],
      position: "사원",
    },
    update: {},
  });
}
```

### 3.3 테스트 콘텐츠 시드 (선택, 시나리오 B용)

부하 사용자가 작성할 게시판 글/작업비고 templates:
- 시나리오 B mix 행동에서 **fresh write**가 발생하면 동시 임베딩 부하도 측정 가능
- 1차는 read-heavy 위주, write는 5% 이하

## 4. k6 시나리오 설계

### 4.1 공통 인증 캐싱

```javascript
// scripts/load-test/lib/auth-cache.js
const tokenCache = new Map();

export function loginCached(user) {
  if (tokenCache.has(user.id)) return tokenCache.get(user.id);
  const r = http.post(`${BASE}/api/v1/auth/login`, JSON.stringify({
    email: user.email, password: user.password,
  }), { headers: { 'Content-Type': 'application/json' } });
  if (r.status !== 200) throw new Error(`login failed: ${user.id}`);
  const token = r.json('accessToken');
  tokenCache.set(user.id, token);
  return token;
}
```

### 4.3 시나리오 B — 평상 mix (10분 soak)

```javascript
// scripts/load-test/scenario-b-mixed.js
const KEYWORDS = ["디버깅", "메인보드", "회의", "분석", "테스트", "프로젝트", "장비"];

export const options = {
  scenarios: {
    mixed: {
      executor: 'constant-vus',
      vus: 60,
      duration: '10m',
    },
  },
  thresholds: {
    'http_req_duration': ['p(95)<500', 'p(99)<2000'],
    'http_req_failed':   ['rate<0.005'],
  },
};

export default function () {
  const u = users[__VU % users.length];
  const token = loginCached(u);
  const auth = { headers: { Authorization: `Bearer ${token}` } };

  const action = Math.random();
  if (action < 0.30) {
    // 30% 게시판 + 검색
    http.get(`${BASE}/api/v1/boards`, auth);
    http.get(`${BASE}/api/v1/posts?categoryCode=notice&limit=20`, auth);
    if (Math.random() < 0.5) {
      const kw = KEYWORDS[Math.floor(Math.random() * KEYWORDS.length)];
      http.get(`${BASE}/api/v1/search?q=${encodeURIComponent(kw)}&limit=20`, auth);
    }
  } else if (action < 0.55) {
    // 25% 작업비고
    http.get(`${BASE}/api/v1/me/work-log-feed?limit=20`, auth);
    if (Math.random() < 0.2) {
      // 5% 신규 작성 (임베딩 부하)
      const taskId = pickRandomTaskId(u);
      if (taskId) {
        http.post(`${BASE}/api/v1/tasks/${taskId}/work-logs`, JSON.stringify({
          content: `부하 테스트 비고 ${Date.now()}`,
          workedAt: new Date().toISOString().slice(0, 10),
        }), { ...auth, tags: { name: 'worklog_write' } });
      }
    }
  } else if (action < 0.75) {
    // 20% 결재
    http.get(`${BASE}/api/v1/approval/inbox?limit=20`, auth);
    http.get(`${BASE}/api/v1/approval/sent?limit=20`, auth);
  } else if (action < 0.90) {
    // 15% 프로젝트
    http.get(`${BASE}/api/v1/projects?limit=10`, auth);
    http.get(`${BASE}/api/v1/me/tasks?limit=20`, auth);
  } else {
    // 10% 대시보드
    http.get(`${BASE}/api/v1/dashboard/personal`, auth);
    http.get(`${BASE}/api/v1/notifications?limit=10`, auth);
  }

  sleep(Math.random() * 5 + 2);  // 2~7초 사용자 think time
}
```

## 6. Cleanup 설계

### 6.1 CASCADE FK 검증

```sql
-- Cleanup 전 어떤 테이블이 영향받는지 확인
SELECT
  tc.constraint_name, tc.table_schema, tc.table_name, kcu.column_name,
  ccu.table_schema AS foreign_table_schema, ccu.table_name AS foreign_table_name,
  rc.delete_rule
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu USING (constraint_name)
JOIN information_schema.constraint_column_usage ccu USING (constraint_name)
JOIN information_schema.referential_constraints rc USING (constraint_name)
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND ccu.table_name = 'auth_users';
```

`auth_users` 참조 모두 `ON DELETE CASCADE` 또는 `SET NULL` 필요. RESTRICT면 cleanup 시 FK 에러 발생 → 사전 확인.

### 6.2 Cleanup 스크립트

```bash
#!/usr/bin/env bash
# scripts/cleanup-load-test.sh
set -euo pipefail
LOAD_TEST_DOMAIN="${LOAD_TEST_DOMAIN:-@erp-ot.load}"
DRY_RUN="${1:-}"

if [ "$DRY_RUN" = "--dry-run" ]; then
  echo "=== DRY RUN — 삭제 대상 ==="
  docker exec erp-ot-postgres psql -U erp_user -d erp_ot -c "
    SELECT '게시글' AS kind, COUNT(*) FROM public.board_posts WHERE author_id LIKE 'loadtest-%'
    UNION ALL SELECT '댓글', COUNT(*) FROM public.board_comments WHERE author_id LIKE 'loadtest-%'
    UNION ALL SELECT '작업비고', COUNT(*) FROM project.work_logs WHERE author_id LIKE 'loadtest-%'
    UNION ALL SELECT '결재문서', COUNT(*) FROM approval.documents WHERE submitter_id LIKE 'loadtest-%'
    UNION ALL SELECT '근태', COUNT(*) FROM attendance.attendance_records WHERE user_id LIKE 'loadtest-%'
    UNION ALL SELECT '리프레시토큰', COUNT(*) FROM public.auth_refresh_tokens WHERE user_id LIKE 'loadtest-%'
    UNION ALL SELECT '사용자', COUNT(*) FROM public.auth_users WHERE id LIKE 'loadtest-%';
  "
  exit 0
fi

read -p "정말 부하 테스트 데이터를 모두 삭제합니까? (yes/no): " CONFIRM
[ "$CONFIRM" = "yes" ] || { echo "취소됨"; exit 1; }

# 사용자 삭제 → CASCADE FK로 종속 데이터 자동 삭제
docker exec erp-ot-postgres psql -U erp_user -d erp_ot -v ON_ERROR_STOP=1 -c "
  DELETE FROM public.auth_users WHERE id LIKE 'loadtest-%';
"
# Resource는 다른 schema이므로 별도
docker exec erp-ot-postgres psql -U erp_user -d erp_ot -v ON_ERROR_STOP=1 -c "
  DELETE FROM project.resources WHERE \"userId\" LIKE 'loadtest-%@erp-ot.load';
"

# 검증
docker exec erp-ot-postgres psql -U erp_user -d erp_ot -c "
  SELECT 'remaining_users' AS k, COUNT(*) FROM public.auth_users WHERE id LIKE 'loadtest-%';
"
```

### 6.3 idempotent 보장

- 시드: `upsert` 사용 → 반복 실행 가능
- Cleanup: `DELETE WHERE LIKE 'loadtest-%'` → 없으면 0 row 영향
- 검증: cleanup 후 SELECT COUNT가 모두 0이어야 함

### 7.3 `run-round.sh` 통합 스크립트

```bash
#!/usr/bin/env bash
# scripts/load-test/run-round.sh
SCENARIO=$1
LABEL=${LABEL:-default}
STRESS=${STRESS:-false}
DURATION=${DURATION:-720}  # 12분 default (10분 soak + 여유)
OUT="docs/04-operation/load-test-results/${LABEL}"

mkdir -p "$OUT"

# 메트릭 캡처 시작
bash scripts/load-test/capture-metrics.sh "$DURATION" "$LABEL" &
METRIC_PID=$!

# k6 실행
docker run --rm --network host \
  -v "$(pwd)/scripts/load-test:/scripts" \
  -e STRESS="$STRESS" \
  grafana/k6 run \
    --out json="/scripts/results.json" \
    --summary-export="/scripts/summary.json" \
    "/scripts/$SCENARIO"

# 결과 이동
mv scripts/load-test/results.json "$OUT/k6-summary.json"
mv scripts/load-test/summary.json "$OUT/k6-summary-export.json"

# 메트릭 캡처 종료
kill $METRIC_PID 2>/dev/null || true

# HTML 리포트
docker run --rm -v "$(pwd):/src" node:20 \
  npx -y k6-html-reporter --json "/src/$OUT/k6-summary.json" \
  --output "/src/$OUT/k6-report.html"

echo "✅ $LABEL 완료 → $OUT/"
```

## 8. Production 빌드 전환 — **별도 PDCA로 분리 (v1.1)**

> 원래 본 PDCA 범위였으나, Next.js prod 빌드가 다음 에러로 실패:
> 1. type error (`apps/web/src/app/equipment/sensors/[id]/page.tsx:79` startDate/endDate 누락 등)
> 2. useSearchParams 페이지 prerender 에러 (`/board/search`, `/approval/new` 등 9개 페이지)
>
> 이는 부하테스트와 독립된 빌드 정리 작업이라 **별도 PDCA `prod-빌드-정리`로 분리**. 본 PDCA는 dev 모드 6 round로 scope 재정의.
>
> 아래 §8.1~8.3 내용은 별도 PDCA 진행 시 참고용으로 보존:


### 8.1 옵션 A — docker-compose override (권장)

```yaml
# docker-compose.prod-test.yml (신규)
services:
  web:
    build:
      target: runner    # 기존 dev stage 대신 production runner
    command: ["node", "/app/server.js"]
    environment:
      NODE_ENV: production
```

기존 web Dockerfile에 `output: standalone` 이미 설정되어 있으므로 활용 가능. 별도 build target 추가:

```dockerfile
# apps/web/Dockerfile (수정)
FROM base AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/apps/web/.next/standalone ./
COPY --from=builder /app/apps/web/.next/static ./apps/web/.next/static
COPY --from=builder /app/apps/web/public ./apps/web/public
EXPOSE 3000
CMD ["node", "apps/web/server.js"]
```

### 8.2 옵션 B — 수동 production 실행

```bash
# 호스트에서
cd apps/web
pnpm build
NODE_ENV=production pnpm start &
# 기존 docker web 컨테이너 stop
docker stop erp-ot-web
```

### 8.3 검증

`x-powered-by` 또는 응답 시간 패턴으로 dev/prod 구분 가능. 첫 요청 응답 시간 비교 (dev: 1~3초 컴파일, prod: <100ms).
