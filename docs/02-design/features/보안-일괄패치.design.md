## 1. Overview

### 1.1 Design Goals

1. **OWASP Top 10 (2021) Critical 0건** 도달 (A01·A02·A07)
2. **사용자 1회 강제 재로그인 외 추가 다운타임 0**
3. **각 Layer 독립 롤백 가능** (단일 Layer 실패 시 이전 상태로 즉시 복구)
4. **회귀 측정**: BASELINE-2026-05-02 대비 ±20% 이내
5. **재발 방지**: shared 모듈로 drift 차단

### 1.2 Design Principles

- **Defense-in-depth**: Zod env 검증 + 코드 fallback 제거 (한 곳만으로는 안전 보장 불가)
- **Single Source of Truth**: `services/shared/`에 미들웨어·JWT·에러·rate-limit 정책 통합
- **Layer-isolated rollback**: 각 Layer가 독립 git commit + 롤백 절차 명세
- **Backward-compatible migration where possible**: Internal API JWT는 X-Internal-Token과 병행 → 검증 → 구방식 제거 (Big Bang 회피)
- **Observable security**: 모든 보안 이벤트 (rate-limit 차단, refresh reuse 감지, internal token 검증 실패) 구조화 로그 + audit
- **Convention-driven**: 신규 정의된 SSOT (에러 포맷, JWT 검증 옵션, rate-limit 정책)는 향후 모든 PDCA의 입력

### 2.2 Data Flow — 로그인 → 토큰 발급 → CSRF → API 호출

```
[1] POST /api/v1/auth/login (email + password)
     ↓
   web proxy → auth-service
     ↓
   bcrypt.compare → JWT issue (HS256, 1h)
     ↓
   $transaction:
     • DELETE refresh_tokens WHERE userId AND deviceId 매치 안 되는 것은 NEW-7로 유지
     • INSERT refresh_tokens(token_hash=sha256(refreshToken), userId, deviceId, expiresAt, lastUsedAt)
     • UPDATE auth_users SET lastLoginAt
     ↓
   Response:
     Set-Cookie: accessToken=jwt; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=3600
     Set-Cookie: refreshToken=jwt; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800
     Set-Cookie: csrfToken=random32; SameSite=Strict; Path=/; Max-Age=3600  ← 더블서브밋
     Body: { user: {...} }  ← accessToken 본문 미포함

[2] GET /api/v1/me/work-log-feed (read-only — CSRF 검증 생략)
     Cookie: accessToken=...
     ↓
   web proxy: cookie + UA + IP forward
     ↓
   auth-service: jwtVerify (algorithms: [HS256])
     ↓
   200 OK

[3] POST /api/v1/tasks/:id/work-logs (state-changing — CSRF 필수)
     Cookie: accessToken, csrfToken
     X-CSRF-Token: <csrfToken 값과 동일>
     ↓
   web proxy: csrf 검증 (cookie csrfToken === header X-CSRF-Token)
     ↓
   project-service: jwtVerify
     ↓
   201 Created

[4] Access 만료 (1h) → silent refresh
   기존 accessToken으로 401 발생
     ↓
   apps/web의 fetch wrapper가 자동 POST /api/v1/auth/refresh (cookie refreshToken)
     ↓
   auth-service:
     • token_hash = sha256(refreshToken)
     • SELECT refresh_tokens WHERE token_hash AND expiresAt > NOW()
     • Reuse detection: 만약 이 token_hash가 이미 사용된 표시(rotated_at != null)면
       → 전 사용자 세션 무효화 + alert
     • Rotate: 새 JWT 발급 + INSERT 새 token_hash + UPDATE 이전 row rotated_at=NOW()
     ↓
   새 Set-Cookie 응답
     ↓
   원 요청 재시도 → 200

[5] 다른 기기에서 로그인 (NEW-7 다중 디바이스)
   기존 deviceId-N의 토큰 유지 + 새 deviceId-M 추가
   사용자 메뉴 "다른 기기 모두 로그아웃" → DELETE refresh_tokens WHERE userId AND deviceId != current
```

### 2.3 Internal API 통신 흐름 (FR-30 — Big Bang 결정 v1.1)

> **v1.1 변경**: 전사 배포 전이라 호환성 부담 없음 → 1주 병행 단계 제거, Layer 5에서 즉시 Bearer JWT만 사용. X-Internal-Token은 service-token 발급 (bootstrap)에만 잔존.

```
[현재 — 본 PDCA Layer 5 적용 전] X-Internal-Token (정적 32자+)

  service-A → service-B
    headers: { "X-Internal-Token": "<INTERNAL_API_TOKEN>" }
    ↓
  service-B requireInternal:
    if (token === env.INTERNAL_API_TOKEN) → req.userId="system", req.userRole="ADMIN"

[변경 후 — Layer 5 적용 즉시] Bearer JWT만 (X-Internal-Token은 bootstrap 전용)

  [부트스트랩 — service-A가 service-token 발급받음]
  service-A → auth-service
    POST /internal/service-token { audience: "project" }
    headers: { "X-Internal-Token": "<INTERNAL_API_TOKEN>" }  ← bootstrap에만 사용
    ↓
  auth-service:
    jwt = sign({ iss: "auth", sub: "service-A", aud: "project", role: "system" },
               JWT_SERVICE_SECRET, { expiresIn: "5m", algorithm: "HS256" })
    Response: { token: jwt, expiresAt: ... }
  ↓
  service-A 인메모리 캐시 (4분, expire-30s buffer)

  [실제 호출 — Bearer만]
  service-A → service-B
    headers: { "Authorization": "Bearer <jwt>" }  ← X-Internal-Token 미포함
    ↓
  service-B requireInternal:
    Bearer 검증 (algorithms: [HS256], audience matches, issuer = "auth")
       → req.userId = "service:" + payload.sub, req.userRole = payload.role
       → 자동 ADMIN 부여 안 함, role 명시
    Bearer 누락/검증 실패 → 401 INTERNAL_TOKEN_INVALID
       (X-Internal-Token fallback 없음)
```

**롤백 시**: Layer 5 git tag로 일괄 revert. 점진 단계가 없으므로 부분 롤백 불필요.

#### 3.1.2 변경 후 스키마

```prisma
model RefreshToken {
  id          String    @id @default(cuid())
  tokenHash   String    @unique  @map("token_hash")     // sha256(token) hex (64자)
  userId      String
  deviceId    String?                                    // NEW-7 다중 디바이스
  userAgent   String?                                    // 디바이스 식별 보조
  ipAddress   String?                                    // audit용
  expiresAt   DateTime
  createdAt   DateTime  @default(now())
  lastUsedAt  DateTime  @default(now()) @map("last_used_at")  // refresh 호출 시점
  rotatedAt   DateTime? @map("rotated_at")               // reuse detection
  rotatedToId String?   @map("rotated_to_id")            // 회전된 다음 토큰 (체인 추적)
  user        User      @relation(...)

  @@index([userId])
  @@index([tokenHash])
  @@index([userId, deviceId])
  @@map("refresh_tokens")
}
```

#### 3.1.3 마이그레이션 SQL

```sql

-- 영향: 모든 활성 세션 무효화 (전 사용자 1회 재로그인)
-- 사전 안내: 작업 시간 외 (이관 전이라 영향 적음)

BEGIN;

-- 1. 새 컬럼 추가 (nullable로 시작)

  ADD COLUMN token_hash    VARCHAR(64),
  ADD COLUMN device_id     VARCHAR(64),
  ADD COLUMN user_agent    TEXT,
  ADD COLUMN ip_address    VARCHAR(45),
  ADD COLUMN last_used_at  TIMESTAMP(3) NOT NULL DEFAULT NOW(),
  ADD COLUMN rotated_at    TIMESTAMP(3),
  ADD COLUMN rotated_to_id VARCHAR(30);

-- 2. 기존 데이터 일괄 무효화 (평문 token을 hash 처리하지 않고 모두 삭제)

DELETE FROM refresh_tokens;

ALTER TABLE refresh_tokens

-- 4. 인덱스 추가
CREATE UNIQUE INDEX refresh_tokens_token_hash_key ON refresh_tokens(token_hash);
CREATE INDEX refresh_tokens_user_id_device_id_idx ON refresh_tokens(user_id, device_id);
```

### 4.2 상세 명세

#### 4.2.1 `POST /api/v1/auth/login`

**Request**:
```json
{ "email": "user@oceant.com", "password": "..." }
```

**Response 200** (Set-Cookie 헤더):
```
Set-Cookie: accessToken=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=3600
Set-Cookie: refreshToken=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=604800
Set-Cookie: csrfToken=<random32>; SameSite=Strict; Path=/; Max-Age=3600
Set-Cookie: deviceId=<uuid>; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=31536000
```
```json
{
  "user": { "id": "...", "email": "...", "name": "...", "role": "..." }
  // accessToken 본문 미포함 (httpOnly cookie로만 전달)
}
```

**Errors**:
- `400 INVALID_INPUT` Zod
- `401 INVALID_CREDENTIALS`
- `403 ACCOUNT_DISABLED`
- `429 TOO_MANY_ATTEMPTS` rate-limit (login 5/min/IP 또는 10/15min/email)

#### 4.2.2 `POST /api/v1/auth/refresh`

**Request**: Cookie refreshToken (자동)

**처리 흐름**:
1. cookie에서 refreshToken 추출, 없으면 `401 NO_REFRESH_TOKEN`
2. `tokenHash = sha256(refreshToken)`
3. `SELECT * FROM refresh_tokens WHERE token_hash = $1`
4. **Reuse detection**:
   - 발견 안 됨 → `401 INVALID_REFRESH`
   - `rotated_at IS NOT NULL` → 재사용 감지
     - alert 발송 (Pino structured log + RabbitMQ `security.alert`)
     - `DELETE FROM refresh_tokens WHERE userId = stored.userId` (전 세션 무효화)
     - `401 REFRESH_REUSE_DETECTED`
   - `expiresAt < NOW()` → `401 REFRESH_EXPIRED`
5. 새 Access JWT 발급 (1h)
6. 새 Refresh JWT 생성 + hash
7. Transaction:
   - `UPDATE refresh_tokens SET rotated_at = NOW(), rotated_to_id = newId WHERE id = stored.id`
   - `INSERT refresh_tokens(id=newId, tokenHash=newHash, userId, deviceId, expiresAt, lastUsedAt=NOW())`
8. Set-Cookie 응답

**Response 200**:
```
Set-Cookie: accessToken=<new>; ...
Set-Cookie: refreshToken=<new>; ...
Set-Cookie: csrfToken=<new>; ...
```

#### 4.2.3 `POST /api/v1/auth/logout-all` (신규, NEW-7)

**Request**: Bearer access token

**처리**: `DELETE FROM refresh_tokens WHERE userId = req.userId AND deviceId != currentDeviceId`

**Response 204**

**Response 200**:

#### 4.2.5 `GET /api/csrf` (신규)

**Response 200**:
```
Set-Cookie: csrfToken=<random32>; SameSite=Strict; Path=/; Max-Age=3600
```

#### 4.2.6 `POST /internal/service-token` (신규, FR-30)

**Request**:
```
X-Internal-Token: <INTERNAL_API_TOKEN>
Content-Type: application/json

{ "audience": "project" }
```

**Response 200**:
```json
{
  "token": "<jwt>",
  "expiresAt": "2026-05-02T07:35:00Z",
```

## 6. Error Handling

### 6.1 신규 에러 코드 매핑

| HTTP | Code | 발생 위치 | 처리 |
|---|---|---|---|
| 400 | INVALID_INPUT | Zod parse 실패 | client toast + form 표시 |
| 401 | UNAUTHORIZED | jwtVerify 실패 | redirect /login |
| 401 | INVALID_CREDENTIALS | bcrypt 불일치 | toast "이메일 또는 비밀번호가 올바르지 않습니다" |
| 401 | NO_REFRESH_TOKEN | cookie 없음 | redirect /login |
| 401 | INVALID_REFRESH | hash 불일치 | redirect /login + toast |
| 401 | REFRESH_EXPIRED | 7d 초과 | redirect /login + toast |
| 401 | REFRESH_REUSE_DETECTED | 회전 감지 | **redirect /login + 보안 경고 페이지** + audit alert |
| 403 | CSRF_TOKEN_INVALID | 더블 서브밋 불일치 | toast + 자동 새로고침 (CSRF 재발급) |
| 403 | FORBIDDEN | requireRole 실패 | toast "권한이 없습니다" |
| 429 | TOO_MANY_ATTEMPTS | rate-limit | toast "{retryAfter}초 후 다시 시도" |
| 401 | INTERNAL_TOKEN_INVALID | X-Internal-Token 불일치 | service log only (사용자 비노출) |

### 6.2 Frontend 에러 처리 패치 (api.ts)

```typescript
// 변경 전:
const data = await res.json();
if (!res.ok) throw new Error(data.error || `${res.status}`);

// 변경 후:
const data = await res.json();
if (!res.ok) {
  const err = data.error || { code: "UNKNOWN", message: `HTTP ${res.status}` };
  throw new ApiError(res.status, err.code, err.message, err.details);
}

// 401 INTERCEPT (silent refresh):
if (res.status === 401 && err.code === "UNAUTHORIZED" && !isRefreshRequest) {
  await refreshToken();
  return retry(originalRequest);
}

// 403 CSRF auto-recovery:
if (res.status === 403 && err.code === "CSRF_TOKEN_INVALID") {
  await fetchCsrfToken();
  return retry(originalRequest);
}
```

### 7.4 CSRF 더블 서브밋 토큰

```
1. GET /api/csrf → Set-Cookie: csrfToken=<random32>; (NOT HttpOnly, JS 읽기 가능)
                  Body: { csrfToken: "<random32>" }

2. State-changing request (POST/PUT/PATCH/DELETE):
   apps/web/src/lib/api.ts에서 자동:
     headers["X-CSRF-Token"] = getCookie("csrfToken")
   web proxy의 csrf 미들웨어:
     if (cookie.csrfToken !== headers["X-CSRF-Token"]) → 403 CSRF_TOKEN_INVALID

3. CSRF 토큰 회전:
   - login/refresh 시 자동 새 csrfToken 발급
   - 1h Max-Age (Access TTL과 동일)
   - 만료 시 401과 동일하게 GET /api/csrf 자동 재발급
```

> SameSite=Strict로 cookie가 cross-site에서 전송 안 됨 → CSRF 1차 방어. 더블 서브밋은 SameSite 미지원 브라우저(IE) 대비 + 같은 origin 내 다른 탭에서의 요청 위조 방어.

### 9.4 신규 파일 — 핵심 인터페이스

#### `services/shared/middleware/require-auth.ts`
```typescript
import { FastifyInstance, FastifyRequest, FastifyReply } from 'fastify';
import fp from 'fastify-plugin';
import { verifyOptions } from '../jwt/verify-options';
import { errorResponse, ErrorCode } from '../errors';

declare module 'fastify' {
  interface FastifyRequest {
    userId: string;
    userEmail: string;
    userName: string;
    userRole: 'ADMIN' | 'MANAGER' | 'OPERATOR' | 'VIEWER';
  }
}

export const requireAuth = fp(async (fastify: FastifyInstance) => {
  fastify.decorateRequest('userId', '');
  fastify.decorateRequest('userEmail', '');
  fastify.decorateRequest('userName', '');
  fastify.decorateRequest('userRole', 'VIEWER');

  fastify.addHook('onRequest', async (req: FastifyRequest, reply: FastifyReply) => {
    if (req.url === '/health') return;
    if (req.url.startsWith('/internal/')) return;  // requireInternal에서 처리

    try {
      const decoded = await req.jwtVerify(verifyOptions) as JwtPayload;
      req.userId = decoded.sub;
      req.userEmail = decoded.email ?? '';
      req.userName = decoded.name ?? '';
      req.userRole = decoded.role;
    } catch {
      reply.code(401).send(errorResponse(ErrorCode.UNAUTHORIZED, '인증이 필요합니다.'));
    }
  });
});
```

#### `services/shared/middleware/require-internal.ts` (v1.1 — Big Bang)
```typescript
// v1.1: Bearer JWT만 검증. X-Internal-Token은 service-token 발급 (POST /internal/service-token)에서만 처리.
export const requireInternal = fp(async (fastify) => {
  fastify.addHook('onRequest', async (req, reply) => {
    if (!req.url.startsWith('/internal/')) return;

    // POST /internal/service-token은 별도 핸들러에서 X-Internal-Token 검증 (bootstrap 전용)
    if (req.url === '/internal/service-token' && req.method === 'POST') return;

    // 그 외 모든 /internal/* 라우트는 Bearer JWT 필수
    const auth = req.headers.authorization;
    if (!auth?.startsWith('Bearer ')) {
      reply.code(401).send(errorResponse(ErrorCode.INTERNAL_TOKEN_INVALID, 'Bearer service token required'));
      return;
    }

    try {
      const decoded = jwt.verify(auth.slice(7), process.env.JWT_SERVICE_SECRET, {
        algorithms: ['HS256'],
        audience: process.env.SERVICE_NAME,  // 자기 서비스 이름
        issuer: 'auth',
      }) as ServiceTokenPayload;
      req.userId = `service:${decoded.sub}`;
      req.userRole = decoded.role;  // 자동 ADMIN 승격 제거 — payload에 명시된 role 사용
    } catch {
      reply.code(401).send(errorResponse(ErrorCode.UNAUTHORIZED, 'Invalid service token'));
    }
  });
});
```

#### `services/shared/jwt/service-token.ts`
```typescript
const cache = new Map<string, { token: string; expiresAt: number }>();

export async function getServiceToken(audience: string): Promise<string> {
  const cached = cache.get(audience);
  if (cached && cached.expiresAt > Date.now() + 30_000) return cached.token;  // 30s buffer

  const res = await fetch(`${process.env.AUTH_SERVICE_URL}/internal/service-token`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Internal-Token': process.env.INTERNAL_API_TOKEN,  // bootstrap 전용
    },
    body: JSON.stringify({ audience }),
  });

  if (!res.ok) throw new Error(`Service token failed: ${res.status}`);
  const data = await res.json();
  cache.set(audience, {
    token: data.token,
    expiresAt: new Date(data.expiresAt).getTime(),
  });
  return data.token;
}
```
