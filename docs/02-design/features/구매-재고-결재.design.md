## 1. 아키텍처 결정

### 1.1 서비스 배치

3개 영역을 2개 서비스에 배치:

| 영역 | 서비스 | 포트 | 이유 |
|------|--------|------|------|
| 구매/회계 + 재고 | equipment-service (기존) | 3005 | 재고번호가 장비와 직결, Part/PurchaseOrder와 구조 유사 |
| 전자결재 | approval-service (신규) | 3006 | 결재 문서량(연 ~10,000건) + 파일 첨부 + PDF 생성이 auth 범위 초과 |

```
equipment-service (port 3005) — 코드 디렉토리 분리
├── src/procurement/     # 구매/회계 (계약, 발주, 원가정산)
├── src/inventory/       # 재고 (InventoryItem, Transaction, AssetCostEvent)
└── src/                 # 기존 장비/수리관리

approval-service (port 3006) — 신규
├── src/api/             # routes
├── src/application/     # services
├── src/domain/          # FSM, entities
└── src/infrastructure/  # file storage, prisma
```

### 1.2 DB 스키마

- 구매/재고: `equipment` 스키마 내 신규 테이블 추가 (기존 26모델 + ~15모델)
- 전자결재: `approval` 스키마 신규 생성

### 1.3 API 경로 + next.config.mjs 리라이트

```
# 기존 (equipment-service)
/api/v1/equipment/**       → equipment-service:3005

# 신규 구매/재고 (equipment-service, 서비스 단위 와일드카드)
/api/v1/procurement/**     → equipment-service:3005
/api/v1/inventory/**       → equipment-service:3005

# 신규 결재 (approval-service)
/api/v1/approval/**        → approval-service:3006
```

**next.config.mjs 리팩토링**: 리소스별 2줄 → 서비스별 와일드카드로 축소 (Plan R-1 대응)

### 1.4 서비스 간 통신

| From | To | 경로 | 용도 |
|------|----|------|------|
| approval-service | auth-service | `/internal/approval-lines/:userId` | 결재선 조회 |
| approval-service | auth-service | `/internal/users/:id` | 결재자 정보 조회 |
| approval-service | equipment-service | `/internal/orders/:id/confirm` | 발주 확정 후처리 |
| approval-service | equipment-service | `/internal/inventory/release` | 출고 후처리 |
| approval-service | equipment-service | `/internal/expenses/:id/follow-up` | 지출결의서 재무 전달 |
| approval-service | attendance-service | `/internal/leave/:id/framework-approve` | 휴가 승인 (Phase 8) |
| approval-service | attendance-service | `/internal/overtime/:id/framework-approve` | OT 승인 (Phase 8) |

모든 내부 통신: `X-Internal-Token` 헤더 검증

## 4. 상태 전이 (FSM)

### 4.1 해외 발주 FSM (OverseasOrder.status)

```
Phase 1 (결재 없음):
  DRAFT → ORDERED → IN_PRODUCTION → SHIPPED → CUSTOMS
    → PARTIALLY_RECEIVED → ARRIVED → CLOSED
    (또는 CUSTOMS → ARRIVED 직행: 전 품목 일괄 입고 시)

Phase 2 (결재 연동):
  DRAFT → PENDING_APPROVAL → APPROVED → ORDERED → ...
         └→ REJECTED (수정 후 재상신)
```

**전이 트리거**:

| 전이 | 트리거 | 자동 처리 |
|------|--------|-----------|
| DRAFT → ORDERED (Ph1) | 사용자 "발주 확정" | - |
| DRAFT → PENDING_APPROVAL (Ph2) | 결재 상신 | ApprovalDocument 생성 |
| APPROVED → ORDERED | 결재 승인 콜백 | postApprovalAction: ORDER_CONFIRM |
| ORDERED → IN_PRODUCTION | 사용자 진행율 최초 입력 | - |
| IN_PRODUCTION → SHIPPED | 실제선적일 입력 | 진행율 100% 권장 |
| SHIPPED → CUSTOMS | 통관일 입력 | - |
| CUSTOMS → PARTIALLY_RECEIVED | 일부 품목 입고 처리 | OrderItem.receivedQuantity 누적 |
| PARTIALLY_RECEIVED → ARRIVED | 전 품목 입고 완료 | 자동 감지 (모든 item FULLY_RECEIVED) |
| CUSTOMS → ARRIVED | 전 품목 일괄 입고 | 위와 동일 |
| ARRIVED → CLOSED | 사용자 "발주 종결" | 계약 내 전 발주 CLOSED 시 계약 COMPLETED |

**부분 입고 규칙**:
```
입고 처리 시:
1. OverseasOrderItem.receivedQuantity += 입고수량
2. receivedQuantity >= quantity → item.receiptStatus = FULLY_RECEIVED
3. receivedQuantity > 0 && < quantity → item.receiptStatus = PARTIALLY_RECEIVED
4. 입고된 품목마다 InventoryItem 생성 (category: IN_TRANSIT → PRODUCT 자동전환, 수동변경 가능)
5. 전체 items 중 하나라도 입고 && 미완료 존재 → order.status = PARTIALLY_RECEIVED
6. 전체 items 모두 FULLY_RECEIVED → order.status = ARRIVED
```

### 4.2 결재 FSM (ApprovalDocument.status)

```
[DRAFT] → [AGREEMENT_PENDING] → [SUBMITTED] → [STEP_1_PENDING] → [STEP_2_PENDING] → [STEP_3_PENDING] → [APPROVED]
               │                     │                │                    │                    │
               └→ RETURNED           │                └── REJECTED ────────┴── REJECTED ────────┘
               (합의자 부동의)         │
                                     └→ 합의자 없으면 바로 SUBMITTED → STEP_1_PENDING
                                     └→ 결재 단계 수는 approvalStepCount에 따라 1~3단 가변
```

**전이 규칙**:

| 전이 | 조건 | 처리 |
|------|------|------|
| DRAFT → AGREEMENT_PENDING | 합의자 존재 | 합의자에게 알림 |
| DRAFT → SUBMITTED | 합의자 없음 | 바로 결재 시작 |
| AGREEMENT_PENDING → SUBMITTED | 전원 동의 | 결재선 매핑, STEP_1_PENDING |
| AGREEMENT_PENDING → RETURNED | 부동의 | 기안자에게 반환 |
| STEP_N_PENDING → STEP_(N+1)_PENDING | N단계 승인 | 다음 결재자 알림 |
| STEP_N_PENDING → APPROVED | 마지막 단계 승인 | postApprovalAction 실행 + 참고 전달 |
| STEP_N_PENDING → REJECTED | 결재 반려 | 기안자에게 알림 |

### 4.3 지출결의서 후속처리 FSM (ExpenseFollowUp.status)

```
[FINANCE_RECEIVED] → [INVENTORY_DECIDED] → [ARRIVED] → [COMPLETED]
                           │
                           └→ isInventoryTarget = false → 바로 COMPLETED
                           └→ isInventoryTarget = true → 입고 대기 → ARRIVED에서 재고번호 부여
```

## 19. v1.6 SKU 확장 (2026-05-13)

> Plan v1.6에 대응하는 신규 설계. §1~§18은 v1.0 컨텍스트 보존용. 충돌 시 §19 우선.

### 19.1 신규 모델 (Prisma equipment 스키마)

#### 19.1.1 ProductMaster 확장 (기존 확장)

```prisma
model ProductMaster {
  id              String    @id @default(cuid())
  name            String
  modelName       String    @map("model_name")
  manufacturer    String
  masterCode      String?   @map("master_code")  // v1.6 신규: SKU code 생성용 짧은 코드 (3~6자)
  keyAttributes   Json?     @map("key_attributes")  // v1.6 신규: ["length", "end_type"] 등 SKU code에 포함할 핵심 축
  commonSpecs     Json?     @map("common_specs")  // 공통 사양 (variant 무관)
  unitOfMeasure   String?   @map("unit_of_measure")  // v1.6 신규: "EA"|"m"|"kg"|"L"|"식"
  defaultCurrency OrderCurrency? @map("default_currency")
  referencePrice  Decimal?  @map("reference_price") @db.Decimal(12, 2)  // 참고용 (사용자가 입력, 자동 갱신 X)

  variants        ProductVariant[]
  inventoryItems  InventoryItem[]
  overseasOrderItems  OverseasOrderItem[] @relation("PMOverseasItems")
  purchaseOrderItems  PurchaseOrderItem[] @relation("PMPurchaseItems")
  bomItems        BomItem[]
  createdAt       DateTime @default(now()) @map("created_at")
  updatedAt       DateTime @updatedAt @map("updated_at")

  @@unique([manufacturer, modelName])
  @@map("product_masters")
  @@schema("equipment")
}
```

#### 19.1.2 ProductVariant (신규)

```prisma
model ProductVariant {
  id              String    @id @default(cuid())
  productMasterId String    @map("product_master_id")
  skuCode         String?   @unique @map("sku_code")  // 의미 코드 (예: "M6F-10M-FE")
  variantSpecs    Json      @default("{}") @map("variant_specs")  // { length: "10m", voltage: "220V", ... }
  isActive        Boolean   @default(true) @map("is_active")

  productMaster   ProductMaster @relation(fields: [productMasterId], references: [id])
  inventoryItems  InventoryItem[]
  bomItems        BomItem[]
  overseasOrderItems  OverseasOrderItem[] @relation("PVOverseasItems")
  purchaseOrderItems  PurchaseOrderItem[] @relation("PVPurchaseItems")
  createdAt       DateTime  @default(now()) @map("created_at")
  updatedAt       DateTime  @updatedAt @map("updated_at")

  @@unique([productMasterId, variantSpecs])  // 같은 마스터 같은 옵션 조합 중복 방지
  @@index([productMasterId])
  @@map("product_variants")
  @@schema("equipment")
}
```

**관련 정책:**
- Variant에 `reference_price` 필드 **없음** (표준가 개념 제거)
- "현재 시세" = `SELECT unit_price FROM inventory_items WHERE variantId=X AND currentStatus=IN_STOCK ORDER BY createdAt DESC LIMIT 1`
- "가격 이력" = `SELECT createdAt, unit_price FROM inventory_items WHERE variantId=X ORDER BY createdAt`

#### 19.1.3 InventoryItem 정정 (v1.6)

```prisma
model InventoryItem {
  id                    String    @id @default(cuid())
  inventoryNo           String    @unique @map("inventory_no")  // INV-YYMM-NNNN (v1.6 새 규칙)
  productMasterId       String?   @map("product_master_id")
  variantId             String?   @map("variant_id")  // v1.6 신규
  itemName              String?   @map("item_name")
  manufacturer          String?
  serialNumber          String?   @map("serial_number")
  trackingMode          InventoryTrackingMode @default(INDIVIDUAL) @map("tracking_mode")
  totalQuantity         Int       @default(1) @map("total_quantity")  // v1.6 정정 (이전: quantity, current_location 제거)
  category              InventoryCategory @default(PRODUCT)
  currentStatus         InventoryStatus @default(IN_STOCK) @map("current_status")
  unitPrice             Decimal?  @map("unit_price") @db.Decimal(12, 2)
  supplyAmount          Decimal?  @map("supply_amount") @db.Decimal(14, 0)
  totalAmount           Decimal?  @map("total_amount") @db.Decimal(14, 0)
```

#### 19.2.1 ProductVariant API

```
GET    /api/v1/product-variants?productMasterId=X
POST   /api/v1/product-variants
   body: { productMasterId, variantSpecs, skuCode? }
PATCH  /api/v1/product-variants/:id
POST   /api/v1/product-variants/:idA/merge/:idB   ← ADMIN 전용 Variant 통합
```

#### 19.2.2 InboundRequest API

```
GET    /api/v1/inbound-requests?status=PENDING
GET    /api/v1/inbound-requests/:id
POST   /api/v1/inbound-requests
POST   /api/v1/inbound-requests/:id/receive   ← 입고 처리 폼 제출
   body: { items: [{ inventoryItemId 또는 신규 생성 정보 }] }
PATCH  /api/v1/inbound-requests/:id/cancel
```

내부 호출 (다른 서비스):
```
POST /internal/inbound-requests  ← 발주·결재·경비 서비스가 호출
   body: { sourceType, sourceId, items[] }
```

#### 19.2.3 BomDefinition / BundleShipment API

```
GET    /api/v1/bom-definitions
POST   /api/v1/bom-definitions
   body: { code, name, items[], referencePrice }
PATCH  /api/v1/bom-definitions/:id

GET    /api/v1/bundle-shipments
POST   /api/v1/bundle-shipments
   body: { bomDefinitionId?, customerId, items[], warrantyUntil }
   * items에 OPTIONAL 슬롯 포함/제외 결정 반영
GET    /api/v1/bundle-shipments/:id/sibling-assets
   → repair_orders에서 형제 자산 조회용
```

#### 19.2.4 Inventory 출고 / TRANSFER API

```
POST   /api/v1/inventory-items/:id/release
   body: { locationId, quantity }
   * 해당 location의 InventoryItemLocation.quantity 차감
   * totalQuantity 차감, 0 되면 currentStatus = RELEASED

POST   /api/v1/inventory-items/:id/transfer
   body: { fromLocationId, toLocationId, quantity }
   * InventoryItemLocation 두 row 수량 조정
   * InventoryTransaction.type = TRANSFER 기록
   * inventory_no는 그대로
```
