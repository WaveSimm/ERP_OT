## 2. project-service 미들웨어 정비

### 2.1 auth.middleware.ts — requireRole 개선

**파일**: `services/project/src/api/middleware/auth.middleware.ts`

현재 `requireRole()` 함수가 존재하나 실제 라우트에 미적용 상태.
함수는 유지하고 **라우트 전체에 적용**하는 것이 핵심.

```typescript
/** 역할 기반 권한 검사 — 지정 role 중 하나 이상이어야 통과 */
export function requireRole(...roles: string[]) {
  return async (request: FastifyRequest, reply: FastifyReply) => {
    if (!roles.includes(request.userRole)) {
      reply.status(403).send({ code: "FORBIDDEN", message: "권한이 없습니다." });
    }
  };
}

/** MANAGER 이상 (MANAGER + ADMIN) */
export function requireManager() {
  return requireRole("ADMIN", "MANAGER");
}

/** ADMIN 전용 */
export function requireAdmin() {
  return requireRole("ADMIN");
}
```

### 2.2 본인/타인 분기 헬퍼

```typescript
/** 본인 여부 확인 후 MANAGER 이상 아니면 403 */
export function requireSelfOrManager(getOwnerId: (req: FastifyRequest) => Promise<string>) {
  return async (request: FastifyRequest, reply: FastifyReply) => {
    if (request.userRole === "ADMIN" || request.userRole === "MANAGER") return;
    const ownerId = await getOwnerId(request);
    if (ownerId !== request.userId) {
      reply.status(403).send({ code: "FORBIDDEN", message: "본인 항목만 수정할 수 있습니다." });
    }
  };
}
```

### 3.2 task.routes.ts

| 메서드+경로 | 권한 | 처리 방식 |
|-------------|------|-----------|
| GET /tasks | 전체 | 없음 |
| GET /tasks/:taskId | 전체 | 없음 |
| POST /tasks | MANAGER+ | `preHandler: requireManager()` |
| DELETE /tasks/:taskId | MANAGER+ | `preHandler: requireManager()` |
| PATCH /tasks/:taskId | 본인 or MANAGER+ | 핸들러 내 분기 처리 |
| POST /tasks/:taskId/assignments | MANAGER+ | `preHandler: requireManager()` |
| DELETE /tasks/:taskId/assignments/:id | MANAGER+ | `preHandler: requireManager()` |

**태스크 수정 분기 처리 (PATCH /tasks/:taskId)**:
```typescript
fastify.patch("/:taskId", async (req, reply) => {
  const { taskId } = req.params as { taskId: string };
  const body = req.body as UpdateTaskDto;

  // MANAGER 이상은 모든 필드 수정 가능
  if (req.userRole === "ADMIN" || req.userRole === "MANAGER") {
    return reply.send(await taskService.updateTask(taskId, body));
  }

  // OPERATOR는 본인 태스크의 진행률만 수정 가능
  const task = await taskService.getTask(taskId);
  if (task.assigneeId !== req.userId) {
    return reply.status(403).send({ code: "FORBIDDEN", message: "본인 태스크만 수정할 수 있습니다." });
  }

  // 허용 필드만 추출 (overallProgress만)
  const allowed = { overallProgress: body.overallProgress };
  return reply.send(await taskService.updateTask(taskId, allowed));
});
```

## 4. 프론트엔드

### 4.1 usePermission 훅

**파일**: `apps/web/src/hooks/usePermission.ts` (신규)

```typescript
import { useAuthStore } from "@/store/auth.store";

export function usePermission() {
  const user = useAuthStore((s) => s.user);

  const isAdmin = user?.role === "ADMIN";
  const isManager = user?.role === "MANAGER" || isAdmin;
  const isOperator = user?.role === "OPERATOR" || isManager;
  const isViewer = !!user; // 로그인만 되어 있으면

  return { isAdmin, isManager, isOperator, isViewer };
}
```

### 4.2 UI 제어 적용 위치

| 컴포넌트 | 조건 | 처리 |
|-----------|------|------|
| 프로젝트 생성 버튼 | `isManager` | false → 숨김 |
| 프로젝트 수정/삭제 버튼 | `isManager` | false → 숨김 |
| 태스크 생성/삭제 버튼 | `isManager` | false → 숨김 |
| 자원 배정 버튼 | `isManager` | false → 숨김 |
| 자원 관리 메뉴 | `isManager` | false → 숨김 |
| 그룹 관리 메뉴 | `isManager` | false → 숨김 |
| 템플릿 관리 메뉴 | `isManager` | false → 숨김 |
| 임계값 설정 | `isManager` | false → 숨김 |
| 사용자 관리 메뉴 | `isAdmin` | false → 숨김 |
| 타인 태스크 수정 | `isManager` | false → 비활성화 |

### 4.3 적용 예시

```tsx
// 프로젝트 페이지
const { isManager, isAdmin } = usePermission();

<Button
  onClick={handleCreateProject}
  className={isManager ? "" : "hidden"}
>
  프로젝트 생성
</Button>

// 태스크 수정 — 본인이거나 MANAGER 이상
const canEditTask = isManager || task.assigneeId === currentUser?.id;
<Button disabled={!canEditTask} onClick={handleEdit}>수정</Button>
```

## 5. 구현 순서 (Phase별)

### Phase 1 — project-service 미들웨어 정비
1. `auth.middleware.ts`: `requireManager()`, `requireAdmin()` 헬퍼 추가
2. 기존 `requireRole` 정리 (변경 없음, 유지)

### Phase 2 — 프로젝트/태스크 라우트 권한 적용
1. `project.routes.ts`: POST/PATCH/DELETE에 `requireManager()` 적용
2. `task.routes.ts`: 생성/삭제/배정에 `requireManager()`, PATCH 분기 처리

### Phase 3 — 자원/그룹/템플릿/대시보드 권한 적용
1. `resource.routes.ts`
2. `group.routes.ts`
3. `template.routes.ts`
4. `dashboard.routes.ts`: 기존 role 문자열 비교 → `requireManager()` 교체

### Phase 4 — 프론트엔드
1. `usePermission.ts` 훅 생성
2. 프로젝트 페이지 적용
3. 태스크/자원/그룹/대시보드 페이지 적용
