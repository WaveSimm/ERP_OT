### 2.2 PostgreSQL CHECK 제약 (raw SQL)

Prisma는 CHECK 제약을 직접 지원하지 않으므로 마이그레이션에 raw SQL 추가:

```sql
-- migrations/YYYYMMDDHHMMSS_milestone_redesign/migration.sql

-- Dependency 무결성: predecessor 1개 + successor 1개
ALTER TABLE project.dependencies
  ADD CONSTRAINT dep_predecessor_xor CHECK (
    (predecessorTaskId IS NOT NULL)::int + (predecessorMilestoneId IS NOT NULL)::int = 1
  );

ALTER TABLE project.dependencies
  ADD CONSTRAINT dep_successor_xor CHECK (
    (successorTaskId IS NOT NULL)::int + (successorMilestoneId IS NOT NULL)::int = 1
  );

-- 자기참조 금지
ALTER TABLE project.dependencies
  ADD CONSTRAINT dep_no_self_task CHECK (
    predecessorTaskId IS NULL OR successorTaskId IS NULL OR predecessorTaskId <> successorTaskId
  );

ALTER TABLE project.dependencies
  ADD CONSTRAINT dep_no_self_milestone CHECK (
    predecessorMilestoneId IS NULL OR successorMilestoneId IS NULL OR predecessorMilestoneId <> successorMilestoneId
  );

-- Milestone status 일관성: ACHIEVED면 achievedDate 필수
ALTER TABLE project.milestones
  ADD CONSTRAINT ms_achieved_date CHECK (
    (status = 'ACHIEVED' AND achievedDate IS NOT NULL) OR status <> 'ACHIEVED'
  );

-- linkedProgress 범위
ALTER TABLE project.milestones
  ADD CONSTRAINT ms_progress_range CHECK (linkedProgress >= 0 AND linkedProgress <= 100);
```

### 2.3 Cycle Detection (App Layer)

DB는 의존성 순환을 막을 수 없으므로 DependencyService에서 검증:

```typescript
async checkNoCycle(newDep: NewDependencyDto): Promise<void> {
  // BFS from successor — if we can reach predecessor, cycle exists
  const visited = new Set<string>();
  const queue: { type: 'TASK' | 'MILESTONE'; id: string }[] = [
    { type: newDep.successorType, id: newDep.successorId }
  ];

  while (queue.length > 0) {
    const node = queue.shift()!;
    const key = `${node.type}:${node.id}`;
    if (visited.has(key)) continue;
    visited.add(key);

    if (node.type === newDep.predecessorType && node.id === newDep.predecessorId) {
      throw new AppError(409, 'DEPENDENCY_CYCLE', '의존성 순환이 발생합니다.');
    }

    // 이 node의 successor를 큐에 추가
    const next = await this.findSuccessors(node);
    queue.push(...next);
  }
}
```

```typescript
export class MilestoneService {
  constructor(
    private readonly prisma: PrismaClient,
    private readonly aggregate: AggregateService,
    private readonly cpm: CPMService,
    private readonly notification: NotificationService,
    private readonly activityLog: ActivityLogService,
  ) {}

  // ─── CRUD ────────────────────────────────────────────────────────
  async create(projectId: string, dto: CreateMilestoneDto, userId: string): Promise<Milestone>;
  async update(id: string, dto: UpdateMilestoneDto, userId: string): Promise<Milestone>;
  async delete(id: string, userId: string): Promise<void>;
  async findByProject(projectId: string): Promise<Milestone[]>;

  // ─── linkedTasks 관리 ────────────────────────────────────────────
  async linkTasks(milestoneId: string, taskIds: string[], userId: string): Promise<void>;
  async unlinkTask(milestoneId: string, taskId: string, userId: string): Promise<void>;

  // ─── 자동 status 갱신 (hook) ─────────────────────────────────────
  /**
   * Task 진행률·status 변경 시 호출. 해당 Task가 link된 모든 milestone의
   * linkedProgress·status 재계산.
   */
  async onTaskProgressChange(taskId: string): Promise<void> {
    const milestones = await this.prisma.milestone.findMany({
      where: { linkedTasks: { some: { id: taskId } } },
      include: { linkedTasks: { select: { overallProgress: true, status: true } } },
    });
    for (const m of milestones) {
      await this.recalculateStatus(m.id);
    }
  }

  /**
   * Milestone status 자동 판정 로직.
   * 1. linkedTasks 평균 progress → linkedProgress 캐시 갱신
   * 2. status 결정:
   *    - achievementCriteria=ALL_LINKED_TASKS_DONE이고 모두 DONE → ACHIEVED
   *    - dueDate 경과 + 미ACHIEVED → MISSED
   *    - dueDate D-7 + linkedProgress < 70 → AT_RISK
   *    - CPM totalFloat < 0 → INFEASIBLE
   *    - 외 → PLANNED
   * 3. 변경 시 NotificationService로 알림 발송 + ActivityLog 기록
   */
  async recalculateStatus(milestoneId: string): Promise<MilestoneStatus>;

  // ─── CPM 결과 반영 ───────────────────────────────────────────────
  async updateCpmResult(milestoneId: string, cpmResult: CpmNodeResult): Promise<void>;
}
```

### 3.3 CPMService 확장

`services/project/src/application/cpm.service.ts`

기존 알고리즘에 다음 변경:

```typescript
// 기존: Task만 노드
// 변경: Task + Milestone 통합 노드 그래프

interface CpmNode {
  type: 'TASK' | 'MILESTONE';
  id: string;
  duration: number;       // Milestone은 0
  earliestStart: Date;
  earliestFinish: Date;
  latestStart: Date;
  latestFinish: Date;
  totalFloat: number;
  isCritical: boolean;
  // Milestone 전용
  dueDateConstraint?: Date;  // (5-β) — latestFinish 상한
}

class CPMService {
  async recalculate(projectId: string): Promise<CpmResult> {
    // 1. Task + Milestone + Dependency 모두 로드
    // 2. 통합 그래프 구성
    // 3. Forward pass (earliestStart/Finish)
    //    - Task: duration = effectiveEndDate - effectiveStartDate
    //    - Milestone: duration = 0
    // 4. Backward pass (latestStart/Finish)
    //    - Milestone with dueDate: latestFinish = MIN(graph_latest, dueDate)
    // 5. totalFloat = latestStart - earliestStart
    //    - Milestone INFEASIBLE: totalFloat < 0
    //    - isCritical: totalFloat = 0
    // 6. 결과 캐시 (Task.totalFloat, Milestone.totalFloat 등)
  }

  // 디바운스 1000ms (OQ-3 결정) — 프로젝트별 키
  private debouncedRecalculate = debounceByKey(
    (projectId: string) => this.recalculate(projectId),
    1000
  );

  /** 외부 호출용 — 변경 hook에서 이걸 사용 */
  triggerRecalculate(projectId: string): void {
    this.debouncedRecalculate(projectId);
  }
}
```

```diff

+ tasks와 milestones를 통합한 행 배열로 렌더
```

## 6. Data Flow

### 6.1 Milestone 생성 → 자동 추적 시작

```
PM이 MilestoneModal에서 입력
    ↓
POST /api/v1/projects/:id/milestones { name, dueDate, linkedTaskIds }
    ↓
MilestoneService.create()
    ├─ Prisma create (linkedTasks connect)
    ├─ recalculateStatus() — 초기 상태 판정
    ├─ ActivityLog: MILESTONE_CREATED
    └─ CPM.recalculate(projectId) — 의존성 그래프 재구성
    ↓
WebSocket: project:{id} → milestone:created
    ↓
모든 클라이언트 GanttChart·SidePanel 갱신 (TanStack Query invalidate)
```

### 6.2 Task DONE → linked Milestone 자동 ACHIEVED

```
Task progress 100% 갱신 (TaskService.updateProgress)
    ↓
AggregateService.onTaskUpdated(taskId)
    ├─ Project.overallProgress 갱신 (기존)
    ├─ MilestoneService.onTaskProgressChange(taskId) ← 신규
    │     └─ for each linkedMilestone:
    │           recalculateStatus(milestoneId)
    │              ├─ linkedProgress 평균 계산
    │              ├─ 모두 DONE → status=ACHIEVED, achievedDate=now()
    │              ├─ NotificationService.send: MILESTONE_ACHIEVED
    │              └─ ActivityLog: MILESTONE_ACHIEVED
    └─ CPMService.recalculate() (debounced)
```

### 6.3 D-7 + 진행률 부족 → AT_RISK 자동 감지

```
IssueDetector cron (또는 Task progress hook)
    ↓
for each milestone where status IN (PLANNED, AT_RISK):
    if dueDate within 7 days AND linkedProgress < 70:
        new_status = AT_RISK
    if dueDate < now AND status != ACHIEVED:
        new_status = MISSED
    if totalFloat < 0:
        new_status = INFEASIBLE
    ↓
status 변경 시 알림 발송 (OQ-7 결정):
    AT_RISK     → PM + linkedTasks 담당자 (priority 2)
    MISSED      → PM + linkedTasks 담당자 + 결재라인 (priority 1)
    INFEASIBLE  → PM + linkedTasks 담당자 (priority 1)
                  ※ 24h 내 동일 milestone 중복 알림 금지
                  ※ 해소(totalFloat ≥ 0) 시 RESOLVED 알림 (priority 3)
    ActivityLog 기록
    Dashboard issue list 추가
```

### 6.4 의존성 추가 → CPM 재계산

```
PM이 갠트에서 의존선 드래그 (Task A → Milestone M1)
    ↓
POST /api/v1/projects/:id/dependencies
    ↓
DependencyService.create()
    ├─ Cycle check (BFS)
    ├─ Prisma create
    └─ CPMService.recalculate(projectId)
        ├─ 통합 그래프 재구성
        ├─ Forward/Backward pass
        ├─ Milestone with dueDate: latestFinish = MIN(graph, dueDate)
        ├─ totalFloat 계산 → INFEASIBLE 판정
        └─ Task·Milestone 캐시 일괄 업데이트
    ↓
변화된 Milestone status 알림
```

### 7.2 마이그레이션 SQL

```sql
-- migration.sql (요약)

-- 1. 신규 enum
CREATE TYPE project."MilestoneStatus" AS ENUM ('PLANNED','AT_RISK','ACHIEVED','MISSED','INFEASIBLE');
CREATE TYPE project."AchievementCriteria" AS ENUM ('ALL_LINKED_TASKS_DONE','MANUAL');

-- 2. milestones 테이블 — 기존 컬럼 삭제 후 재정의
ALTER TABLE project.milestones DROP COLUMN sort_order CASCADE; -- 그룹 의미 제거
ALTER TABLE project.milestones
  ADD COLUMN due_date         TIMESTAMP(3),
  ADD COLUMN achieved_date    TIMESTAMP(3),
  ADD COLUMN status           project."MilestoneStatus" DEFAULT 'PLANNED',
  ADD COLUMN achievement_criteria project."AchievementCriteria" DEFAULT 'ALL_LINKED_TASKS_DONE',
  ADD COLUMN sort_order       INT DEFAULT 0,
  ADD COLUMN display_parent_id VARCHAR,
  ADD COLUMN is_critical      BOOLEAN DEFAULT false,
  ADD COLUMN earliest_start   TIMESTAMP(3),
  ADD COLUMN latest_start     TIMESTAMP(3),
  ADD COLUMN total_float      DOUBLE PRECISION,
  ADD COLUMN linked_progress  DOUBLE PRECISION DEFAULT 0,
  ADD COLUMN created_by       VARCHAR;
-- description, sortOrder는 기존 그대로 유지(이름만 재의미화)

-- 3. linkedTasks 매핑 테이블 (Prisma m:n implicit)
CREATE TABLE project."_MilestoneLinkedTasks" (
  "A" VARCHAR NOT NULL REFERENCES project.milestones(id) ON DELETE CASCADE,
  "B" VARCHAR NOT NULL REFERENCES project.tasks(id) ON DELETE CASCADE,
  PRIMARY KEY ("A","B")
);

-- 4. dependencies 신규 + CHECK 제약
CREATE TABLE project.dependencies (
  id VARCHAR PRIMARY KEY,
  predecessor_task_id VARCHAR REFERENCES project.tasks(id) ON DELETE CASCADE,
  predecessor_milestone_id VARCHAR REFERENCES project.milestones(id) ON DELETE CASCADE,
  successor_task_id VARCHAR REFERENCES project.tasks(id) ON DELETE CASCADE,
  successor_milestone_id VARCHAR REFERENCES project.milestones(id) ON DELETE CASCADE,
  dependency_type project."DependencyType" DEFAULT 'FS',
  lag INT DEFAULT 0,
  created_by VARCHAR NOT NULL,
  created_at TIMESTAMP(3) DEFAULT NOW(),
  CONSTRAINT dep_predecessor_xor CHECK (
    (predecessor_task_id IS NOT NULL)::int + (predecessor_milestone_id IS NOT NULL)::int = 1
  ),
  CONSTRAINT dep_successor_xor CHECK (
    (successor_task_id IS NOT NULL)::int + (successor_milestone_id IS NOT NULL)::int = 1
  )
);

-- 5. 백필: isMilestone Task 2건 → Milestone 레코드
INSERT INTO project.milestones (id, project_id, name, due_date, status, sort_order, created_by, created_at, updated_at)
SELECT
  'msl_' || substring(t.id from 4),         -- 새 ID
  t.project_id,
  t.name,
  s.start_date,                              -- segment의 startDate를 dueDate로
  CASE
    WHEN t.status = 'DONE' THEN 'ACHIEVED'::project."MilestoneStatus"
    ELSE 'PLANNED'::project."MilestoneStatus"
  END,
  t.sort_order,
  t.created_by,
  t.created_at,
  t.updated_at
FROM project.tasks t
JOIN project.task_segments s ON s.task_id = t.id
WHERE t."isMilestone" = true;

-- 6. 백필: TaskDependency → Dependency (OQ-6 결정 매핑)
-- 실측 12건. 컬럼명 신규 통일: lagDays → lag, type → dependencyType
INSERT INTO project.dependencies (
  id, predecessor_task_id, predecessor_milestone_id,
  successor_task_id, successor_milestone_id,
  dependency_type, lag, created_by, created_at
)
SELECT
  'dep_' || substring(td.id from 4),    -- ID prefix 'cl_' → 'dep_'
  td."predecessorId",                    -- predecessor_task_id
  NULL,                                  -- predecessor_milestone_id (현재 마일스톤 없음)
  td."successorId",                      -- successor_task_id
  NULL,                                  -- successor_milestone_id
  td.type,                               -- DependencyType enum 그대로
  td."lagDays",                          -- lagDays → lag
  'system',                              -- OQ-6: sentinel
  NOW()                                  -- OQ-6: 마이그레이션 시각
FROM project.task_dependencies td;
-- 검증: SELECT COUNT(*) FROM project.dependencies; -- 12건 기대
-- 마일스톤 의존성은 마이그레이션 후 사용자가 직접 그림

-- 7. 구버전 제거
ALTER TABLE project.tasks DROP COLUMN "isMilestone";
ALTER TABLE project.tasks DROP COLUMN milestone_id;
DROP TABLE project.task_dependencies;

-- 8. 인덱스
CREATE INDEX milestones_project_sort ON project.milestones(project_id, sort_order);
CREATE INDEX milestones_project_status ON project.milestones(project_id, status);
CREATE INDEX milestones_project_duedate ON project.milestones(project_id, due_date);
CREATE INDEX dependencies_pred_task ON project.dependencies(predecessor_task_id);
CREATE INDEX dependencies_pred_ms ON project.dependencies(predecessor_milestone_id);
CREATE INDEX dependencies_succ_task ON project.dependencies(successor_task_id);
CREATE INDEX dependencies_succ_ms ON project.dependencies(successor_milestone_id);
```

## 10. Decisions (Open Questions Resolved)

| # | 질문 | 결정 사항 |
|---|------|-----------|
| OQ-1 | `displayParentId`가 가리키는 Task가 삭제되면? | **NULL로 set** (Prisma `onDelete: SetNull`). Milestone은 root level로 이동, PM이 SidePanel에서 재배치 |

| OQ-4 | linkedProgress 계산 방식? | **단순평균** `AVG(task.overallProgress)`. weighting은 후속 PDCA "마일스톤-가중진척" |
| OQ-5 | Milestone name을 자연어검색에 포함? | **메인 검색은 후속 PDCA "검색-개선 v2"로 분리** + **MilestoneSidePanel 내부에는 단순 텍스트 필터(client-side ILIKE)** 추가 |
| OQ-6 | TaskDependency 백필 매핑? | **컬럼명 신규 통일** (`lag`, `dependencyType`), **createdBy = `'system'`** sentinel, **createdAt = `NOW()`**. 백필 12건 (실측). 자세한 SQL은 §7.2 |
| OQ-7 | INFEASIBLE 알림 수신자? | **PM + linkedTasks 담당자**. 24h 내 동일 milestone 중복 알림 금지. 결재라인 자동 보고는 후속 PDCA "마일스톤-트리거" |
