## 1. 개요

### 1.1 설계 목표

1. **하이브리드 점수**: 임베딩 코사인 유사도 + 키워드(ILIKE/pg_trgm) 결합
2. **단일 SQL**: 점수 계산은 PostgreSQL 한 번의 쿼리에서 완결 (어플리케이션 후처리 최소화)
3. **운영 중 튜닝 가능**: 7개 환경변수로 가중치·컷오프 조정 (재배포 불필요)
4. **회귀 안전**: 기존 자연어검색 동작 유지 — 임베딩 단독 케이스에서도 동등 결과
5. **비파괴 변경**: Prisma 스키마 변경 없음, extension/index만 추가

### 1.2 설계 원칙

- **공식 단일화**: searchPosts (auth) / semanticSearch (project) 둘 다 **동일한 점수 합산 식** 사용
- **환경변수 SSOT**: 가중치/보너스/컷오프는 `process.env`로만 결정 — 코드에 매직넘버 없음
- **권한 필터 위치 불변**: 기존 `canRead` (post) / `projectIds = ANY(...)` (worklog) 방식 그대로 — 점수 변경이 권한에 영향 없음
- **WorkLog는 title 없음**: title 관련 변수는 WorkLog에서 단순히 0 처리 (조건분기로 SQL 분기 안 함)

## 3. 스키마 변경

### 3.1 PostgreSQL Extension

```sql
-- pgvector 이미지(`pgvector/pgvector:pg16`)에 pg_trgm contrib 모듈 포함됨
-- DB 1회 설치
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

### 3.2 GIN Trigram 인덱스 (3개)

```sql
-- 게시글 제목 (가장 비중 큼)
CREATE INDEX IF NOT EXISTS board_posts_title_trgm_idx
  ON public.board_posts USING gin (title gin_trgm_ops);

-- 게시글 본문
CREATE INDEX IF NOT EXISTS board_posts_content_trgm_idx
  ON public.board_posts USING gin (content gin_trgm_ops);

-- 작업비고 본문
CREATE INDEX IF NOT EXISTS work_logs_content_trgm_idx
  ON project.work_logs USING gin (content gin_trgm_ops);
```

**선택 이유**:
- GIN 트리그램은 `ILIKE '%word%'` 와 `similarity()` 둘 다 가속
- 글/비고는 INSERT/UPDATE 빈도 매우 낮음 → GIN 인덱스 갱신 비용 무시 가능

### 3.3 Prisma 스키마

**변경 없음.** Prisma는 `pg_trgm` 인덱스를 추적할 필요 없음. 마이그레이션은 raw SQL로 분리.

## 4. SQL 설계 (Before / After)

### 4.1 searchPosts — Before (자연어검색 v1.0)

```sql

FROM public.board_posts p

ORDER BY p.embedding <=> $vec::vector
LIMIT $k
```

### 4.2 searchPosts — After (검색개선 v1.0)

```sql
WITH scored AS (
  SELECT
    p.id, p.title, p.content, p.published_at AS "publishedAt",
    p.is_deleted AS "isDeleted",
    p.target_department_id AS "targetDepartmentId",
    p.author_id AS "authorId",
    u.name AS "authorName",
    b.code AS "boardCode", b.name AS "boardName",
    b.write_roles AS "writeRoles",
    b.read_audience AS "readAudience",
    b.audience_target_id AS "audienceTargetId",
    b.allow_comments AS "allowComments",
    c.code AS "catCode",

    -- 임베딩 점수 (0~1)
    (1 - (p.embedding <=> $vec::vector)) AS embed_score,

    -- 정확 일치 보너스 (ILIKE)
    CASE WHEN p.title   ILIKE '%' || $q || '%' THEN $title_exact   ELSE 0 END
    + CASE WHEN p.content ILIKE '%' || $q || '%' THEN $content_exact ELSE 0 END
      AS exact_bonus,

    -- 트라이그램 유사도 점수
    GREATEST(
      similarity(p.title,   $q) * $title_trgm,
      similarity(p.content, $q) * $content_trgm
    ) AS trgm_score
  FROM public.board_posts p
  JOIN public.boards b ON b.id = p.board_id
  JOIN public.board_categories c ON c.id = b.category_id
  JOIN public.auth_users u ON u.id = p.author_id
  WHERE p.embedding IS NOT NULL AND p.is_deleted = false
  ORDER BY p.embedding <=> $vec::vector  -- 1차 ANN top-N (k*3 정도)
  LIMIT $k * 3
),
final AS (
  SELECT *,
    embed_score * $w_embed
    + GREATEST(exact_bonus, trgm_score) * $w_keyword
      AS score
  FROM scored
)
SELECT *
FROM final
WHERE score >= $min_score
ORDER BY score DESC
LIMIT $k;
```

**핵심 변경**:
1. **WITH CTE 2단**: 1차 ANN으로 후보 추림(k*3) → 2차 점수 합산 + 컷오프 + 정렬
2. **`GREATEST(exact_bonus, trgm_score)`**: 정확 일치와 trigram 중 강한 신호 1개만 채택 (이중계산 방지)
3. **k*3 후보**: keyword bonus가 큰 글이 ANN top-K 밖에 있을 수 있음 → 후보 풀 확장

### 4.3 semanticSearch (work-log) — After

```sql
WITH scored AS (
  SELECT
    w.id, w.task_id AS "taskId", w.segment_id AS "segmentId",
    w.author_id AS "authorId", w.author_name AS "authorName",
    w.content, TO_CHAR(w.worked_at, 'YYYY-MM-DD') AS "workedAt",
    t.name AS "taskName", t."projectId" AS "projectId",
    p.name AS "projectName",
    s.name AS "segmentName",

    (1 - (w.embedding <=> $vec::vector)) AS embed_score,

    -- WorkLog는 title 없음 → content만
    CASE WHEN w.content ILIKE '%' || $q || '%' THEN $content_exact ELSE 0 END
      AS exact_bonus,

    similarity(w.content, $q) * $content_trgm AS trgm_score
  FROM project.work_logs w
  JOIN project.tasks t ON t.id = w.task_id
  JOIN project.projects p ON p.id = t."projectId"
  LEFT JOIN project.task_segments s ON s.id = w.segment_id
  WHERE w.embedding IS NOT NULL AND w.is_deleted = false
    AND ($projectIds IS NULL OR t."projectId" = ANY($projectIds))   -- ADMIN/MANAGER vs OPERATOR
  ORDER BY w.embedding <=> $vec::vector
  LIMIT $limit * 3
),
final AS (
  SELECT *,
    embed_score * $w_embed
    + GREATEST(exact_bonus, trgm_score) * $w_keyword
      AS score
  FROM scored
)
SELECT *
FROM final
WHERE score >= $min_score
ORDER BY score DESC
LIMIT $limit;
```

> **참고**: 현재 코드는 `if (projectIds === null) ... else ...` 분기로 두 SQL을 작성. 그대로 두 분기에 같은 CTE 패턴 적용. 단일 SQL로 통합 가능하나 회귀 위험 줄이기 위해 분기 유지.

### 4.4 점수 합산 식 (수식) — v1.1 갱신

```
score = embed_score · W_e
      + max(exact_bonus, trgm_score) · W_k
      + (exact_bonus > 0 ? KEYWORD_MATCH_BONUS : 0)   ← v1.1 추가

where
  embed_score   ∈ [0, 1]    # 1 - cosine distance
  exact_bonus   ∈ {0, B_t,  B_c,  B_t+B_c}      # title/content ILIKE 매칭 합
  trgm_score    ∈ [0, max(W_tt, W_ct)]          # similarity*가중치 max
  W_e + W_k     = 1.0  (검증)

  WHERE score ≥ MIN_SCORE
```

**기본값 (v1.1)**:
- W_e = 0.6, W_k = 0.4
- B_t = 0.30, B_c = 0.15
- W_tt = 0.20, W_ct = 0.10
- KEYWORD_MATCH_BONUS = 0.20  ← v1.1 추가
- MIN_SCORE = 0.35  ← v1.1 변경 (구 0.45)

**v1.1 변경 사유**:
- bge-m3 코사인 점수 분포가 0.4~0.6에 좁게 모이는 특성 + 작은 데이터셋(~10건) 환경에서
  이론치 0.78 도달 어려움 → 절대 가산 보너스로 정확 매칭 글이 cutoff 통과하도록 보강.
- MIN_SCORE 기본값 0.35로 하향 (운영 중 환경변수로 조정 가능).

**해석 (v1.1)**:
- 임베딩 단독으로는 max 0.6 (1.0 × 0.6)
- 정확 일치 단독: max 0.18 (0.45 × 0.4) + 0.20 (match_bonus) = 0.38
- 두 신호 다 강하면 약 0.98까지 도달
- 점수 MIN_SCORE 미만은 컷오프 → 무관 결과 자동 제거

### 5.2 검증 규칙 (서비스 startup 시)

```typescript
// services/auth/src/application/search-config.ts (신규)
function readFloatEnv(name: string, defaultValue: number): number {
  const raw = process.env[name];
  if (!raw) return defaultValue;
  const v = parseFloat(raw);
  if (Number.isNaN(v) || v < 0 || v > 1) {
    throw new Error(`[search-config] ${name} must be float in [0,1], got: ${raw}`);
  }
  return v;
}

export const searchConfig = {
  embedWeight:        readFloatEnv("SEARCH_EMBED_WEIGHT", 0.6),
  keywordWeight:      readFloatEnv("SEARCH_KEYWORD_WEIGHT", 0.4),
```

```typescript
private async searchPosts(
  queryVec: number[],
  q: string,                              // ← 신규
  k: number,
  user: AuthUserContext & { email: string },
): Promise<SearchResultItem[]> {
  const literal = this.embeddingService.toSqlLiteral(queryVec);
  const cfg = searchConfig;
  const trimmedQ = q.trim();

  const rows = await this.prisma.$queryRaw<any[]>`
    WITH scored AS (
      SELECT p.id, p.title, p.content, p.published_at AS "publishedAt",
             p.is_deleted AS "isDeleted",
             p.target_department_id AS "targetDepartmentId",
             p.author_id AS "authorId",
             u.name AS "authorName",
             b.code AS "boardCode", b.name AS "boardName",
             b.write_roles AS "writeRoles",
             b.read_audience AS "readAudience",
             b.audience_target_id AS "audienceTargetId",
             b.allow_comments AS "allowComments",
             c.code AS "catCode",
             (1 - (p.embedding <=> ${literal}::vector)) AS embed_score,
             (CASE WHEN p.title   ILIKE '%' || ${trimmedQ} || '%' THEN ${cfg.titleExactBonus}::float ELSE 0 END
              + CASE WHEN p.content ILIKE '%' || ${trimmedQ} || '%' THEN ${cfg.contentExactBonus}::float ELSE 0 END) AS exact_bonus,
             GREATEST(
               similarity(p.title,   ${trimmedQ}) * ${cfg.titleTrgmWeight}::float,
               similarity(p.content, ${trimmedQ}) * ${cfg.contentTrgmWeight}::float
             ) AS trgm_score
      FROM public.board_posts p
      JOIN public.boards b ON b.id = p.board_id
      JOIN public.board_categories c ON c.id = b.category_id
      JOIN public.auth_users u ON u.id = p.author_id
      WHERE p.embedding IS NOT NULL AND p.is_deleted = false
      ORDER BY p.embedding <=> ${literal}::vector
      LIMIT ${k * 3}

      SELECT *,
        embed_score * ${cfg.embedWeight}::float
        + GREATEST(exact_bonus, trgm_score) * ${cfg.keywordWeight}::float
          AS score
      FROM scored

    SELECT * FROM final
    WHERE score >= ${cfg.minScore}::float
    ORDER BY score DESC
    LIMIT ${k}

  // 권한 필터 (변경 없음)
  const filtered = rows.filter(/* 기존 canRead 로직 */);
  return filtered.map(/* 기존 매핑 */);
}
```

```typescript
  queryVec: number[],
  q: string,                              // ← 신규
  user: { id: string; email: string; role: string },
  limit: number,
): Promise<any[]> {
  if (!queryVec || queryVec.length === 0) return [];
  const cfg = searchConfig;
  const trimmedQ = q.trim();

  // 권한 scope 결정 (기존 로직 유지)
  let projectIds: string[] | null = ...;
  
  const literal = `[${queryVec.join(",")}]`;
  const k3 = limit * 3;

  // 두 분기에 동일 CTE 적용 (간결 표시)
  const baseScored = (whereScope: string) => `
    WITH scored AS (
      SELECT w.id, w.task_id AS "taskId", w.segment_id AS "segmentId",
             w.author_id AS "authorId", w.author_name AS "authorName",
             w.content, TO_CHAR(w.worked_at, 'YYYY-MM-DD') AS "workedAt",
             t.name AS "taskName", t."projectId" AS "projectId",
             p.name AS "projectName", s.name AS "segmentName",
             (1 - (w.embedding <=> '${literal}'::vector)) AS embed_score,
             (CASE WHEN w.content ILIKE '%' || $1 || '%' THEN ${cfg.contentExactBonus} ELSE 0 END) AS exact_bonus,
             similarity(w.content, $1) * ${cfg.contentTrgmWeight} AS trgm_score
      FROM project.work_logs w
      JOIN project.tasks t ON t.id = w.task_id
      JOIN project.projects p ON p.id = t."projectId"
      LEFT JOIN project.task_segments s ON s.id = w.segment_id
      WHERE w.embedding IS NOT NULL AND w.is_deleted = false ${whereScope}
      ORDER BY w.embedding <=> '${literal}'::vector
      LIMIT ${k3}
    ),
    final AS (
      SELECT *, embed_score * ${cfg.embedWeight} + GREATEST(exact_bonus, trgm_score) * ${cfg.keywordWeight} AS score
      FROM scored
    )
    SELECT * FROM final WHERE score >= ${cfg.minScore} ORDER BY score DESC LIMIT ${limit}
  `;
  // 실제 코드는 $queryRaw 템플릿 리터럴 사용 (위는 의사 코드).
  // ...
}
```

> **주의**: $queryRaw에 `${cfg.embedWeight}` 등 숫자를 인터폴레이션하면 Prisma가 `$1`로 바인딩하지만, **CTE 내 `LIMIT`/`OFFSET`은 정수 캐스팅 필요**. 정수는 `${Math.floor(k * 3)}::int`로 명시.

### 7.3 services/project/src/api/routes/internal.routes.ts

```typescript
fastify.post("/work-logs/semantic-search", async (req: any, reply) => {
  const body = req.body as {
    queryVec?: number[];
    q?: string;                       // ← 신규
    userId?: string; userEmail?: string; userRole?: string; limit?: number;
  };
  if (!Array.isArray(body.queryVec) || typeof body.q !== "string"      // ← 검증
      || !body.userId || !body.userEmail || !body.userRole) {
    return reply.status(400).send({ code: "BAD_REQUEST", message: "invalid params" });
  }
  const items = await fastify.workLogService.semanticSearch(
    body.queryVec,
    body.q,                              // ← 전달
    { id: body.userId, email: body.userEmail, role: body.userRole },
    Math.min(50, body.limit ?? 40),
  );
  return reply.send(items);
});
```

## 8. 마이그레이션 전략

### 8.1 SQL 파일 — `scripts/init-pgtrgm.sql`

```sql
-- 검색개선 PDCA — pg_trgm 확장 + GIN trigram 인덱스
-- 실행: docker exec erp-ot-postgres psql -U erp_user -d erp_ot -f /tmp/init-pgtrgm.sql

CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- board_posts (auth)
CREATE INDEX IF NOT EXISTS board_posts_title_trgm_idx
  ON public.board_posts USING gin (title gin_trgm_ops);

CREATE INDEX IF NOT EXISTS board_posts_content_trgm_idx
  ON public.board_posts USING gin (content gin_trgm_ops);

-- work_logs (project)
CREATE INDEX IF NOT EXISTS work_logs_content_trgm_idx
  ON project.work_logs USING gin (content gin_trgm_ops);
```

### 8.2 적용 순서

```bash
# 1. SQL 파일을 컨테이너로 복사
docker cp scripts/init-pgtrgm.sql erp-ot-postgres:/tmp/

# 2. 실행
docker exec erp-ot-postgres psql -U erp_user -d erp_ot -f /tmp/init-pgtrgm.sql

# 3. 확인
docker exec erp-ot-postgres psql -U erp_user -d erp_ot -c "
  SELECT extname, extversion FROM pg_extension WHERE extname='pg_trgm';
  \\di+ public.board_posts*trgm_idx
  \\di+ project.work_logs*trgm_idx
"
```

### 8.3 자동화 (entrypoint 또는 init.sql)

기존 `scripts/init-db.sql` 끝에 include 추가:

```sql
-- 기존 내용...
\i /docker-entrypoint-initdb.d/init-pgtrgm.sql
```

또는 docker-compose에서 `init-pgtrgm.sql`을 `/docker-entrypoint-initdb.d/`에 마운트 (신규 컨테이너에만 적용됨, 기존은 수동 실행 필요).

### 8.4 롤백

```sql
DROP INDEX IF EXISTS public.board_posts_title_trgm_idx;
DROP INDEX IF EXISTS public.board_posts_content_trgm_idx;
DROP INDEX IF EXISTS project.work_logs_content_trgm_idx;
-- extension은 다른 사용자가 있을 수 있으므로 DROP 안 함
```

## 10. 회귀/Sanity 테스트 매트릭스

### 10.1 회귀 테스트 (자연어검색 케이스 보존)

| Query | 자연어검색 결과 (이전) | 검색개선 결과 (목표) |
|-------|----------------------|---------------------|
| "메모리" | top 5 임베딩 점수 0.37~0.56 | 임베딩 0.6 가중치만 적용 시 0.22~0.34 → 컷오프 미달 가능 (예상 결과 0~3건) |
| "회식" | top 5 (무관) 0.37~0.56 | 컷오프로 대부분 제거 예상 (의도된 동작) |
| "납품" | top 5 (무관) 0.30~0.53 | 컷오프로 0~2건 |
| "오류" | top 5 (memory leak 등) 0.46~0.58 | 의미 매칭 유지 — 트러블슈팅 글 0.28~0.35 → 일부 컷오프 |
| "분석" | top 5 0.30~0.57 | 컷오프 후 1~2건 |

> **해석**: 컷오프 0.45 적용 시 임베딩 단독 점수 0.75+ 글만 통과. 데이터 양이 적어 회귀 결과 줄어드는 것은 의도. 키워드 일치하면 보너스로 통과.

### 10.2 Sanity 테스트 (검색개선 핵심 케이스)

| Query | 의도 | 통과 기준 |
|-------|------|----------|
| `DAAU` | 영문 약어 정확 매칭 | "DAAU 메인보드" 글이 1순위, 무관한 글 0건 |
| `메인보드` | 한글 키워드 + ILIKE 정확 매칭 | "메인보드" 단어 포함 글 우선 |
| `디버깅` | 의미 검색 강점 유지 | 트러블슈팅/메모리 누수 글이 임베딩으로 매칭 |
| `회식` | 데이터에 없음 | 0건 (이전엔 무관 5건 나왔던 케이스) |
| `daau` | 대소문자 무시 (ILIKE) | "DAAU" 글 매칭 |

### 10.3 측정 지표

| 지표 | 측정 방법 |
|------|----------|
| 정확도 | sanity 5건 중 의도대로 동작한 비율 |
| 회귀율 | 자연어검색 5건 중 동등/개선 비율 |
| 응답 시간 | took 평균 (목표 < 400ms) |
| 컷오프 효과 | DAAU 검색 시 무관 결과 개수 (목표 0) |

## 12. 보안 / 안정성

| 항목 | 대응 |
|------|------|
| SQL Injection | `${trimmedQ}` 는 Prisma `$queryRaw` 템플릿 리터럴 → 자동 prepared statement (`$1`) |
| pg_trgm 미설치 | 마이그레이션 실패 시 명확한 에러 (`extension "pg_trgm" does not exist`) |
| 환경변수 누락 | `searchConfig` 검증에서 기본값 사용 + 합 검증 실패 시 startup 에러 |
| 가중치 합 ≠ 1.0 | startup 에러 throw |
| similarity() NULL | `COALESCE(similarity(...), 0)` 적용 (안전망) |
| 쿼리 길이 제한 | q.trim().length ≥ 2 (기존 유지) + `${trimmedQ}` 그대로 인덱스 활용 |
